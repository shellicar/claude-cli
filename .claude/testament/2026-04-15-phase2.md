### Implementation details

All three methods follow the same pattern as `getDiagnostics`:
1. Check `#started`, throw if not
2. Resolve file path relative to `#cwd`
3. Open the file via tsserver if not already tracked in `#openFiles` (with 500ms delay)
4. Send the tsserver command
5. Map the response to the return type

**`getHoverInfo`**: sends `quickinfo` command with `{ file, line, offset }`. tsserver's `offset` maps to our `character`. Maps `displayString` to `text`, `documentation` stays (converted to `undefined` if empty string), `kind` stays. Returns `null` if no body or no `displayString`.

**`getReferences`**: sends `references` command. Response body has a `refs` array. Each ref has `file`, `start.line`, `start.offset`, and `lineText`. Maps `lineText` to `text`, `start.offset` to `character`.

**`getDefinition`**: sends `definition` command. Response body is a direct array of spans. Each has `file`, `start.line`, `start.offset`. Maps `start.offset` to `character`.

### tsserver protocol notes

- All three commands take `{ file, line, offset }` where `line` and `offset` are both 1-based
- `quickinfo` returns `{ displayString, documentation, kind, ... }` or fails with `success: false` if no symbol at position
- `references` returns `{ refs: [...], symbolName, ... }` where refs have `{ file, start, end, lineText, isDefinition, ... }`
- `definition` returns an array directly (not wrapped in an object with a named key): `[{ file, start, end, ... }]`
