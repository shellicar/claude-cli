## Phase 6 Maker — revision: persist on send of any user message

The mission was re-dispatched with an SC revision to piece 4. Piece 4 as first built persisted only after the assistant reply (`turn_content`), which leaves a just-sent user message unsaved if the process dies mid-response — the exact interruption this mission fixes. On restart the persisted conversation would lack the trailing user message, and submit-to-resume would have nothing to resume from.

### The seam I chose: `query_summary` (and why it covers both cases with one hook)

The SC asked to persist "at the point a user message is sent, for any user message: end-human text and `tool_result` alike." I traced `QueryRunner.run` (SDK) to find where each user message lands and what CLI-observable event brackets it:

- The end-human user message(s) are pushed at the **start** of `run()`, before the turn loop.
- Each turn's `tool_result` user message is pushed at the **end** of the turn iteration (`this.conversation.push({ role: 'user', content: toolResults })`).
- `query_summary` is published at the **top of every turn iteration**, before the (interruptible) `turnRunner.run`.

So `query_summary` always fires with the just-sent user message already in the `Conversation`, and before the request that can die — for **both** the end-human case (iteration 1) and the tool_result case (iteration N+1). One hook covers both. I added `void this.session.saveConversation().catch(...)` to the `query_summary` case in `AgentMessageHandler`, keeping the existing `turn_content` after-assistant save. Net: the conversation is persisted after every message in both directions, via the same atomic temp-then-rename write.

This is a design choice the SC left open (they named the *what* and the test, not the seam). If a future reader wants a more explicit seam, note that the CLI cannot easily persist "just the user message before the request" from `runAgent`, because the message is pushed into the `Conversation` inside `QueryRunner.run` (async, runs the whole query) — `query_summary` is the CLI's observable pre-request signal. It's the right seam.

### Test

`AgentMessageHandler.spec` gains a `persistence` describe with three tests: send-time save fires on `query_summary` (fake-session count), after-assistant save fires on `turn_content` (unchanged behaviour, kept), and the SC's scenario — a user message sent via `query_summary`, **no** `turn_content` (death before the assistant replies), and a fresh `ConversationSession` reloading from the shared `MemoryFileSystem` still holds that message. To wire the last one I made `makeHandler` accept a real `ConversationSession` override and expose its `conversation` + `fs`, and added a `buildRealSession` helper for the reload side.
