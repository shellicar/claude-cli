## What I confirmed by reading the code

The Anthropic SDK's `BetaMessageStream` emits an `inputJson(partialJson, jsonSnapshot)` event for every `input_json_delta` while a `tool_use`, `server_tool_use`, or `mcp_tool_use` block is streaming. Our `StreamProcessor` does not subscribe to it — that is the gap. Everything else is in place: a long-lived processor with `.on(...)` re-emission, an `sdkChannel` pipeline through `main.ts`, an `AgentMessageHandler` switch over `SdkMessage`, an `AppLayout.appendStreaming` that writes to the active block. Adding the new event is a four-file change along this established lane.

## The inputJson callback signature

The signature is `(partialJson: string, jsonSnapshot: unknown) => void`. Crucially **no index, no name**. The block name is known only from `content_block_start`. That's why the plan caches `index → name` and looks up on each delta. Two readers would otherwise reach for "just emit the JSON" and find no way to label which tool it's for.

## Decision left for the SC

Locked decisions say partial JSON renders "concatenated" with "no transformation" and the existing rendering "does not change". The literal join produces visually-glued output:

```
{"path":"/foo"}🌐 web_search(foo) ✅
```

I picked interpretation (1) — literal — and surfaced (2) — soft, one extra newline — as an open question. The plan ships either way with a single-line change to swap interpretations. I did not invent a third interpretation; that would be the failure mode the role exists to prevent.

## What I deliberately did not investigate

`mcp_tool_use`. The codebase ignores it today (explicit `break` in `StreamProcessor.contentBlock` switch, comment: "Ignored: matches current behavior"). The plan cache covers `tool_use` and `server_tool_use` only, matching that posture. Adding mcp coverage is a separate question and not part of the mission's "all tool calls" scope as the code defines that scope.

# 06:42

Revised plan after supervisor (via Router) returned blocked. Same file path; the previous plan is overwritten.

## What I missed the first time

I treated the visual concern as a separator question — "literal vs soft" between partial JSON and the completion summary. I noted it as an open question with a one-character fix. The supervisor identified the real issue: the **order** is wrong, not the separator. Name belongs first because `content_block_start` carries it before any `input_json_delta` fires. The SDK's lifecycle order is name → JSON → status; the view should follow that order. Once it does, there is no join to debate.

The cue I had — visual oddness from gluing JSON and name together — was itself the signal that the design was wrong upstream. I interrogated the rendering instead of the order. Next time a stop note describes "visually weird" output, the question to ask is "why is the lifecycle ordered like this at all", not "how do I separate the two pieces of text".

## What changed in the revised plan

The corrected design:

- New SDK events `tool_use_start` and `server_tool_use_start` emit at `content_block_start`, carrying the name. Separate events (rather than one event with a discriminator) so the consumer's handler can apply the `🌐 ` prefix in the server arm without inspecting a kind field — matches the codebase's existing pattern of distinct `server_tool_use` vs `tool_use` paths.
- `tool_use_input_delta` no longer carries the name. Just `partialJson`.
- The completion-time `server_tool_use` event/message is removed entirely. Its handler arm (`🌐 ${summary}`) is removed with it.
- `tool_approval_request` resolution narrows: `${summary} ✅\n` becomes ` ✅\n`. Same for `❌` and the catch arm's `💥`.
- `formatToolSummary` and its helpers in `AgentMessageHandler.ts` become unreferenced. The plan flags this and leaves removal out of scope — the helpers may be wanted for the history view (#179).

## Decisions made in this revision

- **Set<number> instead of Map<number, string>** in StreamProcessor for tracking active tool input indices. Name is no longer needed at delta time (the start event placed it); a Set is the minimum needed to gate `inputJson` emission for the right block kinds.
- **Two start events** rather than one event with a `kind` field. Codebase convention separates `server_tool_use` from `tool_use` paths; matching that pattern keeps the new code shape consistent with what's already there.
- **Keep `transitionBlock('tools')` in the `tool_approval_request` arm** even though it's now redundant (the start event has already done it). The existing pattern in this codebase is "every tool-batch entry point ensures it's in the tools block"; removing for purity would diverge from that pattern.

## What I did not do

Did not propose removing `formatToolSummary` and its helpers in this PR. The Cleaner phase isn't in scope for this mission; the Maker explicitly stays out of style territory. Surface and stop.

# 23:00

Router returned again: their previous "completion renders the status marker only" line was their own error. The SC's actual constraint is only about order during streaming. The streaming view is additive; the resolved view stays unchanged. Plan rewritten a third time.

## What the second plan over-cut

I removed the completion-time `server_tool_use` event entirely from `MessageStreamEvents`, `SdkMessage`, the wiring, and the handler. I narrowed the `tool_approval_request` resolution arms from `${summary} ✅` to ` ✅`. Both were based on the Router's now-retracted "narrows to status only" guidance.

The cost of that over-cut is most visible at `Ref` and `Pipe` calls. `Ref` resolves its `id` against the `RefStore` to render `Ref ← ${hint} [${start}–${end} / ${sizeStr}]`. That information is not in the raw JSON — it comes from the store lookup, only available at resolution time. The streaming view shows raw JSON; the resolved view is the only place the hint and total size appear. Cutting the resolved view loses information.

I should have caught this on my own when the supervisor said "narrows to status only" — `formatToolSummary` is non-trivial and tool-specific. Removing it eliminates view shape, not just decoration. The fact that `Ref` and `Pipe` have custom branches was visible in the helper code I read. I treated the Router's instruction as authoritative without holding it against the codebase.

The supervisor verification block in the prompt flagged this exact side effect in their re-review of my second plan. It is the structural cue I should have caught.

## Third plan: purely additive

- `tool_use_start`, `server_tool_use_start`, `tool_use_input_delta` are added.
- `Set<number>` of active tool-input indices gates `inputJson` forwarding.
- Existing `server_tool_use` event, `server_tool_result` event, and `tool_approval_request` resolution are all UNCHANGED.
- `formatToolSummary` and helpers continue to be called from the unchanged arms. Not dead.

Visible output for a server tool: `🌐 web_search{"query":"foo"}🌐 web_search(query: foo) ✅`. The two views coexist. The streaming portion is raw JSON; the resolved portion is the formatted summary. They serve different purposes — streaming for live feedback, resolved for canonical record — and the SC has stated they should coexist.

## Lesson

Treat retraction-shaped instructions as a signal to re-read the codebase, not just to flip a switch in the previous plan. The Router's "narrows to status only" was a translation of the SC's "order during streaming" constraint; the translation was wrong. If I had checked what `formatToolSummary` actually does for `Ref` and `Pipe`, the over-cut would have been visible at write time. Instead I cut against the prose without checking what the prose was abstracting away.

The earlier pattern flag from the supervisor — "stop notes about visual oddness are signals to interrogate the design" — extends here: instructions to remove existing code are signals to re-verify what that code does before removing it.

## What stayed correct from the second plan

The `Set<number>` gating, the two separate start events, the index→deletion pairing on `content_block_stop`. The new SDK additions are unchanged between v2 and v3; only the deletions are reversed.
