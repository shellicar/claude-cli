# 08:34

Plan revision under a block verdict. Three issues to fix, each a recurring
anti-pattern I missed the first time. Worth recording the calibration
because the same shapes will surface again.

## Issue 1 — IIFE for inline validation

I wrapped `--resume` UUID validation in an IIFE because the validation
needed both a guard (`if not a string, return null`) and a fail-exit
(`if invalid, exit`) and a return value. The IIFE felt like the natural
shape for "compute this with branching."

The supervisor's reframe: every neighbour in the same extraction block
is a one-liner. The IIFE was familiar, not fitting. Reach for the local
idiom — one-liner extraction plus a top-level guarded block for the
validation — before reaching for a control structure that buys closure
scope nothing in the local context needs.

The trigger to watch for: "I need branching here, so I'll wrap it in a
function." Question that automatically: do I need a function, or do I
just need a guarded block at the same scope?

## Issue 2 — "Mirrors exactly" is duplication

I wrote in the testament that the new `resume()` method "mirrors the
history-load fragment of `load()` exactly" and presented that as a virtue
("reuses existing behaviour"). The supervisor caught it: mirroring
exactly IS duplication. Reuse is calling the same code, not copying it.

The right shape is what I would have done if I had written `load()` and
`resume()` together from scratch: extract the shared body into a
private helper, both public methods call it. The plan now does that.

The trigger to watch for: my own prose using the words "mirror,"
"parallel to," "matches the shape of," when describing a new method
against an existing one. That language is a signal the bodies are
duplicate and the natural shape is one helper called from two places.

## Issue 3 — includes-as-boolean tests

I wrote rendered-output tests as `expected = true; actual = output.includes('substring')`. The `expected/actual` idiom is supposed to
name what is being tested; "true" is a derived boolean, not the thing
the renderer produces.

The supervisor's reframe: each behaviour gets one test that asserts the
full rendered string. The full string pins the behaviour-under-test AND
the rest of the format simultaneously. `includes` is the wrong primitive
when the unit under test is a renderer.

The trigger to watch for: writing `const expected = true` or
`const expected = false` against an `actual` that comes from `.includes(...)`.
The named `expected` should be the thing the function produces.

## Pattern across all three

Each was reaching for a familiar idiom (IIFE, copy-the-fragment,
includes-as-boolean) without asking whether the idiom fit the local
context. The mission's "lean on existing behaviour" principle has a
line-of-code echo: lean on the local idiom in the file you're editing.

The Maker phase will face the same temptation — if it ships an IIFE, a
duplicated fragment, or an includes-as-boolean test, the calibration
should be applied there too.

move is to re-read the whole prompt first, then act on the verdict.
