## Server tool streaming: two distinct block shapes

`server_tool_use` blocks accumulate input via `input_json_delta`, same as client `tool_use` blocks. Parse the input on `content_block_stop`.

Server tool result blocks (`web_search_tool_result`, `web_fetch_tool_result`, etc.) arrive **fully populated in `content_block_start`** with no deltas. Capture `event.content_block.content` there; `content_block_stop` just fires the emit. Getting this wrong means results are silently lost.

The exhaustive list of result block types (from the Anthropic SDK union):
```
web_search_tool_result, web_fetch_tool_result, code_execution_tool_result,
bash_code_execution_tool_result, text_editor_code_execution_tool_result,
tool_search_tool_result, mcp_tool_result
```

Non-result blocks that hit the inner switch default: `redacted_thinking`, `mcp_tool_use`, `container_upload`. Ignore them silently.

## Use the type map, not string heuristics

The lookup for result block types lives in `SERVER_TOOL_RESULT_NAMES` in `StreamProcessor.ts`. Do not use `endsWith('_tool_result')` or similar; the set of types is known and finite.

## `allowedCallers` and ZDR

`_20260209` versions of `web_search` and `web_fetch` use dynamic filtering internally (run code execution to summarise content). This makes them not ZDR-eligible. Setting `allowed_callers: ['direct']` disables dynamic filtering and restores ZDR. That is what the CLI defaults do.

`AllowedCaller = 'direct' | 'code_execution'`. The code execution version (`code_execution_20250825` or `code_execution_20260120`) is a single global setting from `advancedTools.codeExecutionTool`. `buildServerTools` resolves it at call time; mixing versions per-tool isn't supported.

## `BetaMCPToolset` has no `name`

`BetaToolUnion` includes `BetaMCPToolset` which lacks `name`. `transformTool` in `buildAtuTransform.ts` guards with `isNamedTool(tool)` before touching `tool.name`. Without the guard, TypeScript errors because `BetaMCPToolset` doesn't satisfy `{ name: string }`.

## Session state and server tools

If server tools stop responding mid-session (tool appears called but returns nothing), restart the CLI before debugging code. Long sessions that exhausted code_execution calls can leave server tools silently non-functional. A fresh session fixed it every time during development.

## textEdits pattern matching fails silently when content has drifted

`PreviewEdit` is a preview only — it never writes. The write happens in `EditFile`. If `textEdits` can't find the pattern (because the file drifted from what you expected), `EditFile` applies nothing. This looks like silence but it is a pattern mismatch, not a tool failure.

When a file has been heavily modified in a session and targeted edits aren't landing, rewrite with `CreateFile overwrite: true` rather than fighting the pattern matching.

## Tests must verify what the fix actually changes

The first StreamProcessor tests passed even without the fix because they measured block output, which was identical either way. The correct assertion: check that `server_tool_use` events are emitted, or that text captured after server blocks is non-empty. Test what the code path actually touches, not a downstream value that happens to be the same in both cases.
