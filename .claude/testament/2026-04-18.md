## Server tool streaming — how it actually works

`server_tool_use` blocks accumulate input via `input_json_delta`, same as client `tool_use`. The input starts as `{}` in `content_block_start` and gets built up via deltas. Parse it on `content_block_stop`.

Server tool result blocks (`web_fetch_tool_result` etc.) arrive **fully in `content_block_start` with no deltas**. Capture `event.content_block.content` there. `content_block_stop` just fires the emit.

The exhaustive list of result block types (from the SDK union):
```
web_search_tool_result, web_fetch_tool_result, code_execution_tool_result,
bash_code_execution_tool_result, text_editor_code_execution_tool_result,
tool_search_tool_result, mcp_tool_result
```

Non-result blocks that hit the inner switch default: `redacted_thinking`, `mcp_tool_use`, `container_upload`. These should be silently ignored — do NOT emit events for them.

The name lookup lives in `SERVER_TOOL_RESULT_NAMES` in `StreamProcessor.ts`. Do not use string heuristics (`endsWith('_tool_result')`). The types are known, use the map.

## allowedCallers is conceptual, not versioned

`AllowedCaller = 'direct' | 'code_execution'`. The actual code execution version (`code_execution_20250825` or `code_execution_20260120`) is a single global setting from `advancedTools.codeExecutionTool`. `buildServerTools` resolves it at call time. Mixing versions per-tool makes no sense and isn't supported.

## web_fetch_20260209 / web_search_20260209 and ZDR

The `_20260209` versions use dynamic filtering internally — they run code execution to summarise fetched content. This makes them **not** ZDR-eligible by default. Setting `allowed_callers: ['direct']` disables dynamic filtering and restores ZDR eligibility. This is what the defaults do.

## BetaMCPToolset has no `name` property

`BetaToolUnion` includes `BetaMCPToolset` which has no `name`. The `transformTool` function in `buildAtuTransform.ts` uses `isNamedTool(tool)` to guard before accessing `tool.name`. Without that guard, TypeScript errors because `BetaMCPToolset` doesn't satisfy `{ name: string }`.

## Session state and server tools

If server tools stop working mid-session (tool appears to be called but returns nothing, or fails silently), **restart the CLI before debugging the code**. Long active sessions — especially ones that exhausted code_execution calls — can end up in a state where server tools don't function correctly. A new session fixed it every time.

## Testing: if removing the fix makes tests pass, the tests test nothing

The first version of the StreamProcessor tests passed even without the fix, because the block output (`result.blocks`) was identical whether or not server blocks were tracked. The tests were measuring something that was always true. The correct check: verify what the fix actually changes, e.g. that `server_tool_use` events are emitted, or that text after server blocks is captured.

## CreateFile overwrite is more reliable than textEdits for structural changes

`PreviewEdit` with `textEdits` silently failed twice on StreamProcessor changes — reported success but the file didn't change. When targeted edits fail, rewrite the whole file with `CreateFile overwrite: true`. More reliable for structural changes to files that are being heavily modified in a session.
