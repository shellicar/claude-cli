# 20:54

Apostle re-plan cast. Two new locked design decisions added after the Maker phase committed 9919605.

## What I found

### R1 — Prompt timing (0s duration)

The defect is in `AppLayout.startStreaming()`. The three calls (`transitionBlock('prompt')`, `appendToActive`, `completeActive`) fire in sequence at submit time. Both `createdAt` and `exitedAt` are stamped within the same millisecond.

`waitForInput()` does not touch `ConversationState` at all — the idle time is never recorded.

The fix I proposed: add `markPromptStart()` to `ConversationState`. It stores an `Instant` that `transitionBlock('prompt')` consumes as `createdAt` if present. `AppLayout.waitForInput()` calls `markPromptStart()` as its first step. This avoids opening the prompt block during editor mode (which would create a double-divider problem: `renderConversation` renders the active block, and `AppLayout.render()` independently calls `buildDivider('prompt', cols)` for the editor input area).

The stored-instant pattern is the smallest adjustment that doesn't require renderer changes.

### R2 — Response section on error

Two error paths call `transitionBlock('response')`:
- `AgentMessageHandler.ts` `case 'error'` (line 192)
- `runAgent.ts` catch block (line 62)

The fix: add `'error'` to `BlockType` and route both paths to `transitionBlock('error')`. I chose a new block type over reusing `meta` because `meta`'s label (`query`) and emoji (`ℹ️`) don't read as errors. The new type costs four label-map lines.

The `\n\n` prefix on the error strings was there to separate error content from preceding response text in a shared block. It's removed in the plan because the error now opens its own block.

## Decisions I made

- Stored-instant pattern over early block-open in `waitForInput`. Reason above.
- New `'error'` block type over reusing `meta`. Reason above.
- Removed `\n\n` prefix from error strings. The prefix was for shared-block separation; it's noise in a fresh block.
- No change to `startStreaming`. It already calls `transitionBlock('prompt')`; the stored-instant pattern makes the change entirely in `ConversationState` and `waitForInput`.

# 21:15 — Re-plan correction (r2)

The correct approach: remove the `transitionBlock('response')` call from both error paths. No new block type, no new section. Errors append to whatever block is already active via `appendStreaming`.

# 21:52 — Re-plan correction (r4)

R2 now complete. The r3 two-line deletion was necessary but not sufficient.

The `appendStreaming` no-op when `activeBlock === null` was the gap. Fix: add `bare?: true` to `Block`. When `appendStreaming` is called with no active block, it calls `addBlocks([{ type: 'meta', content: text, bare: true }])`. The bare block enters `sealedBlocks` directly and is flushed through the normal path. The renderer skips the section divider when `block.bare === true`.

Type for bare blocks is `'meta'`. The theoretical concern (a subsequent real meta block treating the bare one as a predecessor and suppressing its own header) doesn't arise in practice: the null-active-block case only occurs when `queryRunner.run` throws before `query_summary` fires, meaning no subsequent query summary follows.

The `render()` call in `appendStreaming` was previously inside the `if (activeBlock)` branch, so it wouldn't fire for the bare-block path. Moved outside both branches — bare block also triggers a render.

# 22:15 — Re-plan correction (r5)

Dropped the bare-block approach entirely. The requirement is "prints in the previous closed section, without opening a new section." That's append-to-last-sealed, not a new headerless block.

New method: `ConversationState.appendToTail(text: string): void`. Appends to active block if open; falls back to last sealed block if not; no-op only if no blocks exist at all.

`AppLayout.appendStreaming` becomes two lines: call `appendToTail`, call `render()`. The conditional is gone. The `render()` call was previously inside the if-branch — moving it outside means the viewport is always updated, including when the fallback path fires.
