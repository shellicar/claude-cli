## The one insight that reframes the whole job

The ESC-to-abort path already exists end to end. `runTurn` (main.ts) makes a fresh `AbortController`
per query, the consumer-channel subscriber calls `currentAbortController.abort()` on cancel, and
`QueryRunner.run` already holds `input.abortController.signal` (it feeds the HTTP request today). The
ONLY gap is that this signal never reaches a tool handler. So this mission is not "add cancellation",
it is "forward an existing signal one level deeper and make Exec honour it". No edit to
`AppLayout.ts`, `runAgent.ts`, or `main.ts`. If a future cast thinks it needs a new controller, it
does not.

## Traps I hit so the next operator doesn't

- **Name collision.** Both `execCommand` and `execPipeline` bind a `close` handler parameter named
  `signal` (the OS signal name, written into the result as `signal: signal ?? null`). Do NOT name the
  AbortSignal `signal` inside the Exec chain or it shadows that. The plan uses `abortSignal`.
- **Scout line numbers drift in big files.** The supervisor flagged this in Phase 1 and it held: I
  re-derived every anchor with grep rather than trusting the Scout's numbers. The Scout's quoted code
  was verbatim-accurate though, so grep on the quoted line finds the truth fast.
- **Two existing test surfaces decide the signature.** `ToolRegistry.spec.ts` calls `resolved.run()`
  / `resolved.run(transform)` at ~14 sites, and `test/helpers.ts` `call`/`callFull` invoke
  `handler(input)` with no signal. Making the new signal **optional** (`signal?: AbortSignal`) keeps
  all of those compiling untouched. Making it **required** forces ~16 test edits for zero runtime gain.

## Decisions I surfaced rather than made

The mission wording ("always take an abort signal", "accept it in their signature") admits a literal
heavier reading and a lighter type-only reading. I wrote the plan to the lighter reading (optional
signal; change the `ToolHandler` type only, leave the 18 non-Exec handlers as `async (input) =>` since
TS lets a fewer-param fn satisfy a wider fn type) and flagged both as Open decisions for the
supervisor. If the supervisor pins the heavier reading, the deltas are written into the plan. The
Maker should build whichever the supervisor confirms at Phase 2 verification.

## Facts worth keeping

- Node v24.14.1. `spawn({ signal })` kills the child natively; `controller.abort()` with no arg gives
  `signal.reason` a `DOMException` with `name === 'AbortError'`. The plan rejects with
  `abortSignal.reason` from guarded `close`/`error` handlers, so ordering of those events does not
  matter. No `AbortError`/`DOMException` precedent exists in active code (one comment in legacy
  `apps/claude-cli`).
- biome enables `noUnusedVariables`/`noUnusedImports` (error) but NOT `noUnusedFunctionParameters`.
  Relevant only if the supervisor wants the param spelled in the 18 handlers.
- Base is 1 behind `origin/main` (`ceb37b9`, HTTP retry/backoff). It touches `TurnRunner.ts`,
  `backoff.ts`, and their tests, none of which this plan modifies. Clean merge expected at Courier.
- Working tree was testament-only dirty at preflight (Scout's `2026-06-05.md` staged, `2026-06-02.md`
  untracked). I am read-only on the code, so it did not affect planning.

## What the lock changed

Cancellation is no longer name-based. A dedicated `ToolCancelledError extends Error` is added to
claude-sdk (`public/ToolCancelledError.ts`, one-liner per the `InvalidAuthorisationCodeError` /
`TokenExchangeFailedError` convention), exported from `index.ts` so claude-sdk-tools can import it.
Exec rejects with `new ToolCancelledError()` at every abort-reachable `close`/`error` branch; the
registry catch and the elapsed-time message gate on `err instanceof ToolCancelledError`, never
`err.name === 'AbortError'`. The Exec test asserts `rejects.toBeInstanceOf(ToolCancelledError)`.

## Things that carried over unchanged

The signal plumbing (ToolHandler/ToolResolveResult/QueryRunner), the `spawn({ signal })` kill
mechanism, the `abortSignal` naming inside Exec (avoids the `close` handler's `signal` param), and the
two open decisions (optional vs required signal; type-only vs 18 handler edits). The lock fixed the
error mechanism, not those.

## instanceof across the package boundary

The registry (claude-sdk) and Exec (claude-sdk-tools) must reference the same `ToolCancelledError`
class identity for `instanceof` to hold. It does, because claude-sdk-tools imports it from
`@shellicar/claude-sdk` (same direction as the existing `defineTool` import, one module instance). The
cancellation test exercises this boundary, so `pnpm test` confirms it rather than leaving it assumed.

# 05:36

Rate-limit retry mission. PR delivers transient error retry with exponential backoff in `TurnRunner`.

## What was built

**`packages/claude-sdk/src/private/backoff.ts`** (new):
- `calculateBackoffDelay(attempt, random)`: `base = min(500 * 2^(attempt-1), 32_000)`; `delay = base + random() * 0.5 * base`. Jitter only ever lengthens a wait (x[1.0, 1.5]).
- `isRetryable(error)`: the single extend-point for the retryable set. Branch order: abort first (never retry); `APIConnectionError` (retry, covers `APIConnectionTimeoutError` subclass); `APIError.type` in {`rate_limit_error`, `overloaded_error`} (retry); else false.
- `defaultSleep(ms, signal)`: resolves immediately if already aborted; races `setTimeout` against the abort signal's event so it returns the instant the signal fires.
- Constants: `BASE_DELAY_MS = 500`, `MAX_DELAY_MS = 32_000`, `MAX_RETRIES = 10`.

**`packages/claude-sdk/src/private/TurnRunner.ts`** (modified):
- Added optional `sleep?` / `random?` constructor params defaulting to real implementations. `main.ts` is untouched.
- Retry loop wraps the stream+process pair only. `conversation.push` and the return stay outside. On exhaustion (11 stream calls, 10 sleeps), the last error re-throws unchanged through `QueryRunner`'s existing catch.
- Post-sleep abort check: `if (abortSignal.aborted) turnInput.abortSignal.throwIfAborted()`. Explicit and deterministic: stops the loop immediately with no further `stream()` call, regardless of what the SDK would throw.

## Error classification

In-stream errors arrive as `APIError` with `status === undefined` (HTTP 200, SSE error event). `overloaded_error` also arrives as HTTP 529 (`InternalServerError extends APIError`, `status 529`). The `.type` check covers both paths.

The SDK's own `maxRetries = 2` handles HTTP 408/409/429/>=500. In-stream errors (inside HTTP 200) are entirely outside SDK retry coverage. HTTP-level errors can double-retry (SDK 2x + our 10x) -- intentionally accepted.

## The abort path

Abort during backoff surfaces via `throwIfAborted()`, which throws `signal.reason` (a `DOMException` when `abort()` is called without a reason). Abort during `stream()` surfaces as `APIUserAbortError` from the SDK -- a different class. This divergence is intentional per the locked decision: do not claim the mechanism mirrors the SDK's internal abort, and do not manufacture `APIUserAbortError` from our layer.

The Apostle plan's Hunk 3 comment contained a false claim that `throwIfAborted()` mirrors the SDK's `makeRequest` abort mechanism (the SDK throws `APIUserAbortError`, not `throwIfAborted()`). The comment was corrected before landing.

## Test trap

`Stream.fromReadableStream` (used by all test helpers via `makeBetaStream`) silently swallows SSE error events. Only the real SSE path (`fromSSEResponse`) throws on `{type:"error"}`. The fake streamer must throw the error directly from `stream()`. Building an error-shaped SSE stream will not work.

## Live verification pending

The retry logic is verified against fakes. A real `overloaded_error` or `rate_limit_error` cannot be triggered on demand. Live verification of the retry path under real API load is pending.

Phase 2 — Apostle. Clock injection plan (four iterations, delivered clean in iteration 4).

## Injection shape

`systemReminders?: string[]` on `RequestBuilderOptions`. The loop in `buildRequestParams` emits one `<system-reminder>` block per entry, after the cache boundary (`cacheLastUserMessage` runs first), uncached. `TurnRunner.run()` is the assembly point: git delta first (turn-1 only, from `turnInput.systemReminder`), clock stamp last (every turn, fresh).

The `string[]` stays on `RequestBuilderOptions` only — does not propagate to `TurnInput` or `PerQueryInput`. `PerQueryInput.systemReminder` is the CLI's external contract; propagating up would require CLI changes (out of scope). `SdkQuerySummary` channel is unaffected.

## Cast type

The injection loop casts `lastUser.content as BetaContentBlockParam[]`, not `BetaTextBlockParam[]`. The content array holds mixed block types on tool-loop turns (text, tool_use, etc.). `BetaContentBlockParam` was absent from `RequestBuilder.ts` and needed to be added to the import.

## Dead branch

The defensive `typeof lastUser.content === 'string'` guard before the push is dead code: `cacheLastUserMessage` arrayifies the content before injection runs. The SDK's no-defensive-code convention applies — dead branches are debt. Push directly.

## Clock injectability at TurnRunner

`clock: Clock = Clock.systemDefaultZone()` was added as a positional constructor parameter. After the merge of #334 (which added `sleep?`/`random?`), the final constructor signature is `(streamer, processor, logger?, sleep?, random?, clock?)`. The `clock` position last means none of the existing call sites or tests need updating. See the `10:41` entry in `2026-06-07.md` for the merge resolution.

## Test fixture date

The locked design example said `Friday, 30 May 2026` — but May 30, 2026 is a Saturday. The test fixture uses `2026-05-29T04:32:15Z`, which renders as `Friday, 29 May 2026 at 14:32:15 Australia/Melbourne (+10:00)`. Don't "correct" the fixture to match the mission text.
