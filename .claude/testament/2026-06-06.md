## The one insight that reframes the whole job

The ESC-to-abort path already exists end to end. `runTurn` (main.ts) makes a fresh `AbortController`
per query, the consumer-channel subscriber calls `currentAbortController.abort()` on cancel, and
`QueryRunner.run` already holds `input.abortController.signal` (it feeds the HTTP request today). The
ONLY gap is that this signal never reaches a tool handler. So this mission is not "add cancellation",
it is "forward an existing signal one level deeper and make Exec honour it". No edit to
`AppLayout.ts`, `runAgent.ts`, or `main.ts`. If a future cast thinks it needs a new controller, it
does not.

## Traps I hit so the next operator doesn't

- **Name collision.** Both `execCommand` and `execPipeline` bind a `close` handler parameter named
  `signal` (the OS signal name, written into the result as `signal: signal ?? null`). Do NOT name the
  AbortSignal `signal` inside the Exec chain or it shadows that. The plan uses `abortSignal`.
- **Scout line numbers drift in big files.** The supervisor flagged this in Phase 1 and it held: I
  re-derived every anchor with grep rather than trusting the Scout's numbers. The Scout's quoted code
  was verbatim-accurate though, so grep on the quoted line finds the truth fast.
- **Two existing test surfaces decide the signature.** `ToolRegistry.spec.ts` calls `resolved.run()`
  / `resolved.run(transform)` at ~14 sites, and `test/helpers.ts` `call`/`callFull` invoke
  `handler(input)` with no signal. Making the new signal **optional** (`signal?: AbortSignal`) keeps
  all of those compiling untouched. Making it **required** forces ~16 test edits for zero runtime gain.

## Decisions I surfaced rather than made

The mission wording ("always take an abort signal", "accept it in their signature") admits a literal
heavier reading and a lighter type-only reading. I wrote the plan to the lighter reading (optional
signal; change the `ToolHandler` type only, leave the 18 non-Exec handlers as `async (input) =>` since
TS lets a fewer-param fn satisfy a wider fn type) and flagged both as Open decisions for the
supervisor. If the supervisor pins the heavier reading, the deltas are written into the plan. The
Maker should build whichever the supervisor confirms at Phase 2 verification.

## Facts worth keeping

- Node v24.14.1. `spawn({ signal })` kills the child natively; `controller.abort()` with no arg gives
  `signal.reason` a `DOMException` with `name === 'AbortError'`. The plan rejects with
  `abortSignal.reason` from guarded `close`/`error` handlers, so ordering of those events does not
  matter. No `AbortError`/`DOMException` precedent exists in active code (one comment in legacy
  `apps/claude-cli`).
- biome enables `noUnusedVariables`/`noUnusedImports` (error) but NOT `noUnusedFunctionParameters`.
  Relevant only if the supervisor wants the param spelled in the 18 handlers.
- Base is 1 behind `origin/main` (`ceb37b9`, HTTP retry/backoff). It touches `TurnRunner.ts`,
  `backoff.ts`, and their tests, none of which this plan modifies. Clean merge expected at Courier.
- Working tree was testament-only dirty at preflight (Scout's `2026-06-05.md` staged, `2026-06-02.md`
  untracked). I am read-only on the code, so it did not affect planning.


# 01:33

Supervisor pinned `## Locked design decisions` after my first plan; revised the plan to match.

## What the lock changed

Cancellation is no longer name-based. A dedicated `ToolCancelledError extends Error` is added to
claude-sdk (`public/ToolCancelledError.ts`, one-liner per the `InvalidAuthorisationCodeError` /
`TokenExchangeFailedError` convention), exported from `index.ts` so claude-sdk-tools can import it.
Exec rejects with `new ToolCancelledError()` at every abort-reachable `close`/`error` branch; the
registry catch and the elapsed-time message gate on `err instanceof ToolCancelledError`, never
`err.name === 'AbortError'`. The Exec test asserts `rejects.toBeInstanceOf(ToolCancelledError)`.

## Things that carried over unchanged

The signal plumbing (ToolHandler/ToolResolveResult/QueryRunner), the `spawn({ signal })` kill
mechanism, the `abortSignal` naming inside Exec (avoids the `close` handler's `signal` param), and the
two open decisions (optional vs required signal; type-only vs 18 handler edits). The lock fixed the
error mechanism, not those.

## instanceof across the package boundary

The registry (claude-sdk) and Exec (claude-sdk-tools) must reference the same `ToolCancelledError`
class identity for `instanceof` to hold. It does, because claude-sdk-tools imports it from
`@shellicar/claude-sdk` (same direction as the existing `defineTool` import, one module instance). The
cancellation test exercises this boundary, so `pnpm test` confirms it rather than leaving it assumed.

## No surprises

The plan was accurate. Code matched the diff bases exactly. The open decisions (optional signal, type-only for non-Exec handlers) resolved to the plan's recommended readings as instructed. The `close` handler's `signal` parameter naming conflict was real — `abortSignal` naming throughout the Exec chain prevents shadowing.

## The `instanceof` boundary

`claude-sdk-tools` imports `ToolCancelledError` from `@shellicar/claude-sdk` via `node_modules`. The registry (claude-sdk) resolves the same path. Single class identity — `instanceof` holds. The cancellation test confirms this at the package boundary rather than leaving it assumed.
