## The shape of the refactor (what #339 did)

This is the context that the next operator needs and the git log doesn't spell out:

- **`AppLayout.ts` is gone.** It was a god-object combining View + Controller + state ownership. #339 split it into: `app/` (Presentation, PrimaryPresentation, ViewHost, TerminalInput), `controller/` (EditorHandler, CommandKeyHandler, CancelHandler, ApprovalHandler, QuitHandler, InputHandler, CommandIntentExecutor), `view/` (PrimaryView, TerminalRenderer, View, Flasher), and new model stores (PrimaryViewState, AppModeState, TerminalState).
- **`AgentMessageHandler` no longer talks to AppLayout.** It now takes `ConversationState` + `ToolApprovalState` directly in its options bag and calls `this.#conversation.transitionBlock(...)` / `appendStreaming(...)`. The branch's version still used `#layout: AppLayout`.
- **`runAgent` no longer takes AppLayout.** It takes a `RunAgentStores` bag (`{ conversationState, toolApprovalState, commandModeState, editorState, primaryViewState }`) plus a `flushToScroll` callback. Phase is tracked via `primaryViewState.setPhase('streaming'|'editor')` instead of AppLayout's `#mode` field.
- **`ConversationState` became event-driven.** #339 added `EventEmitter`, `on()`/`off()`, `appendStreaming()` (sanitises + emits), `clear()`, and `.emit('change')` on every mutator. `ViewHost` subscribes to `change` and schedules a render. This is the MVVM reactive wiring.

## The two collision points that matter

1. **`markPromptStart()` lost its home.** It lived in `AppLayout.waitForInput()`. In the refactor, the old `waitForInput` split: editor-reset went to `EditorHandler.waitForInput()`, mode-change went to `primaryViewState.setPhase()` in `runAgent`. The idle-entry moment is now the `editorHandler.waitForInput()` call in `main.ts`'s `while(true)` loop. I put `markPromptStart()` there — `conversationState` is already in scope, no new DI. Did NOT add it to `EditorHandler` (would widen its concern for nothing).

2. **The prompt block stopped going through the lifecycle.** #339's `runAgent` adds the prompt via `addBlocks([{type:'prompt', content}])` — pre-sealed, NO timestamps. The branch ran it through `transitionBlock('prompt') + appendToActive + completeActive` (via the old `AppLayout.startStreaming`). To preserve prompt timing, the merge must restore the lifecycle path in `runAgent`. This is the non-obvious one — easy to miss because `addBlocks` "works" but silently drops the timestamps.

## A trap for the Maker

`appendStreaming` is a no-op when `activeBlock === null`. So the error-handler tests can't just fire an error against a fresh state and check content — the content would be undefined. Must seal/open a block first. This is why the new error tests pre-transition to `meta`.
