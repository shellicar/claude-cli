Courier — configurable tool permissions (#101). Shipped.

## What was built

The hard-coded `permissions` const in `permissions.ts` is gone. `getPermission` now
takes the permission matrix as a required 4th parameter (`matrix: PermissionConfig`)
and `IFileSystem` as a required 5th (`fs: IFileSystem`). The config schema has a
`permissions` block (two zones: `default` for inside cwd, `outside`; each with
`read`/`write`/`delete` as strings `approve`/`ask`/`deny`).

Live wiring in `main.ts`: `getPermissionMatrix` reads `configLoader.config.permissions`
at call time, maps config strings to `PermissionAction` via `actionFromConfig`, and
returns a `PermissionConfig`. `AgentMessageHandler` stores `getMatrix: () => PermissionConfig`
and calls `this.#getMatrix()` at approval time. Same pattern as `getEffectiveModel`.

When permissions config changes, `conversationState.spliceNotice(formatPermissionsDisplay(config.permissions))`
injects a `🔔 notice` block immediately — mid-turn safe, not deferred to idle.

## Path normalisation: the gate must match the door

Every file tool calls `expandPath(input.path, fs)` before acting. `isInsideCwd` must
see the same expanded path, or `~/...` paths resolve under cwd and get the `default`
zone instead of `outside`. Fix: `getPermission` calls `expandPath(rawPath, fs)` before
`isInsideCwd`. `IFileSystem` is threaded as the 5th param and recursed through Pipe.

## Schema: per-zone defaults matter

`zonePermissionsSchema` was split into `defaultZonePermissionsSchema` (approve/approve/ask)
and `outsideZonePermissionsSchema` (approve/ask/deny). Every field's `.default()` and
`.catch()` equals that field's real value for its zone. Previously all-deny field defaults
meant a partial config (e.g. only `read` specified) silently denied the omitted fields.
The single `deny` remaining in the schema is `outside.delete`, which is its real value.

## Notice: `notice` block type, not `meta`

`meta` maps to the `query` label in `renderConversation.ts`. Injecting as `meta` groups
the notice visually with the turn's query_summary block. Added `'notice'` to `BlockType`,
rendered as `🔔 notice` (`BLOCK_EMOJI`). Use this type for any injected status notices.

## Startup spurious notices

`ConfigLoader` already deduplicates via JSON.stringify equality + 100ms debounce.
`onChange` only fires when the config genuinely changes. No permission-comparison logic
is needed inside the onChange callback — `formatPermissionsDisplay` is always called,
which is fine.

## `schema/sdk-config.schema.json`

A generated JSON-schema mirror of the Zod config schema. Regenerated by
`pnpm --filter @shellicar/claude-sdk-cli run build`. Must be staged alongside every
`schema.ts` change. Forgetting this leaves the mirror advertising the old defaults.

## Required options on AgentMessageHandler

`getMatrix` and `fs` are required options. Changing required options on this class
always breaks the `makeOpts` test factory. Pattern across every phase: add the new
option to `makeOpts` in `AgentMessageHandler.spec.ts` with a test stub.

## Staged testament blocks git merge

A staged-but-uncommitted file blocks `git merge` if the incoming side would update or
delete it. If another merge may follow, either commit the testament or leave it
untracked. Don't leave it staged between phases.

## String-to-enum mapping

Config stores `approve`/`ask`/`deny` (strings). `PermissionAction` is a numeric TypeScript
enum. Mapping lives in `main.ts`'s `actionFromConfig`. Adding new zones or actions
requires updating both `actionFromConfig` and the Zod schema.

## What was built

`@shellicar/core-di-lite` is now the composition root for `apps/claude-sdk-cli`. All object construction lives in `src/setup/container.ts`; `main()` retains activation, event wiring, and the turn loop.

## Container API

- `createServiceCollection()` returns `IServiceCollection`
- `.register(Identifier).to(Implementation)` or `.to(Identifier, factory)`
- Factory form: `(x: IServiceProvider) => x.resolve(Dep)`
- `.buildProvider()` — eager, synchronous, resolves the full DAG at once
- `provider.resolve(Identifier)` — returns the singleton
- Throws `DuplicateRegistrationError` on duplicate registration
- Registration order does not matter (DAG-sorted at build)

## Composition vs activation

**In the container (`src/setup/container.ts`):** all `new X(...)` constructions, ~35 services.

**Stays in `main()` (activation):** `await auth.getCredentials()`, `await tsServer.start()`, session init (`resume/startFresh/load`), `configLoader.start()` (file watcher), signal handlers, event wiring (`client.on`, `processor.on`, channel subscriptions), `renderer.enter()`, history replay, turn loop.

`configLoader.load()` is synchronous I/O; it runs before `buildContainer` so the container stays synchronous. The loaded instance is passed in via `ContainerOptions`.

## New setup/ files

**`SdkChannel` / `ConsumerChannel`**: named subclasses of `ControlChannel<SdkMessage>` and `ControlChannel<ConsumerMessage>`. The container uses the class reference as identifier; registering the same generic class twice throws `DuplicateRegistrationError`. Named subclasses are zero-cost and solve that.

**`ModelOverrides`**: wraps the mutable `{ model, thinking, effort }` overrides object and implements `ModelSettings`. Depends on `StatusState` for `cycleThinking()`/`cycleEffort()` calls.

**`AppToolsService`**: wraps `createAppTools()` so `tools`, `store`, and `refTransform` are one injectable singleton shared by both `ToolRegistry` and `AgentMessageHandler`.

**`DurableConfigFactory`**: replaces the `mapConfig()` closure. Owns the shared mutable `DurableConfig` reference. `QueryRunner` and `AgentMessageHandler` receive `factory.config` (same object). `factory.update()` is called per-turn before each query; `factory.resolveSystemPromptsFor(sessionId)` is called at startup and on session change.

**`container.ts`**: takes `ContainerOptions` (`{ configLoader, systemFlagText, ... }`), returns `IServiceProvider`.

## ReadLine raw-mode timing (open question)

`ReadLine`'s constructor calls `process.stdin.setRawMode(true)`. The container resolves `ReadLine` eagerly at `buildProvider()`, so stdin enters raw mode before `renderer.enter()`. The original code constructed `ReadLine` after `renderer.enter()`. Build/test gates do not exercise interactive TTY ordering, so whether this produces a visible artifact is a runtime question. If you see raw-mode-before-alt-buffer artifacts, `ReadLine` is where to look.

## Permissions display: the on-change comparison is load-bearing (do NOT remove)

**The rule:** the `🔑 Permissions` matrix must be spliced into the conversation ONLY when the permission matrix actually changes. A model change, or any other unrelated config edit, must NOT reprint it.

**Current state (broken on `main` and this branch; unfixed as of writing):** the `onChange` callback in `main.ts` calls `conversationState.spliceNotice(formatPermissionsDisplay(config.permissions))` unconditionally, so the matrix reprints on every genuine config reload, a model-only change included. `ConfigLoader` dedups identical configs (JSON.stringify equality), so it only fires on real changes, but it fires for all of them, not just permission changes. Symptom: repeated identical `🔑 Permissions` blocks after editing `model` in `sdk-config.json`.

**The correct design existed in commit `2c7ea94` ("Add permission change display"):**
- `formatPermissionChange(prev, next): string | null` — returns the matrix only when permissions differ, `null` when unchanged.
- a `stablePermissions` tracker in `onChange` — splice only when `formatPermissionChange` returns non-null.
`#347`'s final form (`1676610`) dropped the comparison and kept only `formatPermissionsDisplay` (full-matrix display), now called unconditionally. `formatPermissionChange` and `permissionsEqual` were deleted from `formatPermissionChange.ts`.

**Why it regressed — the real lesson:** the comparison was deliberate, but it was never documented as deliberate. To a later reader it looked like a redundant call, so it got "simplified" to the unconditional display, and the testament was rewritten to bless it. The `#101` courier entry literally said permission-comparison logic "is not needed... formatPermissionsDisplay is always called, which is fine." That note is WRONG. With no test pinning the behaviour and no testament recording the intent, the regression was invisible and got canonised as correct.

**If you touch this:** the comparison is the feature, not noise. Display fires only on a genuine permissions diff. Do not collapse it back to an unconditional `formatPermissionsDisplay` call. Fix shape: restore `formatPermissionChange` + `permissionsEqual`, seed the baseline from the already-loaded `configLoader.config.permissions` at registration (so even the first reload prints only on a real change, closing the original's "first change always shows the full matrix" quirk), and add a regression test — reload with unchanged permissions → no notice; reload with changed permissions → notice.

**Scope note:** this bug is on `main`, not introduced by the core-di refactor. The refactor faithfully carried the existing (buggy) behaviour forward, which is the correct outcome for a behaviour-preserving refactor. The fix belongs on its own `fix/` branch, not in the core-di PR.
