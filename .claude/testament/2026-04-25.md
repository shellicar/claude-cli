# 00:25

Scout phase for #311 (PDF ReadFile support).

## What I mapped

The full data flow is in the investigation report at `projects/claude-cli/investigation/2026-04-25_311_pdf-readfile-scout.md`. The short version:

**Tool result path:**
1. `ReadFile.handler` returns `ReadFileOutput` (object, not string)
2. `ToolRegistry.run` closure stringifies via `JSON.stringify` → `ToolRunResult { kind: 'success', content: string }`
3. `QueryRunner.#handleTools` wraps in `{ type: 'tool_result', tool_use_id, content: string }` → returns `BetaToolResultBlockParam[]`
4. `QueryRunner.run` at line 160: `this.#conversation.push({ role: 'user', content: toolResults })`
5. `TurnRunner.run` reads via `conversation.cloneForRequest()`, `buildRequestParams()` uses messages as-is

**The injection point:** line 160 in `QueryRunner.ts`. The document block sidecar needs to be appended to the content array alongside the tool results.

**Pipe path:** Pipe calls `handler` directly, bypasses `ToolRegistry`. If ReadFile returns a string for a PDF, Pipe passes it as `content` to the next step, which fails schema validation. Last-step PDFs in a pipe work fine (Pipe returns the string). This matches the design intent.

## The hard part

`IFileSystem.readFile` is UTF-8 only. No binary reading. `NodeFileSystem.readFile` explicitly passes `'utf-8'`. To base64-encode a PDF, you need raw bytes.

Two options: (1) add `readBinary` to `IFileSystem` in `claude-core` — touches abstract class, `NodeFileSystem`, `MemoryFileSystem`; (2) pass a `readBinary` closure in the factory options — scoped to `claude-sdk-tools`, no cross-package changes. I flagged both in the report.

## The signal mechanism

`ToolRunResult` in `public/types.ts` is currently `{ kind: 'success'; content: string } | ...`. No room for a sidecar. Needs extending with `sidecar?: BetaRequestDocumentBlock`. The ToolRegistry detects a special output shape from the handler and populates `sidecar`. The QueryRunner collects sidecars and appends them.

## For the Apostle

The investigation report has specific file paths, line numbers, and type definitions for everything. The key decisions are:
1. Binary reading strategy (option 1 or 2 above)
2. The exact shape of `PdfSidecarResult` that both the tool handler and ToolRegistry share
3. Whether to detect PDF by extension (`.pdf`) or magic bytes (`%PDF-`)
4. What size limit applies to PDFs (the current 500 KB text limit should not apply)

The Apostle can start coding immediately from the report without re-reading the source files.

# 01:07

Apostle phase for #311 (PDF ReadFile support).

## What I designed

Plan is at `projects/claude-cli/investigation/2026-04-25_311_pdf-readfile-plan.md`.

Fifteen files change (three `MemoryFileSystem` fakes, four ReadFile source files, three SDK source files, three test files for ReadFile/ToolRegistry/QueryRunner, plus the SDK index).

## Key decisions

**Binary read:** `IFileSystem.readFile` gains `encoding?: BufferEncoding`. `NodeFileSystem` passes it through to Node's `fs.readFile`. `MemoryFileSystem` handles `'base64'` via `Buffer.from(content).toString('base64')`. This keeps the abstraction consistent across all three MemoryFileSystem copies.

**Signal type:** `ToolResultWithAttachments { __toolAttachments: ToolAttachment[]; content: string }` defined in `claude-sdk/src/public/types.ts` and exported from the package index. ReadFile imports it from `@shellicar/claude-sdk` (the package already depends on it). The double-underscore prefix marks it as an SDK protocol property, avoiding collision with normal handler output shapes.

**ToolRunResult sidecar:** `{ kind: 'success'; content: string; sidecar?: BetaContentBlockParam[] }`. ToolRegistry detects `isToolResultWithAttachments(output)` in the run closure and builds the sidecar blocks. Separate builder function handles `application/pdf` → `BetaRequestDocumentBlock` and `image/*` → `BetaImageBlockParam`.

**QueryRunner:** `#handleTools` return type changes to `{ toolResults, sidecars }`. `ready` list's `run` closures return `{ block, sidecars }`. Both execution paths (approval and sequential) accumulate sidecars alongside tool results. User message becomes `[...toolResults, ...sidecars]`.

## The Pipe emergent case

When ReadFile with `mimeType: 'application/pdf'` is the LAST step in a Pipe:
- Pipe's handler returns `ToolResultWithAttachments`
- ToolRegistry processes Pipe's own result and fires `isToolResultWithAttachments` on Pipe's output
- PDF sidecar is attached — works correctly

When ReadFile is a NON-FINAL step in a Pipe:
- Next step's schema validation fails (`type: 'content'` expected, `ToolResultWithAttachments` has no such field)
- Pipe throws schema error — intended behaviour

The locked decision said "Pipe bypasses the attachment mechanism" — that's true for ReadFile's internal call by Pipe, but Pipe's own output through ToolRegistry does carry the attachment if ReadFile was the last step. This is a pleasant emergent property, not a bug. The plan notes it explicitly for the Maker.

## For the Maker

The plan has complete, copy-pasteable code blocks for every changed file. The Maker should read the plan end to end before touching any files — the file change summary table at the bottom is a checklist. The most complex change is QueryRunner's `#handleTools`: the `ready` list type changes, both execution paths change, and the return type changes. The call site at line 160 also changes. Read these together.

The Pipe emergent case (above) requires no code change but the Maker should know about it.

Tests: MemoryFileSystem handles `'base64'` encoding via Buffer, so the PDF size limit test uses `'x'.repeat(33 * 1024 * 1024)`. MemoryFileSystem `stat` returns `size: content.length` (string length in bytes for ASCII), so this exceeds the 32 MB limit correctly.

# 02:14

Apostle phase (second pass) for #311. Previous plan rejected. Rewrote sections 2 and 3.

## What was wrong

The first plan used a sidecar mechanism: a `__toolAttachments` magic property on the handler output, duck-typed detection in the ToolRegistry, a separate `sidecars` accumulation path through `#handleTools`, and spreading sidecars alongside toolResults in the user message as siblings.

Three things made this wrong:

1. `BetaToolResultBlockParam.content` already accepts `BetaRequestDocumentBlock`. The attachment belongs INSIDE the tool_result.content array, not as a sibling at the user message level. The sibling path was a parallel delivery mechanism solving a problem that didn't exist.

2. Duck typing on `unknown` (`isToolResultWithAttachments`) is wrong when the output is already typed through zod schemas. ReadFile should own its binary output shape in its own schema.

3. Running attachment data through the transform before extraction. The ref-swapper transform would see `{ __toolAttachments: [...10MB base64...] }` and potentially ref-swap the whole object.

## What the corrected plan does

ReadFile's output schema gains `ReadFileBinarySuccessSchema` with `type: 'binary'` discriminant, `path`, `mimeType`, `data` (base64), `sizeKb`. This is ReadFile-owned.

`ToolDefinition`/`AnyToolDefinition` gain optional `buildToolResultContent?: (output: T) => { text, blocks } | null`. ReadFile provides this hook. Returns null for text/error outputs (default stringify path), returns `{ text, blocks }` for binary outputs.

ToolRegistry calls `buildToolResultContent` BEFORE the transform. When non-null: skip transform, return `{ kind: 'success', content: text, blocks }`. Binary data never reaches the transform.

`ToolRunResult` gains optional `blocks?: ToolAttachmentBlock[]`. The run closure inside `QueryRunner.ready.push` assembles `tool_result.content` as an array `[{ type: 'text', text: content }, ...blocks]` when blocks non-empty.

`#handleTools` signature UNCHANGED. Return type UNCHANGED. User message push UNCHANGED. The simplification compared to the rejected plan: three fewer moving parts (no separate sidecar accumulation, no changed #handleTools signature, no changed push).

## Key types

- `ToolAttachmentBlock = BetaRequestDocumentBlock | BetaImageBlockParam` — defined in `claude-sdk/public/types.ts`, exported from package
- `buildToolResultContent` on `AnyToolDefinition` uses `never` parameter (same erasure as `handler: (input: never) => Promise<unknown>`)
- ToolRegistry casts to `(output: unknown) => ...` at call time

## For the Maker

The plan has complete, copy-pasteable code blocks. The magic bytes check uses `Buffer.from(data.slice(0, 20), 'base64')` — decoding only the first 20 base64 chars (~15 binary bytes) avoids a full 32MB decode for validation.

Test fixtures for magic bytes tests must start with the correct bytes: `'%PDF-'` prefix for PDFs, `'\xff\xd8\xff'` for JPEGs, etc. The MemoryFileSystem encodes via `Buffer.from(content)` so the stored string IS the binary content.

The Pipe last-step-ReadFile case: Pipe has no `buildToolResultContent`, so binary data ends up as a JSON string in the tool result. Not useful but not broken. Not in scope.

Check `claude-sdk-tools` package's dependency on `claude-sdk` before writing imports — it already imports `defineTool` from `@shellicar/claude-sdk` so `ToolAttachmentBlock` import is safe.

Apostle phase (third pass) for #311. Code quality corrections applied.

## What changed

**Named type.** `ToolBuildContentResult = { text: string; blocks: ToolAttachmentBlock[] }` replaces the inline type everywhere (definition, ToolRunResult, ToolRegistry cast).

The old `ReadFileBinarySuccessSchema.safeParse(output).success` check (parse, discard result, cast) replaced with `'type' in output && output.type === 'binary'`. TypeScript narrows to `ReadFileBinarySuccess`. No parse, no cast.

**QueryRunner.** The if/else block is now a ternary: `const content = runResult.blocks?.length ? [...] : runResult.content;`.

**Cast in ToolRegistry.** Still needed — same `never → unknown` erasure as `handler`. With the named type it's `entry.definition.buildToolResultContent as ((output: unknown) => ToolBuildContentResult | null) | undefined`.

## `null` meaning

`buildToolResultContent` returns `null` for text outputs (PipeContent) and error outputs (ReadFileOutputFailure). The ToolRegistry's `if (attached !== null)` gates the binary path. Default stringify runs for null. This is correct — ReadFile can produce both binary and text outputs from the same handler depending on `mimeType`.

The comment in the plan and in `ReadFile.ts` explains this: "Returns null for text and error outputs — the ToolRegistry falls through to the default stringify path."

# 05:17

Maker phase for #311 (PDF ReadFile support + output_schema refactor).

## What I built

The plan called for PDF support in ReadFile. I implemented it, but immediately hit a DTS build failure caused by the `handler: ... => Promise<{ textContent: TOutput }>` change exposing a pre-existing TypeScript inference problem with multiple return paths in a handler.

The fix required adding `output_schema: TOutputSchema extends z.ZodType` to `ToolDefinition`. This was the refactor the SC had already planned — it surfaced now because the handler return type change forced it.

## The actual change set (40 source files + 1 test)

**SDK types** (`packages/claude-sdk/`):
- `ToolDefinition<TSchema, TOutputSchema extends z.ZodType>` — second param is now a Zod schema, not a raw type
- `output_schema: TOutputSchema` field added to `ToolDefinition` and `AnyToolDefinition`
- `handler` return type is now `Promise<{ textContent: z.output<TOutputSchema>; attachments?: ToolAttachmentBlock[] }>`
- `ToolAttachmentBlock`, `ToolRunResult` (with `blocks?`), `defineTool` updated accordingly
- `ToolRegistry` now destructures `{ textContent, attachments }` from handler result; attachments go as `blocks` into `ToolRunResult`
- `QueryRunner` builds `tool_result.content` as `[{ type: 'text', text }, ...blocks]` when blocks present

**claude-sdk-tools** (all 20 source tools):
- Every handler now returns `{ textContent: result }` instead of `result` directly
- Every `defineTool` call has `output_schema: <SchemaConstant>`
- 13 tools use concrete output schemas from their schema files; 5 use `z.unknown()` (Pipe, TsDiagnostics, TsHover, TsDefinition, TsReferences)
- `Ref/schema.ts` gains `RefOutputSchema` (discriminated union for the Ref tool output)
- `IFileSystem.readFile` gains `encoding?: BufferEncoding`; `NodeFileSystem` and two `MemoryFileSystem` copies handle base64 encoding
- ReadFile: `mimeType` param, magic bytes validation, binary read path with `attachments`
- `ReadFile/schema.ts`: `SupportedMimeTypeSchema`, `BinaryMimeTypeSchema`, `ReadFileBinarySuccessSchema`

**mcp-exec**: destructures `.textContent` from Exec handler result; `ExecOutput` type updated to `['textContent']`.

**Test infrastructure**:
- `call` helper: `ToolDefinition<T, TOut extends z.ZodType>` returning `z.output<TOut>`
- `callFull` helper: same signature, returns full `{ textContent, attachments? }`
- `AnyToolDefinition` inline objects in test specs need `output_schema: z.unknown()`
- Find.spec.ts and find-symlinks.spec.ts: `as { type: string; values: string[] }` casts needed because `call` now returns the full output union including error branches

## The DTS problem and its resolution

The TS2719 "two different types with this name exist" error from tsup's parallel DTS workers was the underlying issue. Without `output_schema`, TypeScript inferred `TOutput` from the handler's multiple return paths and produced a union type. The parallel DTS workers computed conflicting representations of that union.

The `output_schema: TOutputSchema` field gives TypeScript a single concrete inference site for `TOutputSchema`, computed from the schema value rather than from the handler body. The handler's return type becomes `Promise<{ textContent: z.output<TOutputSchema>; ... }>` where `TOutputSchema` is already resolved.

Two additional literal-type fixes were needed:
1. `type: 'content' as const` / `type: 'files' as const` in Grep/Head/Tail/Range/SearchFiles — TypeScript widens string literals in object returns without `as const`
2. `error: true as const` / `error: false as const` in CreateFile — same widening issue with boolean literals

## For the next operator

The `Ref.ts` edit sequence had a collision: two edits were staged on the same file in sequence, but the second was computed before the first applied. When the second landed, it used the pre-first-edit state as its base and overwrote the first edit's changes. The fix was to apply all changes to the same file in a SINGLE `PreviewEdit` call using `textEdits` (which apply to current file content, not a snapshot). Never stage multiple sequential file edits to the same file as separate PreviewEdit calls — combine them.

The `output_schema` change surfaces in any caller that holds a `ToolDefinition<T, O>` reference where `O` was a plain type. Those callers need updating to `ToolDefinition<T, TOut extends z.ZodType>` with `z.output<TOut>` for the output type.
