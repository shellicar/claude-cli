# PDF ReadFile support (#311)

## What was built

PDF and image support in ReadFile. When `mimeType` is specified, ReadFile reads the file as base64, validates magic bytes, and returns an attachment block. The SDK injects the block inside `tool_result.content` so the API receives the binary content natively.

This PR also includes `output_schema` on `ToolDefinition` — all 20 tool handlers now return `{ textContent: result }` instead of `result`.

## Key decisions

**Attachments go inside `tool_result.content`, not as siblings.** `BetaToolResultBlockParam.content` already accepts `BetaRequestDocumentBlock`. The first Apostle plan built a sidecar mechanism (magic `__toolAttachments` property, duck-typed detection, parallel accumulation path, spreading sidecars alongside tool results in the user message). All unnecessary. The SC corrected it.

**Binary data never enters `textContent`.** Handler returns `{ textContent: TOutput; attachments?: ToolAttachmentBlock[] }`. ToolRegistry destructures before calling `transform`. The ref-swapper sees only the small metadata object in `textContent`, never the 32 MB base64 data.

**`output_schema: TOutputSchema extends z.ZodType` on ToolDefinition.** Without it, tsup's parallel DTS workers computed conflicting representations of handler return unions (TS2719). The field was already planned by the SC; the `{ textContent }` handler return type change forced it in. It gives TypeScript a single concrete inference site resolved from the schema value, not the handler body.

**`blocks` must be absent (not `undefined`) when there are no attachments.** ToolRegistry returns `{ kind: 'success', content }` with no `blocks` key when `attachments === undefined`. Existing `toEqual({ kind: 'success', content: '...' })` assertions fail if `blocks: undefined` is included.

## Traps

**Multiple PreviewEdit calls to the same file.** Each call computes against the current file state. If you stage two calls on the same file sequentially, the second is computed before the first applies and overwrites the first edit's changes when it lands. Combine all changes to the same file into a single PreviewEdit call.

**Literal widening.** TypeScript widens `'content'` to `string` in object returns. Grep, Head, Tail, Range, SearchFiles needed `type: 'content' as const` and `type: 'files' as const`. CreateFile needed `error: true as const` / `error: false as const`. Symptom: DTS build failure.

**`ConfirmEditFile.ts` had an explicit `Promise<EditFileOutput>` return type annotation.** Had to be removed when wrapping the return in `{ textContent }`.

**Inline `AnyToolDefinition` objects in tests need `output_schema: z.unknown()`.** Any tool object literal not created via `defineTool` needs the field or the type check fails.

**Unused type imports left by the Maker block the pre-push hook.** Biome flags them as unsafe fixes. `pnpm ci:fix` handles safe/format fixes; unused imports and missing block statements need manual removal.
