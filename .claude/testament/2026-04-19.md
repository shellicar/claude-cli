## GHSA-458j-xx4x-4375: hono HTML injection in JSX SSR

### The fix

pnpm override in `pnpm-workspace.yaml`. The CVE path is transitive: `claude-cli > @modelcontextprotocol/sdk > hono` and `mcp-exec > @modelcontextprotocol/sdk > hono`. Neither package is a direct dependent of hono; the override is the right tool.

The existing overrides file already had two overlapping hono entries (`hono@<4.12.12` and `hono@>=4.0.0 <=4.12.11`). Phase 1 added a third narrow range (`hono@<4.12.14`) rather than updating the existing ones. This leaves redundant/overlapping overrides in the file. Not a correctness problem (pnpm resolves them all), but the file will accumulate noise over time.

### The placement trap in `changes.jsonl`

The tooling reads top-to-bottom: entries **before** each release marker belong to that release; entries **after the last release marker** become [Unreleased].

Phase 1 inserted the security entry before the `1.0.0-alpha.67` release marker. That would have attributed the fix to an already-released version rather than [Unreleased]. Phase 2 caught this and moved the entry to after the marker before shipping.

The rule is simple but easy to violate when appending near the bottom of a file that ends with a release marker. Always check the tail: if the last non-empty line is a release marker, your new entry must go after it.

### Packages with security entries

Only `apps/claude-cli` and `packages/mcp-exec` were updated. Both depend on `@modelcontextprotocol/sdk`. No other packages in the monorepo have that dependency.

Task: fix #276 — compact should default to disabled.

## What changed

Three places in `apps/claude-sdk-cli/src/cli-config/schema.ts`:
- `enabled` field: `.default(true).catch(true)` → `.default(false).catch(false)`
- outer `.default({...})`: `enabled: true` → `enabled: false`
- outer `.catch({...})`: `enabled: true` → `enabled: false`

The JSON schema at `schema/sdk-config.schema.json` is generated by the build (`pnpm build --filter=@shellicar/claude-sdk-cli`). It updated automatically — compact.enabled now shows `"default": false` there too.

One test in `apps/claude-sdk-cli/test/cli-config.spec.ts` (`returns defaults for empty object`) was asserting the old default of `enabled: true` — updated to `false`.

## What the code doesn't say

The `.catch()` value matters as much as `.default()`. Zod's `.catch()` is the fallback when parsing fails (invalid config file, bad JSON). Without updating `.catch()`, a corrupt config file would silently re-enable compact. That's exactly the bug being fixed — the prompt called it out explicitly and it would have been easy to miss.

The outer object `.default()` and `.catch()` are separate from the field-level ones. Zod uses the outer ones when the entire `compact` key is missing from the config file; the field-level ones apply when the key is present but a field within it is missing or invalid.
