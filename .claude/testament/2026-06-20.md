# Maintenance release: pnpm audit → Dependabot, pnpm 11

Strategy change for claude-cli's dependency-security handling, plus the pnpm 10→11 upgrade. Maker → Courier, two phases. No version bumps, no releases, nothing here touches a published artifact.

## Why the strategy changed (the spine)

claude-cli is a **library**. `pnpm audit` gates the whole resolved tree, but a library ships **declared ranges, not a lockfile** — consumers re-resolve. The hono advisories that triggered this were all transitive (via `@modelcontextprotocol/sdk`), lockfile-only, unfixable from our manifests, and never reach consumers through anything we publish. Confirmed empirically: Dependabot alerts report all 5 against `pnpm-lock.yaml`, **zero** against any `package.json`. So `pnpm audit` as a blocking CI gate was an app-shaped tool on a library — blocking the queue over things we can't fix and don't ship.

New strategy: drop the gate; rely on Dependabot **alerts** for detection (read filtered to `package.json` manifests) and **version-updates** to keep declared floors current — the one thing that actually reaches consumers.

## What was built

- `.github/dependabot.yml`: npm lane over `[/, /apps/*, /packages/*, /scripts]`, weekly, `versioning-strategy: increase` (bumps declared floors = shippable, not lockfile theater), `groups: ["*"]` (one PR, avoids a conflict-storm), `ignore` all majors. No `github-actions` lane — that is freshness, not security, deliberately out of scope.
- `pnpm audit` removed from both CI workflows (`node.js.yml`, `npm-publish.yml`).
- Supply-chain hardening in `pnpm-workspace.yaml`: `ignoreScripts`, `minimumReleaseAge: 1440`, `blockExoticSubdeps`, `trustPolicy: no-downgrade`, plus the chokidar exclusion.
- pnpm 10.33.2 → 11.8.0; `packageManager` SHA re-pinned in root `package.json`.

## Traps for the next operator

- **`trustPolicyExclude` is a TOP-LEVEL key in `pnpm-workspace.yaml`, sibling to `trustPolicy` — NOT nested under `auditConfig`.** The dropped supply-chain skill (and the mission block lifted from it) showed `auditConfig.trustPolicyExclude`; that form silently does nothing and install keeps failing `ERR_PNPM_TRUST_DOWNGRADE` on chokidar@4.0.3. The file on disk has the correct top-level placement. **Do not "fix" it back to auditConfig.** Flag to the skill owner.
- **pnpm 11 install wants a TTY** to confirm purging node_modules — run with `CI=true` in a non-interactive cast.
- Dependabot ignore-all-majors form is validated: `dependency-name: "*"` + `update-types: ["version-update:semver-major"]` is documented and only affects version updates, never security updates.

## Decisions (SC)

- Keep the supply-chain config as standalone install-hardening (not tied to the audit approach).
- Ignore majors covers the breaking-change concern, so no separate `syncpack` ignore.
- `pnpm-lock.yaml` was already up to date — install clean, nothing to stage there despite the mission listing it.
