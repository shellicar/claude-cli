# Dependency-security model for claude-cli (read before any CVE / dep mission)

This repo changed how it handles dependency security. If you arrive on a future mission about CVEs, audits, or dependency updates, start here ‚Äî the old instincts are now wrong.

## The model (why pnpm audit is gone)

claude-cli is a **library**. It ships **declared ranges, not a lockfile** ‚Äî consumers re-resolve against their own trees. `pnpm audit` gates the whole *resolved* tree, so it blocks CI on transitive, lockfile-only advisories that can't be fixed from our manifests and never reach anyone consuming the package. It was an app-shaped tool on a library.

So the gate is gone from both CI workflows. The replacement, two lanes:

- **Detection** = Dependabot **alerts**. Read them filtered to `package.json` manifests; an alert that only hits `pnpm-lock.yaml` is lockfile-only and does not reach consumers. (Empirically, the 5 hono advisories that triggered this work reported against `pnpm-lock.yaml` and zero against any `package.json`.)
- **Currency** = Dependabot **version-updates** (`.github/dependabot.yml`): weekly npm lane, grouped into one PR, `versioning-strategy: increase` (bumps declared floors = the thing that actually reaches consumers), majors ignored (deliberate manual call). No `github-actions` lane ‚Äî freshness, not security.

## `pnpm-workspace.yaml overrides` ‚Äî handle with care

The `overrides:` block pins transitive versions **in our lockfile only**. It does not change published bytes, so an override alone reaches no consumer and earns no changelog entry / release.

This block had accumulated a CVE-by-CVE pile from the old audit-chase workflow (one or more lines per advisory). The SC pruned it to a minimal set (`esbuild`, `hono@<4.12.25: ^4.12.25`). **Do not reflexively add an override line per new advisory** ‚Äî that pile is exactly the lockfile theater this mission moved away from. The hono line is what patches the triggering advisories locally; keep the set minimal.

## Traps

- **`trustPolicyExclude` is a TOP-LEVEL key in `pnpm-workspace.yaml`, sibling to `trustPolicy` ‚Äî NOT under `auditConfig`.** The nested form silently does nothing and install keeps failing `ERR_PNPM_TRUST_DOWNGRADE` on chokidar@4.0.3. The dropped supply-chain skill had it wrong; if a brief tells you to nest it, the brief is wrong.
- **pnpm 11 install wants a TTY** to confirm purging node_modules. In a non-interactive cast, run with `CI=true`.
- Dependabot ignore-all-majors form (`dependency-name: "*"` + `update-types: ["version-update:semver-major"]`) is valid and only affects version updates, never security updates.

## What this mission shipped

Drop `pnpm audit` from CI; add `.github/dependabot.yml`; add standalone supply-chain hardening to `pnpm-workspace.yaml` (`ignoreScripts`, `minimumReleaseAge: 1440`, `blockExoticSubdeps`, `trustPolicy: no-downgrade`); upgrade pnpm 10 ‚Üí 11. No version bumps, no releases ‚Äî nothing here touches a published artifact. SC follow-ups after the Courier handoff: removed a defunct `onlyBuiltDependencies: sharp` allowance and collapsed the overrides pile.
