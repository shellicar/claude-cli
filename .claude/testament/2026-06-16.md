# 01:43 AppendFile tool (Scaffolder, Builder, Courier)

`AppendFile` shipped: the additive sibling of `CreateFile` in `@shellicar/claude-sdk-tools`. Same `{ path, content }` input, but it asserts only a suffix, so it has no `overwrite` flag, no existence errors, and no PreviewEdit/EditFile pair. The shape and contract are in the code; below is what the code does not tell you.

## Adding a tool to this repo: the wiring

A new tool reaches the live tool set through three points. The brief named one; miss the other two and the tool either fails to resolve or never loads. Mirror an existing sibling (`CreateFile`) at each:

1. `src/entry/<Tool>.ts`, the entry. tsup globs `src/entry/*.ts`, so this builds automatically with no config change.
2. `packages/claude-sdk-tools/package.json` `exports`: add a `./<Tool>` block. Without it the `@shellicar/claude-sdk-tools/<Tool>` subpath import does not resolve.
3. `apps/claude-sdk-cli/src/createAppTools.ts`: import the tool and put it in the `otherTools` array. This is what actually loads it into the CLI.

## Write-tool design: lean on the fs and the runner

The whole AppendFile handler is `expandPath`, then `fs.appendFile`, then a success return. No try/catch and no existence check, deliberately. Two repo facts make that correct, and they hold for any write tool here:

- `fs.appendFile` creates the file and any missing parent dirs when absent (`NodeFileSystem` runs `mkdir(dirname, { recursive: true })` first; `MemoryFileSystem` concatenates onto an empty buffer). So a missing file is a normal write, not an error to guard.
- The SDK tool runner catches thrown errors and surfaces the message to the model. A hand-written try/catch only restates the OS error (EISDIR, EACCES), and risks misdescribing it.

Add neither guard unless the tool needs behaviour the runner and fs do not already give you.

## Traps

- The worktree is not bootstrapped. A fresh worktree has root `node_modules` but no `@shellicar/*` workspace links and no built `dist/` for `claude-core` or `claude-sdk`. Until you run `pnpm install --frozen-lockfile && pnpm build`, every `type-check` and `test` fails with `Cannot find module '@shellicar/...'` and spurious TS7006. Frozen is correct: only the `package.json` `exports` map changed, no deps.
- biome does not flag an unused factory param. A scaffold stub can keep `createAppendFile(fs)` with `fs` unused and biome stays clean, so there is no need for a `_fs` workaround.

## For the next Courier: the union testament merge bites

`.claude/testament/*.md` uses `merge=union` (see `.gitattributes`), so dated files from concurrent missions auto-combine. Two things this cast taught:

- A branch behind main still needs the merge. This branch was cut at `a33082e`; PR #353 and the retry fix had since added entries to `2026-06-15.md`. Merge `origin/main` before shipping, or you silently drop them from the dated file.
- Union duplicates a seam if you break the trailing newline. When you remove your mission's fragment from a shared dated file, keep the file's final newline. I left `2026-06-15.md` without it, and the union driver then read the last line as changed on both sides and kept both copies, duplicating it. The fix is a one-line delete after the merge, but it is invisible unless you diff the merged file against `origin/main`, so do that.
