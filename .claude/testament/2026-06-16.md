# 01:43 AppendFile tool (Scaffolder, Builder, Courier)

`AppendFile` shipped: the additive sibling of `CreateFile` in `@shellicar/claude-sdk-tools`. Same `{ path, content }` input, but it asserts only a suffix, so it has no `overwrite` flag, no existence errors, and no PreviewEdit/EditFile pair. The shape and contract are in the code; below is what the code does not tell you.

## Adding a tool to this repo: the wiring

A new tool reaches the live tool set through three points. The brief named one; miss the other two and the tool either fails to resolve or never loads. Mirror an existing sibling (`CreateFile`) at each:

1. `src/entry/<Tool>.ts`, the entry. tsup globs `src/entry/*.ts`, so this builds automatically with no config change.
2. `packages/claude-sdk-tools/package.json` `exports`: add a `./<Tool>` block. Without it the `@shellicar/claude-sdk-tools/<Tool>` subpath import does not resolve.
3. `apps/claude-sdk-cli/src/createAppTools.ts`: import the tool and put it in the `otherTools` array. This is what actually loads it into the CLI.

## Write-tool design: lean on the fs and the runner

The whole AppendFile handler is `expandPath`, then `fs.appendFile`, then a success return. No try/catch and no existence check, deliberately. Two repo facts make that correct, and they hold for any write tool here:

- `fs.appendFile` creates the file and any missing parent dirs when absent (`NodeFileSystem` runs `mkdir(dirname, { recursive: true })` first; `MemoryFileSystem` concatenates onto an empty buffer). So a missing file is a normal write, not an error to guard.
- The SDK tool runner catches thrown errors and surfaces the message to the model. A hand-written try/catch only restates the OS error (EISDIR, EACCES), and risks misdescribing it.

Add neither guard unless the tool needs behaviour the runner and fs do not already give you.

## Traps

- The worktree is not bootstrapped. A fresh worktree has root `node_modules` but no `@shellicar/*` workspace links and no built `dist/` for `claude-core` or `claude-sdk`. Until you run `pnpm install --frozen-lockfile && pnpm build`, every `type-check` and `test` fails with `Cannot find module '@shellicar/...'` and spurious TS7006. Frozen is correct: only the `package.json` `exports` map changed, no deps.
- biome does not flag an unused factory param. A scaffold stub can keep `createAppendFile(fs)` with `fs` unused and biome stays clean, so there is no need for a `_fs` workaround.

## For the next Courier: the union testament merge bites

`.claude/testament/*.md` uses `merge=union` (see `.gitattributes`), so dated files from concurrent missions auto-combine. Two things this cast taught:

- A branch behind main still needs the merge. This branch was cut at `a33082e`; PR #353 and the retry fix had since added entries to `2026-06-15.md`. Merge `origin/main` before shipping, or you silently drop them from the dated file.
- Union duplicates a seam if you break the trailing newline. When you remove your mission's fragment from a shared dated file, keep the file's final newline. I left `2026-06-15.md` without it, and the union driver then read the last line as changed on both sides and kept both copies, duplicating it. The fix is a one-line delete after the merge, but it is invisible unless you diff the merged file against `origin/main`, so do that.

## The finding that decides scope

The SDK runtime reaches **three** places, not one:

1. **Transport + stream object.** `AnthropicClient` → `TokenRefreshingAnthropic extends Anthropic` → `beta.messages.stream()` returns a `BetaMessageStream`. (`AnthropicClient.ts:76`, `MessageStreamer.ts:5`.)
2. **The stream-consumer boundary.** `StreamProcessor.process(stream: BetaMessageStream)` consumes the SDK's *assembled* event vocabulary (`streamEvent`/`text`/`thinking`/`inputJson`/`contentBlock`/`end`) **and** `finalMessage()`. It does NOT work from raw SSE, and it does NOT work from an already-finished result. The `contentBlock` payload and `finalMessage()` are the SDK's accumulation output (snapshot mutation + partial-JSON tool-input parsing via the SDK's vendored `partialParse`). This is the real measure of the work.
3. **The retry classifier.** `backoff.ts:1` imports `APIError`/`APIConnectionError`/`APIUserAbortError` as runtime values and `instanceof`-checks them in `isRetryable`. This is a coupling *outside* the transport — "change nothing else" has its sharpest edge here.

Everything else importing `@anthropic-ai/sdk` is `import type`. Only TWO runtime value-imports exist in `claude-sdk/src`: `TokenRefreshingAnthropic.ts` (`Anthropic`) and `backoff.ts` (the three error classes).

## What is already clean (cuts the work down)

- **Downstream of StreamProcessor is fully insulated.** `MessageStreamResult` + `MessageStreamEvents` (`private/types.ts:21-48`) are built only from our own `ContentBlock` + primitives. `QueryRunner`, the channel, the consumer never see an SDK runtime object. The SDK-stream coupling is contained in `StreamProcessor.process`'s parameter + body and `AnthropicClient`/`IMessageStreamer`.
- **Auth split in two.** Acquisition/refresh (`Client/Auth/`) is plain `fetch` — `refreshCredentials.ts` POSTs to the token URL; trigger is an expiry check (`isExpired.ts`: `Date.now() >= expiresAt`), NO 401 path. Untouched by the swap. The SDK-coupled part is only `TokenRefreshingAnthropic`'s `bearerAuth`/`apiKeyAuth` overrides, which exist solely to subclass `Anthropic`. A from-scratch transport sets `Authorization: Bearer` directly and **deletes `TokenRefreshingAnthropic.ts` + `sdkInternals.ts` entirely**.
- **Q4 confirmed.** `claude-sdk-tools` imports `@anthropic-ai/sdk` zero times. `claude-sdk-cli` imports it `import type` only; its runtime imports are from `@shellicar/claude-sdk`. A runtime swap leaves both untouched in source.

## Traps / things the code doesn't say

- **`sdkInternals.ts:3` says "Mirrored from 0.82.0" but installed is 0.92.0.** Stale by-hand mirror of `@anthropic-ai/sdk/internal/*` (those paths aren't in the package `exports`). Irrelevant if the swap deletes the subclass; a live risk if any subclassing path survives.
- **Fresh worktree had no `node_modules`** (root probe failed). `pnpm install --frozen-lockfile` fixed it — "Already up to date", nothing tracked changed. The SDK source is NOT at the workspace root; it's under `node_modules/.pnpm/@anthropic-ai+sdk@0.92.0_zod@4.4.3/node_modules/@anthropic-ai/sdk/`. The reference surfaces: `lib/BetaMessageStream.mjs` (accumulation), `core/streaming.mjs` (SSE framing), `resources/beta/messages/messages.mjs` (`stream`→`create`→`post('/v1/messages?beta=true')`, streaming timeout default 600000ms), `client.mjs` (`authHeaders`/`bearerAuth`, `makeStatusError` on non-2xx).
- **SDK does its own retries internally; `TurnRunner` also retries on top** (`MAX_RETRIES=10`). A swap changes who owns retry/timeout — flagged in the report as a decision, not measured in full.
- **Worktree, isolated** (preflight `worktree:true`) — co-working not loaded. Tree was clean at start; I only created the testament here and the report in the fleet repo.

## Addendum findings (streaming error handling)
- **Two error sites.** (a) Connect-phase HTTP errors thrown before any SSE body: conn failure/timeout caught `client.mjs:244` → `APIConnectionTimeoutError`/`APIConnectionError` (`:273`/`:275`); non-2xx `if(!response.ok)` (`:282`) → `makeStatusError` throw (`:311`). - **SDK internal retry (`makeRequest`) covers ONLY the connect phase.** Default `maxRetries`=2 (`client.mjs:42`/`:70`), not overridden by `AnthropicClient`. `shouldRetry` (`:366`-`:386`): `x-should-retry` header, else 408/409/429/≥500. `retryRequest` (`:388`-`:417`): honours `retry-after-ms` then `retry-after` (seconds or HTTP-date) **UNCAPPED** — sleeps exactly the header (`:415`), the 24h+ behaviour; else default backoff `min(0.5×2^n,8)`s.
- **Mid-stream SSE errors bypass SDK retry entirely** (200 OK already returned) — only `TurnRunner` retries them, and it never reads `retry-after`.
- **TurnRunner layers its own loop on top:** 11 attempts (`backoff.ts:5`), `isRetryable` on SDK error classes + `error.type`, `calculateBackoffDelay` (500ms→32s, 0-50% jitter). Net: ~2×11 connect-phase HTTP attempts; `retry-after` consulted only inside the SDK's inner 2.
- Report's reproduce/drop/do-instead: drop the uncapped `retry-after` and the double retry layer; do a single bounded-retry owner.
