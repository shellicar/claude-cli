# 21:42 — exec-core review (Phase 1 Reviewer, iteration 1)

Fresh-eyes review of `feature/exec-core` (baseline `7a19e0a`, on worktree branch
`feature/exec-core-review`). The change consolidates process spawning behind a new
`@shellicar/exec-core` leaf (`Executor.run(CommandSpec, SpawnOpts) → ExitStatus`,
streams as fds 0/1/2), migrates Exec (V1) and ExecV2 onto it, and spawns children
`detached: true` so they cannot reach the operator's tty.

## Bottom line

The consolidation is sound on everything the characterization suites cover. The full
suite is green (470 tests) once the workspace is built, and that suite is thorough —
all the V1 and V2 scenarios (chaining, pipes, redirects, merge, stdin, cwd/env,
timeout, stripAnsi, blocked rules, cancellation) still hold. The detach change
achieves its intent (new session/process-group leader → `process.kill(-pid)` group
kill works; piped stdin means no tty), and the spawn is genuinely consolidated into
one leaf — no second spawn path left to drift.

Two real behaviour drifts exist, both in **uncovered edge combinations**, both narrow,
and both arguably move toward more shell-correct behaviour. They are proven red, not
asserted. The SC decides whether either is worth fixing.

## Findings (each proven by a red test in `test/Exec/review-findings.spec.ts`)

**Finding 1 — `merge_stderr` + `redirect:stdout` drops the merged stderr.**
- Where: `src/exec-shared.ts` `resolveSinks` (evaluates `merge_stderr` before redirect)
  vs old `src/Exec/execCommand.ts` on main.
- Old: stderr was merged into the captured `result.stdout` while the child's stdout
  flowed to the redirect file. New: `merge_stderr` wins first, so `stderr = stdout =
  file`; the merged stderr lands in the file and `result.stdout` is `''`.
- Red test: `Finding 1 — merge_stderr + redirect:stdout > captures the merged stderr
  into stdout (main behaviour)`. Actual on new code: `stdout === ''`.
- Note: new behaviour matches real shell `cmd > file 2>&1` (both to file, nothing
  captured) — debatably more correct than main. Unusual input combo.

**Finding 2 — bad `cwd` on a non-final pipeline stage returns 127, not 126.**
- Where: `src/Exec/execPipeline.ts` (`intermediateSpawnFail` treats a stage's 126 as a
  spawn failure → overall 127) vs old execPipeline pre-checking every command's cwd and
  returning 126.
- Old: 126. New: 127. stderr still contains "Working directory not found"; `success` is
  false either way. Only the exit code differs.
- Red test: `Finding 2 — bad cwd on a non-final pipeline stage > reports exit code 126
  (main behaviour)`. Actual on new code: `127`.
- Standalone bad-cwd is still 126 (scenario ER2, green) — drift is specific to a
  non-final pipeline stage.

## Things I checked and cleared (so the next cast need not re-walk them)

- **Cancellation / `ToolCancelledError`.** Old execCommand/execPipeline *rejected*
  `ToolCancelledError` on abort; exec-core has no such concept and resolves normally.
  This is not a regression: both tools now throw `ToolCancelledError` at the handler
  level (`if (signal?.aborted)`) after `execute`/`executeTree`. `Exec.spec.ts`
  cancellation test and TO1 timeout tests are green.
- **Empty pipeline crash.** `execPipeline` lost its `commands.length === 0` guard and
  would crash on `statuses[-1]` — but `execStep` only calls it with ≥2 commands, so it
  is unreachable. Not a bug.
- **stdin.** `Readable.from(string)` emits the whole string as one chunk (Node special-
  cases strings/Buffers), so piping replaces the old `child.stdin.write()` faithfully.
  F1 green.
- **Deadlock / backpressure.** Capture runs concurrently with the run via `Promise.all`
  (the 1MB `execCommand.spec` test guards this). Sinks piped with `end:false`;
  `endSinks` de-dupes a shared Writable so merge doesn't double-close. Sound.
- **`base.json` blast radius.** The change adds `"types": ["node"]` to the *shared*
  TypeScript base, not just exec-core — repo-wide. But full `pnpm type-check` passes for
  every existing package (the only failure was my own test before I added `append`), so
  it disturbs nothing. Flagging it only because the radius is wider than the feature.
- **Wiring.** Both tools share one module-level `executor` singleton (`exec-shared.ts`),
  so the `process.on('exit')` group-kill sweep registers once. Tests that `new
  Executor()` each leak an exit listener (never disposed); cosmetic, not a tool bug.

## What changed

- **Finding 2 (FIX)** — `src/Exec/execPipeline.ts`. The old logic forced the whole
  pipeline's exit to `127` whenever any non-final stage exited `126` or `127`

  Replaced `.some(...)` with `.find(...)` and carry the failing stage's *own* exit
  code through. Bad cwd now reports `126` in a pipeline exactly as standalone;
  genuine command-not-found stays `127`. The red test `Finding 2 ... reports exit
  code 126` is now green; standalone bad-cwd (ER2) and command-not-found unchanged.

- **Finding 1 (no impl change, test recharacterized)** — `test/Exec/review-findings.spec.ts`.
  Rewrote the old red test (asserted `result.stdout` contained `ERR`) into two green
  assertions: `result.stdout` empty and `result.stderr` empty. That is the precise
  observable contrast of the new behaviour — old code captured the merged stderr into
  `result.stdout`, new code captures nothing because merge_stderr + redirect:stdout
  sends both streams to the redirect sink.

## The trap I hit and backed out of — read this if you touch redirect-to-file

My first cut of the Finding 1 test wrote the redirect to a real temp file and asserted
the file contained both `OUT` and `ERR` (the literal "merged output goes to the redirect
target" wording in the brief). It passed in isolation but **failed intermittently in the
full suite** — the file held only `OUT\n`, stderr lost.

It is not a double-close (exec-core wires both pipes `{ end: false }` and `endSinks`
ends each distinct sink once — that part is sound). It is a **flush race**: `endSinks`
calls `file.end()` but the run promise resolves without awaiting the file stream's
`finish` event, so a `readFileSync` immediately after races the OS flush. Under load the
file is read before stderr (sometimes before stdout) lands.

This is a real, pre-existing limitation of redirect-to-file in exec-core, surfaced only
by reading the file programmatically right after the run. In production it is mostly
benign: the tools return `result.stdout` (empty on a stdout redirect), not the file
contents, so nobody reads the file in the same tick. The existing scenario suite always
redirects to `/dev/null` and never reads it back, which is why it was never caught. I did
not fix it — out of Phase 2 scope (Finding 1 is "no implementation change"). Surfaced to
the SC. The recharacterized test deliberately asserts the *result* (deterministic), not
the file (racy).

## The flush race — fixed red-first (the headline)

`Executor.run` resolved before its output sinks finished flushing: `endSinks` called
`sink.end()` and `run` resolved without awaiting the sink's `finish`. A caller reading a
redirect file the moment `run` resolved raced the OS flush — exactly what made my
iteration-1 file-read test flaky.

- **Red proof first.** `packages/exec-core/test/Executor.spec.ts` passes `run` a
  `Writable` whose `_final` is delayed (setTimeout) and asserts `finished === true` when
  `run` resolves. It asserts the *ordering* of two events, not a file's contents (a
  file read would pass even with the bug present — a false guard, per the brief). Ran it
  before any fix: **RED**, `expected false to be true` — recorded here as the proof.
- **Fix.** `Executor.run` now awaits each distinct sink's flush before resolving, via the
  promise form of `node:stream/promises` `finished` (resolves on finish, `.catch(() => {})`
  swallows error so a broken/errored sink cannot hang or fail the await). Refactored the
  old `endSinks` dedupe into a shared `distinctSinks` helper used by both `endSinks` and
  the new `flushSinks`. `run` is now `async`; both resolution paths flush — the main
  close-path and the early bad-cwd return (I made the early path flush too, for
  consistency, so no second racy path is left behind; it does not change the observed
  exit code or stderr). After the fix: **GREEN**.

## Editor papercut: node: imports in tests

VS Code binds a file to the nearest `tsconfig.json`. Every package's `tsconfig.json`
had `include: ["src/**/*.ts"]`, so test files (in `test/`) fell outside the project and
landed in an inferred project that lacks `types: ["node"]` and the bundler resolution —
so `node:*` builtin imports failed to resolve in the editor (`@types` that leak through
still worked, hence "types work, node:* doesn't"). `pnpm type-check` was always green
because it uses `tsconfig.check.json`, which includes tests. The ecosystem repo never
hit this only because its tests don't import `node:*`.

Fix: every `tsconfig.json` include broadened to `["**/*.ts"]`, matching the
`tsconfig.check.json` include so the two share one file set. **Verified empirically that
this does not pollute the build**: tsup emits from `entry: ['src/entry/*.ts']` and its
`dts:true` pass follows that entry graph, not the tsconfig `include` — `dist` output is
byte-identical with the broadened include. Applied to all seven: exec-core,
claude-sdk-tools, claude-core, claude-sdk, mcp-exec, apps/claude-sdk-cli, scripts.

## exec-core tsconfig.check.json → symlink

It was the only package whose `tsconfig.check.json` was a real file; the other five are
symlinks to the root `tsconfig.check.json`. Replaced it with the same symlink
(`../../tsconfig.check.json`). The `extends: "./tsconfig.json"` resolves relative to the
symlink's package dir, so the shared root check file picks up each package's own
`tsconfig.json`.

## Deleted NodeDirectoryWatcher.spec.ts (flaky)

`packages/claude-core/test/NodeDirectoryWatcher.spec.ts` created a real tmp dir, wrote a
file, and waited a fixed 150ms for the OS watch event — testing `node:fs` `watch` + the
kernel, not our code, and flaking when the event missed the window. SC's call: delete it
rather than fake the watch primitive (which would only re-assert our thin glue —
tautological). Implementation untouched.
