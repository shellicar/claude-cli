# exec-core review & ship

This branch (`feature/exec-core-review`, cut from local `feature/exec-core`, baseline
`7a19e0a`) carries the whole exec-core consolidation plus the review fixes. When it
merges, all of it lands against main, not just the fix edits.

## What exec-core is

`@shellicar/exec-core` is a new leaf package: one shared way to spawn a single process.
`Executor.run(CommandSpec, SpawnOpts) → ExitStatus`, with stdin/stdout/stderr as fds
0/1/2 (`Writable` sinks). Exec (V1) and ExecV2 both migrate onto it, so there is no
longer a second spawn path to drift. Children spawn `detached: true` — a new
session/process-group leader — so they cannot reach the operator's tty (a command that
wants a terminal, e.g. `sudo`, fails cleanly instead of grabbing it), and `process.kill(-pid)`
group-kill works. Piped stdin means no tty either way.

Merging stderr into stdout is done by handing the same `Writable` to both fds.

## The consolidation is sound

The characterization suite (V1 + V2: chaining, pipes, redirects, merge, stdin, cwd/env,
timeout, stripAnsi, blocked rules, cancellation) stays green and is thorough. Two real
behaviour drifts were found in uncovered edge combinations — both proven with red tests,
both narrow. The SC gated them:

- **Finding 2 — FIXED.** Bad `cwd` on a *non-final* pipeline stage returned exit `127`;
  it now returns `126`, consistent with the standalone case. `execPipeline.ts` was
  promoting any non-final stage's `126`/`127` to `127` (`intermediateSpawnFail ? 127`);
  the fix carries the failing stage's *own* exit code through (`.find` over `.some`).
  Standalone bad-cwd (ER2) stays `126`, genuine command-not-found (ER3) stays `127`.
- **Finding 1 — NOT a bug, kept.** With `merge_stderr: true` + `redirect: stdout → file`,
  the new code sends both streams to the file and leaves `result.stdout` empty. That is
  shell-correct (`> file 2>&1`); the old code wrongly dropped the merge and returned the
  stderr. No implementation change; its test was rewritten to assert the new behaviour.

## The trap worth knowing — redirect-to-file flush race (now fixed)

`Executor.run` used to resolve before its output sinks finished flushing: `endSinks`
called `sink.end()` and `run` resolved without awaiting the sink's `finish`. A caller
that reads a redirect file the instant `run` resolves raced the OS flush and could see
partial or empty content. Benign for the current tools (they read `result.stdout`, never
the file), but a real latent bug for any caller that reads the file — **ExecV3 will**.

Fixed red-first: `run` is now `async` and awaits each distinct sink's flush before
resolving, via the promise form of `node:stream/promises` `finished` (resolves on finish
*or* error, with a swallow, so a broken sink cannot hang the await). Both resolution
paths flush — the main close-path and the early bad-cwd return — so no second racy path
is left. Proof is `packages/exec-core/test/Executor.spec.ts`: it asserts the *ordering*
(sink finished before run resolved) using a `Writable` with a delayed `_final`, not a
file read. A file-read test passes even with the bug present, so it is a false guard —
do not write one. This test is also exec-core's first, which is why the root `pnpm test`
no longer fails on an empty suite (no `--passWithNoTests` needed).

## tsconfig consolidation (SC-directed, wider than exec)

Editor papercut: VS Code binds a file to the nearest `tsconfig.json`, whose include was
`src/**/*.ts`. Test files in `test/` fell outside the project, into an inferred one
lacking `types: ["node"]` and bundler resolution — so `node:*` imports failed *in the
editor* while `pnpm type-check` (which uses `tsconfig.check.json`, includes tests) stayed
green. Fix: every package's `tsconfig.json` include broadened to `**/*.ts`, matching
`tsconfig.check.json`. Verified empirically this does not pollute `dist` — tsup emits
from its `entry` graph, not the tsconfig include, so output is byte-identical. exec-core's
`tsconfig.check.json` was the only real file among the packages; converted to the shared
symlink to root like the others.

Also deleted `packages/claude-core/test/NodeDirectoryWatcher.spec.ts` — flaky, it waited
a fixed 150ms for an OS `fs.watch` event, testing the kernel not our code. SC chose
deletion over faking the watch primitive (which would only re-assert thin glue).

`base.json` gained `"types": ["node"]` on the *shared* TS base (repo-wide reach), but a
full `pnpm type-check` passes for every package, so it disturbs nothing.

## Cleared, so you need not re-walk

- **Cancellation** is handled at the handler level (`if (signal?.aborted)` after
  `execute`/`executeTree` throws `ToolCancelledError`); exec-core has no cancel concept
  and resolves normally. Not a regression.
- **Empty-pipeline guard** was dropped from `execPipeline`, but `execStep` only calls it
  with ≥2 commands, so the `statuses[-1]` path is unreachable.
- **Backpressure** — capture runs concurrently with the run (`Promise.all`); sinks piped
  `{ end: false }`; `endSinks`/`flushSinks` de-dupe a shared `Writable` so a merge does
  not double-close.
