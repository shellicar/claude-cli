# esbuild CVE maintenance release (2026-06-15)

Branch `security/audit-2026-06-15`, PR #352. Two esbuild advisories, both patched at `>=0.28.1`: GHSA-gv7w-rqvm-qjhr (high, RCE via `NPM_CONFIG_REGISTRY` in the Deno install path) and GHSA-g7r4-m6w7-qqqr (low, Windows dev-server path traversal). The Phase 0 audit found nothing else.

## The fix changes nothing downstream

esbuild is dev/build-only across the whole repo: a devDependency in the five packages, otherwise transitive through tsup, vite, build-clean, build-version. It never enters a published bundle, so the CVE is consumer-invisible. The fix is a root `pnpm-workspace.yaml` override (the repo's established CVE mechanism, same as vite, hono, qs, minimatch), which touches no published package's own `package.json`. So no `changes.jsonl`, no version bumps, no release. The SC chose override-only with no patch sweep; nothing else in the repo had a runtime patch waiting.

## Traps for the next operator

- **`fix-audit.sh` over-nukes the lockfile.** It applies the pnpm#6774 lockfile rebuild unconditionally, but #6774 only matters for *chained* overrides (one override feeding another). This CVE has no chaining (one package, two ranges, both resolving to `>=0.28.1`), so the nuke was not needed and floated unrelated dev tooling up within its `^` ranges (biome, turbo, knip, and friends). The SC redid it surgically: restore the lockfile, keep the override, plain `pnpm install`. The float vanished and the diffstat dropped from ~1122 to 598 lines. For a single-package, non-chained CVE, skip the nuke or gate it on real chaining.
- **pnpm 10 misreports where it writes the override.** `pnpm audit --fix` says "added to package.json" but actually writes to `pnpm-workspace.yaml`, where this repo keeps overrides. Verify the location, do not trust the message.
- **Testaments are tracked and live in the worktree.** claude-cli whitelists `.claude/*/**/*.md` in `.gitignore`, so testaments belong in the worktree on the branch and ship with the PR, not in the main checkout. The mission's full-path testament line was a scaffold bug (`substituteTestamentPath` forces the full-path form and preempts the `check-ignore` probe); flagged for a separate fleet fix.
- **`.claude/audit/<date>.json` is committed by convention.** Prior security PRs (#332, #328) shipped their audit captures, so the Phase 0 capture correctly rides into the fix commit.

## Ship

PR #352 carries the override, lockfile, audit capture, and this testament. Root-only change, so labels `dependencies` and `security` with no `pkg:` label. CI was clean bar checks still running at hand-off.

# 23:09 — Phase 1 Scaffolder (AppendFile tool)

Mission: add `AppendFile` to `@shellicar/claude-sdk-tools`, the additive sibling of `CreateFile` (same path+content shape, but asserts only a suffix: appends when the file exists, creates it when it doesn't, no `overwrite`, no existence errors). Scaffold phase only: a stub handler that throws, plus six failing tests that are the contract for the Builder.

## Biggest trap: the worktree was not bootstrapped

This worktree had root `node_modules` but no `@shellicar/*` workspace links and no built `dist/` for `claude-core`/`claude-sdk`. Until that is fixed, BOTH `type-check` and `test` fail on every file (mine and pre-existing) with `Cannot find module '@shellicar/claude-sdk'` / `@shellicar/claude-core/...`, and a cascade of spurious `TS7006` "implicitly any" on every handler param. Fix once per cast:

    pnpm install --frozen-lockfile && pnpm build

Frozen because no dependencies changed, only the package.json `exports` map. After the build, type-check is clean (exit 0) and 459 existing tests pass. The Builder will need this if it starts from a fresh worktree.

## What I built

Three source files + one test file, mirroring the CreateFile siblings:

- `src/AppendFile/schema.ts` — `AppendFileInputSchema` (path + content; **content is required, no default, and there is no `overwrite` field**), `AppendFileOutputSchema` (same success/error discriminated union as CreateFile).
- `src/AppendFile/AppendFile.ts` — `createAppendFile(fs)` factory returning `defineTool({...})`, `operation: 'write'`. Handler is the stub `throw new Error('not implemented')`. It does NOT import `expandPath` yet — that import is the Builder's (an unused import in the scaffold would be a biome `noUnusedImports` error).
- `src/entry/AppendFile.ts` — `export const AppendFile = createAppendFile(nodeFs);`

## Wiring: registration took TWO files, not one

The mission's stage block implies one "registration" file. CreateFile actually reaches the tool set through three points, and AppendFile mirrors all three:

1. **tsup entry — automatic.** `tsup.config.ts` globs `src/entry/*.ts`, so `entry/AppendFile.ts` builds to `dist/{esm,cjs}/AppendFile.*` with no config change.
2. **`packages/claude-sdk-tools/package.json` `exports`** — added an `./AppendFile` block mirroring `./CreateFile`. Without it the `@shellicar/claude-sdk-tools/AppendFile` subpath import does not resolve.
3. **`apps/claude-sdk-cli/src/createAppTools.ts`** — added `import { AppendFile } from '@shellicar/claude-sdk-tools/AppendFile'` and placed `AppendFile` in the `otherTools` array next to `CreateFile`.

Staged set = the 4 named files + `package.json` + `createAppTools.ts`.

## The test contract (the Builder makes these pass)

All six fail against the stub with `Error: not implemented` — failing for the right reason (missing behaviour), unsatisfiable without the real handler. `MemoryFileSystem.appendFile` already concatenates onto an empty buffer when the key is absent (creates implicitly), and `NodeFileSystem.appendFile` does `mkdir(dirname, {recursive:true})` then `appendFile` — so the "creates file" and "creates parent dirs" contracts hold for the real implementation.

```ts
import { describe, expect, it } from 'vitest';
import { createAppendFile } from '../src/AppendFile/AppendFile';
import { call } from './helpers';
import { MemoryFileSystem } from './MemoryFileSystem';

describe('createAppendFile — appending to existing files', () => {
  it('appends content verbatim at the end of an existing file', async () => {
    const fs = new MemoryFileSystem({ '/log.txt': 'first\n' });
    const AppendFile = createAppendFile(fs);
    const expected = 'first\nsecond\n';

    await call(AppendFile, { path: '/log.txt', content: 'second\n' });
    const actual = await fs.readFile('/log.txt');

    expect(actual).toBe(expected);
  });

  it('leaves existing contents unchanged when appending empty content', async () => {
    const fs = new MemoryFileSystem({ '/log.txt': 'existing' });
    const AppendFile = createAppendFile(fs);
    const expected = 'existing';

    await call(AppendFile, { path: '/log.txt', content: '' });
    const actual = await fs.readFile('/log.txt');

    expect(actual).toBe(expected);
  });
});

describe('createAppendFile — creating files that do not exist', () => {
  it('creates the file when it does not exist', async () => {
    const fs = new MemoryFileSystem();
    const AppendFile = createAppendFile(fs);
    const expected = 'hello';

    await call(AppendFile, { path: '/new.txt', content: 'hello' });
    const actual = await fs.readFile('/new.txt');

    expect(actual).toBe(expected);
  });

  it('creates missing parent directories', async () => {
    const fs = new MemoryFileSystem();
    const AppendFile = createAppendFile(fs);
    const expected = 'nested';

    await call(AppendFile, { path: '/a/b/c.txt', content: 'nested' });
    const actual = await fs.readFile('/a/b/c.txt');

    expect(actual).toBe(expected);
  });
});

describe('createAppendFile — result shape', () => {
  it('returns a success result carrying the expanded path', async () => {
    const fs = new MemoryFileSystem();
    const AppendFile = createAppendFile(fs);
    const expected = { error: false, path: '/new.txt' };

    const actual = await call(AppendFile, { path: '/new.txt', content: 'hello' });

    expect(actual).toEqual(expected);
  });

  it('expands ~ in the returned path', async () => {
    const fs = new MemoryFileSystem({}, '/home/testuser');
    const AppendFile = createAppendFile(fs);
    const expected = '/home/testuser/log.txt';

    const actual = await call(AppendFile, { path: '~/log.txt', content: 'hello' });

    expect(actual.path).toBe(expected);
  });
});
```

## Notes for the Builder

- The whole handler is the three lines in the Phase 2 brief (`expandPath` + `fs.appendFile` + success return). No try/catch, no existence check.
- biome `check` is CLEAN on the scaffold — the unused `fs` factory param is NOT flagged, so the factory stays `createAppendFile(fs)` as specified (no `_fs` workaround). Pre-commit/pre-push biome will not block.
- Verify state at hand-off: `type-check` exit 0; `test` exit 1 because the 6 new tests fail by design (459 other tests green).
