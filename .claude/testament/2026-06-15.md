### The one fact that shapes Phases 2–4: nothing ships

esbuild is a **dev/build-only** dependency everywhere in this repo — `devDependencies` in five packages (claude-cli legacy, claude-sdk-cli, claude-sdk, claude-sdk-tools, mcp-exec), otherwise transitive through `tsup`/`vite`/`@shellicar/build-clean`/`build-version`. It never enters a published bundle, and devDependencies are not installed by consumers. So the CVE fix is consumer-invisible.

The fix is a root `pnpm-workspace.yaml` override (the repo's established CVE mechanism — vite, hono, qs, minimatch all done this way). A root override changes **no published package's own `package.json`**. By the skill's Phase 4.5 / 6.2 test that means: **no `changes.jsonl` entries, no version bumps, no releases.**

**Phase 2 (github-version) Maker: expect to find no published package's own `package.json` changed. Record "nothing to version" and stop.** The only tracked changes are `pnpm-workspace.yaml` (+2 override lines) and `pnpm-lock.yaml`. Don't manufacture a version bump for a dev-only override.

### What the SC chose at the Phase 2 gate

Two advisories, both esbuild, both patched `>=0.28.1`: GHSA-gv7w-rqvm-qjhr (high, RCE via NPM_CONFIG_REGISTRY) and GHSA-g7r4-m6w7-qqqr (low, Windows dev-server file read). Scenario A (high CVE present). The SC chose **A1: override only, no patch sweep.** Notably every available update in this repo is dev-only — no prod/runtime dep had a patch waiting (the prod deps @anthropic-ai/sdk, @js-joda/*, diff, sharp only offered minor/major, all skipped). So A1 = override + lockfile, nothing else.

### Override landed

```
esbuild@>=0.17.0 <0.28.1: '>=0.28.1'
esbuild@>=0.27.3 <0.28.1: '>=0.28.1'
```

Two lines (one per advisory) as `pnpm audit --fix` generated them. Despite its "added to package.json" message, pnpm 10 wrote them to `pnpm-workspace.yaml` (where this repo keeps overrides) — verify location, don't trust the message. Audit clean after. Both installed esbuild copies (0.27.7 + 0.28.0) collapse to a single 0.28.1.

### fix-audit.sh over-reached — surgical lockfile needed (fleet lesson)

`fix-audit.sh` applies the pnpm#6774 lockfile-nuke **unconditionally**. #6774 is about *chained* overrides (one override's resolution feeding a second). This CVE has no chaining — one package, both overrides → `>=0.28.1`. The nuke wasn't needed, and the clean re-resolution **floated dev tooling up within `^` ranges**: biome 2.4.16→2.5.0, turbo 2.9.16→2.9.18, knip 6.15.0→6.16.1, ncu 22.2.2→22.2.3, syncpack 14.3.0→14.3.1, @types/node 25.9.1→25.9.3. That churn cut against the SC's minimal-A1 intent.

The SC redid it surgically (restore lockfile, keep override, plain `pnpm install` — no nuke). After redo the float was gone: all those tools back to original, only esbuild moves (plus the peer-key ripple — lockfile keys that embed esbuild's version, e.g. `build-clean(esbuild@0.28.1)`, `vite`, `vitest`, get rewritten; package versions unchanged). Diffstat dropped 1122→598 lines.

**Fleet takeaway:** for a single-package, non-chained CVE, the nuke is overkill and introduces unrelated within-range float. Consider gating fix-audit.sh's lockfile-nuke on actual chaining, or use a plain `pnpm install` after the override.
