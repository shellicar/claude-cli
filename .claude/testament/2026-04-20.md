## input_examples gating — why it matters

The `toWireTool` comment said `buildRequestParams` gates `input_examples` on ATU being enabled, but the gating was never written. The first Cleaner pass removed the claim from the comment rather than implementing the gating. SC caught it.

**Why gate it:** `input_examples` are only meaningful in ATU mode. Sending them when ATU is off is wasteful and exposes implementation detail to the API unnecessarily.

**Where it belongs:** CLI concern, not SDK concern. The SDK defines the shape (`toWireTool` always includes `input_examples`). The CLI's `transformTool` hook decides what reaches the wire. `buildRequestParams` should not contain logic about `input_examples`.

**Decision:** `buildAtuTransform` now always returns a function. When ATU is disabled the function strips `input_examples` using the same type guard pattern (`isToolWithInputExamples`) already used for `isNamedTool` in the file. No `as unknown as` anywhere.

## Show cwd basename in status bar (issue #279)

The model line now always shows the cwd basename alongside the model name. Layout: ` ⚡ Sonnet  claude-cli-status-cwd`. Before a model is set: ` claude-cli-status-cwd`.

**Revised after SC feedback:** The initial approach called `path.basename(process.cwd())` directly in `renderStatus.ts`. SC pointed out there is a filesystem abstraction (`IFileSystem`) that must be used instead.

Final shape: `StatusState` takes `IFileSystem` in its constructor (matching the pattern used by `ClaudeMdLoader`, `AuditWriter`, `ConversationSession`). It calls `path.basename(fs.cwd())` once at construction and exposes it via `cwdBasename`. The renderer reads `state.cwdBasename` — no direct `path`/`process` calls in the renderer.

`main.ts` passes `nodeFs` to `new StatusState(nodeFs)`. Tests use `MemoryFileSystem({}, '/home/user', '/repos/my-project')` so assertions against `'my-project'` are deterministic and independent of the actual process cwd.

Branch is one commit behind `origin/main`. No conflict with this work.
