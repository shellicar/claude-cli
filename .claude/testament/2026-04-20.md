## input_examples gating — the real fix

The first testament entry had the wrong fix for `toWireTool`. The comment said `buildRequestParams` gates `input_examples` on ATU being enabled — but the gating code was never written. An earlier Claude removed the claim from the comment instead of implementing the gating. SC caught it.

**Why the comment was right:** `input_examples` should only reach the API in ATU mode. Sending them when ATU is off is wasteful at best.

**Where the fix belongs:** This is a CLI concern, not an SDK concern. The SDK defines the shape (`toWireTool` always includes `input_examples`). The CLI's `transformTool` hook decides what actually goes on the wire. `buildRequestParams` should not contain any logic about `input_examples`.

**How it was fixed:** `buildAtuTransform` now always returns a function (never `undefined`). When ATU is disabled, the function strips `input_examples` via the established type pattern already in the file:

```typescript
type ToolWithInputExamples = Extract<BetaToolUnion, { input_examples?: unknown }>;

function isToolWithInputExamples(tool: BetaToolUnion): tool is ToolWithInputExamples {
  return 'input_examples' in tool;
}
```

Then the strip:
```typescript
if (!isToolWithInputExamples(tool)) return tool;
const { input_examples: _, ...base } = tool;
return base;
```

This mirrors the `NamedTool`/`isNamedTool` pattern already at the top of the file. No `as unknown as` anywhere.

**Tests:** `apps/claude-sdk-cli/test/buildAtuTransform.spec.ts` (new file, 4 tests) — strips `input_examples` when ATU disabled, keeps them when ATU enabled, preserves tool name in both paths.

`test/cli.spec.ts` spawns `tsx` by name via `StdioClientTransport`. Works under `pnpm test` because `node_modules/.bin` is on PATH. Fails in VS Code vitest extension which doesn't inherit that PATH.
