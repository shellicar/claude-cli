### Key structural observation

The `MessageChannel` was used bidirectionally:
- SDK → consumer: `port1.postMessage(sdkMsg)` → `port2.on('message', handler)`
- Consumer → SDK: `port2.postMessage(consumerMsg)` → `port1.on('message', listener)` → `channel.on('message', listener)`

The new design replaces both directions with JavaScript-level mechanisms:
- SDK → consumer: `channel.subscribe(handler)` with async-ordered pump per subscriber
- Consumer → SDK: `channel.postConsumerMessage(msg)` → synchronously invokes registered listeners

This removes the `node:worker_threads` dependency entirely from `ControlChannel.ts`.

### Apostle decisions

1. **Multiple subscribers**, each with their own independent FIFO queue and pump. Single is sufficient for now but multiple is no harder and avoids an API break later.

2. **Swallow errors**: handler rejects → log (implicitly: silently for now) → continue with next message. The existing consumer's `handle()` never throws.

3. **`send` after `close()`**: no-op. Throws would require try/catch around every `channel.send()` in QueryRunner.

4. **`drain()` after `close()`**: same semantics. No new messages enqueue; resolves when in-flight ones settle.

5. **`drain()` multiple times**: yes, all resolve together when idle.

### Consumer-side changes

Three files in `apps/claude-sdk-cli/`:
- `main.ts`: `consumerPort.on(...)` → `channel.subscribe(...)`, `port: channel.consumerPort` → `channel: channel`, `runAgent(..., channel.consumerPort, ...)` → `runAgent(..., channel, ...)`
- `runAgent.ts`: `consumerPort: MessagePort` → `channel: { postConsumerMessage }`, `postMessage` → `postConsumerMessage`
- `AgentMessageHandler.ts`: `port: MessagePort` → `channel: MessageSink` (narrow type defined locally), `postMessage` → `postConsumerMessage`, add `turn_content` no-op case

`IControlChannel` is NOT exported from the package. The consumer uses concrete `ControlChannel` in `main.ts` and narrow `MessageSink` interface elsewhere.

### `turn_content` message

`SdkTurnContent = { type: 'turn_content'; blocks: ContentBlock[] }`

Sent in `QueryRunner.run()` after `message_usage` and before `done` (or before the next turn's tool dispatch). `ContentBlock` must be re-exported from `public/types.ts` since `SdkTurnContent` is public and references it.

### Test impact

Two existing test files updated:
- `QueryRunner.spec.ts`: `FakeControlChannel` loses `consumerPort`, gains `subscribe`, `postConsumerMessage`, `drain`; `deliverConsumerMessage` → `postConsumerMessage`; add `turn_content` ordering tests
- `AgentMessageHandler.spec.ts`: `makeOpts.port` → `makeOpts.channel` (plain object satisfying `MessageSink`); remove `MessageChannel` import

One new test file: `packages/claude-sdk/test/ControlChannel.spec.ts` — 14 tests covering all section 1 contract points with deferred promises (no `setTimeout`).

### What the code doesn't say

The `#runPump` logic: when `send()` is called from inside a handler (re-entrant), the message goes to the end of the queue and will be processed after the current handler completes. This is correct FIFO behavior and falls naturally out of the pump loop design. It is not obvious from the code — the test for it documents the guarantee explicitly.

The `drain()` waiter check happens after `sub.running = false` (not before). Timing matters: a waiter added between `sub.queue.length === 0` and `sub.running = false` would be missed. The implementation avoids this race because `#isIdle()` is called after the flag is set.

## Phase 1 Apostle — revised plan (third Apostle cast)

### Why this cast was needed

The 20:31 Apostle (from the same date) documented the rejected plan's decisions in the testament. The plan on disk was still the rejected version — it had `void | Promise<void>` union handlers and a single `IControlChannel` abstract class. Phase 1 status had been reset to `ready` by the PM.

This cast read all the same files (ground truth confirms they haven't changed) and wrote a fully revised plan that honours all three SC-pinned bullets the first Apostle missed.

### What changed from the rejected plan

**Three structural differences:**

1. **Handler type**: `void | Promise<void>` → `Promise<void>`. No union. All tests and handler types in the new plan use the strict signature.

2. **Interface architecture**: The rejected plan updated `IControlChannel` as a single abstract class with `send`, `subscribe`, `on`, `postConsumerMessage`, `close`, `drain`. The new plan removes `IControlChannel` entirely. In its place: two generic interfaces `IPublisher<T>` (send, close, drain) and `ISubscriber<T>` (subscribe), implemented by one generic concrete class `ControlChannel<T>`.

3. **Two channel instances**: The rejected plan kept a single combined channel object. The new plan creates two `ControlChannel` instances in `main.ts`: `sdkChannel = new ControlChannel<SdkMessage>()` and `consumerChannel = new ControlChannel<ConsumerMessage>()`. Each is passed to the appropriate call site as the interface it needs. `QueryRunner` takes `IPublisher<SdkMessage>` (not `IControlChannel`).

### Key decisions the Apostle made

**`drain()` and `close()` on `IPublisher<T>`**: The producer owns the channel lifecycle. Subscribers are passive. Lifecycle on the subscriber interface would let subscribers affect the send path.

**`IControlChannel` is gone**: `QueryRunner` only calls `send()`. It never called `channel.on('message', ...)` — that was `main.ts`. So `QueryRunner` needs only `IPublisher<SdkMessage>`. Removing the combined abstraction makes each call site explicit about what it holds.

**`deliverConsumerMessage` → `approval.handle()`**: The old test helper simulated the consumer routing a `ConsumerMessage` to the `ApprovalCoordinator`. With the new design, `QueryRunner` no longer holds anything on the consumer direction. The test calls `w.approval.handle(msg)` directly. Cleaner, no fake consumer channel needed in `QueryRunner.spec.ts`.

**Two-subscriber test is one `it` with two `expect` calls**: Both assertions test the same claim. Noted this as an exception with justification in the plan.

### What the Maker needs to know

- The `ControlChannel<T>` generic is the one file. `IPublisher<T>` and `ISubscriber<T>` live in the same file. `IControlChannel` is deleted from that file.
- `IPublisher` and `ISubscriber` must be exported from `packages/claude-sdk/src/index.ts`. The consumer app imports `IPublisher<ConsumerMessage>` for `AgentMessageHandler.channel`.
- `node:worker_threads` import is removed from `ControlChannel.ts`. No `MessagePort` anywhere in the SDK package after this change.
- The drain timing note in the plan is important: `#isIdle()` is checked after `sub.running = false`, not before. The test for "drain resolves after all handlers settle" covers this ordering.

## Ship Cast — async control channel

### What was built

The `MessageChannel`-backed `ControlChannel` (node:worker_threads) is replaced with a custom async-ordered pub/sub. Two generic interfaces `IPublisher<T>` and `ISubscriber<T>`, one concrete class `ControlChannel<T>` implementing both. The node:worker_threads dependency is gone from the SDK package.

After each turn, the SDK now emits a `turn_content` message carrying the full assembled `ContentBlock[]`. This is the canonical per-turn content; the consumer can swap its partially-built buffer for this authoritative version.

### Architecture decisions (from the Apostle)

**Two channel instances, not one.** `main.ts` creates `sdkChannel = new ControlChannel<SdkMessage>()` and `consumerChannel = new ControlChannel<ConsumerMessage>()`. Each call site gets only the interface it needs: `QueryRunner` takes `IPublisher<SdkMessage>`, `AgentMessageHandler` takes `IPublisher<ConsumerMessage>`.

**`drain()` and `close()` on `IPublisher<T>` only.** The producer owns the channel lifecycle. Subscribers are passive; giving them lifecycle methods would let them affect the send path.

**Handler errors are swallowed.** The pump catches and discards handler rejections and continues with the next message. The existing consumer's `handle()` never throws, so this is safe for now.

**`send` after `close()` throws.** The Apostle originally decided no-op, but the SC overrode this: a no-op makes the producer believe the message went out when it did not. Throwing puts the diagnostic at the call site.

**Multiple subscribers, independent FIFO queues.** Each subscriber has its own queue and pump. Messages are delivered to all subscribers in send order, but one slow subscriber does not block another.

### The pump -- things the code doesn't say

The `msg === undefined` guard in `#runPump` is unreachable in practice. The while condition `sub.queue.length > 0` guarantees at least one item before each `shift()`. The guard exists purely to satisfy the TypeScript type system without a non-null assertion (`!` is forbidden by biome's `noNonNullAssertion` rule).

The drain timing guarantee: `#isIdle()` is checked AFTER `sub.running = false`, not before. If idle were checked before the flag was set, a `drain()` call arriving between the last `shift()` and `sub.running = false` would never resolve.

### Consumer-side changes

Three files in `apps/claude-sdk-cli/`:
- `main.ts`: two `ControlChannel` instances replace the single `IControlChannel`. `channel.on('message', ...)` becomes `consumerChannel.subscribe(async (msg) => ...)`. `sdkChannel.subscribe(...)` handles inbound SDK messages.
- `runAgent.ts`: `consumerPort: MessagePort` becomes `consumerChannel: IPublisher<ConsumerMessage>`. `postMessage` becomes `send`.
- `AgentMessageHandler.ts`: `port: MessagePort` becomes `channel: IPublisher<ConsumerMessage>`. `postMessage` becomes `send`. `turn_content` no-op case added to the switch.

### turn_content placement

Emitted in `QueryRunner.run()` after `message_usage` and before the stop-reason branch (tool dispatch or terminal). `result.blocks` is in scope from `turnRunner.run()`. The consumer gets authoritative assembled content before the runner either dispatches the next tool call or signals done.

### Test surface

14 new tests in `ControlChannel.spec.ts` cover the full contract: FIFO ordering, handler serialisation, drain semantics, close behaviour. All use deferred promises (no `setTimeout`). 4 new tests in `QueryRunner.spec.ts` cover `turn_content` placement and ordering.

One structural note on drain tests: "drain resolves only after all queued messages are delivered" and "drain resolves after all handlers settle" each have two `expect` calls. Both assertions test the same claim (before/after the drain resolves). The Apostle noted this as a justified exception to the one-assertion-per-test rule; the plan documents it explicitly.

### Known documentation bug

The JSDoc on `IPublisher.close()` still says "Subsequent `send()` calls are no-ops." This is wrong now that `send` throws after close. It is a documentation bug, not a behavioural bug. The Builder who applied the throw correction was instructed not to touch files outside the two authorised for that cast. Worth fixing in a follow-up.
