# 23:55

Apostle, iteration 3 of the AppLayout refactor. The mission gained decision 5 ("each handler claims only its own keys; ctrl+c is the only key a handler shared across ALL chains may claim; a shared handler must not claim command-mode or cancel keys") and clarified decision 14 (preserve what the user sees, not AppLayout's internal cascade, where reproducing it re-fuses concerns). The block was scoped to ONE construct: my `AppKeyHandler` was the shared handler and it claimed ctrl+/ and escape (command + cancel concerns), which decision 5 now forbids. Everything else carried forward.

## The fix and why it cascaded

`AppKeyHandler` dissolved into:
- `QuitHandler` — ctrl+c only (the only key a fully-shared handler may claim).
- `CommandKeyHandler` — gains ctrl+/ (toggle) and escape-to-close, alongside command keys. Editor chain only.
- `CancelHandler` — escape → injected cancel callback. The send is inert with no running turn (consumer already guards on currentAbortController), so NO turn-state read needed.
- `EditorHandler` — editor keys; no longer needs a mode/command gate (it's editor-chain-only and the command handler precedes it).

The non-obvious consequence: decision 5's gating must come from CHAIN COMPOSITION ("a chain without command mode lets ctrl+/ pass through"), not from handlers reading a mode flag. That forces editor/streaming to be two distinct CHAINS, which makes them the AppModeState values (`editor` | `streaming`, future `history`), both bound to the same PrimaryView. So:

- **PrimaryViewState is DELETED.** Its `mode` → AppModeState; its `cancelFn` → the callback injected into CancelHandler.
- This REVERSES my iteration-2 call that editor/streaming was a sub-state. Decision 5 forced the reversal — the boundary moved, the design followed.

Chains:
- editor: `[quit, approval, command, cancel, editor]`
- streaming: `[quit, approval, cancel]`

I walked every key through both and confirmed today's observable precedence holds. Crux checks:
- ctrl+/ in streaming → command handler absent → falls through → nothing (matches today's streaming swallow).
- escape in editor (command closed) → cancel claims → inert (no turn). escape in streaming → cancel → abort.
- command mode open → command handler claims ALL keys (editor never sees them), so EditorHandler needs no command gate.

## CancelHandler / consumer guard — the load-bearing subtlety

The cancel callback `() => consumerChannel.send({type:'cancel'})` is CONSTANT and safe to call anytime: the consumer subscription only aborts `if (currentAbortController)`. So escape in the editor chain posts a cancel that's discarded when no turn runs. This is why CancelHandler can sit in BOTH chains with no turn-state read and no cross-concern coupling. If a future reviewer questions "why does the editor chain have a cancel handler," this is the answer: chain membership + the existing guard reproduce today's null-cancelFn no-op.

I kept CancelHandler in both chains to match the verdict's suggested precedence `[quit, approval, command, cancel, editor]`. Omitting it from the editor chain gives the identical visible result (escape→nothing); noted as an alternative in the plan's stop notes.

## EditorHandler.#submit touches commandModeState

ctrl+enter (an editor key) unifies editor text + pending attachments into UserInput, so #submit reads/takes from commandModeState. Decision 5's "touches only its concern's state" is about SHARED handlers leaking concerns; EditorHandler is editor-chain-only, not shared, and the submit genuinely owns input-assembly. Flagged in the plan so it's not mistaken for a violation. If the SC wants attachment-gathering elsewhere, that's a further refinement, not what decision 5 targets.

## runAgent change vs iteration 2

runAgent now switches `appModeState` editor↔streaming (was primaryViewState.setMode) and does NOT touch cancel at all (CancelHandler owns it via injection + chain composition). resetExpanded moved into runAgent's turn-end finally (was on waitForInput) so EditorHandler.waitForInput touches only editorState. consumerChannel may become unused in runAgent — Maker drops the param if the compiler flags it.

## Process note (for whoever routes next)

This is the THIRD consecutive block where the plan was sound at its snapshot and the mission boundary moved underneath it (8→13 decisions, then new decision 5). The verdict itself flagged the churn is in the contract, not the execution. Each re-cut has been contained. Not a complaint — just: the design is converging, and the remaining risk is more boundary additions, not flaws in what's there.

Apostle, iteration 4 of the AppLayout refactor. One contained fix to the iteration-3 plan, not a re-cut.

## What was wrong

Iteration 3 kept `CancelHandler` in BOTH chains, arguing escape-in-editor posts a cancel that's "inert" because the consumer subscription guards on `currentAbortController`. An independent check found that reasoning is WRONG: `ApprovalCoordinator.handle` sets `#cancelled = true` on any `cancel` message, regardless of `currentAbortController`. So an editor-mode escape DOES mutate coordinator state — it diverges from today (which posts nothing), masked only because the next query calls `reset()`. The "inert" claim was the flaw.

## The fix (the alternative iteration 3 already named)

`CancelHandler` → streaming chain only.
- editor chain: `[quit, approval, commandKey, editor]`
- streaming chain: `[quit, approval, cancel]`

Editor-mode escape (command closed) is now unclaimed → posts nothing → faithful to today. This is cleaner anyway: the cancel concern lives only where a turn can be cancelled, gated purely by chain composition (no turn-state read, no reliance on a downstream guard).

Lesson for future me: "inert because a downstream guard discards it" is a fragile preservation argument — it assumes the guard is the ONLY consumer. Here there were two consumers of the cancel message (the abort guard AND the ApprovalCoordinator). Don't claim a side-effecting post is a no-op without checking every subscriber. The two-chain split (don't post at all) is strictly safer than post-and-rely-on-discard.

# 04:32 (iteration 5)

Apostle, iteration 5. Decision 3 was rewritten and blocked the prior revision's flat mode model. Re-grounded on `ceb37b9` (was 083d58c in the header — stale; the actual code I'd been reading since iteration 2 already included #331's launch flags).

## What decision 3 now demands

App-level mode names ONLY the presentation (`primary` | `history`). A presentation owns its sub-states and selects its own chain internally. `editor`/`streaming` is the primary's turn phase, NESTED inside primary, never a top-level mode. The two axes are orthogonal (a turn streams under any presentation) and cannot share one key or one flat map.

This blocks iteration 3/4's `AppModeKey = 'editor' | 'streaming' | 'history'` flat map — it collapsed both axes into one enum.

## The re-cut (scoped to the mode-selection seam)

- `AppModeState` → `'primary' | 'history'` (presentation axis).
- `PrimaryViewState` RETURNS, now holding `phase: 'editor' | 'streaming'` + `setPhase` (turn axis). (In iteration 2 it held mode+cancelFn; now just phase — cancelFn is still the injected CancelHandler callback from iteration 3/4.)
- New `Presentation` interface: `{ view: View; activeChain(): readonly InputHandler[] }`. The render-only View is the render half; the Presentation bundles it with chain-selection.
- New `PrimaryPresentation`: holds PrimaryView + primaryViewState + editorChain + streamingChain; `activeChain()` picks by phase.
- `ViewHost` resolves TWO layers: `appModeState.active` → Presentation (from registry) → `presentation.activeChain()`. Registry is `Map<AppModeKey, Presentation>`.
- `PrimaryView.render` reads `primaryViewState.phase === 'editor'` (was `appModeState.active === 'editor'`).
- `ViewModel`: carries `primaryViewState` (views read phase), NOT `appModeState` (ViewHost holds it separately — app mode is the coordinator's concern, not a view's).
- `runAgent` switches `primaryViewState.setPhase` (was `appModeState.setActive`).

The handler split (Quit/Approval/CommandKey/Cancel/Editor) and the chains themselves (`editor: [quit, approval, command, editor]`, `streaming: [quit, approval, cancel]`) are UNCHANGED. Only their grouping (into a PrimaryPresentation keyed on phase) and how ViewHost resolves them changed.

## The distinction that's now load-bearing

Two separate stores, two separate axes:
- AppModeState = which presentation (primary/history). The coordinator reads it to pick the Presentation.
- PrimaryViewState.phase = the primary's editor/streaming. The PrimaryPresentation reads it to pick its chain; PrimaryView reads it for the editor region.

A turn can stream (phase=streaming) while history is shown (appMode=history) — orthogonal. runAgent sets phase regardless of app mode. The history view, when added, reads its OWN sub-state for its OWN chain; primary is untouched (decision 6 now satisfied at the Presentation layer, not a flat registry entry).
