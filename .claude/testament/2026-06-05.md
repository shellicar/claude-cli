Scout phase for rate-limit retry mission. Branch `feature/rate-limit-retry` [61d4fb6].

## What the investigation found

**Error type**: `APIError` from `@anthropic-ai/sdk` with `.type === 'rate_limit_error'` and `.status === undefined`. NOT a 429 RateLimitError — the in-stream error comes through the SSE `event: error` path in the SDK's `streaming.ts`, which constructs `new APIError(undefined, body, undefined, headers, type)`. The status is undefined because the HTTP response is a 200.

**Retry integration point**: `TurnRunner.run()` — wrap the `stream + process` pair with a retry loop. Both `body` and `requestOptions` are local variables in scope at that point.

**Raw-surface path preserved**: `QueryRunner.run()` catch block sends `{ type: 'error', message: err.message }` on the publisher. After retries exhausted, `TurnRunner.run()` re-throws the last error unchanged, hitting exactly this path.

**New file**: `packages/claude-sdk/src/private/backoff.ts` — exports `calculateBackoffDelay(attempt, random)`, `BASE_DELAY_MS`, `MAX_DELAY_MS`, `MAX_RETRIES`.

**Injectable I/O**: `sleep` and `random` as optional `TurnRunner` constructor parameters (after the existing `logger` param). `main.ts` constructs `new TurnRunner(client, processor, logger)` — no changes needed there since both are optional.

**Test locations**: `packages/claude-sdk/test/backoff.spec.ts` (new, for the pure delay function) + new describe block in `packages/claude-sdk/test/TurnRunner.spec.ts`.

**`FakeMessageStreamer` extension for error tests**: the existing `FakeMessageStreamer` needs to support `Error` as a scripted response (alongside `BetaRawMessageStreamEvent[]`). When the next response is an `Error`, `stream()` throws it directly — the try block in the retry loop catches it the same way as an error from `process()`.

## SDK gotcha

`APIError` is exported from `@anthropic-ai/sdk` (confirmed). The SDK version in `packages/claude-sdk` is `^0.92.0` — the version in the main worktree's node_modules is `0.92.0`.

## What the code doesn't say

The `emptyToolUseRetries` in `QueryRunner.ts` looks like a retry pattern but has nothing to do with the HTTP request — it retries the turn loop when the model returns `stop_reason: tool_use` with no tool uses. Different beast entirely.

The `customFetch` already logs full request/headers at info level — no additional logging needed for the retry path.

# 20:15

Scout phase iteration 2 — scope broadened to all transient errors via `isRetryable`.

## What changed

**Q1 correction (Supervisor flag)**: `err.message` is NOT `'Rate limited'`. `APIError.makeMessage(undefined, body, undefined)` falls back to `JSON.stringify(body)` when there's no top-level `message` field on the body. The surfaced message is the full JSON-stringified payload. Detection is via `error.type`, not `error.message`.

**fromSSEResponse vs fromReadableStream**: Critical distinction I verified from the SDK source. The real HTTP path uses `Stream.fromSSEResponse` (has the `sse.event === 'error'` handler that throws `APIError`). The test helper `makeBetaStream` uses `Stream.fromReadableStream` (yields raw JSON lines, NO SSE error handling). An `{type:"error"}` line through `fromReadableStream` is silently yielded and does not cause `finalMessage()` to throw. This confirms why the fake must throw from `stream()` directly.

**Q7**: `APIUserAbortError`, `APIConnectionError`, `APIConnectionTimeoutError` — all confirmed from `src/core/error.ts` and exported from `src/index.ts`. `APIConnectionTimeoutError extends APIConnectionError` — one instanceof check covers both. `APIUserAbortError` is NOT in the `APIConnectionError` hierarchy.

**Q8**: `overloaded_error` arrives both in-stream (HTTP 200, `status: undefined`) and as HTTP 529 (after SDK's own 2 retries, `status: 529`, `InternalServerError extends APIError`). The 22× log occurrences are in-stream. `isRetryable` checks `error.type === 'overloaded_error'` regardless of status — covers both.

**Q9**: SDK has its own `maxRetries=2` for HTTP 408/409/429/5xx and connection failures. `TokenRefreshingAnthropic` does not override `maxRetries`. In-stream errors (inside HTTP 200) are entirely outside the SDK's retry coverage — that's the primary target. HTTP-level errors can double-retry (SDK 2× + our 10×), acceptable.

**Q10**: `isRetryable` lives in `backoff.ts` (same file as `calculateBackoffDelay` and constants). Check order: `APIUserAbortError` first (never retry) → `APIConnectionError` (retry) → `APIError` with type in `{rate_limit_error, overloaded_error}` (retry) → everything else (never retry).

## Key fact: all errors surface from process(), not stream()

`AnthropicClient.stream()` returns immediately — the HTTP request runs in the stream's `_run` executor asynchronously. ALL errors surface from `this.#processor.process(stream)` via `stream.finalMessage()`. For test fakes, throwing from `stream()` is equivalent since the try block wraps both.

## Decisions I made that the mission did not specify

**Stub `defaultSleep` uses `setTimeout` not `Promise.resolve()`**: the mission says "stub that compiles and the tests can bind to it, but does not yet produce correct delays." Using `Promise.resolve()` would make the abort-awareness tests accidentally pass in Red phase (they resolve immediately, same as the abort-aware real impl). Using `setTimeout` means the tests actually fail because elapsed time exceeds the assertion thresholds. This makes the Red/Green distinction meaningful.

**`APIError` construction in test helpers uses `undefined as any` for headers**: the SDK's `APIError` constructor takes `Headers | null | undefined`. In practice `undefined` may or may not satisfy the type depending on strictness. The Scout used `as any`; I carried that forward. If `null` works, that's cleaner — the Scaffolder should try `null` first and fall back to `undefined as any`.

**`FakeSleep` stores only `ms` values, not the signal**: the abort-awareness is tested via `defaultSleep` in `backoff.spec.ts`. In `TurnRunner.spec.ts`, the abort path is covered by "does not retry APIUserAbortError" — the second `stream()` call would throw abort. Capturing the signal in `FakeSleep` would add noise without additional coverage.

**Retry loop uses `attempt > MAX_RETRIES` not `attempt === MAX_RETRIES + 1`**: both express the same condition given the loop bounds. `attempt > MAX_RETRIES` reads as the general rule ("we've used all retries") without tying it to the specific iteration count, which is clearer when `MAX_RETRIES` changes.

## What the code doesn't say (Scaffolder needs to know)

The `FakeMessageStreamer` modification changes the constructor type. TypeScript will accept existing callers because `BetaRawMessageStreamEvent[]` satisfies `BetaRawMessageStreamEvent[] | Error`. But the Scaffolder should verify this compiles without errors after the change.

The `let result!: MessageStreamResult` pattern (definite assignment assertion) is valid TypeScript but unusual in this codebase. If the Scaffolder's type-check flags it, the alternative is to extract the stream+process pair into a private helper method and use `| undefined` with a post-loop assertion. The `!` is the cleaner choice; the alternative is more defensive.

The `defaultSleep` tests involve real time (50ms, 200ms). Vitest's default timeout is 5000ms. These will not timeout, but they do add ~250ms to the test run in Red phase. Acceptable.

# 23:50

Apostle phase iteration 2 — deterministic abort cancellation.

## What changed from iteration 1

The locked decisions gained one requirement: stopping must be **explicit and deterministic**. When the abort signal is set, the retry loop stops at once with no further `stream()` call, and stopping must NOT depend on the SDK surfacing a particular error class. The iteration-1 plan relied on the next `stream()` call throwing `APIUserAbortError` — implicit, fragile if abort ever surfaces as a connection error (which `isRetryable` treats as retryable).

## The fix (two things)

**Loop change** (`TurnRunner.ts` Hunk 3): after `await this.#sleep(...)`, add `if (turnInput.abortSignal.aborted) { throw err; }`. The loop stops with no further stream() call, regardless of what the SDK would have thrown.

**Test change**: the abort test (`'makes no further stream() call when abort fires during backoff'`) now scripts TWO responses — `[rate_limit_error, makeTextStream('ok')]`. This makes it discriminating: without the explicit check, the loop makes a second call, receives the success, and returns. `streamer.calls.length === 2` → test fails. With the explicit check, the loop stops after the first call. `streamer.calls.length === 1` → test passes. In Red phase (no loop at all), 1 call → passes accidentally, which is acceptable.

**`FakeSleep` gains an optional `onCall` callback** to trigger the abort during the sleep, simulating abort firing mid-backoff.

## What did NOT change

`backoff.ts` — unchanged from iteration 1. `defaultSleep` is still abort-aware. `isRetryable` and `calculateBackoffDelay` are unchanged. All other tests unchanged.

# 00:15

Apostle phase iteration 3 — abort surfaces as abort, not as a stale transient error.

## What changed from iteration 2

The locked decision added: when the loop stops on abort it must surface as an abort (the way a direct Ctrl-C does), not the stale transient error it was retrying. `throw err` in iteration 2 surfaced a `rate_limit_error` as a failure. The new requirement is `throwIfAborted()`, which throws `signal.reason` — exactly what the SDK throws when an abort fires on the first `stream()` call.

## The fix

**`TurnRunner.ts` Hunk 3**: replace `throw err` with `turnInput.abortSignal.throwIfAborted()` inside the `if (aborted)` block. `throwIfAborted()` throws `signal.reason` (set by whoever called `abort(reason)`).

**New test**: `'surfaces the abort signal reason when abort fires during backoff'` — uses `abort.abort(abortReason)` with an explicit `new Error('aborted')` reason, then asserts `rejects.toBe(abortReason)`. The explicit reason makes the assertion precise and independent of Node version defaults. This test fails in Red (stub throws the stale rate_limit_error, not abortReason) and passes in Green.

## Note: `throwIfAborted()` Node version dependency

Requires Node.js 17.3+ (added to `@types/node` in v18.7). If the Scaffolder's type-check fails, the fallback is `throw turnInput.abortSignal.reason ?? new DOMException('The operation was aborted', 'AbortError')`. The test uses `abort.abort(abortReason)` which sets an explicit reason, so both forms throw `abortReason` — the test holds either way.
