## ts_diagnostics POC (#263)

Branch: `feature/ts-tools-poc`

### ITypeScriptService interface (as it ended up)

```typescript
type DiagnosticSeverity = 'error' | 'warning' | 'suggestion';

type Diagnostic = {
  file: string;
  line: number;
  character: number;
  message: string;
  code: number;
  severity: DiagnosticSeverity;
};

type DiagnosticsOptions = {
  file: string;
  severity?: DiagnosticSeverity | 'all';
};

abstract class ITypeScriptService {
  abstract getDiagnostics(options: DiagnosticsOptions): Promise<Diagnostic[]>;
}
```

The `DiagnosticSeverity` type is extracted separately because it maps directly to tsserver's `category` field and will be reused across tool schemas. The `'all'` option on `DiagnosticsOptions.severity` is a filter hint, not a tsserver concept: tsserver always returns everything, and the tool filters before returning.

### How tsserver communication works

tsserver uses a custom framing protocol, not standard LSP JSON-RPC:
- **Outbound** (stdin): `JSON.stringify({ seq, type: "request", command, arguments }) + "\n"`
- **Inbound** (stdout): `Content-Length: N\r\n\r\n{json}` framing. Must buffer and parse incrementally because responses can arrive in multiple chunks.

The key flow for diagnostics:
1. `open` command registers a file with tsserver. Includes `projectRootPath` so tsserver finds the right `tsconfig.json`.
2. Wait ~500ms for tsserver to process (it walks the project, loads config, analyses). Without this delay, the first diagnostics query returns empty.
3. `semanticDiagnosticsSync` and `syntacticDiagnosticsSync` return diagnostics. These are sync variants (they block until results are ready) so no need for event-based polling.

Request/response matching uses `seq`/`request_seq`. Events (type `"event"`) are ignored.

### What surprised me

**Nothing broke.** The composable pattern (factory function taking `ITypeScriptService`, same as `createFind` takes `IFileSystem`) worked on the first try. tsup automatically picked up the new entry point. The main risk was tsserver process management but the `start()`/`stop()` lifecycle pattern is straightforward.

**`createRequire` for tsserver resolution.** Can't use `import.meta.resolve` because tsserver isn't the package entry point. `createRequire` resolves `typescript` to find the lib directory, then derives `tsserver.js` from there. This means the consumer needs `typescript` installed, which is expected.

### What the interface needs for the other three tools

All three need the file opened first (same as diagnostics), so `open` handling stays internal to the service. The interface additions:

**`ts_hover`** needs:
```typescript
type HoverInfo = { text: string; documentation?: string; kind: string };
type HoverOptions = { file: string; line: number; character: number };
abstract getHoverInfo(options: HoverOptions): Promise<HoverInfo | null>;
```
Maps to tsserver's `quickinfo` command. Returns null if no symbol at position.

**`ts_references`** needs:
```typescript
type Reference = { file: string; line: number; character: number; text: string };
type ReferencesOptions = { file: string; line: number; character: number };
abstract getReferences(options: ReferencesOptions): Promise<Reference[]>;
```
Maps to tsserver's `references` command. Returns all references across the project.

**`ts_definition`** needs:
```typescript
type Definition = { file: string; line: number; character: number };
type DefinitionOptions = { file: string; line: number; character: number };
abstract getDefinition(options: DefinitionOptions): Promise<Definition[]>;
```
Maps to tsserver's `definition` command. Returns an array because a symbol can have multiple definitions (overloads, declaration merging).

All three share a `{ file, line, character }` position pattern. Worth extracting a `Position` type when they're implemented.

### Process lifecycle consideration

`TsServerService` has explicit `start()`/`stop()`. The consumer owns the lifecycle. For the CLI, this means creating the service once and keeping it alive across turns. The `open` call per-file is tracked internally (via `#openFiles` set) so repeat queries on the same file don't re-open.

The 500ms delay after `open` is the weakest part. It works for the POC but is fragile for large projects. A proper approach would be to send a `geterr` request and wait for the diagnostic event, but that's a performance optimization, not a correctness issue.
