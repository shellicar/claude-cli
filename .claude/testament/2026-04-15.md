# TypeScript Language Tools (#263)

Branch: `feature/ts-tools`. Four tools: `ts_diagnostics`, `ts_hover`, `ts_references`, `ts_definition`.

## tsserver protocol

tsserver uses custom framing, not LSP JSON-RPC. Outbound: `JSON.stringify({ seq, type: "request", command, arguments }) + "\n"` on stdin. Inbound: `Content-Length: N\r\n\r\n{json}` on stdout. Buffer and parse incrementally because responses arrive in chunks. Match responses to requests via `seq`/`request_seq`. Ignore messages with `type: "event"`.

All position-based commands take `{ file, line, offset }` where both `line` and `offset` are 1-based. The interface uses `character` externally, mapped to `offset` internally.

A file must be `open`ed with tsserver before any query. The first open needs a ~500ms delay for tsserver to walk the project. This delay is the weakest part of the design: it works but is fragile for large projects. A proper fix would use `geterr` and wait for the diagnostic event, but that is a performance concern, not a correctness one.

## Command-to-response mapping

| Command | Response shape | Key fields |
|---------|---------------|------------|
| `semanticDiagnosticsSync` | `Diagnostic[]` | `start.line`, `start.offset`, `text`, `code`, `category` |
| `quickinfo` | object or fail | `displayString` (→ `text`), `documentation` (empty string → undefined), `kind` |
| `references` | `{ refs: [...] }` | `ref.file`, `ref.start.line`, `ref.start.offset`, `ref.lineText` (→ `text`) |
| `definition` | array directly (not wrapped) | `span.file`, `span.start.line`, `span.start.offset` |

The `references` response wraps results in `{ refs }`. The `definition` response is a bare array. This inconsistency is worth knowing because it changes how you extract the body.

## `createRequire` for tsserver resolution

Cannot use `import.meta.resolve` because tsserver is not the package entry point. `createRequire` resolves `typescript` to find the lib directory, then derives `tsserver.js` from there. The consumer must have `typescript` installed.

The initial POC used `createRequire(path.resolve(cwd, '__placeholder.js'))` which breaks when `cwd` has no `node_modules`. The fix uses `createRequire(import.meta.url)` so resolution starts from the package itself.

## Test fixture

`test/fixtures/ts-project/` has `tsconfig.json`, `greeter.ts` (class with JSDoc, `prefix` property, `greet` method), and `main.ts` (imports Greeter, has an `_unused` variable for "no references" test). Tests start a real `TsServerService` per suite.

Fixture files needed biome compliance: `public` modifiers on class members, `_unused` underscore prefix, `biome-ignore` for `console.log`.

## CLI wiring

`TsServerService` is created once in `main.ts`, started eagerly at app init (alongside config watcher), and stopped in the cleanup handler before watcher dispose and layout exit. No `cwd` argument: defaults to `process.cwd()`, same as the rest of the CLI.

The four tools are factory functions that take the shared service instance. They go into `otherTools` which feeds `ToolRegistry`. No changes needed outside `main.ts`.

## Package exports

Each tool needs a `./TsXxx` export in `package.json` (import + require, types + default). Used `pnpm pkg set` for each. Entry files in `src/entry/` are picked up by tsup automatically.
