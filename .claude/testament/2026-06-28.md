# 16:42 — core-di-lite rework: the whole arc (Courier consolidation)

This replaces the per-phase entries for the `feature/core-di-rework` PR. It is the understanding
that does not live in the diff: the principle the SC drove the design to, the traps that cost real
time, and the library semantics you must know before touching `container.ts` again.

## What shipped

Full adoption of `@shellicar/core-di-lite` across all four packages — `claude-core`, `claude-sdk`,
`claude-sdk-tools`, `apps/claude-sdk-cli`. The hand-wired `register(T).to(T, x => new T(x.resolve(...)))`
punch-in form is gone; classes declare dependencies as `@dependsOn` injected properties and the
container resolves the graph. `main` no longer composes anything. `pnpm build`, `type-check`, and
`test` are green across the monorepo (the app alone has 926 tests).

## The load-bearing principle (this is the whole point)

`core-di-lite` is **single, eager resolution, and that is its entire value.** `buildProvider()`
constructs every registered service and runs `injectDependencies` on each, at startup. So every
construction or config error surfaces at boot, not mid-session. Two rules fall out of that:

- **Never a `??=` or lazy getter to dodge a constructor.** It throws the eager guarantee away: a bad
  config or a failed DB open then fails on first *touch*, not at boot. "Anything with `??=` is likely
  wrong" — the SC.
- **Composition lives in `container.ts`, the composition root — not pushed down into the classes.** A
  class that injects raw deps and then builds itself in a getter is doing the composition root's job in
  the wrong place. Classes are *given* finished things. Where there is genuine setup (read+parse, open a
  resource, subscribe, start a watch), that is a **small factory in the root, run eagerly**, registering
  the finished value. Property injection (`.to(Class)`, constructor deleted) everywhere else.

The decision rule for any site: class only receives deps and uses them in methods → property injection.
Genuine setup work → a small single-purpose factory. Never a lazy getter.

## The trap that got iteration 1 aborted — read this

The first Apostle plan PASSED supervisor review and was then **aborted mid-Maker by the SC**, because
it kept forbidden constructs under relabels: a memoised `??=` getter to relocate constructor work, a
`start()` for view subscribers, and `ConfigLoader.load()`/`start()` — all justified as "activation
lifecycle / runtime." Decision 3 was hardened in response: **a reason for a forbidden construct IS the
defect.** If you ever write "this is not the lazy/init thing decision 3 forbids, because…", stop — that
sentence is the smell. Relabeling `load`/`start`/`??=` as "lifecycle" or "activation" does not make it
permitted.

The one genuine exception, sanctioned by the SC: `TsServerService.start()` spawns the tsserver child
process — real async runtime I/O over an already-constructed object, not graph wiring. It is held in
`runApp` as `using ts = provider.resolve(TsServerService); await ts.start();`, and the class
`implements Disposable` (`[Symbol.dispose]() { this.stop(); }`) so teardown runs on scope exit. The
distinction: `load`/`start`/`??=` that dodge DI construction ordering = forbidden; genuine async I/O
over a constructed object = allowed, but **surface it, never relabel-and-proceed**.

## core-di-lite cache semantics — the latent bug to avoid in `container.ts`

The singleton cache key is `descriptor.factory ?? descriptor.implementation`. Consequences when sharing
one instance across a contract token and its concrete:

- **Zero-arg / property-injected concrete:** `register(C).to(C)` + `register(IContract).to(C)` — both
  descriptors have no factory, so both key on `C` → they share one instance. The plain form works.
- **Factory-built concrete:** `register(C).to(C, factoryFn)` keys on `factoryFn`. Then
  `register(IContract).to(C)` keys on `C` *and* its createInstance is `() => new C()` — so resolving the
  contract calls `new C()` **zero-arg** (wrong, C needs ctor args) and does not share. Use the alias
  form instead: `register(IContract).to(IContract, (x) => x.resolve(C))`. This bit `IToolProvider`/
  `AppToolsService` and `IDurableConfigProvider`/`DurableConfigFactory` (both factory-built for setup).

Also: every concrete that `runApp` resolves directly must be registered as its own concrete token, not
only as a contract. The plan registered some as contracts only; they had to be dual-registered.

## How the pieces are shaped now

- **Config (claude-core):** split with no load/start. `readConfig(options, reader, fs)` is a pure
  read+merge+parse. `ConfigLoader` is a holder (state + `onChange` + `apply()`), factory-built from one
  eager `readConfig`. `ConfigReloader` is pure property injection. `ConfigWatchHandle` is a factory whose
  body **is** the single `watcher.watch(...)` call (the only watch call site in the repo) and returns the
  handle as the `using` lifetime; it became an abstract class with `[Symbol.dispose]()`. The SC's test
  for whether a factory earns its place: if it only resolves-and-passes deps, it is hiding a
  side-effecting constructor — collapse it to property injection. A factory must *do* one nameable thing.
- **Database (app):** `DatabaseFactory.getDatabase(name)` with a `Map<string, DatabaseSync>` cache.
  File-vs-memory is the factory's concern via `IDatabaseOptions { inMemory }` (false in `runApp`, true in
  `runVerify`), not two classes and not a `:memory:` magic name. DB names are config
  (`config.persistence.database`, default `persistence.db`). The cache Map is the **legitimate** face of
  memoisation — it does not defer the graph (first `getDatabase` runs when the owning store resolves, at
  boot), so open errors still surface eagerly. Each named db has one owning store: two `:memory:` opens
  are isolated, so single-owner prevents disk/memory divergence under verify.
- **Entry (app, decisions 13/14):** `src/main.ts` exports `main()` (argv parse, early-exit branches,
  builds the four options objects) and `runApp` (the old `main` body, no `new`). `src/entry/main.ts` is
  just `import { main } from '../main.js'; await main();` — its only import-time effect. `entry/verify.ts`
  takes the options objects with `databaseOptions {inMemory:true}`. The subtle part: SIGINT →
  `process.exit` bypasses `using`, so the SIGINT cleanup disposes the watch explicitly via
  `provider.resolve(ConfigWatchHandle)[Symbol.dispose]()`.

## The toolchain constraint: vitest must stay on Vite 7

Documented in `CLAUDE.md` and pinned in the lockfile. The `@dependsOn` decorators are TC39 **standard**
decorators (`experimentalDecorators` off). Vite 7 transforms with esbuild, which lowers them; Vite 8
switched to oxc, which does not lower stage-3 decorators, so on Vite 8 every decorated spec fails to
**load** with `SyntaxError: Invalid or unexpected token`. The build (tsup → esbuild) lowers them
independently, so this constrains the vitest transform only. If you ever see that SyntaxError, check the
resolved vite version before anything else — `vitest` peer-allows `^6 || ^7 || ^8`, so a careless
`pnpm update` can pull v8 and break the whole suite.

## Smaller corrections worth keeping

- **The rework changes WIRING, never CAPABILITY.** "Nothing calls it" is not licence to delete. Stripping
  the `'manual'` auth redirect, `GitStateMonitor.takeSnapshot`, or the `nodeFs` export to "tidy" was
  caught and reversed. Any class whose ctor params are all optional-with-defaults is already
  container-compatible — leave it alone.
- **QuitHandler stays closure-based**, not field-injected. Biome enforces `controller/ ↛ view/`;
  injecting `TerminalRenderer` (in `view/`) breaks the boundary. The container (allowed to import view)
  wires `() => x.resolve(TerminalRenderer).exit()`. This only surfaces under `pnpm ci:fix`, not
  type-check — run `ci:fix` before calling a phase done.
- **AgentMessageHandler injects the concrete `AppToolsService`, not `IToolProvider`** — it needs `.store`,
  which the SDK contract does not expose.
- **No chaining** as a style rule: never `x.resolve(T).foo`. Bind first, then access.

## Test gotchas (cost real time)

- `buildProvider()` resolves the **whole graph** eagerly and runs `injectDependencies` on every instance.
  A container that registers a class which `@dependsOn(IClockProvider)` must also register
  `IClockProvider` — even if the class is supplied as an instance via a factory. Otherwise
  `UnregisteredServiceError` at `buildProvider`.
- `new ConversationState()` **compiles but breaks at runtime** — the clock is now `@dependsOn` and
  undefined unless DI-built. Type-check does not catch it; any clock-using method throws. Build these
  through a real `createServiceCollection()` container with fakes, the way the SDK specs do.

## Standing repo traps (still live)

- Shared worktree: another session can stage under you. `git status` before any stage/commit.
- Build order is bottom-up: `claude-core` → `claude-sdk`/`claude-sdk-tools` → app; workspace packages
  resolve via `exports` to `dist/esm`, so rebuild after adding files.
- Biome import-sort churns only under `pnpm ci:fix`; the rework added many imports.
- `AccountLimitNotice` → `spliceNotice` delivery is deliberately **not** through the control channel; the
  `AccountLimitListener` abstract-class conversion preserves that path.

## The painting bug was tmux, not us. Full diagnosis so nobody re-chases it.

**Root cause: a tmux 3.6b rendering bug with certain variation-selector (U+FE0F)
emoji.** The orphaned-character / doubled-arrow ghost in the TUI headers (e.g.
`ℹ️ query — 21:38 →→ 21:38`) is tmux mishandling the cell for specific emoji:
`ℹ️ ✏️ ☁️ ❤️` ghost, **`⚠️` does not**. Not us. Fixed by **tmux 3.7** (in-pane
content is clean). A residual last-column / pane-edge bleed remains even on tmux
**master (next-3.7)** and is reproducible by echoing `ℹ️` at a bare shell prompt
inside tmux — the CLI never running. So the residual is also purely tmux; report
upstream if ever worth it (minimal repro: `printf '\u2139\ufe0f\n'` in a tmux pane).

### How it was proven (do not re-run these — they're done)

- **Bare terminal: clean. Under tmux: ghosts.** Both iTerm2 *and* Terminal.app
  break identically under tmux → terminal-independent → tmux is the variable.
- **Width is not the cause.** A CSI 6n probe (`/tmp/width-probe.mjs`) showed `ℹ️`
  reports width 2 in every layer. `⚠️` (also width 2) is clean. Same width,
  opposite behaviour → not a width disagreement. The plan's "dormant width
  disagreement" lead is a dead end.
- **Writing every cell does not fix it.** A maximal-explicit renderer (every cell
  its own absolute cursor move, whole frame reverse, per-cell full SGR) still
  ghosted. An `_`-for-blanks coverage probe proved we write every cell, yet stale
  spaces survived — tmux receives our writes but does not forward the cells to the
  terminal (its diff thinks they're unchanged). The corruption is below the app.
- **tmux `codepoint-widths` can't fix it** (it sets cell *count*, not how the
  glyph is drawn; the bug isn't width).
- A from-source **tmux 3.7** build fixed the in-pane ghost outright.

### What shipped (state 2) and why nothing else did

The mission deliverable is the small one: in `ScreenBuffer.diffToWrites`, drop the
row-skip and **full-repaint every row every frame** (`prev`→`_prev`), plus the
identical-grids spec flip and `repaintEveryCell.spec.ts`. Cause-agnostic
robustness, kept as defense-in-depth. Everything else built during the night —
the width-proof writer, the maximal-explicit per-cell reverse renderer with
per-cell SGR, the F3 debug scaffolding (`toggleDebugAscii`/`isDebugAscii`,
blanks-as-`_`, test headers, `dumpBuffer`, the `f3` key added to `claude-core`) —
was all fighting the tmux bug and has been reverted.

### Do-not-relearn

- **In the row-string render model, a wide grapheme's continuation cell MUST stay
  `''`, not a space.** The terminal advances the cursor across the glyph, so a
  space there shifts the whole row right. "Write spaces on empty cells" only
  applies to an absolute-per-cell renderer, which we abandoned. Don't re-add
  continuation-space handling to the row-string path; it's a regression.
- **The CLI runs from `dist` (`node dist/main.js`).** Source edits do nothing
  until `pnpm --filter claude-sdk-cli build`. Testing a stale dist cost an hour.
- **Building tmux from source**: needs `autoconf`/`automake`/`pkg-config` (brew).
  The utf8proc configure check insists on pkg-config; bypass with
  `LIBUTF8PROC_CFLAGS`/`LIBUTF8PROC_LIBS` env. Build in `/tmp`, run with
  `tmux -L <socket> new-session` from *outside* tmux so it doesn't hit the running
  3.6b server.

### Render pipeline (for whoever touches rendering next)

State change → `ViewHost.scheduleRender` (setImmediate coalesces a burst) →
`renderNow` → `PrimaryView.render(model)` → `renderer.paint(rows)`. `PrimaryView`
rebuilds the **entire transcript every frame** (`renderConversation`; sealed-block
*content* is cached per block via WeakMap, dividers rebuilt each frame),
bottom-aligns by slicing to the content height, appends the status bar. `paint`:
rows → `buildGrid` (`layoutRow` measures graphemes via `string-width`; a wide glyph
is one glyph cell + `''` continuation) → `diffToWrites` → bytes inside a `?2026`
synchronized-output block. Note: `writeToScroll`/`flushSealedToScroll` was found
**commented out** (the SC did that deliberately to prove the whole transcript is in
the alt buffer and repainted) — relevant if anyone restores scroll-flush.
