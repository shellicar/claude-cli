## What the previous plan got wrong (and why it passed review anyway)

Its spine was: "core-di-lite has no post-injection init hook, so relocate constructor work into a **memoised `??=` getter**." That, plus a `start()` for the four view subscribers and `ConfigLoader.load()`/`start()`, all justified as "activation lifecycle / runtime." Every one of those is the lazy/two-phase decision 3 forbids, and **a justification for a forbidden construct IS the defect** (decision 3 hardened: "a reason is not an exception"). The relabel passed review because it read as thoughtful prose. It is not. If you ever find yourself writing "this is not the X decision N forbids, because…", stop — that sentence is the smell.

## The principle the SC drove me to (this is the load-bearing part)

He taught it Socratically over many turns. The model, in order of how it clicked:

1. **`core-di-lite` is single, EAGER resolution. That is the entire value.** Every construction/initialisation error surfaces at `buildProvider()`, at startup. A `??=` lazy getter throws that away — a bad config or failed DB open then fails on first *touch* mid-session, not at boot. **Any `??=` to dodge the constructor is wrong.** ("anything with ??= is *likely* wrong" — SC.)

2. **A factory is for WHEN you do config/setup.** The SC is NOT against factories (he wrote the lib, uses them constantly). His objection to PR#350 was factories-by-DEFAULT for *everything*, which hides what each is doing. So: factory where there is genuine setup; plain `.to(Class)` property injection where the class only receives deps and uses them in methods. Decision 8 ("factories only for argv") **can be relaxed** — its real intent was "don't reach for a factory unless there's no other way / unless there's real setup."

3. **`container.ts` IS the composition root. Composition/configuration live THERE, not pushed down into classes.** The previous plan's instinct (and mine) was to push build/config logic down into the classes (a class injects raw deps then builds itself via a getter). That is the class doing the composition root's job. Same code, wrong place. Classes should be *given* finished things, not configure themselves.

4. **AppToolsService was the tell.** It's a class wrapping `createAppTools` — taking individual deps just to call a function. `createAppTools` is itself a hand-wired sub-container (news RefStore, builds tools from deps, reaches for the `nodeFs` singleton). Tools are NOT dependencies — `AppTools` is a tool-registry *value*. So: the composition root's factory calls `createAppTools` directly (injecting fs/tsServer/objects/config), and registers the finished `AppTools`. No wrapper-class-around-a-function.

5. **Factories INJECT their options; source is isolated from use** (the Redis example). A factory never reaches inline for `process.argv`/`process.cwd()`/env/`nodeFs`. Runtime values from `main` ride on registered options objects (`IConfigOptions`, `IRuntimeOptions`, `ITsServerOptions`); the factory resolves the object. "More small, single-purpose factories is better than fewer big ones — the test is what each factory *does*."

## The rule the plan is built on (use this verbatim next time)

- Class only receives deps + uses them in methods → **property injection**, `.to(Class)`, constructor deleted.
- Genuine config/setup work (read+parse, open+configure a resource, build a value, subscribe, start a watch) → **a small factory** in the composition root, run **eagerly** at buildProvider, registering the finished thing.
- **Never** a `??=`/lazy getter to dodge the constructor.
- Plain getters are fine ONLY for cheap derivations that cannot fail (e.g. `path.basename(this.fs.cwd())`) — they defer no error.

## 03:14 — addendum: config watch redesigned with the SC (the ConfigWatcher class is gone)

The SC kept pushing on the config-watch shape until it was right; the final design (plan §3.1) is **not** what an earlier draft had. Record it so nobody reintroduces the bad version:

- A `ConfigWatcher` *class* whose constructor calls `watcher.watch(...)` is **wrong** — its factory does nothing but resolve deps and pass them, laundering a side-effecting constructor. The SC's test: "everything IS a dependency → it's property injection; the factory shouldn't be needed." If a factory only resolves-and-passes, you've hidden a side effect in a constructor.
- Grounding that settled it: `NodeDirectoryWatcher` is **stateless** — `watch(paths, onChange)` opens fs watches and **returns the handle**; the handle IS the whole lifetime. And there is exactly **one** call site of `watcher.watch` in the repo (the old `ConfigLoader.start`). So there's nothing to wrap.
- Final shape (4 parts):
  1. `readConfig(options, reader, fs)` — pure fn (load).
  2. `ConfigLoader` — holder: state + `onChange` + `apply()`. Factory-built with the initial `readConfig(...)` result (eager read, fails at boot). Pure receipt.
  3. `ConfigReloader` — pure property injection (options, reader, fs, ConfigLoader, logger). `scheduleReload()`/`reload()`. No watch, no ctor side effect.
  4. `ConfigWatchHandle` factory — the one irreducible `watcher.watch(options.paths, () => reloader.scheduleReload())` call, returns the handle as the `using` lifetime. `ConfigWatchHandle` becomes an **abstract class** with `[Symbol.dispose]()` (was a `type` with `dispose()`) so it's a DI identifier + `using`-able; `NodeDirectoryWatcher`'s returned literal renames `dispose()` → `[Symbol.dispose]()`.
- The principle the SC drilled, generalised: *something* has to call a side-effecting operation (`watch`, a subscribe). Without putting it in a constructor, the **only** honest home is a factory whose body **is** that call. Pushing it into a class's constructor via `@dependsOn` is "constructor injection by another name" — it just hides the call again. Two factories, each doing one nameable thing (initial read; start the watch); everything else is property injection; no factory merely resolves-and-passes. That rule decides the four view subscribers too (§3.7 — their ctor subscribe stays, factory-registered, eager).


## 03:52 — addendum: database factory redesigned + a hard style rule

Two more things the SC drilled in, both now in plan §2.3/§3.5/§4/§5:

**The database factory is a SERVICE, not a one-shot.** My first cut (`productionDatabaseFactory`/`memoryDatabaseFactory` closures + an `IDatabase` wrapper, selected by registration) was wrong. The SC's design:
- `DatabaseFactory` — one class, `getDatabase(name): DatabaseSync`. Plain property injection (`@dependsOn(IFileSystem)`, `@dependsOn(IDatabaseOptions)`). Holds a `Map<string, DatabaseSync>` cache.
- File-vs-memory is the **factory's** concern via `IDatabaseOptions { inMemory }` (set once: runApp false, runVerify true) — NOT two classes, NOT a `:memory:` magic name in config, NOT a per-store flag. `if (this.options.inMemory) return new DatabaseSync(':memory:')`.
- DB **names are config** (`config.persistence.database`, default `'persistence.db'`). Each store reads its own configured name and asks `factory.getDatabase(name)`. The memory/testament tool will do the same with its own name. The factory never grows; config does.
- `IDatabase` wrapper deleted — the store takes a `DatabaseSync` directly. Connection pragmas live in the factory; the table schema lives in the store.
- Verified empirically (`node --experimental-sqlite`, Node v24): two `new DatabaseSync(':memory:')` are **isolated**. So the cache is what makes same-name callers share one connection (matching disk). **Design rule:** each named db has ONE owning store (DI singleton); two independent opens of a name would share a file on disk but get isolated memory dbs under verify — single-owner prevents that divergence.
- The cache `Map` is the **legitimate** face of lazy (the SC said so explicitly): it memoises per name but doesn't defer the graph's construction — the first `getDatabase` runs when the owning store resolves, at buildProvider, so open errors still surface at boot. This is NOT the forbidden `??=`.

**No chaining — a hard, global style rule.** Never `x.resolve(T).foo` / `x.resolve(T).method()`. Bind first:
```ts
const factory = x.resolve(DatabaseFactory);
const db = factory.getDatabase(name);
return new SqliteObjectStore(db);
```
Direct args (`new T(x.resolve(A), x.resolve(B))`) are fine — it's the resolve-then-access/call that's banned. Every factory in plan §3/§4 was swept for this. Apply it to all generated code, not just the DB.

- `ServiceBuilder.to(impl, factory?)` → `createInstance = factory ?? (() => new impl())`.
- `resolveInternal(id)`: `cacheKey = descriptor.factory ?? descriptor.implementation`; if `singletons.has(cacheKey)` return it; else `instance = createInstance(this)`, then `injectDependencies(instance)`, cache under `cacheKey`.

Consequence for **sharing one instance across a contract + concrete**:
- **Zero-arg / property-injected concrete** (`register(C).to(C)` + `register(IContract).to(C)`): both descriptors have `factory=undefined`, so both cacheKey = `C` → **they share** one instance. ✅ plan literal works (SdkChannel/ISdkMessagePublisher, AccountLimitNotice/AccountLimitListener, ModelOverrides/ModelSettings, StreamProcessor/IStreamProcessor, TurnRunner/ITurnRunner, etc.).

  services.register(IToolProvider).to(IToolProvider, (x) => x.resolve(AppToolsService));
  services.register(IDurableConfigProvider).to(IDurableConfigProvider, (x) => x.resolve(DurableConfigFactory));

  Affected factory-built shared pairs: **IToolProvider/AppToolsService** and **IDurableConfigProvider/DurableConfigFactory** (both are factory-registered because they do setup). Use `(x) => x.resolve(Concrete)`, NOT `.to(Concrete)`.
