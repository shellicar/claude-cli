# core-di-lite adoption — testament for `feature/core-di-rework`

Written after the PR shipped. This is the understanding the diff and the git log don't carry:
why the design is shaped the way it is, the traps that cost real time, and the one piece of work
deliberately left for next.

Orientation: this PR is the full adoption of `@shellicar/core-di-lite` across all four packages
(`claude-core`, `claude-sdk`, `claude-sdk-tools`, `apps/claude-sdk-cli`). The old hand-wired
`register(T).to(T, x => new T(x.resolve(...)))` punch-in form is gone — classes declare dependencies
as `@dependsOn` properties and the container builds the graph. `main` composes nothing; `container.ts`
is the single composition root.

## The one principle everything rests on

core-di-lite is **single, eager resolution** — and that *is* its value. `buildProvider()` constructs
every registered service and runs `injectDependencies` on each, at startup, so every construction or
config error surfaces at boot, not mid-session. Two rules fall straight out:

- **No `??=` / lazy getter to dodge a constructor.** It throws the eager guarantee away — a bad config
  or a failed DB open then fails on first *touch*, not at boot.
- **Composition belongs in `container.ts`, not pushed into the classes.** Classes are *given* finished
  things. Genuine setup (read+parse, open a resource, subscribe, start a watch) is a small,
  single-purpose factory in the root, run eagerly. Property injection everywhere else.

Decision rule for any new site: receives deps and uses them in methods → property injection; real setup
work → a small factory; never a lazy getter.

## The rule the SC hardened — read before touching `container.ts`

An earlier plan passed review and was aborted mid-build because it kept forbidden constructs under
relabels: a memoised `??=` getter, a `start()` for view subscribers, `ConfigLoader.load()`/`start()` —
all dressed as "lifecycle / activation / runtime." The hardened rule: **a reason for a forbidden
construct IS the defect.** If you find yourself writing "this isn't the lazy/init thing decision 3
forbids, because…", that sentence is the smell. Relabelling `load`/`start`/`??=` does not license it.

One sanctioned exception: `TsServerService.start()` spawns the tsserver child process — real async I/O
over an already-constructed object, not graph wiring. Held in `runApp` as
`using ts = provider.resolve(TsServerService); await ts.start();`, and it `implements Disposable`. The
line: async I/O over a built object is fine; a `start`/`load`/`??=` that dodges construction ordering is
not. Surface it, never relabel-and-proceed.

## core-di-lite cache semantics — the latent bug in `container.ts`

The singleton cache key is `descriptor.factory ?? descriptor.implementation`. Whether one instance is
shared across a contract token and its concrete depends entirely on this:

- **Property-injected concrete:** `register(C).to(C)` + `register(IContract).to(C)` — neither has a
  factory, both key on `C`, so they share one instance. The plain form just works.
- **Factory-built concrete:** `register(C).to(C, fn)` keys on `fn`. A plain `register(IContract).to(C)`
  then keys on `C` and `new C()`s it zero-arg (wrong). Use the alias form:
  `register(IContract).to(IContract, x => x.resolve(C))`.

This is why, once `DurableConfigFactory` became property-injected, its registration could drop the alias
for the plain shared form. `AppToolsService`/`IToolProvider` is still factory-built, so it keeps the alias.

## How the graph is shaped

- **Config (claude-core):** no load/start. `readConfig(options, reader, fs)` is a pure read+merge+parse.
  `ConfigLoader` is a holder (state + `onChange` + `apply()`), built by a factory from one eager
  `readConfig`. `ConfigReloader` is pure property injection. `ConfigWatchHandle` is a factory whose body
  *is* the single `watcher.watch(...)` call (the only watch site in the repo); it returns the handle as a
  `using` lifetime and `implements Disposable`. The test for whether a factory earns its place: if it only
  resolves-and-passes deps it is hiding a side-effecting constructor — collapse it to property injection.
  A factory must *do* one nameable thing.
- **Database (app):** `DatabaseFactory.getDatabase(name)` with a `Map<string, DatabaseSync>` cache.
  File-vs-memory is the factory's concern via `IDatabaseOptions { inMemory }` (false in `runApp`, true in
  `runVerify`) — not two classes, not a `:memory:` magic name. DB name is config
  (`config.persistence.database`). The cache Map is *legitimate* memoisation: it does not defer the graph
  (first open runs when the owning store resolves, at boot), so open errors still surface eagerly.
- **Clock:** js-joda's `Clock` is the DI token directly — `register(Clock).to(Clock, () => Clock.systemUTC())`,
  `@dependsOn(Clock)`, no wrapper. Needs a *value* import of `Clock`, not `import type`. The principle:
  wrap a cross-cutting dep in a provider only when the platform gives no seam of its own. js-joda gives
  one (`Clock`, `Clock.fixed`), so the wrapper was redundant; stdlib sleep/random don't, so
  `ISleepProvider`/`IRandomProvider` stay. Tests use `Clock.fixed(Instant.ofEpochMilli(0), ZoneId.UTC)`,
  never a live clock.
- **DurableConfigFactory:** property-injected, `extends IDurableConfigProvider`. `config` is a **getter
  that derives on each read** (`{ ...this.#build(), cachedReminders: this.#cachedReminders }`) — no shared
  mutable object, no `readonly` lie, no per-turn `Object.assign`. `update()` only stores the claude.md
  input; the rest refreshes through the getter. Consumers already read `provider.config` fresh at point of
  use, so the getter is a drop-in.
- **Entry (decisions 13/14):** `src/main.ts` exports `main()` (argv parse, early-exit branches, builds the
  four options objects) and `runApp` (the old body, no `new`). `src/entry/main.ts` is a thin invoker.
  SIGINT → `process.exit` bypasses `using`, so the SIGINT cleanup disposes the watch explicitly
  (`provider.resolve(ConfigWatchHandle)[Symbol.dispose]()`).

## Errors after startup vanish from the console — know this

The winston `logger` writes to a **file only** (`claude-sdk-cli.log`), no console transport. `runApp`
registers `process.on('uncaughtException'|'unhandledRejection', ...)` that route to that file logger.
Registering a handler also **suppresses Node's default stderr trace**. So any error that becomes an
unhandled rejection *after* those handlers install (and after `renderer.enter()` switches to the alt
buffer) produces zero console output — the process just exits to a clean prompt. Errors *before* the
handlers (argv parse, the eager `buildProvider()` where real DI failures surface) still print, which is
why it looks like it "depends where" the error is.

`entry/main.ts` now wraps `await main()` in a try/catch → `console.error` + `exit(1)`, so a fatal
*startup* error is visible again. The steady-state `process.on` handlers are still file-only; if you make
them print, exit the alt buffer first or you scribble over the live TUI.

## Test gotchas (these cost real time)

- `buildProvider()` runs `injectDependencies` on **every** instance — including one you supply via a
  factory `() => existingInstance`. So a container that builds a class which `@dependsOn(X)` must also
  register `X`, even when the class is handed in pre-built; otherwise `UnregisteredServiceError` at
  `buildProvider`. Cleanest path: build the whole graph through one container with fakes (the way the SDK
  specs and `ThinkingRequestWiring.spec` do) rather than `new`-ing instances and registering them.
- `new ConversationState()` **compiles but throws at runtime** — `clock` is `@dependsOn` and undefined
  unless DI-built. Type-check will not catch it. Build it through `createServiceCollection()`.

## Standing repo traps

- **vitest must stay on Vite 7.** The `@dependsOn` decorators are TC39 *standard* decorators
  (`experimentalDecorators` off). Vite 7 lowers them (esbuild); Vite 8 (oxc) does not, so every decorated
  spec fails to *load* with `SyntaxError: Invalid or unexpected token`. `vitest` peer-allows
  `^6 || ^7 || ^8`, so a careless `pnpm update` can pull v8 and break the whole suite. See that
  SyntaxError → check the resolved vite version first. (Also in `CLAUDE.md`.)
- **`ci:fix`, not type-check, is the real gate.** Biome's `controller/ ↛ view/` boundary and import-sort
  surface only under `pnpm ci:fix`. Run it before calling anything done. This is why `QuitHandler` stays
  closure-based: it cannot field-inject `TerminalRenderer` from `view/`, so the container wires
  `() => x.resolve(TerminalRenderer).exit()` instead.
- **The CLI runs from `dist`** (`node apps/claude-sdk-cli/dist/main.js`). Source edits do nothing until
  `pnpm --filter @shellicar/claude-sdk-cli build`. Testing a stale dist has cost hours.
- **The rework changes wiring, never capability.** "Nothing calls it" is not licence to delete — stripping
  the `'manual'` auth redirect / `GitStateMonitor.takeSnapshot` / the `nodeFs` export to "tidy" was caught
  and reversed. A class whose ctor params are all optional-with-defaults is already container-compatible;
  leave it.
- **Shared worktree:** another session can stage under you. `git status` before any stage or commit.
- **`TimeoutSleepProvider` removes its abort listener on the timeout path too, not only via `{once:true}`.**
  `{once:true}` detaches the listener when it fires (an abort); a sleep that ends by timeout must detach it
  explicitly, which the timeout callback does with `removeEventListener`. Drop that and the retry loop's
  reused `AbortSignal` piles listeners up (`MaxListenersExceededWarning` past 10, plus retained closures).
  `TimeoutSleepProvider.spec.ts` reuses one signal across 20 sleeps and asserts
  `getEventListeners(signal, 'abort')` stays empty.
- `AgentMessageHandler` injects the concrete `AppToolsService` (it needs `.store`, which the `IToolProvider`
  contract does not expose). `AccountLimitNotice`→`spliceNotice` delivery is deliberately *not* through the
  control channel; the `AccountLimitListener` abstract-class conversion preserves that path.

## The one thing left for next: collapse the two prompt-loading paths

The deliberate deferral. claude.md and the system prompt are **the same operation** — load instruction
text from the same four locations (user/project/projectClaude/local) and write it to the conversation once
at the start — implemented two different ways:

- **claude.md:** `runTurn` loads via `ClaudeMdLoader.getContent()` and pushes into `update()`; the factory
  is a dumb sink (`cachedReminders`), injected by QueryRunner only when there are no user messages
  (fresh / post-compaction).
- **system prompt:** `SystemPromptLoader` is injected *into* the factory, which owns the enabled-check,
  composition, the async `resolveSystemPromptsFor`, and session state (`#resolvedSystemPrompts`,
  `#systemPromptSessionId`, `needsSystemPromptResolve`).

The SC's point: **updating an in-memory value and using it are separate concerns.** Both inputs should
stay fresh in memory the same cheap way; `config` derives on read (already true after this PR); the
consumer decides *when* to use it at the point of use. The async `resolveSystemPromptsFor` (the
"activation" smell-word, now reworded out of the doc-comment) is a symptom of fusing "consumed once per
session" with "guard the update" — the gating belongs at *use*, not *update*. Target: both collapse to one
push path, no session-prompt gating in the factory. Sized as its own piece of work. (bananabot's
suggestion to move `#build` into the container factory is a dead end — `#build` re-runs per turn, so it
cannot live in a run-once factory; the reshape above is the real fix.)

## Rendering / tmux — unrelated to this PR, kept so it is not re-chased

The variation-selector emoji ghost in TUI headers (`ℹ️ ✏️ ☁️ ❤️` doubled or orphaned; `⚠️` is fine) is a
**tmux 3.6b bug, not the app** — fixed in tmux 3.7, reproducible by `printf '\u2139\ufe0f\n'` at a bare
shell inside tmux with the CLI never running. Do not re-chase it in the renderer.

Render path, for whoever touches it: state change → `ViewHost.scheduleRender` (setImmediate coalesces) →
`PrimaryView.render` rebuilds the whole transcript each frame → `renderer.paint(rows)` → `buildGrid`
(`string-width` per grapheme) → `diffToWrites` inside a `?2026` synchronized-output block. **A wide
grapheme's continuation cell must stay `''`, never a space** — a space there shifts the row right;
re-adding continuation-space handling to the row-string path is a regression.

# 03:11 (30th) — ExecV3 shipped: what the next session should know

ExecV3 is built and merged into the #380 ship. The phase-by-phase story (who walked what, which
supervisor said PASS, the G1 back-and-forth) is in the git log and not worth your time. This is
the part the log doesn't carry: how the thing actually works, where it bites, and the traps that
cost real time.

## What ExecV3 is

`packages/claude-sdk-tools/src/ExecV3/`. A structured exec tool: a flat `commands` list joined by a
forward-pointing `op` (`;` `&&` `||` `|`), no shell. It replaces ExecV2's recursive `{op,left,right}`
tree, which fought construction. The call-time field is **`intent`** (Claude's goal for the run), not
`description` — that rename went across all the exec tools, V1 Exec included.

The pieces: `schema.ts` (Zod input/command/result + `superRefine` for the cross-command rejects),
`group.ts` (flat list → maximal `|` pipelines + connectors), `engine.ts` (the fold), `runPipeline.ts`
(one pipeline → one result per stage), `normalise.ts` (path expansion via `IFileSystem`), `ExecV3.ts`
(tool factory + the six bash-equivalent examples the model reads at invocation).

Built on **`@shellicar/exec-core`** (`packages/exec-core/`): `IExecutor.run(cmd, opts)` runs ONE
process and captures nothing — the caller wires PassThrough streams and collects via `fromStream`.
The exit-code edges (126 bad cwd, 127 ENOENT, null+SIGTERM when already aborted) live inside
`Executor.run`, so ExecV3 inherits them for free.

## The engine model (worth holding in your head)

Bash list semantics, reproduced by a single left-to-right fold because `&&`/`||`/sequential are all
left-associative. `evaluate()` walks the pipelines, gating each connector on the carried `$?`. The one
thing that is easy to get wrong: short-circuit **skips and continues**, it does not `break`. `false &&
echo a` then a sequential `echo b` still runs `echo b` — bash does. The fold that `break`s is the bug
that was found and fixed; don't reintroduce it.

Results are **position-aligned**: `(CommandResult | null)[]`, length always equals `commands.length`,
`results[i] ↔ commands[i]`. A skipped (short-circuited) command is `null`. Read defensively:
`results[i]?.exitCode`. `exitCode: null` therefore means signal-kill ONLY. Pipeline exit status is the
**last stage**, no `pipefail`; the per-stage truth is in the `results` array.

## Known bash-divergences — untested, real, and the most valuable thing here

CI is green and the 64 ExecV3 tests pass, but these paths have no coverage and diverge from the
"bash-exact" claim. Nobody will re-find them from a failing test, so they're written down:

1. **Redirect file errors are swallowed → false success.** `runPipeline`'s sink resolution does
   `file.on('error', () => {})`, and the reported `exitCode` is the program's, not the redirect's. So
   `echo hi > /nonexistent/dir/f` comes back `success: true`; bash returns failure. This is the
   sharpest divergence and the easiest to fix.
2. **Pipe early-exit has no SIGPIPE → hang.** `find / | head` in bash: `head` takes its lines and
   exits, `find` gets SIGPIPE and dies instantly. In ExecV3 the stages are bridged by a PassThrough;
   when `head` exits nothing kills `find`, it fills the pipe buffer, stalls on backpressure, and
   because `runPipeline` does `Promise.all` over all stages the call hangs on the live producer until
   the timeout fires. `… | head`, `… | grep -m1`, `cat big | head` all run to completion-or-timeout
   instead of short-circuiting. The fix is the hard one: propagate downstream-exit upward to kill the
   producer — the thing SIGPIPE does for free.
3. **Two streams, one file → truncate-and-interleave.** `redirect: { stdout: f, stderr: f }` (same
   path, not `stderr: '&1'`) opens two `'w'` streams to one file; they truncate each other and
   interleave. The schema makes "two destinations, one stream" unrepresentable but not the reverse.

Checked and **solid** (don't re-audit): dangling-operator and pipe/redirect rejects in `superRefine`,
pipe-stdin handling, and signal-kill exit treatment in the engine.

## Wiring — five touch points

`entry/ExecV3.ts` (tsup globs `entry/*.ts`, no config change), the hand-maintained `./ExecV3` block in
`package.json` `exports`, `createAppTools`, and the `execV3` flag in `cli-config/schema.ts`. ExecV3
ships **default-on**, ExecV2 flipped **default-off**. The changelog entry is the Courier's.

## Traps that cost real time

- **Build ordering.** `claude-core → exec-core → claude-sdk → claude-sdk-tools`, or just run the root
  `pnpm build`. Out of order, the DTS step fails with `Cannot find module '@shellicar/claude-core/...'`.
  The mission's verify commands assume deps are already built.
- **`createAppTools` was a merge landmine and is no longer.** It had a 6-positional-parameter signature
  that grew every time a tool gained a dependency — and that is literally what made the #380 merge
  conflict against main's Memory work unresolvable-by-git. It's now an options object
  (`CreateAppToolsOptions`) with a push-based tool array. Lesson worth carrying: positional growth on a
  composition-root function is a collision waiting to happen.
- **mcp-exec wraps V1 Exec.** When the `description → intent` rename went through, `mcp-exec`'s test and
  README still called it the old way and CI went red on a package the ExecV3 mission never named. Claude
  is the consumer of mcp-exec, not external programs, so renaming the input field is fine — just update
  the callers. If you touch an exec tool's input schema, grep `mcp-exec` for the field.
- **The daily testament is a merge hazard.** This file (`2026-06-28.md`) is per-DAY, not per-branch, and
  it got mangled merging main: main's core-di-lite commit had rewritten the same dated file, so git
  three-way-merged two unrelated versions into a truncated, orphaned mess with no conflict markers. If
  you merge main and the dated testament collides, take main's version of the file wholesale and
  re-append your own entries — don't let git interleave them.
