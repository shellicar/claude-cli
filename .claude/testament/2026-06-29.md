## The DI pattern on the new main (#378) â€” read this before resolving any conflict here

`@shellicar/core-di-lite` property injection is the house style **everywhere** on main: every
handler, model, SDK class, the config layer, and `DatabaseFactory` itself declare
`@dependsOn(Token) private readonly x!: T` fields and register with a bare `.to(Impl)`. Grep
`@dependsOn` on origin/main and you'll see it's the norm, not the exception.

The deciding rule (the SC's words):
- **If it is configuration or wiring, it lives in `container.ts`.** That is the entire point of
  that file â€” create, resolve, AND configure the classes there.
- **A constructor must hide no dependencies.** No reaching out to a singleton, no reading config,
  no `new`-ing a dependency inside a constructor. Whatever a class needs is handed to it.
- **A factory closure (`.to(Token, (x) => new Impl(...))`) is allowed only when a constructor
  argument is not itself a resolvable token** â€” a runtime/derived value: a `db` handle, a
  cwd-derived string, a config slice, an assembled array/map. When every dep is "resolve this
  token," use `@dependsOn` + bare `.to(Impl)`.
- **Eager initialisation in a constructor is fine** â€” opening/preparing a db, running DDL/
  migrations on a db that was *passed in*. That hides nothing; it works on what it was handed.
  This is why `@dependsOn` (property injection, constructor runs before fields are set) cannot do
  DB work in its constructor, but a factory-built class (args passed in) can.

## How Memory wires under that rule

- `SqliteMemoryStore` and `GitMemoryEnvironmentProvider` are `@dependsOn` classes â€” bare
  `.to(Impl)`. The store keeps `@dependsOn(SqliteMemoryEngine)` and
  `@dependsOn(IMemoryEnvironmentProvider)` exactly as the SC wrote it. **Do not rewrite the store
  to a plain constructor / factory** â€” that was my mistake; `@dependsOn` is correct and matches
  everything around it.
- `SqliteMemoryEngine` is the **one** factory: its db is not a token, and the db-file name from
  `config.memory.tenantId` is *configuration*, so it is computed in `container.ts`
  (`tenantId == null ? 'memory.db' : 'memory.<id>.db'`), the db pulled from
  `DatabaseFactory.getDatabase(name)`, and handed to `new SqliteMemoryEngine(db, clock)`. The
  engine runs its DDL/migrations on that db in its constructor (eager init). This mirrors exactly
  how `SqliteObjectStore` is wired on main.
