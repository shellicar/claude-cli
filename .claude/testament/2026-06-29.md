# Memory tool — the consolidated record

The persistent, shared, relevance-searchable Memory tool: built in Phase 2, then rebased onto the
`core-di-lite` #378 refactor. This is the single durable record — it folds in the Phase 2 build notes
and supersedes the earlier "DI constraint" section (written pre-#378, now wrong). What the code and
the git log can't tell you lives here.

## What it is, and how it's built

- **`packages/claude-core/src/memory/`** — the contract: `IMemoryStore`, the types, the pure
  `toFtsMatch` search helper, and `IMemoryEnvironmentProvider`. Timestamps are ISO strings on the
  contract; `@js-joda` stays confined to the stores, never the library runtime.
- **`SqliteMemoryEngine`** (app, `persistence/`) — the real store, a plain class: constructor
  `(db, clock)`, all the SQL, DDL in the constructor. One **ordinary** FTS5 table (not contentless,
  not external-content, so `UPDATE`/`DELETE` work for the soft-delete) with `title/body/keywords`
  indexed and `id/type/keywords_json/environment/created_at` UNINDEXED. `keywords` is stored twice on
  purpose: indexed text (ranks) plus `keywords_json` (returns the array faithfully). bm25 weights
  `10.0, 1.0, 4.0` give title > keywords > body; column order fixes the weight positions. This is the
  class the integration tests run over `:memory:`.
- **`SqliteMemoryStore`** (app, `persistence/`) — the injectable `IMemoryStore` adapter:
  `@dependsOn(SqliteMemoryEngine)` + `@dependsOn(IMemoryEnvironmentProvider)`, resolves environment
  per write, delegates the rest. (No lazy open — the engine is built and injected by the container.)
- **`GitMemoryEnvironmentProvider`** — resolves `org`/`project`/`repo` from the live git remote and
  `fleet` from config, **on every write**, so a mid-session remote/cwd change is reflected.
- **`packages/claude-sdk-tools/src/Memory/`** — the five tools (`WriteMemory`, `ReadMemory`,
  `SearchMemory`, `DeleteMemory`, `MemoryTypes`). Thin handlers that map and shape; no store logic.
- **Rendering** consumes the existing client `tool_result` SDK event (on main from history-view #364)
  via `AgentMessageHandler`; `ToolObject.setResultLine` carries SearchMemory's count + top title.

## The load-bearing safety piece

`toFtsMatch` (pure, claude-core): match `/[\p{L}\p{N}]+/gu`, wrap each token in double quotes, join
with ` OR `. OR (not AND) because the goal is relevance and recall, not exact conjunction. No-token
input returns `null`, the store returns `[]`, never errors. Every FTS5 operator a user might type
(`hello -world`, quoted phrases, `OR`/`AND`/`NEAR`, `star*`, `(paren)`, bare `-`) is reduced to
quoted literals, so none survives to reach the engine. A bareword `OR` becomes the search term
`"OR"`, which is the point. The full adversarial input-safety set runs green against the real engine.

## DI: when `@dependsOn`, when a factory — the rule (the SC's)

`core-di-lite` is property injection, used everywhere on main (grep `@dependsOn`): the container does
`new Impl()`, *then* sets the `@dependsOn(Token) private readonly x!: T` fields, and registration is a
bare `.to(Impl)`. Because fields are set after construction, a constructor cannot read them.

- **Configuration and wiring live in `container.ts`.** That is the file's whole purpose: create,
  resolve, configure. A class never sources its own dependency or config — no singleton import, no
  config read, no `new`-ing a dependency in a constructor. If you see that, it belongs in the container.
- **`@dependsOn` + bare `.to(Impl)`** when every dependency is a resolvable token. This is the default.
- **A factory closure `.to(Token, (x) => new Impl(...))` only when a constructor argument is not a
  token** — a runtime/derived value: a db handle, a cwd-derived string, a config slice, an assembled
  array/map. That, and only that, earns a factory.
- **Eager init in a constructor is fine and good** — opening a db, running DDL/migrations — *as long
  as it acts on what was handed in*; it hides nothing. This is why a factory-built class can do DB work
  in its constructor and an `@dependsOn` class cannot: the latter's fields aren't set yet.

So Memory wires this way: the store and the environment provider are `@dependsOn` classes (bare
`.to(Impl)`); `SqliteMemoryEngine` is the **single** factory, because its db is not a token and the
db-file name (`config.memory.tenantId` → `memory.db` / `memory.<id>.db`) is configuration, computed
in `container.ts` and handed in. Same shape as `SqliteObjectStore`. Don't "simplify" the store to a
plain constructor — `@dependsOn` is what matches everything around it.

## The test-double lesson (the most important one on this mission)

A test double with behaviour means your unit test tests the double, not the unit. An earlier
`FakeMemoryStore` was a working substring store, and the handler "unit tests" asserted its filtering —
testing the fake. The fix: `RecordingMemoryStore` is a **spy** that records the call argument and
returns a canned value, nothing else. Handler tests assert only **mapping** (the captured arg proves
`intent` is dropped and the fields forwarded) and **shaping** (canned return → output shape). Every
search/filter/rank/count assertion lives in the integration tier against the real engine. One ranking
implementation, so never a parity test. The double's form (stub/spy/fake) is the operator's call.

## Asserting on log output is an anti-pattern of the worst kind

`migrate.spec.ts` / `SqliteMemoryEngine.spec.ts` had once been written to inject an `ILogger` and
assert on `logger.entries` — testing the narration instead of the behaviour. The `RecordingLogger`
they imported was never committed beside them; it lives in `stash@{6}`, but that stash is a *separate,
broad ILogger-injection refactor* — never paired with migrate, so don't resurrect it here. Resolution:
no injected logger (migrate is `(db, migrations, name)`, engine is `(db, clock)`); the tests that read
logs now assert behaviour. If a test says "it logged X," it is testing the wrong thing; assert what the
system *did*. (`migrate.ts` itself still logs through an imported `logger` singleton — the
hidden-dependency reach the DI rule would lift into the container. Left as-is, flagged.)

## SC design decisions the code doesn't explain

- **`intent`, not `description`.** The per-call narration field (cross-tool convention, as `Exec`
  carries) where Claude narrates the call to the SC for visibility: required input, **not stored, not
  searched, not ranked**. Present on write/read/search/delete, absent on `MemoryTypes`.
- **Environment resolves per write, never at startup.** The git remote can change mid-session, so a
  launch-time snapshot would stamp a stale origin. The provider resolves live at write time.
- **Schema migrations** (`migrate.ts`) are an append-only list, versioned in `PRAGMA user_version`
  (`major*1000 + minor`; minor = additive/tolerated, major = destructive/refused). The version is
  re-checked *inside* the `BEGIN IMMEDIATE` write lock so two CLIs racing the same migration can't
  double-apply — the whole reason the shared `~/.claude` store needs the lock. See CLAUDE.md
  "Database Schema & Migrations".

## Traps that cost real time

- **biome runs only in `pnpm run ci`, never in `type-check` or `test`.** It will fail CI on
  import-sort, formatting, barrel files, and member-accessibility that a green gate sailed past — it
  flagged 18 errors after a clean type-check/test once. Hand-fix the non-auto-fixable ones (barrel
  file, `public` modifiers), then `pnpm ci:fix` (never `biome check --write` directly). Run before pushing.
- **`createMemoryTools` returns a tuple, not an array.** `return [Write, Read, …]` infers as
  `Array<unionOfAllFive>`, so tests calling a specific tool fail type-check. Cast to the explicit tuple
  of `typeof` each tool.
- **No `export *` barrel.** Biome's `noBarrelFile`/`noReExportAll` reject it at error level; the entry
  file uses explicit named re-exports, matching the existing `Exec.ts` shape.
- **FTS5 + bm25 + porter confirmed on this machine's Node** via the engine integration tests (the
  `CREATE VIRTUAL TABLE … USING fts5` in the constructor is the earliest-signal check, and it passed in
  a real run). The bundled SEA's Node is the one remaining binding risk; construct-on-`:memory:` re-checks it.
- **To take main's side of a conflicted file:** `git show origin/main:path > path`. `git checkout
  --ours` and `git restore` are blocked by safe-operations.
- **The dated testament files are `merge=union` and shared across concurrent missions.** A rebase can
  make your copy *look* like it replaced another mission's entries; the union driver recombines both
  sides at merge. Leave others' entries alone; before shipping run
  `git diff origin/main...HEAD -- .claude/testament` to confirm you only added your own. (This is why
  `2026-06-27.md` also carries an unrelated repaint-every-cell entry — it's main's, folded in by union.)

## Phase 5 Apostle — interruption-resilience implementation plan

Wrote `implementation-plan.md` in the Handler mission dir (not this repo). It
covers all four pieces with real code blocks, grounded in the files at `f6e7ad8`.
This is the rebuild design after Phase 4 (Maker) was reverted for putting wake-lock
config + platform logic in `container.ts`, snapshotting dynamic config, and
registering a concrete to itself.

### The load-bearing design decisions (so a re-run doesn't re-derive them)

- **The wake-lock and the reconnect signal both attach inside `TurnRunner.run`,
  not at the CLI layer.** The request lifecycle (connect + backoff + stream) lives
  entirely in `TurnRunner.ts:103–143`. The only CLI-visible per-turn boundaries
  are SDK events, and `message_start` only fires *after* the connection succeeds —
  too late to hold a lock during connect/backoff. So the SDK must call injected
  interfaces. This is the **exact `AccountLimitListener` precedent**: a CLI concern
  (display / sleep) surfaced into `TurnRunner` via a small injected interface,
  because `TurnRunner` has no channel publisher (QueryRunner does). Mirror it; don't
  invent a new mechanism.

- **Wake-lock `enabled` is dynamic (read per-acquire); platform + PATH availability
  are static (decided once at startup).** This is the reconciliation of pinned
  principles 3 (read mechanism at use-time) and 4 (bind real-vs-noop through DI).
  The factory `createWakeLock` picks `CaffeinateWakeLock` vs `NoopWakeLock` from
  `process.platform` + an on-PATH probe at startup; `CaffeinateWakeLock.acquire()`
  reads `config.preventSleep` live each call via an injected `() => config.preventSleep`
  accessor. The accessor (not the whole ConfigLoader) is what keeps it testable AND
  use-time. Don't pass ConfigLoader into the wake-lock class — pass the accessor.

- **The JSON config schema is GENERATED.** `apps/claude-sdk-cli/build.ts:42–43`
  writes `schema/sdk-config.schema.json` from the zod `sdkConfigSchema` at build
  time. Edit `cli-config/schema.ts` only; `pnpm build` regenerates the JSON. The
  brief's "add to the JSON schema" is satisfied by the build, not by hand-editing.
  Hand-editing drifts and gets overwritten.

- **`cli-config.spec.ts` has a strict `toEqual` on the full default config object.**
  Adding `preventSleep` to the schema breaks it unless the expected object is
  updated. Easy to miss — it's a whole-object equality, not per-field.

- **`terminated` classification goes in `transport.ts`, not `sse.ts`.** Convert
  `yield* parseSse(...)` to a manual `for await` loop with a `catch` that mirrors
  the connect-phase catch's abort/timeout/TransportError/else discrimination. The
  raw undici `terminated` is the `else` → wrap as `StreamInterruptedError` (its own
  class extending `TransportError`, NOT `ConnectionError`, so it bypasses the
  generic exponential `isRetryable` path and gets its own short-fixed strategy
  branch in `TurnRunner`).

- **Submit-to-resume sends `messages: []`.** `QueryRunner.run` with an empty
  messages array pushes nothing and loops against the existing trailing user
  message — no SDK change. The CLI changes are: `EditorHandler` gains a `Conversation`
  dep and allows empty submit when `messages.at(-1)?.role === 'user'`; `RunAgentInput.message`
  becomes nullable; `runAgent` skips the prompt block and sends `[]` on resume.

- **Persist on `turn_content`** (per-SDK-turn, fired at `QueryRunner.ts:149`). At
  that instant the conversation ends on the completed assistant message (consistent).
  Inject a `saveConversation` callback into `AgentMessageHandler` (it has no session
  today). Atomic write needs a new `IFileSystem.rename` — added to the abstract class,
  so EVERY `extends IFileSystem` must implement it (NodeFileSystem + two test
  MemoryFileSystems are the known set; grep `extends IFileSystem`).

### Three forks I surfaced rather than decided (plan §7)

1. `TurnRunner` constructor is now 9 positional params (two new trailing optionals +
   `undefined` test-seam placeholders at the wiring site). Options-object refactor
   would read better but is unrequested scope. Left positional.
2. The brief says the display reset "also fixes transient mid-stream garble." I wired
   `reconnecting()` only to the stream-interrupt branch (the required case). The
   generic transient branch fires for both connect-phase (no garble) and mid-stream
   `ApiStreamError` (garbles); firing `reconnecting()` there would show a spurious

3. `caffeinate -i -w <pid>` — `-w` is an anti-leak measure (crash without release
   can't strand a process holding the machine awake). Brief literally said `-i` only.

## Phase 5 Apostle — iteration 2 (SC correction on the wake-lock)

The SC caught a defect my iteration-1 plan had and the supervisor missed:
**core-di-lite registrations are all singletons.** My iteration-1 design used a
startup factory (`createWakeLock`) to pick `CaffeinateWakeLock` vs `NoopWakeLock`
from config + a PATH probe — but a singleton registration runs that factory once,
freezing the choice and snapshotting config at registration. The SC's point, exactly:
*`WakeLock` is already a factory — `acquire()` returns a handle — so every dynamic
decision belongs inside `acquire()` on the one singleton, not at registration.*

### What changed (scoped to §1 — wake-lock registration/lifecycle only)

- **One `WakeLock` implementation, `CaffeinateWakeLock`, registered as a plain
  singleton** (`register(WakeLock).to(CaffeinateWakeLock, factory)`). No
  `createWakeLock` factory, no separately-bound `NoopWakeLock` class — both deleted
  from the plan.
- **Every dynamic decision moved inside `acquire()`:** read `enabled` (live),
  resolve the platform command (live), check availability, then spawn or return a
  no-op handle. The no-op is now a *handle* (`NOOP_HANDLE`) returned per-call, not a
  bound implementation.
- **Availability probe is lazy + memoised per command** (a `Map<string, boolean>`
  on the instance), probed on first acquire — NOT at startup. Key insight: a
  startup probe would have to read `config.preventSleep.platforms` at registration,
  which is the exact frozen-config problem the correction rules out. Lazy-memoised
  satisfies "probe once" (per command) without any registration-time config read.
- `isExecutableOnPath` survives as the injected probe (`WakeLockProbe` type now on
  `CaffeinateWakeLock`). `resolvePlatformCommand` and `spawnWakeLock` unchanged.

### For whoever builds this: the load-bearing constraint

**core-di-lite is singleton-only.** Any design that needs a per-use decision must
put that decision in a method called per-use, not in the registration factory. The
registration runs once. The `() => x.resolve(ConfigLoader).config.preventSleep`
accessor works because the *closure* is what's stored; it reads live config each
time `acquire()` calls it. This is the general pattern for "dynamic config in a
singleton DI graph" in this codebase.

Everything else in the plan (pieces 2–4, the §0 SDK seams, the test approach for
the other pieces) was untouched — the SC scoped the correction to the wake-lock.
The three §7 forks still stand. No code in the worktree; the plan in the Handler
repo is the only deliverable. Testament re-staged.
