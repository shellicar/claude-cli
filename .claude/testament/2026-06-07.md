# 14:02

Courier for cancel-tool-use, Phase 6. Branch `feature/cancel-tool-use`. What this PR ships and what the next operator needs to know.

## What was built

ESC can now cancel a running Exec subprocess and deliver the cancellation to Claude. Three behaviours:

1. ESC while a tool is running: cancels the tool, delivers a `tool_result` with elapsed time and a cancellation message to Claude, query continues so Claude can decide what to do next.
2. ESC while the model is thinking or streaming (no tool running): ends the query, returns to the prompt. Existing behaviour, unchanged.
3. Double-ESC while a tool is running: ends the query immediately.

**What landed across the three commits:**

Phase 3 (plumbing):
- `ToolCancelledError extends Error {}` in `packages/claude-sdk/src/public/ToolCancelledError.ts`, exported from `index.ts`
- Tool handlers receive `signal?: AbortSignal` (optional); the 18 non-Exec handlers are not respelled since TypeScript allows fewer-parameter functions to satisfy a wider type
- Exec chain threads `abortSignal` through `Exec.ts -> execute.ts -> execStep.ts -> execCommand.ts / execPipeline.ts`; `spawn({ signal: abortSignal })` kills the child natively; every abort-reachable `close` and `error` terminal handler rejects with `new ToolCancelledError()`
- `ToolRegistry` captures `startMs` at run start; on `instanceof ToolCancelledError`, builds "Tool execution cancelled by user after X.Xs" and returns it as the `tool_result`

Phase 5 (cancel-separation):
- `ApprovalCoordinator` gains `#toolController` and `#toolCancelled` fields; `toolRunStarted(controller)` and `toolRunFinished()` register/deregister the running tool's controller; `handle()` now returns `CancelOutcome` (`'none' | 'tool_cancel' | 'query_cancel'`)
- `QueryRunner.#handleTools` creates its own `toolController` per batch, wraps each `run` call with `toolRunStarted`/`finally toolRunFinished`, passes `toolController.signal` to the handler (not the query signal)
- `main.ts` subscriber reads `handle()`'s return value; aborts the query `AbortController` only on `query_cancel`

SC fix (between phases, committed separately): moved `@shellicar/claude-sdk` from `devDependencies` to `dependencies` in `packages/claude-sdk-tools/package.json`.

## Key decisions

**Route in the SDK, not the TUI.** The routing (tool-cancel vs query-cancel) happens in `ApprovalCoordinator.handle()`. The TUI has no window on when a tool handler is executing - `AgentMessageHandler` sees `tool_approval_request` before the handler and prints its summary immediately after sending the approval response, not after the handler finishes. `QueryRunner` is the only place that knows the execution window precisely, so `ApprovalCoordinator` became the routing brain. ESC keeps sending `{ type: 'cancel' }` unchanged; no new `ConsumerMessage`, no `AppLayout` or `runAgent` edit, no new SDK events.

**Double-ESC falls out for free.** `#toolCancelled` records whether this run already got a cancel (reset via `toolRunStarted`). Second cancel while the tool is still registered escalates to `query_cancel`. If the tool already died and the delivery turn is streaming, `#toolController` is null, which also routes to `query_cancel`. No ESC-counting needed anywhere.

**`CancelOutcome` is not exported from `index.ts`.** `main.ts` gets the type by type inference from the return of `approval.handle()`. If type-check complains in future, exporting from `index.ts` is the fallback.

## Traps

**Naming collision in execCommand/execPipeline.** The `close` event handler has its own parameter named `signal` (the OS exit signal name, written into the result as `signal: signal ?? null`). The abort signal must be named `abortSignal` throughout the Exec chain or it shadows the OS signal name and corrupts the result object. This is wired correctly; do not rename it.

**devDependency bundling makes `instanceof` fail across packages.** This is the Phase 5 Iteration 2 BLOCK and the biggest trap in the whole feature. tsup (`bundle: true`) inlines everything from `devDependencies` but externalizes `dependencies` and `peerDependencies`. When `@shellicar/claude-sdk` was a `devDependency` of `claude-sdk-tools`, tsup bundled its own inline copy of `ToolCancelledError` into `packages/claude-sdk-tools/dist/`. The registry (`packages/claude-sdk/dist/`) held a different copy. Two class objects at runtime, `instanceof` returns false, `isCancelled` is false, the error falls through to `err.message`, and `new ToolCancelledError().message` is `''`. The cancellation `tool_result` arrived at Claude with empty text. Fixed by moving the dependency to `dependencies`. The general lesson: cross-package `instanceof` only works if both packages resolve to the same module instance at runtime. Bundling breaks this for `devDependencies`. Always verify with the built dist, not just the vitest source run.

**The loop/delivery bug (Phase 3 root cause).** Phase 3 wired the tool to honour the query `AbortController`'s signal. ESC calls `approval.handle({ type: 'cancel' })`, which sets `#cancelled = true`, and also aborts the query controller. Both together: `while (!this.#approval.cancelled)` exits before the delivery turn runs, and the delivery turn's `input.abortController.signal` is already aborted anyway. The cancellation `tool_result` was pushed to the conversation array but never delivered to Claude. The fix: a tool-cancel aborts only `#toolController` (never the query controller), never sets `#cancelled`, and the query loop continues untouched for the delivery turn.

**Tests cover source, not built dist.** vitest runs through the source module graph. Neither the Phase 3 Exec unit test nor the Phase 5 `QueryRunner.spec.ts` integration test loads the bundled dist, so neither surfaces the class-duplication bug. The live test caught both defects (the loop bug and the empty message). A built-path regression guard was considered and deferred.
