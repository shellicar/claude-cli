Apostle cast for the `--config '{...}'` override. Plan written to the PM repo at `projects/claude-cli/investigation/2026-06-07_config-override-plan.md`. No code landed. Two decisions flagged for the supervisor.

## The one fact that shapes everything

`--model` does **not** flow through the config object. It sits in a standalone mutable slot (`main.ts:177` `const overrides: { model: string | null }`) and is resolved by `getEffectiveModel()` (`main.ts:181`), which layers the slot over `configLoader.config.model`. Every *other* config value comes straight from `configLoader.config`. So "override any config value" means influencing the config resolution, not the model slot. Two separate override mechanisms exist and the plan keeps them separate.

A free win falls out of this: `getEffectiveModel()` already makes `--model` beat `configLoader.config.model`, so `--config '{"model":...}'` (which sets the config value) is naturally overridden by `--model` when both are present. The "precedence" requirement needs zero new code if you accept that behaviour. That's flagged decision #2.

## The central tension (flagged decision #1)

The config schema is permissive *by design*: every field has `.catch()` (wrong type → default, never throws) and the top-level `z.object` strips unknown keys silently. `test/cli-config.spec.ts` locks this in (`it('ignores unknown fields')`, `parse({model:123})` → default). The locked requirement says `--config` must *reject* unknown/mistyped keys. These conflict head-on. You cannot reuse `sdkConfigSchema` to validate the override and also reject typos.

Plan's resolution: a separate strict gate (`parseConfigOverride`) that JSON-parses and rejects unknown **top-level** keys against `sdkConfigSchema.shape`, then hands the raw object to `ConfigLoader` as a highest-precedence override layer where the permissive schema validates the merged whole. This catches the `--mdl`-style top-level typo the brief names. It does **not** catch nested typos or wrong value types — that needs a full parallel strict schema (drift cost), presented as Option B. Recommended Option A. The supervisor picks; the tests assume A.

## Codebase facts the next operator should not have to re-derive

- `sdkConfigSchema` resolves to **`any`** at the type level — `.meta()` widens it in this Zod 4.4.3 / TS setup. `ResolvedSdkConfig` is an index-signature type, `configLoader.config` is effectively `any` everywhere. Pre-existing, out of scope, do not try to fix. It's why the unknown-key check reads `.shape` at runtime rather than leaning on static types.
- **Only `main.ts` constructs `ConfigLoader`** in production (rest is tests). Adding an optional `overrides?` to `ConfigLoaderOptions` is backward-compatible. The legacy `claude-cli` app does not use it.
- `ConfigLoader.#readAll()` folds raw layers with `mergeRawConfigs` then validates once (`ConfigLoader.ts:155`, `:38`). The override layer goes in there as `[...raws, overrides]`, so it survives reloads with no call-site wiring.
- `mergeRawConfigs` (`config.ts:16`) deep-merges **only two levels** (top key + one sub-key level); deeper is replace-then-schema-default-refill. The `--config` layer inherits this exactly — no new merge semantics invented.
- Error pipe for a mistyped flag = `process.stderr.write(...)` + `process.exit(1)` (`main.ts:63-65`). The `--resume` validator (`main.ts:105-110`) is the closer analog: a value that parsed as a string but is semantically invalid, same pipe, no `printUsage`. `--config` is that category.
- The repo pattern for testable CLI logic: extract a pure function + spec (`decodePromptEscapes.ts` + `.spec.ts`). The plan follows it with `parseConfigOverride`. The inline `main.ts` exit path stays untested, matching how `--resume` is handled.

Apostle cast for configurable system prompts (#284). Plan written to the PM repo at `projects/claude-cli/investigation/2026-06-05_system-prompts-plan.md`. No code landed in the target repo. One decision flagged for the supervisor.

## The one decision that shapes the plan

The locked design says each source contributes its own API `system` block (`system: [ {…}, {…}, … ]`). The current SDK does **not** do that: `RequestBuilder.ts:118-120` JOINS all `options.systemPrompts` into a single wire block (pinned by `RequestBuilder.spec.ts:108-113`, which expects `['prefix','\nsecond\n\nthird']`). So "wire in where the array feeds the call today" (the brief's framing) does not produce per-source blocks. Honouring the literal decision means changing the SDK AND the cache strategy, because Anthropic caps `cache_control` at **4 breakpoints/request** and the current request already uses 4 (2 system + last tool + last user). Splitting system into N source blocks each cache-controlled blows the cap.

Plan surfaces two options with code: **A (recommended)** leave the SDK joining, `durableConfig.systemPrompts` carries one ordered element per source, zero SDK blast radius, model sees identical text; **B** change RequestBuilder to one block per source with `cache_control` only on the last system block. A delivers the configurability fully; B matches the literal notation. SC's call. Everything else in the plan is identical under either.

## Codebase facts the next operator should not re-derive

- **The system prompt is `durableConfig.systemPrompts`** (not `cachedReminders`). CLAUDE.md is `cachedReminders` and does not change. Confirmed via `main.ts` `mapConfig` + `RequestBuilder`.
- **`NodeConfigWatcher` is used only in `main.ts`** (line 7 import, line 169 instantiation). Legacy `claude-cli` does not use it. Safe to replace with a directory-based `NodeDirectoryWatcher`.
- **The two watcher defects are real** (`NodeConfigWatcher.ts:13-18`): `watch(filePath)` throws on a missing file (swallowed), and binds to the inode (vim swap leaves a stale watch). Directory watching + basename filter fixes both. Residual: a missing *directory* still can't be watched (out of scope).
- **`ConfigLoader` migration is a wiring swap, not a rewrite.** It depends on the `IConfigWatcher` abstraction, so only `main.ts` changes the concrete class. `MemoryConfigWatcher` / `MemoryConfigFileReader` fakes stay valid; `ConfigLoader.spec.ts` unchanged. That's the DI design working, not a gap.
- **`query_summary` count is `1 + durable.systemPrompts.length`** (`QueryRunner.ts:126`), array-length driven, so no change under either option.
- **`SystemPromptLoader` must NOT mirror `ClaudeMdLoader` exactly**: no `INSTRUCTION_PREFIX`, no per-section labels, and it returns an ordered `string[]` (one entry per file), because these are real system blocks composed additively, not reminder framing. That difference is why it's a separate class.
- **Once-per-session timing** is cleanly done with a session-id gate in `runTurn` (`session.id !== systemPromptSessionId`). `Ctrl+/ n` → `ConversationSession.createNew()` changes `session.id`, so the next turn re-resolves with no `AppLayout` change. The loader reads live; the gate is the policy. Mirrors the `getEffectiveModel`/`overrides` pattern.
- **Schema is permissive by design** (`.catch()` everywhere, unknown keys stripped). The new `systemPrompt` section inherits this. Adding it BREAKS the `'returns defaults for empty object'` snapshot in `cli-config.spec.ts:11-25` — that test must be updated (plan shows the diff).
- **`--system` reuses `decodePromptEscapes`** exactly like `--prompt`; no new decode logic, no new flag test (main.ts flag parsing is untested by existing convention).

## The one fact that makes this easy

The bridge already exists in pattern form. `renderModel(this.#statusState, cols, this.#session.id)` (`AppLayout.ts:395`) already passes a conversation-session-derived value (the id) into a pure renderer as a trailing arg. `renderStatus` is the very next line (`:396`). The turn count is the identical shape: derive on the conversation, expose via `ConversationSession`, pass as a third param. No new mechanism, just a second instance of an established one. The locked "derived not maintained" decision falls out for free because the value is read at render time, never stored on `StatusState`.

## What a turn is, in code

One assistant-role message. `TurnRunner` pushes exactly one assistant message per turn (only when content is non-empty). `QueryRunner.ts:122` already computes `messages.filter((m) => m.role === 'assistant').length` for its `query_summary` event, so the derivation is a concept the SDK already uses inline. Assistant messages are never merged (only consecutive *user* messages merge in `Conversation.push`), so counting them is stable. Compaction messages are assistant-role and count as a turn, which is correct (a compaction is its own API cycle). I pinned that with a test so it is a visible choice.

## The fork I did not resolve

Where the `turnCount` getter lives: on `Conversation` (SDK, Option A, recommended) or CLI-only in `ConversationSession` computing from the public `messages` array (Option B). The real tradeoff is blast radius: Option A bumps the published `claude-sdk` package under lockstep versioning and needs its own changelog entry; Option B keeps the change to `claude-sdk-cli` alone but duplicates the filter. I recommended A for encapsulation (mirrors `get messages()`), wrote the tests/stub/files against A, and gave the exact delta for B. Supervisor's call.

## The ripple the next operator must not miss

`renderStatus` gaining a required third param breaks all 19 existing `renderStatus(...)` call sites in `renderStatus.spec.ts`. This is exactly what `renderModel`'s required `conversationId` did: every `renderModel(...)` call in that same spec passes `''`. The fix mirrors it: append `, 0` to each existing `renderStatus` call. The Scaffolder lands this with the stub so the suite compiles and only the new turns tests fail.

## Codebase facts worth not re-deriving

- `Conversation` is already exported from `packages/claude-sdk/src/index.ts:88`. Adding a getter is transparent; `index.ts` needs no change.
- `renderStatus` has exactly one production caller (`AppLayout.ts:396`) plus the spec. Clean signature change.
- The empty-state guard in `renderStatus` (`:43-45`) is the visibility control. Putting `turns:` after it keeps a hidden line hidden, satisfying the locked visibility decision.
