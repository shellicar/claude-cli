# 09:04

Apostle for #239, iteration 7 — ordered tool-object list for both server and client tools.

## New locked architecture

The Router brought a fundamental change from the SC: the mark/flag machinery is gone entirely.
Both server and client tools are self-rendering objects in an ordered list, keyed by tool id.
Every state change rebuilds the block from the list. The mark protected "server tool text
ahead of the client batch" — with server tools as objects in the same list, there is nothing
to protect against and the mark has no purpose.

## How the id problem is solved

The `inputJson` event from the Anthropic SDK carries no block id. The previous approach used
a `Set<number>` gated on block index. The new approach uses a `currentToolId: string | null`
cursor that is set at `content_block_start` and cleared at `content_block_stop`. Since the
SDK guarantees no interleaving between a block's start and stop, the cursor is always the id
of the block currently producing deltas. The cursor value is included in every emitted event.
Two Sets are gone; two simple local variables replace them.

## What changes everywhere

Six SDK event types gain an `id` parameter (six public types, six private event signatures,
six `main.ts` forwards). `AppLayout` loses three members (`#streamingMark`, `markStreamingPosition`,
`replaceFromMark`) and gains one (`setActiveBlockContent`). `AgentMessageHandler` loses two
fields (`#clientBatchStarted`, `#clientBatchCleared`) and gains two (`#toolObjects`,
`#toolOrder`) plus a new `#redrawTools` helper.

## Key design observations

- `SdkToolApprovalRequest.requestId` is the `toulu_...` id, matching the `id` emitted by
  `tool_use_start`. Direct map lookup in the `tool_approval_request` arm — no queue needed.
- `server_tool_use` (completion) arm loses its defensive `transitionBlock` and `usageBeforeTools`
  snapshot. Those now live exclusively in `server_tool_use_start`.
- The `message_usage` tests fire `tool_approval_request` without prior `tool_use_start`. The
  map lookup returns `undefined`, `#redrawTools()` writes `''`, and `appendToLastSealed`
  assertions are unaffected. No changes to those tests needed.
- The `tool_approval_request` arm still has its own defensive `usageBeforeTools` snapshot.
  This exists for the test infrastructure (which tests `usageBeforeTools` via
  `tool_approval_request` alone, not `tool_use_start`). Keeping it does no harm.

## What I caught on re-read

The `tool_approval_request` tests in the plan use `requestId: 'toulu_01'` (typo: missing
`c`). The correct Anthropic format is `toolu_...` but the tests use the same string for
both `tool_use_start.id` and `tool_approval_request.requestId` — as long as they match, the
map lookup works. The Maker should use `toolu_01` for both.

## What I did not surface as a stop note

The `#toolOrder.map(id => #toolObjects.get(id)?.render() ?? '').join('')` composition: the
`?.` fallback to `''` means a missing id produces an empty string silently. This is safe
because the only place ids are added to `#toolOrder` is immediately after `#toolObjects.set`,
and both are cleared together in `message_usage`. The map will never be missing an id from
the order array in production code.
