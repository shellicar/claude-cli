Courier cast for the turns status line (PR: `feature/turns-status-line`).

## What was built

A `turns:` field on the `claude-sdk-cli` status line, showing the number of API turns in the current conversation.

**Option B was chosen (CLI-only).** The getter lives on `ConversationSession`, not on `Conversation` in `packages/claude-sdk`. Nothing in `packages/claude-sdk` changed.

Two source edits:

- `apps/claude-sdk-cli/src/model/ConversationSession.ts`: `turnCount` getter — `this.#conversation.messages.filter((m) => m.role === 'assistant').length`
- `apps/claude-sdk-cli/src/view/renderStatus.ts`: `turns` parameter added to signature; `b.text(` turns: ${turns}`)` appended after the ctx block

`AppLayout.ts` passes `this.#session.turnCount` to `renderStatus` (was wired by the Scaffolder, not the Builder).

## How it works

One assistant message = one turn. `TurnRunner` pushes exactly one assistant message per API cycle. Counting them is stable because assistant messages are never merged (only consecutive user messages merge in `Conversation.push`). `createNew()` calls `setHistory([])`, so `turnCount` reads 0 after a new conversation with no new code in the `n` path.

Compaction messages are assistant-role and count as turns. That's intentional — each compaction is its own API cycle.

The value is read at render time from the conversation, never stored on `StatusState`. The "derived, not maintained" locked decision costs nothing here.

## Render behaviour

`turns:` renders unconditionally in the non-empty path (after the ctx block). There is no `turns > 0` guard. A fresh conversation with usage-but-no-turns would show `turns: 0`. This is the minimal-change decision; the test `omits turns when no usage recorded` is satisfied by the empty-state guard at `renderStatus.ts:43-45`, not a turns-specific guard. If `turns: 0` is unwanted, `if (turns > 0)` is trivial to add.

## The empty-state guard is the visibility control

`renderStatus.ts:43-45` returns `''` when all token counts are zero. `turns:` sits after this guard, so a hidden status line stays hidden. The locked "do not change status-line visibility" decision holds without extra logic.
