## What was built

`AppLayout` (542 lines, owned everything) was dissolved into:

- **`TerminalRenderer`** — owns `Screen`, sync framing, cursor, stdout write. Presentation produces rows; renderer draws them.
- **`ViewHost`** — coordinator. Holds the presentation registry, subscribes to all seven stores, batches re-renders via `setImmediate`, dispatches keys to the active handler chain.
- **`PrimaryPresentation`** — implements `Presentation { view, activeChain() }`. Holds `PrimaryView` + the two handler chains; `activeChain()` picks by `primaryViewState.phase`.
- **`PrimaryView`** — render only: `render(model) → string[]`. No key handling.
- **Five handlers**: `QuitHandler` (ctrl+c, every chain), `ApprovalHandler` (Y/N + tool nav), `CommandKeyHandler` (ctrl+/, command keys, escape-to-close; editor chain only), `CancelHandler` (escape → cancel; streaming chain only), `EditorHandler` (text input, up/down, ctrl+enter).
- **`CommandIntentExecutor`** + `AttachmentSource`/`NodeAttachmentSource` — intent/execution split for clipboard, path resolution, media detection.
- **`Flasher`** — subscribes to `ToolApprovalState.change`, starts/stops flash timer independently.
- **`flushSealedToScroll`** — free function called at turn boundaries from `runAgent`.
- **`AppModeState`** (`'primary'` | future `'history'`) and **`PrimaryViewState`** (`phase: 'editor' | 'streaming'`) — two separate stores, two orthogonal axes.
- All five model stores gained `EventEmitter<{change:[]}>` + `on`/`off`.
- `AgentMessageHandler` and `runAgent` now call stores directly. `AppLayout.ts` deleted.

All verify gates green: 631 tests / 40 files, type-check, build (forced, no cache).

## The two-axis mode model — the load-bearing design decision

The mode design has TWO orthogonal axes that must never be flattened into one:

- **`AppModeState.active`**: which presentation is on screen (`'primary'`, future `'history'`). The app-level coordinator reads this.
- **`PrimaryViewState.phase`**: the primary view's turn phase (`'editor'` | `'streaming'`). `PrimaryPresentation` reads this to pick its handler chain. `PrimaryView` reads it for the editor region.

They're orthogonal: a turn can be streaming (`phase === 'streaming'`) while history is shown (`appMode === 'history'`). `runAgent` sets `phase` without touching `appMode`. The flat-registry form (both axes in one enum: `'editor' | 'streaming' | 'history'`) was explicitly blocked — it collapses concerns that must stay separate.

## The history view seam

`ViewHost` holds `Map<AppModeKey, Presentation>`, currently `Map([['primary', primaryPresentation]])`. To add the history view: create `HistoryPresentation` (owns its view, its handler chain, its state), add `['history', historyPresentation]`, and call `appModeState.setActive('history')` to switch. `PrimaryPresentation`, `PrimaryView`, `PrimaryViewState`, existing handlers, and `ViewHost` are untouched. This is what decisions 3 and 6 require.

## CancelHandler placement — streaming chain only, never editor chain

CancelHandler is in the streaming chain only: `[quit, approval, cancel]`. The editor chain is `[quit, approval, command, editor]` with no CancelHandler.

Why: in editor mode, escape must post nothing. If CancelHandler were in the editor chain, escape would post a cancel message that mutates `ApprovalCoordinator.#cancelled = true` even when no turn is running. That diverges from today's behaviour (which posted nothing). The fix is chain composition: don't post at all. The "inert because the consumer guards on currentAbortController" reasoning was wrong — there are two consumers of the cancel message (the abort guard AND `ApprovalCoordinator`). "Inert because a downstream guard discards it" is fragile when there are multiple subscribers.

## main.ts ordering

`CancelHandler` closes over `consumerChannel`. So the handler/presentation/ViewHost/Flasher/ReadLine construction block must sit after `consumerChannel.subscribe(...)`. `TerminalRenderer` is constructed earlier (where `AppLayout` was) because `using renderer` is needed for `cleanup()`. `renderer.enter()` and `host.renderNow()` are in the later block.

## Dropped `consumerChannel` param from `runAgent`

The plan kept it "unless the compiler flags it unused" — it is unused (cancel moved to CancelHandler). `noUnusedParameters` would error. Dropped. The main.ts call site drops the arg too. This is intentional, not an oversight.

## TypeScript closure-assignment `never` trap

`let x: T | null = null; promise.then(v => { x = v; }); x?.prop` fails with "Property does not exist on never." CFA narrows `x` to `null`, then `?.` makes the non-null branch `never`. Fix: capture the needed field inside the callback (`resolvedText = v.text`), not the whole object.

## `advanceFlushedCount` does not emit

Intentional. Scroll is a display concern; rendered content is unchanged when the scroll index advances. The non-emit is documented in the implementation.
