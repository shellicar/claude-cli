## The core insight

Phase 3 made the tool honour the *query* signal. That is the whole bug: one controller, one
`#cancelled` flag, one meaning for ESC. The fix is not new machinery bolted on, it is *splitting the
one signal into two*. Give the running tool its own `AbortController`, separate from the query's. Then
tool-cancel aborts only the tool; the query loop and the delivery turn keep the query controller, which
is never aborted on a tool-cancel. Both of the supervisor's blockers dissolve at once:
- Blocker A (loop ends): tool-cancel never sets `#cancelled`, so `while (!cancelled)` stays true.
- Blocker B (dead delivery signal): the delivery turn reuses `input.abortController.signal`, which a
  tool-cancel never touches.

## The decision that shaped everything: route in the SDK, not the TUI

The routing needs to know "is a tool running right now". I checked hard whether the TUI can know this.
It cannot, cleanly. `AgentMessageHandler` only sees `tool_approval_request` (before the handler) and
prints its own summary line immediately after sending the approval response (NOT after the handler
finishes). There is no event marking the handler's execution window. That window is known precisely in
`QueryRunner` (between calling a tool's `run` closure and it returning). So the `ApprovalCoordinator`
(shared between main.ts and QueryRunner, already receives every cancel) becomes the routing brain:
QueryRunner registers the tool's controller via `toolRunStarted`/`toolRunFinished`; `handle` reads
`#toolController != null` as "a tool is running".

Consequence: ESC keeps sending `{ type: 'cancel' }` unchanged. No new ConsumerMessage, no AppLayout or
runAgent edit, no new SDK events. `handle` returns a `CancelOutcome` (`'none' | 'tool_cancel' |
'query_cancel'`); main.ts aborts the query controller only on `query_cancel`. That is the entire wiring
change in the consumer.

This is Open Decision 1 in the plan, flagged because the Phase 3 BLOCK explicitly made "tool-cancel
meaning vs a distinct consumer message" the SC's call. I recommended SDK-routed and showed the
TUI-routed alternative is a strict superset (it needs the same coordinator mechanism PLUS new events +
TUI state). If the SC pins TUI-routed, the plan says what extra is needed.

## Double-ESC falls out for free

`#toolCancelled` (reset per run in `toolRunStarted`) records "this run already got a cancel". Second
cancel while the tool is still registered -> escalates to `query_cancel`. If the tool already died and
the delivery turn is streaming, `#toolController` is null -> plain `query_cancel`. Either path ends the
query. No ESC-counting needed anywhere.

## Tests: the gap the Phase 3 unit test couldn't reach

The Phase 3 Exec test called `Exec.handler` directly and never touched the loop. The
`QueryRunner.spec.ts` harness (`makeWiring`) runs the REAL TurnRunner + StreamProcessor + ToolRegistry
+ ApprovalCoordinator against scripted HTTP responses. That is the right surface for behaviours 1 and 3
(does the loop continue / deliver / exit). Key trick: a `makeCancellableTool` that blocks until its
signal aborts then rejects `ToolCancelledError`, exposing a `started` promise so the test fires the
cancel exactly when the handler is executing (no setImmediate guessing). The two synchronous `handle`
calls in the escalation test both land before the handler's abort listener (a later microtask), so the
second sees the tool still registered. Behaviour 2's routing is a pure ApprovalCoordinator unit test
(new spec file); its HTTP-abort-ends-loop mechanics are unchanged from today.

## No surprises

The plan was accurate. Every diff base matched the code exactly. The only non-plan moment: the `append` tool call included literal surrounding quotes from the parameter value; caught in preview, fixed with `replace_text` on the last block instead.

## The key insight (worth keeping)

The separation works because `CancelOutcome` travels from `handle` to `main.ts` as a return value. The subscriber was previously calling `handle` for its side effects and separately checking `msg.type === 'cancel'` to abort the query controller. Merging those into a single call that routes and reports is what makes the contexts distinguishable at the consumer level without any new message types.
