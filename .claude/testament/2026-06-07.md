# 08:37

Garbled tool_use rollback. Three source files changed, one test suite extended. Shipped as a single commit.

## The bug

The API can return `stop_reason: tool_use` with no `tool_use` block in the content -- the content is a text block instead. The `TurnRunner` appends this corrupt assistant turn to the conversation. When the retry loop tries to resend, the conversation ends on an assistant message, which the API treats as prompt prefill and rejects with 400.

## What was built

- `Conversation.ts` -- `removeLast()` pops the last message (no ID needed). `TurnRunner` pushes assistant turns without IDs, so `remove(id)` could not target them. That is why a new method was needed rather than using the existing `remove(id)`.
- `QueryRunner.ts` -- in the `toolUses.length === 0` retry branch, guard on `messages.at(-1)?.role === 'assistant'` then call `removeLast()` before `continue`. The role guard protects the edge case where the garble produced empty content and nothing was appended -- in that edge case the last message is a user/tool_result turn and must not be dropped. The give-up path (exhausted retries) is untouched; fail-forward is the locked decision.
- `QueryRunner.spec.ts` -- three single-assertion tests in a new `describe('QueryRunner — garbled tool_use rollback', ...)` block. The `makeGarbledToolUseStream` helper produces a text block under `stop_reason: tool_use`.

## Test strategy

The brief originally said fake `ITurnRunner` + `ConversationSession.spec.ts` style. The existing `QueryRunner.spec.ts` convention uses the real `TurnRunner` + real `StreamProcessor` with scripted SSE via `FakeMessageStreamer` (`makeWiring`). The plan followed the package convention (better: exercises the real append/rollback pair end to end) and flagged the deviation. The supervisor did not veto.

## Trap to know

The corrupt turn is always last when the rollback branch runs: nothing pushes between `TurnRunner` appending the assistant turn and the `toolUses.length === 0` branch. "Last message" == "the garbled turn" there. The role guard is not load-bearing in the common path; it exists for the edge where nothing was appended.
# 04:29

Phase 4 Apostle. Plan for the backoff clock-seam fix.

## The problem

`defaultSleep` hardcodes `setTimeout`. The two `defaultSleep` tests assert correctness by measuring wall-clock elapsed time (`Date.now()` deltas with `< 10` and `< 100` bounds). On a loaded CI runner, the already-aborted case took 11ms against the 10ms bound: `expected 11 to be less than 10`. The implementation is correct; the test instrument is wrong.

## The seam

Added `ScheduleTimer = (ms: number, onExpiry: () => void) => () => void` type. Added unexported `defaultTimer` (wraps `setTimeout`/`clearTimeout`, returning a cancel function). Added optional third parameter `schedule: ScheduleTimer = defaultTimer` to `defaultSleep`. Body replaces direct `setTimeout` call with `schedule(ms, resolve)` and the returned `cancel` function in the abort handler.

The cancel-function shape hides the `setTimeout` handle entirely from the seam. Tests never need to construct a `NodeJS.Timeout` or cast to satisfy `clearTimeout`'s type.

## The tests

`neverFires = (_ms, _onExpiry) => () => {}` — never calls the callback, no-op cancel.

With `neverFires`, the only path to `resolve()` is the abort-signal listener. Both tests assert resolution deterministically:
- Already-aborted: `defaultSleep` returns `Promise.resolve()` immediately; `schedule` is never called.
- Fires during sleep: enters Promise body, `neverFires` registers nothing, `controller.abort()` triggers the listener.

Both use `.then(() => 'resolved', () => 'rejected')` with `expected`/`actual` constants per the TDD skill.

## Callers

Only one caller outside tests: `TurnRunner.ts` line 57 (`this.#sleep = sleep ?? defaultSleep`). Field typed as `(ms, signal) => Promise<void>`. The updated `defaultSleep` with optional third param is assignable — no change to `TurnRunner.ts`. `TurnRunner.spec.ts` injects its own sleep stub at constructor level; unaffected.

## What the Maker needs to know

- Lines 113–147 of `backoff.spec.ts` are replaced wholesale (both tests + preamble comment).
- The `ScheduleTimer` type is local to `backoff.ts`, not exported — it's an implementation detail of the seam, not public API.
- The two test names change: "resolves before the delay when..." → "resolves when..." (the behaviour, not the timing).
