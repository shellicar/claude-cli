### The one decision worth knowing
The plan's verbatim test source does **not** pass the repo's Biome gate, even
though the mission requires `ci:fix` clean. Two clashes:
1. `noNonNullAssertion` — the plan used `this.#grid[row]![col]`. The app tsconfig
   has `noUncheckedIndexedAccess` off, so `string[][]` indexing already yields
   `string[]`; the `!` is both forbidden and unnecessary. Dropped it. This is the
   established convention — see `test/VirtualScreen.ts`, which indexes
   `#grid[row][col]` bare.
2. `noControlCharactersInRegex` — the plan put a `biome-ignore` on only one of
   the three control-char regexes. Added the missing two (CURSOR_RE and the
   inline `/\u001b/`), matching VirtualScreen's exact ignore comment.

Behaviour of the test is identical; only lint-conformance changed. I made this
call rather than stopping because it's convention-following inside the test, not
a reshape of the deliverable. Surfaced in the debrief.

The persistent, shared, relevance-searchable Memory tool. Built, gated green, driven live by the
SC, shipped on `feature/testament-tool`. This entry carries what the code and the mission don't.

## What was built and how it works

- **`packages/claude-core/src/memory/`** — the contract: `IMemoryStore`, the types, the pure
  `toFtsMatch` search helper, and `IMemoryEnvironmentProvider`. Timestamps are ISO strings on the
  contract; `@js-joda` stays confined to the stores, never the library runtime.
- **`apps/claude-sdk-cli/src/persistence/SqliteMemoryEngine.ts`** — the real store, a plain class
  (constructor `(db, clock)`, all the SQL, DDL in constructor). One **ordinary** FTS5 table (not
  contentless, not external-content, so `UPDATE`/`DELETE` work for the soft-delete) with
  `title/body/keywords` indexed and `id/type/keywords_json/environment/created_at/deleted_at`
  UNINDEXED. `keywords` is stored twice on purpose: indexed text (ranks) plus `keywords_json`
  (returns the array faithfully). bm25 weights `10.0, 1.0, 4.0` give title > keywords > body; column
  order fixes the weight positions. This is the class the integration tests run over `:memory:`.
- **`apps/claude-sdk-cli/src/persistence/SqliteMemoryStore.ts`** — the injectable `IMemoryStore`
  adapter. Empty constructor, lazy engine open on first use, resolves environment per write then
  delegates.
- **`GitMemoryEnvironmentProvider`** — resolves `org`/`project`/`repo` from the live git remote and
  `fleet` from config, **on every write**, so a mid-session remote/cwd change is reflected.
- **`packages/claude-sdk-tools/src/Memory/`** — the five tools (`WriteMemory`, `ReadMemory`,
  `SearchMemory`, `DeleteMemory`, `MemoryTypes`). Thin handlers that map and shape; no store logic.
- **Rendering** consumes the existing client `tool_result` SDK event (already on main from
  history-view #364) via `AgentMessageHandler`; `ToolObject.setResultLine` carries SearchMemory's
  count + top title on the Primary row.

## The load-bearing safety piece

`toFtsMatch` (pure, claude-core): match `/[\p{L}\p{N}]+/gu`, wrap each token in double quotes, join
with ` OR `. OR (not AND) because the goal is relevance and recall, not exact conjunction. No-token
input returns `null`, the store returns `[]`, never errors. Every FTS5 operator a user might type
(`hello -world`, quoted phrases, `OR`/`AND`/`NEAR`, `star*`, `(paren)`, bare `-`) is reduced to
quoted literals, so none survives to reach the engine. A bareword `OR` becomes the search term
`"OR"`, which is the point. The full adversarial input-safety set runs green against the real engine.

## The DI constraint that shapes everything

`@shellicar/core-di-lite`'s `@dependsOn` is **property injection**: the provider does `new Impl()`
then sets the dependency fields. So a constructor cannot read injected deps, the injected field must
be **public** (`public foo!: T`, the `!` is the library's requirement, not a null-safety dodge), and
registration is `.register(Id).to(Impl)` single-arg. This **forces** the two-class split: you cannot
make one class be both a no-constructor-arg DI citizen and a constructor-injected testable unit. The
engine (constructor-injected, testable) carries all behaviour; the adapter (DI-managed, empty
constructor, lazy open) is thin glue. Do **not** copy the factory-closure `new X({ a: x.resolve(…) })`
pattern threaded through the rest of `container.ts` — that pattern is what this mission steered away
from. `AppToolsService` stays factory-built and handed `memory: x.resolve(IMemoryStore)`; converting
it to a registry is the larger refactor the mission scoped OUT. Leave it.

## SC corrections, and the lessons behind them

- **`intent`, not `description`.** The SC renamed the per-call narration field from `description` to
  `intent` directly in the worktree (commit `adf0c4b`). It is the cross-tool convention (as `Exec`
  carries) where Claude narrates the call to the SC for visibility: required input, **not stored, not
  searched, not ranked**. Present on write/read/search/delete, absent on `MemoryTypes`.
- **Environment resolves per write, never at startup.** An earlier design froze the environment bag
  in `main.ts`; the git remote can change mid-session, so a memory would be stamped with a stale
  origin. The provider resolves live at write time instead.
- **The test-double lesson (the most important one on this mission).** A test double with behaviour
  means your unit test tests the double, not the unit. An earlier `FakeMemoryStore` was a working
  substring store, and the handler "unit tests" asserted its filtering — the same shape as the
  struck ranking-fake. The fix: `RecordingMemoryStore` is a **spy** that records the call argument
  and returns a canned value, nothing else. Handler tests assert only **mapping** (the captured arg
  proves `intent` is dropped and the fields forwarded) and **shaping** (canned return → output
  shape). Every search/filter/rank/count assertion lives in the integration tier against the real
  engine. There is one ranking implementation, so there is never a parity test. The form of the
  double (stub/spy/fake) is the operator's call; the mission states only the constraint.

## Traps that cost real time

- **Biome is not in the type-check or test gate.** Only `pnpm run ci` (`biome ci`) catches
  import-sort, formatting, barrel-file, and member-accessibility — it flagged 18 errors after a clean
  type-check/test. Fix the non-auto-fixable ones by hand (barrel file, `public` modifiers), then
  `pnpm ci:fix` (never `biome check --write` directly), then re-run. Run it before pushing.
- **`createMemoryTools` returns a tuple, not an array.** `return [Write, Read, …]` infers as
  `Array<unionOfAllFive>`, so tests that call a specific tool fail type-check. Cast to the explicit
  tuple of `typeof` each tool.
- **No `export *` barrel.** Biome's `noBarrelFile`/`noReExportAll` reject it at error level; the
  entry file uses explicit named re-exports, matching the existing `Exec.ts` shape.
- **FTS5 + bm25 + porter confirmed on this machine's Node** via the engine integration tests (the
  `CREATE VIRTUAL TABLE … USING fts5` in the constructor is the earliest-signal check, and it passed
  in a real run). The bundled SEA's Node is the one remaining binding risk; the construct-on-`:memory:`
  check is what re-confirms it there.
- **The dated testament files are `merge=union` and shared across concurrent missions.** A rebase can
  make your copy *look* like it replaced another mission's entries, but the union driver recombines
  both sides at merge time — leave the other entries alone. When shipping, diff each touched
  testament three-dot against `origin/main` to confirm you are only adding your own.
