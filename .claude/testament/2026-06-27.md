## The one fact that drives the whole rework

`@shellicar/core-di-lite@1.0.2` (read its source, not the existing factory usage — that usage is what
the mission deletes). `buildProvider()` does, per registration:

1. `new Impl()` with **zero args** (default factory) — the constructor runs with injected fields still `undefined`.
2. `injectDependencies(instance, …)` — sets each `@dependsOn` field **after** construction, by assignment.

**There is no post-injection init hook.** So:
- assign-only constructors are deleted; params become `@dependsOn private readonly x!: T` fields.
- constructors that did *one-time work using a dependency* (StatusState `path.basename(fs.cwd())`,
  SqliteObjectStore pragmas+prepare, ToolRegistry wire-map, DurableConfigFactory `config` build) have
  **nowhere** to do that work, because the deps aren't there at construction. It relocates to a
  **memoised getter** on the injected field. This is the spine of the mission and it is everywhere.

Decision 3's "no lazy/no init/no load" means no lazy *DI resolution* (the `x.resolve(...)` deferral
closures in `container.ts`), NOT "a class may not derive a value in a getter." The getter-relocation
is within the rules. Confirmed by re-reading decision 3 against the lib's actual capability.

Canonical patterns (from the SC's own `~/repos/@shellicar/ecosystem/examples/core-di/readme/src/`):
- `@dependsOn(IDep) private readonly dep!: IDep;` (plain field, `!`, no ctor) + `register(IDep).to(Concrete)`.
- contract = `abstract class IThing {}`; impl `class Concrete extends IThing {}`.
- factory `register(T).to(T, x => …)` only for runtime/options values.

Decorator config needs **no tsconfig change**: shared base (`@shellicar/typescript-config/base.json`)
is `target es2024`, no `experimentalDecorators` → native TC39 standard decorators, which is exactly
what `dependsOn` (typed against `ClassFieldDecoratorContext`) requires.

`core-di-lite` is **not** `core-di`: no `createScope`, no `registerModules`, no `resolveAll`, no
lifetimes. Singletons, eager, that's it. Don't reach for the full-lib idioms in the readme (scopes,
`using scope`, modules) — they aren't in lite.

## The three decisions I surfaced (in the plan's §6; none is a blocker, each has a within-rules answer)

1. **(Was wrong, corrected.)** I first claimed an "alias gap": that registering both `SdkChannel→SdkChannel`
   and `ISdkMessagePublisher→SdkChannel` yields two instances, so `QueryRunner` (sends on the publisher)
   and `runApp` (subscribes to the channel) would see different objects. **That is wrong.** lite keys its
   singleton cache by `descriptor.factory ?? descriptor.implementation` — the *implementation*, not the
   identifier. Two identifiers pointing at the same no-factory concrete resolve to ONE shared instance.
   So register both the contract and the concrete to the one class (plain `.to(Concrete)`) and they share.
   Same for IToolProvider/AppToolsService and IDurableConfigProvider/DurableConfigFactory. The only way to
   split into two instances is a `() => new X()` factory (keys the cache by the factory fn). No SC ruling
   needed. The SC caught this immediately — it is his library. Lesson: derive registration-identity claims
   from the cache-key line, not from intuition about "two registrations."
2. The getter-relocation of synchronous setup work (above). Pervasive; stated plainly so it's not a surprise.
3. `AnthropicClient.authToken` closure → injected `AnthropicAuth` + a private method (behaviour identical).
   `AnthropicAuth` itself is LEFT UNTOUCHED — its optional-options constructor already defaults `redirect` to
   `'local'`, so `.to(AnthropicAuth)` is zero-arg-compatible with nothing removed.

   I first proposed DELETING the `'manual'` redirect option "because no production code calls it." The SC was
   furious, and right: that deletes a real auth flow (the manual paste-the-code path, used when no local
   callback server can run) to tidy a class. **The DI rework changes WIRING, never CAPABILITY.** Any class
   whose only constructor parameters are optional-with-defaults is already container-compatible — leave it
   alone, do not strip it. Same trap caught and reversed on GitStateMonitor (kept its `takeSnapshot` ctor)
   and the `nodeFs` export (kept). "Nothing calls it" is never a licence to delete.

No async-construction case exists (decision 12's permitted stop did not arise): getCredentials / TsServerService.start
/ SYSTEM.md reads / configLoader.load (sync) are all **activation** runApp drives after resolution.

## The load() ordering inversion (the decision-13 "diffs reveal ordering bugs" case)

Today `configLoader.load()` runs BEFORE `buildContainer` because the eager graph reads `config` at
construction (AppToolsService, DurableConfigFactory, PermissionsNoticeGate). Once those read `config`
in getters instead (§3.1/3.2), nothing touches `config` during `buildProvider`, so `load()` moves to
just after `buildContainer`. The Maker MUST verify no construction-time `config` read survives — one
of them re-breaks the order silently. The §5.2 diff shows the `using renderer/host/_flasher/_` block
order, which must not move.

## Where the abstractions go (decision 6)

Chain is `claude-core` → `claude-sdk`/`claude-sdk-tools` → `apps/claude-sdk-cli`. So shared
abstractions live in **claude-core**: the three providers (IClockProvider/ISleepProvider/IRandomProvider),
ILogger (moved out of claude-sdk's public/types `type`; SDK re-exports from core for API stability),
IFileSystem (already there). `claude-core`, `claude-sdk`, `claude-sdk-tools` each gain a runtime dep on
`@shellicar/core-di-lite` (today only the app has it) — add via `pnpm --filter <pkg> add`, not a hand-edit.

One clock provider serves both consumers: `Instant.now(clock)` is zone-independent, so ConversationState
(UTC today) and TurnRunner (default-zone, for the human stamp) share `SystemClockProvider` returning
`Clock.systemDefaultZone()`.

## Constructor subscriptions — the second init-hook consequence (§7.6)

Beyond derived VALUES (memoised getters), four classes SUBSCRIBE to injected objects' events / set
process raw-mode IN THEIR CONSTRUCTORS: TerminalRenderer (terminalState.setSize + screen.onResize),
Flasher (state.on('change')), ViewHost (nine model.X.on('change')), ReadLine (setRawMode +
setupKeypressHandler). A side EFFECT, not a value, so the getter idiom doesn't fit. They move to an
activation start() (TerminalRenderer folds into its existing enter()) that runApp calls after the `using`
resolve, paired with the existing Dispose. This is activation lifecycle (like configLoader.start /
tsServer.start / renderer.enter), NOT the two-phase DI wiring decision 3 forbids. Order in runApp:
enter(); host.start(); flasher.start(); readline.start(); renderNow().
