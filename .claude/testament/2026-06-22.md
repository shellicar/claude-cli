# beta.10 release arc (2026-06-22)

**Status:** version bumped to `1.0.0-beta.10` and PR opened. Next: cut the release (Phase 3) so npm-publish ships the macOS SEA. The SEA portability work itself is done and merged (#363); this arc is just getting beta.10 out, because beta.9 is already published and can't be re-released.

## Phase 3's job: cutting the release (read before you cut)

The release path is the one part not yet exercised in anger ‚Äî the pipeline's first real fire is the first release.

- **One entrypoint:** `.github/workflows/npm-publish.yml`, `on: release` with no tag filter (every release fires it). A `prepare` job parses the tag into `is-cli`; jobs branch: `publish-generic` when `is-cli == false`, else `platform` then `main` for the SEA path.
- **Order is load-bearing.** `platform` (matrix; only `macos-14`/`darwin-arm64` active, Linux/Windows commented as the seam) publishes the platform package *before* `main` publishes the launcher (`needs: [prepare, platform]`), or the launcher's `optionalDependency` can't resolve.
- **Versions must equal the tag before cutting.** `publish-package` verifies each `package.json` against the tag, so both `apps/claude-sdk-cli/package.json` and `platforms/claude-sdk-cli-darwin-arm64/package.json` must read the tag's version. `workspace:*` rewrites to the exact version on publish (not a caret ‚Äî prerelease caret matching is unreliable).
- **Auth is npm OIDC trusted publishing** (`id-token: write` + `--provenance`), no token. Pre-1.0 publishes land on the `beta` dist-tag and leave `latest` unmanaged ‚Äî do **not** run a separate `dist-tag` step (it fails `E401`). `workflow_dispatch` runs the same path with `--dry-run` as a rehearsal.
- Tag form is the monorepo short name, no `v`: `claude-sdk-cli@1.0.0-beta.10`. Release targets the merge commit by SHA (`--target "$MAIN_SHA"`), pre-release (hyphen) ‚áí `--prerelease`, notes come from the `[Unreleased]` CHANGELOG section.

## The acceptance bar (the gap that bit us before)

`npm i -g @shellicar/claude-sdk-cli@beta` must install a binary that **launches the full interactive CLI** on macOS ‚Äî not just `--version`. Twice in this arc a proxy hid a real failure: `--version` passed green while the full launch crashed (`Cannot find module 'typescript'`), and a stale platform binary made a fixed source still crash because the running binary predated the fix. Verify the real boot path. `--verify` (`src/entry/verify.ts`) is built for exactly this: non-interactive, no TTY, boots config + container + store + checks `resolveTsServerPath()`. Exit 0 clean / 2 degraded / 1 hard-fail. The full interactive launch past the eager `start()` needs a TTY + auth, so `--verify` is the scriptable proxy for it ‚Äî but the *install-and-launch* on a clean machine is the only thing that confirms OIDC publish, the macOS runner, the `gitversion-osx` asset name, and Gatekeeper all actually work.

If Gatekeeper blocks the binary (ad-hoc signing is expected to suffice, since npm's tar extraction sets no quarantine xattr so the notarization gate never fires) ‚Äî that needs the SC's Apple Developer cert. Stop and raise it, don't work around it.

## Durable facts for future store / SEA work

- **The store connection is injected, not constructed** (an SC correction). `SqliteObjectStore` takes a `node:sqlite` `DatabaseSync`, configures it (WAL, `synchronous=NORMAL`, `busy_timeout=5000`, creates the `objects` table) and uses it ‚Äî no path, no `mkdirSync`, no getter exposing the raw handle. Creation lives in `setup/container.ts`. Don't widen the production API for a test (the original `db` getter was rejected at the root).
- **`node:sqlite` quirks:** `.exec`/`.prepare`/`.run`/`.get` carry over from better-sqlite3, but there's no `.pragma()` helper ‚Äî pragmas go through `.exec('PRAGMA x = y')`, and reading back is `db.prepare('PRAGMA busy_timeout').get().timeout`. Unflagged on Node 24.17 and 26 (flagged on 22).
- **ESM SEA needs Node 26+, never CJS.** Entry gated on `"mainFormat": "module"` in `sea-config.json` (the value is `"module"`, not `"esm"`). Node 24.17 silently ignores it and runs ESM as CJS, dying with `Cannot use import statement outside a module`. macOS signing order is load-bearing: strip-signature ‚Üí `postject` inject ‚Üí ad-hoc re-sign (injection invalidates the signature; wrong order, OS kills the binary). Recipe lives in `build-sea.sh`.
- **typescript can't be bundled** (dynamic resolve + spawned `tsserver.js` process reading disk). The launcher resolves typescript on the user's Node and passes the path to the SEA via `CLAUDE_SDK_CLI_TSSERVER_PATH`; inside the SEA `import.meta.url` is virtual so it can't resolve alone. Missing typescript degrades gracefully (TS tools gated out) rather than crashing.
- **build-sea.sh writes the signed binary straight into `platforms/claude-sdk-cli-<platform>/`** (platform derived from the host, never hardcoded ‚Äî a hardcoded `darwin-arm64` was rejected for breaking the per-platform matrix seam). Building is what makes the launcher runnable; there's no separate copy step.
- **The launcher forwards signals** via a standalone `runForeground` (async spawn, zero deps): relay SIGTERM/SIGHUP to the child, swallow SIGINT/SIGQUIT (the terminal already delivers those to the whole foreground group; re-forwarding double-delivers and trips the CLI's force-quit). The child stays in the launcher's foreground group (the TUI needs raw-mode I/O), so it can't be `detached`.

Version bumps don't touch `pnpm-lock.yaml` (`workspace:*` deps). Pre-release stays under `[Unreleased]`, no version heading.
