## SQLite-portability: claude-sdk-cli ships as a per-platform SEA (Courier — PR open)

The CLI now runs as a Single Executable Application: the store moved off the native
`better-sqlite3` addon to builtin `node:sqlite`, the CLI bundles its own Node runtime
(ESM), and it publishes per-platform via npm. Effect: no more `NODE_MODULE_VERSION`
crash when a repo pins a different Node — the store runs on the bundled runtime, not
the shell's. The next phase is the release Maker cutting the beta; this is what it needs.

### How a release fires

- One entrypoint: `.github/workflows/npm-publish.yml`. `on: release` has no tag filter
  (every release fires every listener), so a `prepare` job parses the tag → `is-cli`
  and jobs branch: `publish-generic` (`is-cli == false`); `platform` then `main`
  (`is-cli == true`) for the SEA path.
- **Order is load-bearing.** `platform` (matrix; only `macos-14`/`darwin-arm64` active,
  Linux/Windows commented as the seam) publishes the platform package *before* `main`
  publishes the launcher (`needs: [prepare, platform]`) — the launcher's
  `optionalDependency` won't resolve otherwise.
- **Versions must equal the tag before cutting.** `publish-package` verifies
  package.json against the tag, so `apps/claude-sdk-cli/package.json` and
  `platforms/claude-sdk-cli-darwin-arm64/package.json` both must match
  `claude-sdk-cli@<version>`. `workspace:*` rewrites to the exact version on publish
  (not a caret — prerelease caret matching is unreliable).
- Auth is npm OIDC trusted publishing (`id-token: write` + `--provenance`), no token.
  `workflow_dispatch` runs the same path with `--dry-run` — the rehearsal.
- Shared boilerplate is in two local composite actions (`install-gitversion`,
  `publish-package`); a local `uses: ./.github/actions/...` needs `actions/checkout`
  first and pnpm/Node set up before it.

### Constraints that cost time to find

- **The SEA runtime must be Node 26+, not 24.** ESM SEA entry is gated on
  `"mainFormat": "module"` in `sea-config.json` (value is `"module"`, not `"esm"`).
  Node 24.17 silently ignores it and runs the ESM entry as CJS — dies with `Cannot use
  import statement outside a module`. `build-sea.sh` reads `SEA_NODE` (a 26 binary) in
  CI, falls back to `fnm exec --using=26` locally. ESM is a hard requirement, never CJS.
- **macOS signing is mandatory.** postject injection invalidates the signature; the
  strip → inject → ad-hoc-sign bracket must run in that order or the OS kills the binary.
- **Notarization is not required** (settled with the SC): npm extracts via tar, sets no
  `com.apple.quarantine` xattr, so Gatekeeper's notarization gate never fires; ad-hoc
  signing suffices for an npm-delivered binary. Only confirmable by `npm i -g …@beta` on
  a clean machine. If Gatekeeper *does* block, notarization needs the SC's Apple
  Developer cert — stop and raise it, don't work around it.

### The shape on disk

- `platforms/claude-sdk-cli-darwin-arm64/` — thin binary carrier (`os: [darwin]`,
  `cpu: [arm64]`); the SEA binary is built in CI and dropped in before publish,
  gitignored locally. New `platforms/*` workspace glob.
- `apps/claude-sdk-cli/bin/launcher.mjs` — the package `bin`, runs on the user's Node
  (no `node:sqlite`, safe anywhere), resolves and `spawnSync`s the platform binary. Only
  the resolved SEA needs Node 26. Extensionless resolve works because the platform
  package has no `exports` map; add one and you must whitelist the binary path.
- The main package's `files` dropped `dist` (the SEA embeds the bundle). `build.ts`
  bundles deps in so `dist/main.js` is one self-contained ESM file — an ESM SEA resolves
  only builtins, everything else must be bundled.

### Store behaviour preserved, and what's unverified

`node:sqlite`'s `DatabaseSync` kept the schema, queries and pragmas: WAL,
`synchronous = NORMAL`, `busy_timeout = 5000` (machine-wide store, concurrent CLIs are
concurrent writers — wait for the lock rather than throwing `SQLITE_BUSY`). The
connection is injected into `SqliteObjectStore`, built in `setup/container.ts`; pragmas
go through `.exec('PRAGMA ...')` (`node:sqlite` has no `.pragma()`). The CI run is
correct-by-construction but unexercised — the macOS runner, the `gitversion-osx` asset
name, the OIDC publish, and the real `npm i -g …@beta` all happen at the first release.
