## SQLite-portability: the store, the SEA, and the release path

This is the consolidated legacy for the whole arc — the persistence store that crashed
on the wrong Node, made to run, then carried portably. The CLI now runs as a Single
Executable Application: the store is on builtin `node:sqlite` (no native addon), the CLI
bundles its own Node runtime (ESM), and it publishes per-platform via npm. Effect: no
more `NODE_MODULE_VERSION` crash when a repo pins a different Node — the store runs on the
bundled runtime, not the shell's. The PR is open with checks green; the next phase cuts
the beta release.

## The store (durable — future store work inherits this)

- **The connection is injected, not constructed (an SC correction).** `SqliteObjectStore`
  takes a `node:sqlite` `DatabaseSync` in its constructor, *configures* it (WAL /
  synchronous / busy_timeout pragmas, creates the `objects` table) and uses it — no path,
  no `mkdirSync`, no `new`, no getter exposing the raw handle. The better-sqlite3-era
  first cut punched a `db` getter through the two-method interface so a test could read a
  pragma back; the SC rejected widening the production API for a test and fixed it at the
  root. Creation lives in `setup/container.ts`: it builds the `~/.claude/persistence.db`
  path, `mkdirSync(dirname, { recursive: true })` (the dir must exist before the
  connection opens), news up `DatabaseSync`, and injects it. The store import is
  `import type`.
- **`node:sqlite` mapping.** `DatabaseSync`/`StatementSync`. `.exec`/`.prepare`/`.run`/
  `.get` carry over from better-sqlite3 unchanged, but there is **no `.pragma()` helper** —
  pragmas go through `.exec('PRAGMA x = y')`, and reading `busy_timeout` back is
  `db.prepare('PRAGMA busy_timeout').get().timeout` (the column is `timeout`).
- **Semantics preserved exactly:** WAL, `synchronous = NORMAL`, `busy_timeout = 5000`. The
  store is machine-wide, so two CLIs at once are two concurrent writers; the timeout makes
  the second wait for the lock instead of throwing `SQLITE_BUSY`. No fallback or caching
  layer — SQLite errors surface to Claude as ordinary tool errors (the SC's call; revisit
  only if failures prove frequent).
- **Test the real on-disk store.** `SqliteObjectStore.spec.ts` exercises the concrete
  store against a temp file: construction proves the binding/runtime resolves, set/get
  round-trips, missing id → `undefined`, `busy_timeout` reads back 5000. This reverses an
  earlier "the store is not unit-tested" decision — that omission is exactly what let a
  store that never loaded ship green, because only the in-memory `MemoryObjectStore` was
  ever exercised.

## The ESM SEA (the crux — don't re-derive)

- **The runtime must be Node 26+, not 24.** ESM SEA entry is gated on
  `"mainFormat": "module"` in `sea-config.json` — the value is `"module"`, not `"esm"`
  (`"esm"` errors). Node 24.17 *silently ignores* `mainFormat` and runs the ESM entry as
  CJS, dying with `Cannot use import statement outside a module`. **ESM is a hard
  requirement, never CJS** — the project is heading ESM-only and MCP is ESM-only; the
  earlier reach for CJS to dodge top-level `await` was a dead end the SC closed.
- **macOS signing is load-bearing, in order:** strip-signature → `postject` inject →
  ad-hoc re-sign. postject injection invalidates the signature; out of order the OS kills
  the binary on launch. `build-sea.sh` carries the recipe; it reads `SEA_NODE` (a Node 26
  binary) in CI, falls back to `fnm exec --using=26` locally.
- **Notarization is not required** (settled with the SC): npm extracts tarballs via tar,
  which sets no `com.apple.quarantine` xattr, so Gatekeeper's notarization gate never
  fires — ad-hoc signing suffices for an npm-delivered binary (same reason esbuild ships
  unsigned binaries). Only truly confirmable by `npm i -g …@beta` on a clean machine. If
  Gatekeeper *does* block, notarization needs the SC's Apple Developer cert — stop and
  raise it, don't work around it.
- `dist/main.js` is bundled self-contained (`build.ts`: deps bundled in, `splitting:
  false`) — an ESM SEA resolves only builtins, so everything else must be in the bundle.
  `node:sqlite` is unflagged on 24.17 and 26 (flagged on 22), so the store path stays clean.

## The release path (Phase 4 needs this)

- **One entrypoint:** `.github/workflows/npm-publish.yml`. `on: release` has no tag filter
  (every release fires every listener), so a `prepare` job parses the tag → `is-cli` and
  jobs branch: `publish-generic` (`is-cli == false`); `platform` then `main`
  (`is-cli == true`) for the SEA path.
- **Order is load-bearing.** `platform` (matrix; only `macos-14`/`darwin-arm64` active,
  Linux/Windows commented as the seam) publishes the platform package *before* `main`
  publishes the launcher (`needs: [prepare, platform]`) — the launcher's
  `optionalDependency` won't resolve otherwise.
- **Versions must equal the tag before cutting.** `publish-package` verifies package.json
  against the tag, so `apps/claude-sdk-cli/package.json` and
  `platforms/claude-sdk-cli-darwin-arm64/package.json` both must match
  `claude-sdk-cli@<version>`. `workspace:*` rewrites to the exact version on publish (not a
  caret — prerelease caret matching is unreliable).
- Auth is npm OIDC trusted publishing (`id-token: write` + `--provenance`), no token.
  `workflow_dispatch` runs the same path with `--dry-run` — the rehearsal. Shared
  boilerplate is in two local composite actions (`install-gitversion`, `publish-package`);
  a local `uses: ./.github/actions/...` needs `actions/checkout` first and pnpm/Node set up
  before it.
- **Shape:** `platforms/claude-sdk-cli-darwin-arm64/` is a thin binary carrier (`os:
  [darwin]`, `cpu: [arm64]`); the SEA binary is built in CI and dropped in before publish,
  gitignored locally. `apps/claude-sdk-cli/bin/launcher.mjs` is the package `bin` — it runs
  on the user's Node (no `node:sqlite`, safe anywhere), resolves and `spawnSync`s the
  platform binary. Extensionless resolve works because the platform package has no
  `exports` map; add one and you must whitelist the binary path.

# 04:03 — Phase 4: typescript-on-disk, graceful degradation, --verify

The blocker (top of this file + the `-blocker-typescript-under-sea` file) is fixed. Three deliverables, all proven on the actual SEA binary.

## What the fix actually is

- **`resolveTsServerPath()` (TsServerService.ts) now returns `string | null`** and is the single source of truth for TS availability. It reads `CLAUDE_SDK_CLI_TSSERVER_PATH` first (existsSync-checked), then falls back to `require.resolve('typescript')` relative to the module. Exported (with `TSSERVER_PATH_ENV`) from the `TsService` barrel.
- **The launcher (`bin/launcher.mjs`) resolves typescript on the user's Node and passes the tsserver.js path to the spawned SEA via that env var.** typescript is already a prod dep of the launcher package (#266), so npm puts it on disk; the launcher just hands the binary the path.
- **Graceful degradation:** `start()` returns early (sets `#available=false`) instead of throwing when the path is null. The TS tools are gated out at *container-build* time — `container.ts` computes `tsServerPath` once and passes `tsServerPath != null` to `AppToolsService` → `createAppTools(..., tsAvailable)`. So when typescript is absent, the four TS tools are never registered, and `main.ts`'s eager `start()` is also wrapped in try/catch.
- **`--verify`** (`src/entry/verify.ts`): non-interactive, bypasses the `!isTTY` gate (guarded with `!values.verify`), handled early in `main()` before auth. Boots config+container+store, then checks `resolveTsServerPath()`. Exit codes: **0 clean / 2 degraded / 1 hard-failure** (`VERIFY_EXIT`).

## The thing the blocker testament got subtly wrong (worth knowing)

The blocker said the SEA's `import.meta.url` is "virtual" so `require.resolve('typescript')` *cannot* work inside the binary. On Node 26's SEA that's not quite true: `require.resolve` walks up from the **binary's real on-disk location**. When the binary sits inside the monorepo (`dist/claude-sdk-cli`), it finds the hoisted typescript with **no env var at all** — even when run from `/tmp`. To see genuine degradation you must place the binary where no `node_modules` sits above it (I copied it to an isolated `/tmp` dir). So the env var is the *reliable* contract (an installed platform binary lives in `node_modules/@shellicar/...` with typescript hoisted near the launcher, which passes the var), but the cwd-independent fallback is a real second path that happens to cover the in-repo case.

## Proof (actual binary, built via build-sea.sh, Node 26.3.1)

- Isolated dir, no typescript: `--verify` → `DEGRADED`, `store: ok`, exit **2**. This is the exact scenario that previously crashed with `Cannot find module 'typescript'`; it now boots.
- Env var supplied (as the launcher does): `--verify` → `OK`, `store: ok`, typescript resolved, exit **0**.
- The `node:sqlite` store boots inside the SEA in both cases (`store: ok`).

## Gates

type-check green, build green, biome (`pnpm run ci`) green after `pnpm ci:fix`. Tests: **one pre-existing failure unrelated to this work** — `Exec.spec.ts > runs steps concurrently` asserts two overlapping 200ms sleeps finish in `<350ms`; measured 416–511ms under load at 4am. It's a timing-fragile test in code I never touched; failed identically before and after. Flagged, not chased.

## Couldn't fully prove

The *full interactive* launch (past the eager `start()` in `main()`) needs a TTY **and** auth, so it can't be driven non-interactively. `--verify` is the mission's designated proxy for exactly this and it's green. The `!isTTY` gate is at module top-level *before* `main()`, so a bare non-TTY run exits at the TTY check before reaching the TS init anyway — another reason `--verify` (which bypasses that gate) is the right vehicle.

# 04:46 Phase 4 cont.: launcher signal forwarding + build-into-platform-package

Two things landed after the typescript fix, both about making the *installed* binary actually work, not just the source resolving.

## The stale-binary trap (and the build fix)

The launcher resolves and runs `platforms/claude-sdk-cli-<platform>-<arch>/claude-sdk-cli` (the optionalDependency), not `apps/claude-sdk-cli/dist/claude-sdk-cli`. build-sea.sh used to build into `dist/`, and a separate CI step (`cp dist -> platforms/${matrix.platform}`) staged it. Locally there was no such step, so `./bin/launcher.mjs` ran whatever stale binary sat in the platform package. That is how a "fixed" source still crashed on launch: the running binary predated the fix. I missed it because I tested the fresh `dist` binary directly, never through the launcher chain; the SC caught it on a real launch.

Fix: build-sea.sh now writes the signed binary straight into `platforms/claude-sdk-cli-${PLATFORM}/`, where PLATFORM is derived from the host (`process.platform + "-" + process.arch`), never hardcoded. (I first hardcoded `darwin-arm64`; the SC rejected it hard, because it breaks the per-platform matrix seam where adding a row is meant to be uncomment-only. Deriving from the host equals `matrix.platform` on every runner and keeps the seam.) Consequences:
- CI's separate `cp` staging step is gone (coupled: dist/claude-sdk-cli no longer exists).
- build.ts dropped the `@shellicar/build-clean` plugin (`destructive: true`); it was deleting dist artifacts and fought the new flow.
- Building is now what makes the launcher runnable. No stale-copy gap.

## Launcher signal forwarding (runForeground)

The launcher was `spawnSync(binary, args, { stdio: 'inherit' })`. Gap: a signal sent directly to the launcher PID (an orchestrator's abort path) does not reach the SEA child, and spawnSync cannot forward it, because it blocks the event loop so the launcher's own signal handlers never run until the child has already exited. So a killed wrapper orphans the SEA. (Same wrapper->grandchild shape as an exec hang the SC had hit.)

Replaced with a standalone `runForeground` (async spawn, zero deps; the SC explicitly did not want foreground-child/signal-exit, partly over recent isaacs bracket-expansion CVEs). Policy:
- RELAY SIGTERM/SIGHUP to the child. These reach the launcher only via a direct PID kill (the terminal never sends SIGTERM to the foreground group), so relaying prevents the orphan with no double-delivery.
- SWALLOW SIGINT/SIGQUIT on the launcher (noop handler). The terminal delivers these to the whole foreground group, so the child already gets them directly. Re-forwarding would double-deliver, and a doubled SIGINT trips the CLI's "second press = force quit", hard-killing mid-cleanup. Swallowing also stops the launcher dying before the child finishes cleanup. This reproduces spawnSync's old parent-ignores-SIGINT behaviour on purpose.
- On child exit: remove the handlers, then mirror. Re-raise the terminating signal (default action restored once the listener is gone) or `process.exit(code)`.

The child must stay in the launcher's foreground group (the TUI needs raw-mode terminal I/O), so we cannot isolate it with `detached`. That is why the policy is per-signal rather than "forward everything."

Proven with an A/B negative control (old spawnSync vs new runForeground, SIGTERM to the wrapper PID only): old gives ORPHAN (child survives), new gives OK (child reaped, wrapper mirrors the signal).

## Meta-lesson for the next operator

Twice this phase I verified a proxy instead of the real thing: the typescript fix against `dist` rather than the launcher chain (missed the stale binary), and no negative control for the signal bug until the SC asked. Prove against the real path (the launcher, the installed shape), and show the bug exists before claiming the fix. Verifying a proxy is the exact gap that produced the original blocker (`--version` green while the full launch crashed).
