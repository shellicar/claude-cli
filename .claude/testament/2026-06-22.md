## Phase 2 Maker — ship the SEA CLI to npm (per-platform packages)

Phase 1 already proved the SEA and landed `build-sea.sh`, `sea-config.json`, and the `build.ts` deps-bundled edits (commit `6035bad`, "Finish SEA POC"). Phase 2 turned that into a publishable shape and a release pipeline.

### The packaging decision (settled with the SC)

The SC's "build it as a matrix, uncomment a row to add a platform" framing points at the **esbuild/turbo per-platform-package pattern**, not a single fat package. I surfaced #1 (single package, bin→binary) vs #2 (per-platform packages via optionalDependencies) and the SC picked **#2**. Also settled: **notarization not required** — npm extracts tarballs via tar, so no `com.apple.quarantine` xattr is set and Gatekeeper's notarization gate doesn't fire; ad-hoc signing is enough for an npm-delivered binary (same reason esbuild ships unsigned binaries fine). Only truly confirmable by `npm i -g …@beta` on a clean machine, post-merge.

### What the shape is

- `platforms/claude-sdk-cli-darwin-arm64/` — new published package, a thin binary carrier. `os: [darwin]`, `cpu: [arm64]`, `files: [claude-sdk-cli, README.md]`. The binary is built in CI and dropped in before publish; it's gitignored locally. Added `platforms/*` to the pnpm workspace glob.
- `apps/claude-sdk-cli/bin/launcher.mjs` — new. The main package's `bin` now points here instead of `dist/main.js`. It `require.resolve`s `@shellicar/claude-sdk-cli-${platform}-${arch}/claude-sdk-cli` and `spawnSync`s it with `stdio: inherit`. Runs on the user's Node (touches no node:sqlite), so safe on any Node; only the resolved SEA binary needs the bundled 26.
- `apps/claude-sdk-cli/package.json` — `bin`→launcher, `files`: `dist`→`bin` (the SEA embeds dist/main.js, so the npm package ships only the shim), `optionalDependencies: { "@shellicar/claude-sdk-cli-darwin-arm64": "workspace:*" }`. `workspace:*` rewrites to the **exact** version on publish (exact pin, esbuild-style) — `^` prerelease caret matching is unreliable, don't use it.

### Things not to re-derive

- **`require.resolve` of an extensionless subpath works** without an `exports` map in the platform package. With no `exports`, Node allows full directory access; `require.resolve('pkg/claude-sdk-cli')` hits the exact file. If you ever add `exports` to the platform package you must whitelist the binary path.
- **`build-sea.sh` is now CI-portable.** It reads `SEA_NODE` (path to a Node 26 binary) if set, else falls back to `fnm exec --using=26` locally. Added a `MAJOR < 26` guard. CI sets `SEA_NODE="$(node -e 'process.stdout.write(process.execPath)')"` after `setup-node@v4` with `node-version: 26`.
- **Two workflows, not one.** `release-cli.yml` (new) owns `claude-sdk-cli@*` tags: a `prepare` job gates on the tag being the CLI, a `platform` matrix job (macOS row active; linux/windows rows commented as the seam) builds+publishes each platform package, then a `main` job (`needs: [prepare, platform]`) publishes the launcher package. Order matters: platform packages must hit the registry before the main package, or its optionalDependency resolves to nothing. `npm-publish.yml` (generic, all other packages) got a `skip-publish` guard so it doesn't race to publish the CLI tag — it still runs the build/test as a redundant check, just skips the Publish step.
- **Auth is npm OIDC trusted publishing**, not a token. The existing publish step has no `NODE_AUTH_TOKEN` — just `id-token: write` + `--provenance`. I matched that; don't add a token env.
- **GitVersion on macOS**: the version plugin runs `gitversion` at build time, so the macOS runner installs `gitversion-osx-arm64-6.6.0.tar.gz` (matrix carries the asset suffix per row). Unverified against the real runner — first CI run will confirm the asset name.

### Proven locally (the heart of it)

Built the SEA via `build-sea.sh` (v26.3.1), copied the binary into the platform package, ran `node apps/claude-sdk-cli/bin/launcher.mjs --version` → `1.0.0-sea-poc.1`, exit 0. The launcher → optional-dependency-binary → SEA-runtime chain works end to end through the pnpm workspace symlink.

### Verify gates

type-check, build, test all green. `pnpm run ci` (biome) first failed on **pre-existing** import-sort errors in `SqliteObjectStore.ts` and `container.ts` (from the Phase-1 binding swap, committed) — fixed with the sanctioned `pnpm ci:fix` (the only files it touched). `pnpm --filter scripts run validate` green.

### What's unverifiable here

The actual GitHub Actions run: macOS arm64 runner, gitversion-osx asset name, OIDC publish, and the real `npm i -g …@beta` acceptance. All post-merge. The pipeline is correct-by-construction; the first release exercises it.

### Decisions I made (not spelled out in the brief)

- Put platform packages under a new `platforms/*` workspace glob (not `apps/*`) — semantically they're not apps, and it makes "add a platform" a self-contained dir.
- Dropped `dist` from the main package's `files` — the SEA embeds the bundle, so the npm package is just the launcher shim. Reduces the published main package to a few KB.
- `workspace:*` (exact pin) for the optionalDependency rather than a caret.
- Made `npm-publish.yml` still build/test the CLI tag (redundant green check) rather than bailing the whole job — only the publish is skipped.

## Consolidated the two release workflows into one branching entrypoint

The SC asked the right question about my two-workflow design: `on: release` has **no tag filter**, so every workflow listening on `release: created` fires on every release — there's no way to target one. My first cut had `release-cli.yml` + a `skip-publish` guard on `npm-publish.yml`, which meant the generic workflow woke and ran a full build/test on every CLI release just to skip the publish. Wasteful.

Folded both into a single `npm-publish.yml`: a `prepare` job parses the tag once and outputs `is-cli`; jobs branch on it — `publish-generic` (`if: is-cli == 'false'`) for every other package, `platform` + `main` (`if: is-cli == 'true'`) for the SEA path. Deleted `release-cli.yml`, dropped the guard. Exactly one branch runs per release, the fork is visible in one file.

Validated the YAML parses (`ruby -ryaml`). The gates (type-check/build/test/ci/validate) don't lint workflow YAML, so they were unaffected — still green from the prior run.
