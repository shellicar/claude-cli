## Resolver placement

The single point I had to decide: where does `getEffectiveModel` live in `main.ts`?

Constraints:
- After `configLoader.load()` so `configLoader.config` exists at first call.
- Before `configLoader.onChange((config) => { ... })` registration so the callback can
  reference the resolver in its body. (Closures capture by reference, so technically
  the resolver could be defined later — but defining-before-use makes the file read
  top-to-bottom without surprise.)

Insertion point: between line 130 (`configLoader.load();`) and line 131
(`configLoader.onChange((config) => {`). Same block carries `overrides` declaration so
the slot and its reader sit together.

This also collapses the prior Maker note about `overrides.model ?? config.model` vs
`overrides.model ?? configLoader.config.model` — both forms become `getEffectiveModel()`.
The note is gone from the recast plan.

## Test convention discovered

The existing `renderStatus.spec.ts` mixes two styles:
- Inline `expect(actual).toContain('my-project')` in the `renderModel — empty state`
  describe block.
- Named-variable `expected = true; actual = ...includes('Sonnet'); expect(actual).toBe(expected)`
  in the `renderModel — model abbreviation` block.

The `tdd` skill calls for the named-variable form. The `model abbreviation` block
already uses it, so the new `session name override` block follows that precedent. This
keeps the new code consistent with the dominant convention in the same file rather than
introducing a third style.
