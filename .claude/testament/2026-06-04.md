## The design call that matters

The clock stamp goes through `clockStamp?: string` on `RequestBuilderOptions`, NOT through `TurnInput.systemReminder`. Two reasons:
1. `TurnInput.systemReminder` is the one-shot git-delta field — cleared by `QueryRunner` after turn 1. The clock must fire every turn.
2. The mission explicitly says not to couple them.

`TurnRunner.run()` calls `formatClockStamp()` and assigns `clockStamp: formatClockStamp()` in `builderOptions`. This runs fresh every call to `run()`.

## The injection point

The `clockStamp` block is appended immediately after the `systemReminder` block in `buildRequestParams`, before the `lastTool` cache_control assignment. Both land in the same uncached slot (after `cacheLastUserMessage`). They are separate `<system-reminder>` blocks in the user message content array.

By the time `clockStamp` injection runs, `cacheLastUserMessage` has already converted string content to arrays. The defensive `typeof lastUser.content === 'string'` branch is dead code at that point but mirrors the `systemReminder` pattern for consistency.

## Test strategy

No TurnRunner-level integration test for the clock. The reason: `TurnRunner.run()` calls `formatClockStamp()` without a clock parameter (system default). A TurnRunner test would assert against wall time unless the clock is injected into `TurnRunner`. The mission design ("keep it minimal") ruled out injecting a clock into `TurnRunner`.

The two test locations instead:
1. `clockStamp.spec.ts` — tests `formatClockStamp(fixedClock)` against a fixed instant and zone. Deterministic.
2. `RequestBuilder.spec.ts` — tests that passing `clockStamp` (a pre-computed string) through `buildRequestParams` produces the right block, in the right position, uncached, even when `systemReminder` is absent. This covers the "turn 2+" requirement without needing wall time.

## js-joda dependency state

None of `@js-joda/core`, `@js-joda/locale_en`, `@js-joda/timezone` are in `packages/claude-sdk/package.json` today. All three are needed at runtime:
- `@js-joda/timezone`: side-effect import for `VV` (named zone) resolution
- `@js-joda/locale_en`: for `EEEE` (Friday) and `MMMM` (May) patterns
- `@js-joda/core`: the core types

The `pnpm add` command installs them as production dependencies.

Phase 2, iteration 2. Injection direction changed.

## What changed

The separate `clockStamp?: string` field approach (iteration 1) was replaced with generalising `systemReminder?: string` → `systemReminders?: string[]`. The loop in `buildRequestParams` emits one `<system-reminder>` block per entry. TurnRunner assembles the list.

## The wiring decision I made

The `string[]` does not propagate to `TurnInput` or `PerQueryInput`. Reasons:

1. `PerQueryInput.systemReminder` is the CLI's contract — changing it means touching the CLI (out of scope).
2. `TurnInput` doesn't need to carry the clock; the clock is SDK-internal, computed at the TurnRunner layer.
3. Keeping the `string[]` on `RequestBuilderOptions` only means `SdkQuerySummary.systemReminder` and the QueryRunner channel send are untouched.

Assembly in `TurnRunner.run()`:
- `if (turnInput.systemReminder != null) systemReminders.push(turnInput.systemReminder)`
- `systemReminders.push(formatClockStamp())`
- Git delta first (turn 1 only), clock last (every turn).

## Test coverage change

The old `describe('buildRequestParams — systemReminder', ...)` block had two tests using the now-removed `systemReminder` field. TypeScript would error on them if left. The plan replaces the whole block with a `systemReminders` block covering: single entry, multi-entry ordering, turn-2+ (clock only), uncached, empty/undefined.

## TurnRunner tests — still fine

The existing TurnRunner test passes `systemReminder: 'stay focused'` to `TurnInput` — that field is unchanged. The test uses `.find()` to check for any `<system-reminder>` block. After the change there will be two blocks (git delta + clock), but `.find()` still returns the first. No changes needed to TurnRunner tests.

# 23:30

Phase 2, iteration 3. Dead branch removed — hard block.

## What changed

The `typeof lastUser.content === 'string'` branch in the injection loop was dead code. `cacheLastUserMessage` arrayifies the content before injection runs, so the string case is unreachable. Iteration 2 carried it anyway ("defensive consistency") — that framing was wrong. Internal SDK code reflects actual invariants; speculation dressed as safety is debt.

## The fix

The loop body in section 3b now pushes directly:

```typescript
const content = lastUser.content as BetaTextBlockParam[];
for (const reminder of options.systemReminders) {
  content.push({ type: 'text', text: `<system-reminder>\n${reminder}\n</system-reminder>` });
}
```

`BetaTextBlockParam` is already imported in `RequestBuilder.ts` — no new import needed. `as BetaTextBlockParam[]` is a TypeScript invariant assertion (the type checker cannot see through `cacheLastUserMessage`'s mutation), not a runtime defence.

## Why `BetaTextBlockParam[]` not `BetaContentBlockParam[]`

`BetaTextBlockParam` is already in the existing import statement in `RequestBuilder.ts`. `BetaContentBlockParam` is not. Using `BetaTextBlockParam[]` requires no import change. The slight imprecision (the array might contain other block types) is irrelevant for a `.push()` call.

## Cast type: BetaContentBlockParam not BetaTextBlockParam

A user message's content array is `BetaContentBlockParam[]` — it holds mixed block types on tool-loop turns (text, tool_use, etc.), not only text. `BetaTextBlockParam[]` was technically wrong for the cast. `BetaContentBlockParam` needed to be added to the import in `RequestBuilder.ts` line 3 (it is already imported in `TurnRunner.ts` line 2 but was absent from `RequestBuilder.ts`).

## Clock injection at TurnRunner

The constructor now takes an optional `clock: Clock = Clock.systemDefaultZone()` as its 4th parameter (after `logger?`). Existing call sites compile unchanged. `#clock` field added to the class. `run()` calls `formatClockStamp(this.#clock)`.

The `clock` position after `logger?` was deliberate: it lets tests pass `new TurnRunner(streamer, processor, undefined, fixedClock)` without touching main.ts (which passes `(streamer, processor, logger)` — the 4th arg defaults to system clock).
