# abort fix on the exec-core consolidation

The PR-arc testament is `2026-06-25.md` (exec-core, the two findings, the flush race,
the tsconfig consolidation). This entry covers the one thing that landed after it:
the already-aborted abort guard, flagged by PR #371's review bot and shipped via
PR #371 (not a new PR).

## The bug

`Executor.run` wired cancellation only via `opts.signal?.addEventListener('abort', ‚Ä¶)`.
That listener never fires for a signal that is **already aborted** when `run` is called,
and there was no `aborted` check before `spawn`. So once one command is cancelled, a
chained command that inherits the already-aborted signal still spawns and runs ‚Äî
defeating ESC-cancel. Reachable through `;` (right side always runs), `||` (a killed
left exits non-zero, so the right qualifies), and V1's sequential loop. `&&` /
`bail_on_error` are incidentally spared because a killed left short-circuits them.

Why the 2026-06-25 review cleared cancellation as "not a regression": it checked the
handler-level `signal?.aborted` after `execute`/`executeTree`, but missed that
`Executor.run` itself spawns unconditionally. Different layers ‚Äî the handler guard and
the spawn guard are not the same check. Likely a regression: `main` passed the signal
straight to `spawn` (which refuses an already-aborted signal); the consolidation moved
to a manual listener + group-kill and dropped that guard.

## The fix

One guard at the top of `run()`, before `spawn`: if `opts.signal?.aborted`, close the
sinks (matching the existing bad-cwd early-return path, so the flush contract from the
06-25 race fix still holds) and return `{ exitCode: null, signal: 'SIGTERM' }` ‚Äî the
same killed shape the group-kill path produces. One place covers V1 steps, V2 trees,
and pipelines because every caller spawns through `run`. Nothing else changed.

Proof: `Executor.run already-aborted signal > does not spawn when the signal is already
aborted` in `packages/exec-core/test/Executor.spec.ts` ‚Äî abort before `run`, run
`echo hi` with the aborted signal, assert collected stdout is empty.

## Known CI flake

A load-sensitive concurrency timing assertion in `claude-sdk-tools` (~350ms threshold)
can red spuriously under load and passes in isolation. Unrelated to this work ‚Äî if CI
reds only on that, it is the flake, not a regression.
