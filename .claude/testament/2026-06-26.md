# Ghost-text: the wrap-boundary render defect, fixed by a cell-grid renderer

Consolidated arc for the `fix/ghost-text` PR (Investigator → Maker → Courier). This is the
ghost-text rendering work; the SQLite/SEA legacy in `2026-06-22.md` is a different arc and stays
there.

## The bug

Content near the right margin was duplicated: a stale fragment from the previous frame survived at
the wrap boundary. First seen as input echoed into the response region one keypress behind, later on
streaming response/tool output too, so it is **not** input-specific.

`TerminalRenderer.paint` wrote each row as `\r clearLine row \n` and trusted the bare `\n` to advance
exactly one physical line. A response line that fills the terminal width (a long unbroken token that
`wrapLine` emits as an exactly-`cols`-wide line), written into an autowrap-on terminal, can consume
an extra physical line; the `\n` then over-advances and strands a line still holding the prior
frame's content. The bet that `\n` advances one physical line is terminal-dependent, which is why the
ghost was intermittent and why "less often now" tracks `string-width` improving.

**Trap for anyone reading the old investigation notes:** the trigger is a wrapped **response** line,
not the input/editor line. The editor line is short and never wraps (`wrapLine` early-returns short
lines, `reflow.ts`). An early investigation iteration pinned it on the editor line and was wrong.

## What shipped

A cell-grid renderer where the ghost cannot occur by construction:

- **`apps/claude-sdk-cli/src/view/ScreenBuffer.ts`** — the screen as a cell grid. `layoutRow`
  (styled row → cells, ANSI carried as a zero-width prefix onto the next glyph, wide graphemes get a
  continuation cell, clip at margin), `buildGrid`, `diffToWrites`, `gridToLines`. Pure functions.
- **`src/view/TerminalRenderer.ts`** — rewired onto the grid: holds the previous grid, builds the
  next, writes only changed rows at absolute `cursorAt` positions with **autowrap off**. Resets the
  buffer to null on resize, alt-buffer enter, and inline write-through. No reliance on cursor advance
  or margin wrap, so the ghost class is gone.
- **`packages/claude-core/src/ansi.ts`** — `disableAutowrap`/`enableAutowrap` (`?7l`/`?7h`).
- **Test infra:** `test/VirtualScreen.ts` — a terminal-grid emulator implementing the `Screen`
  interface, so a real `TerminalRenderer` runs against it unmocked. Models only the ANSI `paint`
  emits plus last-column autowrap, with the `?7` toggle honoured. Built because `paint` had **no**
  test and there was no virtual screen in the codebase (the brief's premise that one existed was
  false; the Maker surfaced it and the SC approved building it as scope).
- **Tests:** `ScreenBuffer.spec.ts`, `VirtualScreen.spec.ts`, `TerminalRenderer.spec.ts`. 24 green,
  type-check clean.

Two commits on the branch: `0f55167` (Tier 1b — absolute-addressing the `\n`-based paint) then
`c39167e` (Tier 3 — the cell grid, superseding Tier 1b's paint). Same branch, squash-merges.

## Two non-obvious truths (do not relearn the hard way)

1. **`diffToWrites` MUST be row-granular, not cell-granular.** ANSI colour is cumulative across a
   row — a cell's colour is set by an escape in an earlier cell. A cell-granular diff that writes
   only the changed cell drops that SGR state, so a partial update renders in the wrong colour (this
   was a real command-mode styling glitch in the first Tier-3 cut). The fix: rewrite the whole row
   from column 1 whenever it changes. A test proves it — revert to cell-granular and it goes red
   (`\x1b[1;2HC\x1b[0m`, no colour prefix) vs green (`\x1b[1;1H\x1b[31mAC\x1b[0m`). Do not "optimise"
   back to cell-granular without first solving SGR-state carry.

2. **Copy-as-newlines is architectural, not a bug.** Because the app wraps lines itself and writes
   each wrapped row as its own grid line, the terminal/tmux see separate lines and copy inserts a
   newline at every wrap point. Preserving a soft-wrap on copy means letting the *terminal* wrap
   (autowrap on, full logical lines written) — which reintroduces the ghost class. It is the cost of
   owning the layout; the old `\n`-based paint had it too.

# exec-core ship: the traps and the changelog tooling

The exec-core arc lives in `2026-06-25.md` (the leaf, the two findings, the flush
race, the tsconfig consolidation). Read it first. This entry is the durable guidance
that came out of shipping it.

## The temporal-fix rule (most likely to bite you)

The whole exec-core PR is unreleased and squash-merges to main as one change. So the
baseline for any changelog entry or PR bullet is **released main**, never the branch's
own history. A bug the consolidation introduced and then repaired on the same branch
is net-zero against main: no consumer ever saw it, so it is not a "fix" to document.

Three of these were deliberately kept OUT of both the changelog and the PR body:

- the already-aborted abort guard (released main passed the signal to `spawn`, which
  refuses an already-aborted one, so main already stopped chained commands after a
  cancel; the consolidation dropped that and the guard restored it),
- bad cwd staying exit 126 in a pipeline (was 127 on the branch only),
- the sink flush race (introduced by exec-core's own sink handling).

The reflex to write "now fixes X" is strong and wrong here. Ask: did released main have
X? If not, it is temporal. Leave it out. The SC caught two of these; do not make him
catch a third.

## Changelog tooling (new, and now enforced)

`changes.jsonl` and `CHANGELOG.md` must never drift, and two things now hold that:

- `pnpm --filter scripts run changelog` regenerates every package's CHANGELOG from its
  changes.jsonl in one shot. It wraps the unchanged `generate-changelog.ts`.
- CI (`node.js.yml`) regenerates and runs `git diff --exit-code` on the changelogs, so
  any drift reds the build.

Workflow: edit a changes.jsonl, run the script, commit both together. A hand
regeneration earlier missed `mcp-exec` entirely; the all-script exists so you never
have to remember the full package list.

## Vocabulary

"operator" is fleet terminology. It has no home in anything a consumer reads (PR bodies,
changelogs, docs). The exec-core code comments say "operator's tty" internally; do not
carry that outward. The terminal belongs to whoever runs the CLI.

## Shared worktree

This worktree is shared with co-working sessions. Mid-cast, another session rebased the
branch onto a freshly-advanced main and staged/unstaged files under me. Before any
commit or push, run `git status` and resolve refs explicitly (`git rev-parse`, reflog)
rather than trusting the last SHA you saw. The reflog is the authoritative record of who
moved HEAD.
# messages-api-layer: owned HTTP/SSE transport (mission legacy)

Consolidated legacy for the whole mission, Investigator through Courier. The PR is `feature/messages-api-layer`. It replaced the `@anthropic-ai/sdk` *runtime* in `@shellicar/claude-sdk` with our own transport, keeping the SDK as a types-only dependency. The reason we own it: the SDK honoured a `retry-after` of 24h+ with an uncapped, unabortable wait that froze the CLI. Owning the request, the SSE stream, and the error/retry path is the only way to take that control.

## Design decisions that aren't obvious from the code

- **Boundary: raw events assembled in StreamProcessor, not a reproduced SDK emitter.** The first plan reproduced the SDK's `MessageStream` event emitter; the SC redirected to the smaller change. What shipped: `IMessageStream = AsyncIterable<BetaRawMessageStreamEvent>`, and `StreamProcessor.process` is one `for await` loop that assembles via `MessageAccumulator`. Everything downstream of StreamProcessor was already insulated (it builds only our own `ContentBlock` plus primitives), so only StreamProcessor changed shape. Useful fact: the SDK's `BetaMessageStream` *is* an `AsyncIterable<BetaRawMessageStreamEvent>`, so older fakes that return it still satisfy the interface.
- **No partial-JSON parser.** StreamProcessor forwards the raw `input_json_delta` string and parsed tool input is only read at block stop, so the accumulator just buffers the delta strings and `JSON.parse`s once at `content_block_stop`. This is the simplification that made owning the accumulation small.
- **Retry and give-up live in TurnRunner, on purpose.** Keeping the loop there is what let QueryRunner and the control channel stay (mostly) untouched. v1 behaviour: every honoured `retry-after` capped at `RETRY_AFTER_CAP_MS` (60s); a 429 whose `retry-after` exceeds the cap is the non-transient account-limit case; it retries within `ACCOUNT_LIMIT_BUDGET_MS` (10 min, checked *before* each wait) then gives up; a *separate* transient counter means account-limit waits don't burn `MAX_RETRIES`; ESC aborts the wait.
- **Give-up is one signal, not two.** TurnRunner throws a dedicated `AccountLimitStoppedError` (not a raw 429), and QueryRunner's catch has one clause that ends the query with no `error` published, so the user sees only a single `🛑`. The notice (`⏳`/`🛑`) is delivered through an injected `AccountLimitListener` to `AccountLimitNotice` to `ConversationState.spliceNotice`, deliberately *not* through the control channel. The SC ruled a display-layer guard on the raw error a hack; the fix lives in the termination path where the error is never created.

## SC corrections future work must respect

- **Every 429 raises the notice, on every retry. No de-duplication.** An earlier brief carried a fabricated "one persistent notice, refreshed, not a new line each minute" requirement the SC never gave. It became a gate that made the lockout nearly invisible. Removed. Do not reintroduce a gate.
- **Foundational wire-parsing is pinned at the unit level.** `parseFrame`, `findFrameBoundary`, and `parseSse` are exported from `sse.ts` and tested directly and exhaustively, because this is the API-communication code and the SC's standard is that it must work perfectly. Keep them exported and keep the direct tests.

## SDK 0.92.0 gotchas (cost real time)

- `context_management` is on the `message_delta` *event*, not on its `delta`.
- `message_delta` usage fields are nullable, so cumulative usage must coalesce per-field, never spread wholesale.
- `compaction_delta` summary text must be concatenated onto the `compaction` block (mirrors the SDK's old `BetaMessageStream.mjs`). This is *not* in the build plan; it is carried only by tests. Without it, compaction blocks surface empty.
- `StreamProcessor.mapBlocks` guards a sparse-content hole (`if (block == null) continue;`): a server-tool-result opens a block at index 1 with no index 0. Production streams are contiguous-from-0, but assembly must not crash on the hole.

## Wire facts and what was dropped

Base URL `https://api.anthropic.com`, header `anthropic-version: 2023-06-01`, `POST /v1/messages?beta=true` with `stream:true`, streaming read timeout 600000ms, SSE framing on the double-newline, `retry-after-ms` preferred over `retry-after`. The `X-Stainless-*` telemetry headers were dropped on purpose. `TokenRefreshingAnthropic.ts` and `sdkInternals.ts` are deleted; auth acquisition and refresh (`Client/Auth/`) is plain `fetch` on an expiry check (no 401 path) and was untouched by the swap.

## Build and CI gates that bite

- **Manual API testing is the merge gate.** Build and type-check do not exercise the wire. The SC drove the account-limit path with a `SIMULATE_429` block in `transport.ts` plus shrunk caps in `backoff.ts`; that scaffolding must never ship (it was stripped before the feature commits). If you reintroduce it to test, strip it again before committing.
- **`pnpm run type-check` / `test` abort under no-TTY** (`ERR_PNPM_ABORTED_REMOVE_MODULES_DIR_NO_TTY`, pnpm's verify-deps purge gate). Drive the toolchain directly: per package, `node_modules/.bin/tsc -p tsconfig.check.json` and `node_modules/.bin/vitest run`. The CLI type-check needs the workspace deps' `dist/` built first (`tsup`). The root `tsconfig.check.json` is broken standalone; use the per-package one.
- **biome import-sort is not in the type-check/test gate.** Only `pnpm run ci` (`biome ci`) catches `organizeImports`, and it failed CI here. Run `pnpm ci:fix` before pushing, never `biome check --write` directly.
- **Changelog drift gate.** A `changes.jsonl` entry without a regenerated `CHANGELOG.md` reds the build. Run `pnpm --filter scripts run changelog` and commit both. The generator was changed during this mission to sort entries deterministically within category, so `changes.jsonl` line order (which rebases and merges reshuffle) no longer causes spurious drift.

## A hazard about these testament files

The dated testament files are `merge=union` and shared across concurrent missions. A rebase makes a branch's copy of a dated file *look* like it replaced another mission's, but the union driver recombines both sides at merge time, so leave the other entries alone. When shipping, diff each touched testament against `origin/main` (three-dot shows only what your PR adds) and confirm you are only adding your own entries.

## SSE hardening after code review (same PR, not yet shipped)

The reviewer's code-review pass flagged three small things in the SSE layer; all three were taken. No `changes.jsonl` entry: this transport has never shipped, so the baseline is unreleased and there is nothing for a consumer to have seen.

- `parseSse` carried a dead `_signal` parameter. Abort is enforced by the signal handed to `fetch` (the body reader inherits it, so an abort rejects `reader.read()` and the generator throws), so `parseSse` never consulted it. Dropped the parameter and the call-site argument.
- A malformed `data:` frame threw a bare `SyntaxError` (which `isRetryable` does not match, so it surfaced as a hard, non-retryable error) while the `error` frame wrapped via `safeJsonParse`. `handleFrame` now wraps the data-frame `JSON.parse` in `ApiStreamError`, so a corrupt stream is classified like any other stream failure.
- The `finally` did `reader.releaseLock()`, which leaves the underlying stream and its socket open if the consumer stops early. Now `await reader.cancel().catch(() => {})`, which closes them and is a harmless no-op once the stream has finished.

Test trap worth keeping: to prove the cancel, the stream must still be OPEN when the consumer stops. Cancelling an already-closed `ReadableStream` does not invoke the source's `cancel` hook (per spec, a closed stream's cancel is a no-op), so a test that enqueues everything and `close()`s the stream stays red even with the fix in place. Enqueue one frame and leave the stream open.
# 21:59 — Apostle (Phase 1, iteration 1): Memory tool plan replanned against current main

# 22:53 — Apostle (Phase 1, iteration 2): DI + per-write environment corrections folded in

The SC corrected two things after iteration 1; revised plan §1, §3, §4, §5, §7, §9, §10, §11. The rest stands.

## The two corrections and how they landed

1. **Environment resolved per write, not frozen at startup.** Iteration 1 resolved the bag once in `main.ts` and threaded it through `ContainerOptions`. Wrong — the git remote/cwd/config can change mid-session. Now: an injectable `IMemoryEnvironmentProvider` (abstract in claude-core) with `GitMemoryEnvironmentProvider` (app) whose `resolve()` reads config + runs `git config --get remote.origin.url` **on every write**. The store calls it inside each `write`. `main.ts` and `ContainerOptions` are no longer touched at all — which also dissolved the iteration-1 verify.ts risk.

2. **`@dependsOn` property injection, not factory closures.** I read `@shellicar/core-di-lite@1.0.2` source directly (the codebase has ZERO existing `@dependsOn` usage — the whole container.ts is factory-closures). Grounded facts the Maker must respect:
   - `dependsOn(Id)` is a **standard TS5 field decorator**. Provider does `new Impl()` then `instance[key] = resolve(...)` — **constructs first, injects after**.
   - **Constructor cannot read injected deps** (they're undefined during construction) → empty constructor, lazy resource open.
   - **Injected field MUST be public** — `instance[key] = ...` bracket-set can't reach `#private`. So `public foo!: T` with definite-assignment.
   - Register with **`.register(Id).to(Impl)`** single-arg (no factory). Abstract-class-as-identifier is the established `IObjectStore` shape.
   - **Decorators compile here**: no `experimentalDecorators` in the tsconfig chain → standard decorators; esbuild `target: node26`; core-di-lite uses its own WeakMap metadata so NO reflect-metadata / `emitDecoratorMetadata` needed.

## The design consequence the Maker most needs to grok

The DI constraint forces a **two-class split**, and it's not gold-plating — it's forced:
- **`SqliteMemoryEngine`** (app, NOT DI-managed): plain class, constructor `(db, clock)`, all the SQL, DDL in constructor. This is the analog of `SqliteObjectStore` and is the "real store over :memory:" for the integration tests. `write(draft, environment)` takes env as a param.
- **`SqliteMemoryStore`** (app, the injectable `IMemoryStore`): `@dependsOn(ConfigLoader)` + `@dependsOn(IMemoryEnvironmentProvider)`, empty constructor, `#ready()` lazy-opens the engine on first use, `write()` resolves env per call then delegates.

You cannot make one class be both a no-constructor-arg DI citizen AND a constructor-injected testable unit. That's why the split exists. The engine carries all asserted behaviour (ranking, soft-delete, input-safety, types, per-write stamping via the env param). The adapter is thin glue, deliberately NOT unit-tested (it opens real disk + delegates) — covered by the live acceptance test. I flagged that choice in §11.

## Other deltas worth carrying

- **No js-joda change anywhere.** The app already deps `@js-joda/core` (used by ConversationState); the engine/adapter use it there. The trivial `FakeMemoryStore` uses a constant string → claude-sdk-tools needs NO js-joda devDep (iteration 1 wrongly added it; dropped).
- **No `pnpm-lock.yaml` change** — no new dependency (core-di-lite + js-joda both already present).
- `AppToolsService` stays factory-closure-built and is handed `memory: x.resolve(IMemoryStore)` — the mission explicitly scopes its conversion-to-registry OUT. Do not refactor it. The new injectables are the store + provider only.
- The one place to **stop and report** rather than work around: if `@dependsOn` misbehaves under `verbatimModuleSyntax`/`useDefineForClassFields` at build/type-check. The fallback (factory closure for these two) is the exact pattern the SC is steering away from, so don't silently take it.


# 01:37 (27 Jun) — Apostle: testing-double correction (§3 + §9 redone)

The SC caught a real error, and it's the most important lesson on this mission. Recording it plainly because it's the exact trap that got a previous Handler removed.

## The mistake

My iteration-1/2 `FakeMemoryStore` was a working substring store — it filtered by type, sliced by limit, grouped for `types()`. Then my handler "unit tests" asserted *that* behaviour (e.g. "SearchMemory passes type through" wrote two memories and asserted count===1 — which tests the fake's filter, not the handler). That is the struck ranking-fake's smaller cousin: **a test double with behaviour means the handler tests test the double, not the handler.** The mission's whole testing note exists to kill this. I reproduced it without noticing.

Second mistake, when first corrected: I jumped to "use a stub" — swapping one prescription for another. The *form* of the double (stub/spy/fake) is the operator's call; dictating it is reaching into the wrong lane. The mission was then corrected to state only the **constraint**, leaving form to me.

## The fix (now in the plan)

- The double (`RecordingMemoryStore`, renamed from `FakeMemoryStore` because it's a **spy**, not a fake in Meszaros's terms) carries NO store behaviour: each method records the arg it was called with and returns a canned value the test sets. Nothing else.
- Handler unit tests (§9) assert ONLY the handler's own logic:
  - **mapping**: the captured store-call argument (`store.writeArg` toEqual `{title,body,type,keywords}` proves description dropped + fields forwarded; `store.searchArg` toEqual `{query,type,limit}`).
  - **shaping**: set a canned return, assert the handler's output shape (found/not-found discriminator, `{count,results}` wrap, literal delete, environment surfaced on read).
- ALL search / filter / rank / count is asserted only in the integration tier against the real `SqliteMemoryEngine` over `:memory:`. Unchanged.

## The durable rule for whoever's next

If a test double has logic in it, your unit test is probably testing the double. A handler is a thin mapper — test the mapping (captured call args) and the shaping (canned return → output), nothing that needs a working store. Anything that needs real behaviour is an integration test against the real thing. There is one ranking implementation, so there is never a parity test.

Everything else in the plan (engine, FTS5 search + safety, @dependsOn / per-write-environment wiring, the five tools, rendering, integration tests) stands from iteration 2.
