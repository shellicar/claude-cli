## Phase 4 (Maker) — already-aborted abort guard in exec-core

PR #371's review bot flagged, and the Courier confirmed, a real abort gap the
consolidation introduced. Note this contradicts the 2026-06-25 testament, which
cleared cancellation as "not a regression" — the reviewer checked the handler-level
`signal?.aborted` after `execute`/`executeTree`, but missed that `Executor.run`
itself spawns unconditionally. The two are different layers.

### The bug

`Executor.run` wired cancellation only via `opts.signal?.addEventListener('abort', …)`.
That listener never fires for a signal that is **already aborted** when `run` is
called, and there was no `aborted` check before `spawn`. So once one command is
cancelled, a chained command that inherits the already-aborted signal still spawns
and runs. Reachable through `;` (right side always runs), `||` (a killed left exits
non-zero, so the right qualifies), and V1's sequential loop. `&&` / `bail_on_error`
are incidentally spared because a killed left short-circuits them.

### Red-first

Added `Executor.run already-aborted signal > does not spawn when the signal is
already aborted` to `packages/exec-core/test/Executor.spec.ts`: abort an
`AbortController` before `run`, run `echo hi` with the aborted signal, assert the
collected stdout is empty. Confirmed RED before the fix:
`AssertionError: expected 'hi\n' to be ''`.

### Fix

One guard at the top of `run()`, before `spawn`: if `opts.signal?.aborted`, close
the sinks (matching the existing bad-cwd early-return path, so the flush contract
holds) and return `{ exitCode: null, signal: 'SIGTERM' }` — the same killed shape
the group-kill path produces. One place covers V1 steps, V2 trees, and pipelines
because every caller spawns through `run`. Test green after; nothing else changed.

### Verify

`pnpm test` and `pnpm type-check` both green at root (turbo: 11 test tasks, 10
type-check tasks). Existing cancellation/timeout tests unaffected.

### Suggested commit

`Guard exec-core run against an already-aborted signal`
