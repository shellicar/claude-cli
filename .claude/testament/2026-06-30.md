## Composable toolset redesign — shipped (Apostle → Maker → Courier)

### What the redesign is
No composable tool is ever both a pipe *source* and a pipe *stage*. That straddle forced a conditional schema — a field required in one pipe position and absent in another — which a memoryless model can't read. The fix: atomic single-role tools over typed, grouped streams, threaded by a rewritten `Pipe` that validates `out == next.in` by type before running. The old `{ ...step.input, content }` graft (`Pipe/Pipe.ts`) is gone — the typed stream is threaded as its own object and never enters `step.input`.

### The shape that carries it
- `composable.ts` — the three-face `ComposableTool`: `model` (pure Zod, model-facing), `pipe: {in,out}` (string edge tokens), `run` (canonical only). The three faces never reference each other; that separation is the whole point. Sources adapt to the standalone `ToolDefinition` via `toStandalone`; stages are pipe-only (transducers are ~never used standalone).
- `stream.ts` — the `Files` and `Content` typed streams, grouped, **path on the file**. `Content` flattens to a path header + `lineno:text` lines; `Files` flattens to one `path … size` line per record (dir trailing `/`, symlink as `name -> target`).
- **`Files` carries `path` as a first-class field on every record.** That is the addressable value the downstream *paths* mission consumes. The marking/normalisation/permission-running is NOT in this mission — the field is there to be marked, nothing more.
- Tool set: `Find` (discovery source), `Paths` (explicit-path source), `Read` (read stage), `Match` (filter), `Head`/`Tail`/`Range` (windowers), `Pipe`. `Grep` and `SearchFiles` are deleted — their jobs are now `Match` and `Paths | Read`. `ReadFile` is the **non-pipe** side of reading and is never a pipe step; reading named files in a pipe is `Paths | Read`. The boundary is deliberate.

### claude-core reaches (scope was relaxed mid-mission to allow this)
- `walk` returns `FileRecord[]` (was `string[]`) — it kept the type it already computed per entry instead of discarding it. Mid-traversal `readdir` failures are swallowed at the recursive call site (you never named those dirs); a top-level failure throws (fatal).
- `IFileSystem.find` returns records; `readlink` added across the interface, `NodeFileSystem`, and the test fakes. `FileRecord`/`FileType` live in claude-core (`fs/records.ts`) because `walk` must return them.
- Symlink target is the **one-hop** `readlink`, not `realpath` (which resolves the whole chain — wrong shape for `name -> target`).

### The critical defect the gates missed — pipe stages auto-denied (found in use, not by tests)
A pipe containing any stage was silently auto-denied with no prompt; only source-only pipes worked. `getPermission` (`apps/.../permissions.ts`) looks every pipe step up in the registered tool list, which holds only sources — stages live solely in the pipe registry, so the lookup missed and `Deny` won the `Math.max`. Fix: a permission-only `permissionTools` list = registered tools + stages via `toStandalone`, used for resolution only and never put on the wire. The name-lookup is the right mechanism (a future `Delete` terminal must still be gated), so the fix makes the lookup *find* the stages rather than weakening the guard.

**The insidious half:** a missed lookup was surfaced to the model as *"Rejected by user, do not reattempt"* — a human decision that never happened, so the model reasoned as if told no. `QueryRunner.ts` forwards the approval `reason` verbatim; the fix is to send a truthful one. Added `PermissionAction.NotFound = 3` (dominates the pipe `Math.max`); a genuinely-unknown tool now returns `Tool not found: <names>. This is a tool-lookup failure, not a user rejection.` A real `Deny` still sends no reason → the SDK's "Rejected by user" default, correct only for an actual deny.

### Post-build changes the SC made after using the tools
- **`Slice` deleted.** All four windowers shipped first on purpose, to gather usage data on which the model reaches for; the benchmark settled it — windowers are `Head`/`Tail`/`Range`.
- **`Range` inverted bounds is a zod `.refine` schema error**, caught pre-flight, not a runtime throw.
- **Emptied files are dropped** (`windowGrain`): a file left with no content after a per-file stage produces no output — grep parity, no empty path header.
- **`Match` follows grep:** dropped `context`, added `before`/`after` (`-B`/`-A`), each `int().min(0).default(0)`.
- **Regex validation via one shared schema.** `regexSchema` = `z.string().min(1).superRefine(...)` building `new RegExp` in a try/catch and surfacing the caught message via `ctx.addIssue` — the value stays a string, the handler builds the RegExp at use. A `regexPattern(purpose, examples)` factory bakes the JS/ECMAScript-flavour note in once and attaches per-field `examples`. Applied to `Match.pattern`, `Find.pattern`, and `EditFile`'s `regex_text` (the shared `min(1)` dissolved the earlier fork that kept EditFile separate). The shared-schema form was a correction: an object-level `superRefine` doesn't reuse across fields.

### Deviations from the plan (compiler-led / codebase-fact, all legitimate)
- **No per-tool `reconcile` field.** TS can't infer `TIn` from a `StreamFor<TIn>` parameter, so a per-tool generic field wouldn't assign. One central `reconcile(model, upstream)` in `composable.ts`, applied by the Pipe and `toStandalone` — the plan's "sole bridge" intent, more literally.
- **`pipe.ts` kept, trimmed to `PipeFilesSchema`.** The plan said delete it; `DeleteFile`/`DeleteDirectory` import `PipeFilesSchema` as their model input and Delete is out of scope.
- **`Read` drops the binary branch** — binary-in-a-pipe is fatal.

### Traps for the next cast
- **`TsDiagnostics` serves stale results mid-refactor** — it kept citing a removed field with byte-identical messages. Trust the real `tsc`/`tsup` build, not it.
- **turbo `type-check` gives a false green after app-only edits** (FULL TURBO cache). Run the package tsc directly: `pnpm --filter @shellicar/claude-sdk-cli exec tsc -p tsconfig.check.json`.
- **Don't run `test` and `build` concurrently against the same package** — two tsup builds race on `dist` (`ENOENT unlink …js.map`). Build alone is clean.
- The mission's `## Agents` points at `fleet/agents/operators/*.md`, which does not exist. The role text was already in the system prompt, so nothing was lost — standing brief defect, non-misdirecting.

## Phase 5 Apostle — iteration 3 (SC pinned the wake-lock structure)

Third revision of the wake-lock design in `implementation-plan.md` (Handler repo).
The SC pinned the structure after iteration 2 was still wrong. The other three
pieces (auto-retry, submit-to-resume, persist) are untouched and stand.

### Why iterations 1 and 2 both failed — the through-line

Both were **static resolutions where the decision must be live**:
- **Iter 1:** a startup factory (`createWakeLock`) read config at *registration* to
  pick real-vs-noop. core-di-lite registrations are all singletons → the factory
  runs once → the choice is frozen, can't track dynamic config.
- **Iter 2:** moved the decision into `acquire()` (good) but kept ONE
  mechanism-named class `CaffeinateWakeLock` that spawned `<command> -i -w <pid>`.
  Those are *caffeinate's* flags — hardcoded in a generic class. linux/windows
  mechanisms take entirely different arguments, so it could never extend. A
  command-string resolver (`resolvePlatformCommand` returning just the binary name)
  hid this: the binary varies by config, but the *invocation* is platform-specific
  and can't be parameterised by a string.

### The pinned structure (iteration 3) — locked, do not re-derive

- **DI token = `I`-prefixed abstract class `IWakeLock`.** The `I` is the *preferred*
  convention (matches `IStreamProcessor`/`ITurnRunner`). Existing no-`I` code in the
  repo is **drift** — do NOT copy it, do NOT "correct" the new `I` to match. The SC
  said the supervisor should not flag the `I` against the no-`I` code.
- **DI knows only `IWakeLock → PlatformWakeLock` (one impl).** That impl's
  `acquire()` IS the factory — it produces the `WakeLockHandle`.
- **The platform branch is a `switch` in `acquire()`, in code, NEVER in DI.** Adding
  linux/windows = a new `case` + a new mechanism, NOT a new registration and NOT a
  different bound implementation.
- **Each platform's mechanism owns its own invocation.** caffeinate's `-i -w <pid>`
  lives in `caffeinateWakeLock` (a function in this plan) and nowhere else.
- **Config read live inside `acquire()`** via the injected `() => config.preventSleep`
  accessor.
- **Class-vs-function for the mechanism is the operator's call** (SC: do not
  over-constrain). The plan shows a function; a class via `@dependsOn` is equally
  fine. The fixed rule: DI *provides* the mechanism, never *chooses* the platform.

### Files in the revised plan

`PlatformWakeLock` (the bound `IWakeLock`, platform switch in acquire), `caffeinateWakeLock`
(macOS mechanism fn, owns the flags), `spawnWakeLock` (killable spawn seam, swallows
ENOENT). Removed from the plan vs iter 2: `CaffeinateWakeLock`, `NoopWakeLock`,
`createWakeLock`, `resolvePlatformCommand`, `isExecutableOnPath`.

### My decision worth flagging

I **dropped the startup PATH probe** (the Phase 4 brief asked "probe once at
startup"). Reason: a probe must read `config.platforms` to know what to probe, and
doing that at startup re-introduces a config snapshot; doing it per-acquire defeats
"once." The swallowed spawn `error` (ENOENT) already delivers the real requirement —
"never an error if caffeinate absent." Cost: a doomed spawn per request on a macOS
box without caffeinate (rare, cheap). Surfaced as §7 fork #4 for the SC to rule on,
not silently dropped.

### Process note for whoever handles the next design rejection

The Handler's delivery note (2026-06-30) records the lesson: after two design
rejections, the SC stopped bouncing guesses off the Apostle and **pinned the design
in conversation first, then relayed it** as locked constraints. That's why this
iteration's brief is prescriptive (`I`-prefix, one-impl, branch-in-acquire) rather
than principle-shaped. Build to the locked constraints; the genuine latitude left is
small and named (class-vs-function, the probe fork).

### THE load-bearing fact for anyone touching this codebase

**Everything is `@dependsOn` property injection now.** Classes declare deps as
decorated fields — `@dependsOn(IMessageStreamer) private readonly streamer!: IMessageStreamer;`
— with **no constructor**. `container.ts` registers `abstraction → concrete` with a
bare `.to(Impl)` and the container auto-wires; hand-written `(x) => new X(...)`
factories survive ONLY where there's real composition-root work (assembling
handler chains, reading config to pick a db file). Do NOT write `new TurnRunner(...)`
call sites or `undefined` placeholder args — that shape is gone.

Verify wiring against the ACTUAL `container.ts` every time. The SC has now caught
DI-convention misses three times; grounding against a remembered shape is the
recurring trap. Read the file.

### The exact mirror to copy (it's all `AccountLimitListener`)

- `AccountLimitListener` is an **abstract class** in `packages/claude-sdk/src/public/types.ts`
  (~line 220), value-exported from `index.ts`.
- `AccountLimitNotice extends AccountLimitListener` with `@dependsOn(ConversationState) private readonly conversation!: ...`, no constructor.
- Container: **two** registrations — `register(AccountLimitNotice).to(AccountLimitNotice)`
  AND `register(AccountLimitListener).to(AccountLimitNotice)`.
- TurnRunner injects it via `@dependsOn(AccountLimitListener)`.

I mirrored this for BOTH new seams: `IWakeLock` (abstract token in
`public/interfaces.ts` beside `IStreamProcessor`/`ITurnRunner`) and

### Test approach for property-injected classes (confirmed from TurnRunner.spec)

There's a `buildTurnRunner(...)` helper that creates a real `createServiceCollection()`,
registers test fakes against the abstractions (`ISleepProvider` as `{ sleep }`,
`IRandomProvider` as `{ next }`, etc.), then `buildProvider().resolve(TurnRunner)`.
That's THE pattern — to add deps to a property-injected class you extend its
build-helper to register the new contracts, or existing tests stop resolving. My
plan notes extending `buildTurnRunner` to register `IWakeLock`/`StreamInterruptListener`.
