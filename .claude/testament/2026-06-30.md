**The render path, settled.** Three sinks all read `ConversationState` and pass through
one choke point for sealed content: `renderBlockContent` (renderConversation.ts:56) →
`blockContentLines` (blockLayout.ts:27). Sinks: live alt-buffer (PrimaryView.render →
renderConversation, full repaint/frame, PR #377), scrollback flush (flushSealedToScroll
→ renderBlocksToString, turn boundaries), History (HistoryView.#blockCard). The
replacement point for raw→markdown is `renderBlockContent`, routed by
`block.type === 'response' && cfg.markdown.enabled`.

**The one real constraint: the count-preservation invariant.** blockLayout.ts:8-11 —
code `decorate` returns one line per source line, so the History plain measurement
(historyContentExtent → plainCode) agrees with the view's highlighted render. The
markdown layout must mirror blockContentLines' shape: a layout fn taking a
count-preserving `decorate` for code bodies. Then measure/render agree because only
code-body colour differs. Put markdownLayout in model/ (like blockLayout) so both view
and measurement reach it; cli-highlight stays injected from view (getHighlighted).

**marked owning the parse is clean but SCOPED, not global.** CODE_FENCE_RE
(blockLayout.ts:5) serves every block type (tool JSON, notice, thinking, response).
Markdown is response-only. So marked replaces CODE_FENCE_RE on the response path;
the fence splitter stays for the rest. Global removal would regress fence highlighting
in tool output/thinking. No stop — the scoping makes it smaller.

**Streaming needs no special seam.** Primary repaints from state every frame, so
completeActive (ConversationState.ts:228, keeps raw content) auto-re-renders the sealed
block next frame. Alt buffer makes the changed line count safe; flush at turn boundary
goes through the same flag. Active block rendered inline at renderConversation.ts:170-181
(splits \n, no fence detection, not cached — cache is sealed-only by design); streaming
flag gates markdown there.

**Config.** schema.ts sdkConfigSchema (:200), copy thinkingSchema's shape (:154); loader
readConfig wired container.ts:127, hot-reload via ConfigReloader. Threading: render fns
are pure — add ConfigLoader to ViewModel (View.ts:24, assembled container.ts:227) so

is in scope.

**Things worth knowing for whoever follows.**
- `configLoader.config` resolves to `any` through the DI (confirmed via TsHover), which is why threading the typed `MarkdownConfig` compiles without friction. ViewModel types the field `ConfigLoader<typeof sdkConfigSchema>`.
- The active streaming block keeps its emoji prefix by swapping the first markdown line's indent for the emoji (`mdLines[0].slice(activeIndent.length)`) — markdownContentLines indents every line, so this preserves the existing `📝 ` marker.
- marked's `space` token must emit one blank line (`''`) — the boundary spec case (`# Hello` / blank / ```` ```md ````) requires the blank between heading and code box. The plan had `break`; that would have dropped it.
- Tables: marked parses GFM tables natively; the `default:` case emits `t.raw` verbatim → matches the out-of-scope passthrough spec exactly. No special handling needed.
- The build (`tsx build.ts`) regenerates `schema/sdk-config.schema.json` from the Zod schema; it picked up `markdown` automatically. Staged it.
- Three ViewModel test makeModel()s (PrimaryView/ViewHost/HistoryView spec) needed a `configLoader` stub — added in each file's own cast style. cli-config.spec defaults snapshot needed the `markdown` group.
- Code-box test cases drive the real `getHighlighted` on both sides — verifies the renderer routes the fence to the box + label without coupling the assertion to cli-highlight's exact colours (which are cli-highlight's own contract, and which the spec's colour codes came from).
