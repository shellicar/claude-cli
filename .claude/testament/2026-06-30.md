## Composable toolset redesign — Phase 2 Maker, iteration 2 (permission defect fix)

The mission was re-issued after iteration 1 passed supervision but the SC found a defect in use: a pipe containing any *stage* (`Read`, `Match`, `Head`, `Tail`, `Range`, `Slice`) was silently auto-denied — only `Paths`/`Find`-only pipes worked.

### Root cause (confirmed against the code)
`getPermission` (`apps/claude-sdk-cli/src/permissions.ts`) resolves a `Pipe` by walking each step and looking it up: `allTools.find(t => t.name === s.tool)`, with `if (!definition) return PermissionAction.Deny`. The redesign registers only the *sources* standalone (`...sources.map(toStandalone)` in `createAppTools`); stages live solely in the pipe registry, so the lookup missed every stage → `Deny` won the `Math.max`. The redesign changed *what is in the registered tool list*, and the permission path that assumed every pipe step is findable there was not brought along.

### Fix — a permission-only tool list, kept off the wire
Added `permissionTools` = `[...tools, ...stages.map(toStandalone)]` to `createAppTools`'s return. It is the registered set **plus** the stages, used *only* for permission resolution. The wire/registry `tools` is unchanged (stages stay pipe-only; the model never sees them as top-level tools). Threaded it: `AppTools` type → `AppToolsService.permissionTools` → container passes it in the `AgentMessageHandler` opts → handler resolves permissions against `this.#permissionTools` instead of `this.#config.tools`.

Why this shape: the stages are legitimately not in `allTools`, and a name-lookup is the right mechanism (a future `Delete` terminal is `operation: 'delete'` and *must* still be gated — so "treat unknown pipe step as read" would be wrong). The fix makes the lookup find the stages rather than weakening the guard.

### Tests
- `createAppTools.spec.ts` (the real regression pin): `permissionTools` contains every stage name; a pipe `Find | Read | Match` resolves to `Approve`, not `Deny`.
- `permissions.spec.ts`: a pipe whose steps include a stage (present in the list) is not auto-denied.
- Fixed two pre-existing app tests that fed the permission list via `config.tools`: `AgentMessageHandler.spec`'s `makeOpts` now mirrors `config.tools` into `permissionTools` (the 7 approval-status tests went red because the lookup moved field); `ThinkingRequestWiring.spec`'s `AppToolsService` stub gained `permissionTools: []`.

### Gotcha — turbo type-check cache is unreliable here
`pnpm run type-check` reported FULL TURBO (all cached) and did **not** re-run `tsc` after my app `src` edits, so it gave a false green while a test was actually failing. The honest check is to run the package's `tsc` directly: `pnpm --filter @shellicar/claude-sdk-cli exec tsc -p tsconfig.check.json` (exit 0, clean). Don't trust the cached turbo type-check after app-only edits — run the package tsc directly.

## Permission defect — second half: a missed lookup must not masquerade as a user rejection

The mission was extended: the deeper bug is that when a pipe step's lookup misses, `Deny` is surfaced to the model as *"Rejected by user, do not reattempt"* — a human decision that never happened. The model then reasons as if it were told no.

### Where the lie comes from (SDK, confirmed)
`QueryRunner.ts` (~280): on `!response.approved`, the model is sent `response.reason ?? 'Rejected by user, do not reattempt'`. So the `reason` field on the approval response is already forwarded to the model verbatim. No SDK change needed — the fix is to send a truthful `reason`.

Also confirmed: a genuinely-unknown *top-level* tool never reaches this path — the SDK's `ToolRegistry.resolve` returns `not_found` and emits `Tool not found: <name>` directly (QueryRunner ~221), before any approval. So the only place a not-found leaks into the approval flow as a fake rejection is a **pipe step** (the permission recursion), because the SDK can't see pipe steps.

### Fix (app-side)
- `permissions.ts`: added `PermissionAction.NotFound = 3` (highest, so it dominates the pipe's `Math.max` — any unfound step makes the whole pipe not-found). `getPermission` returns `NotFound` instead of `Deny` for a missing definition. Added `findUnknownTools(tool, allTools)` — names the missing top-level tool, or for a pipe, each unfound step.
- `AgentMessageHandler`: on `NotFound`, send the approval response with `approved: false` **and** a `reason` of `Tool not found: <names>. This is a tool-lookup failure, not a user rejection.` — so the model sees the true cause. A real `Deny` still sends no reason → the SDK's "Rejected by user" default, which is correct for an actual deny. The not-found tool is shown locally as an error (💥), not a denial (❌).

### Tests
- `permissions.spec`: unknown tool → `NotFound` (was Deny); a pipe with an unknown step → `NotFound` (dominates a known read step); `findUnknownTools` names the missing step.
- `AgentMessageHandler.spec`: gave `makeOpts` an `onSend` capture; the old "records auto-denied status for an unknown tool" (asserted the ❌ glyph) is replaced by two channel-level pins — an unknown tool, and a pipe with an unknown step, each asserting the response is `approved:false` with a `Tool not found: …` reason (so the model is never told the user rejected it).
