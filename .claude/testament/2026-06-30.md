# 19:05

## Composable toolset redesign ŌĆö shipped (Apostle ŌåÆ Maker ŌåÆ Courier)

### What the redesign is
No composable tool is ever both a pipe *source* and a pipe *stage*. That straddle forced a conditional schema ŌĆö a field required in one pipe position and absent in another ŌĆö which a memoryless model can't read. The fix: atomic single-role tools over typed, grouped streams, threaded by a rewritten `Pipe` that validates `out == next.in` by type before running. The old `{ ...step.input, content }` graft (`Pipe/Pipe.ts`) is gone ŌĆö the typed stream is threaded as its own object and never enters `step.input`.

### The shape that carries it
- `composable.ts` ŌĆö the three-face `ComposableTool`: `model` (pure Zod, model-facing), `pipe: {in,out}` (string edge tokens), `run` (canonical only). The three faces never reference each other; that separation is the whole point. Sources adapt to the standalone `ToolDefinition` via `toStandalone`; stages are pipe-only (transducers are ~never used standalone).
- `stream.ts` ŌĆö the `Files` and `Content` typed streams, grouped, **path on the file**. `Content` flattens to a path header + `lineno:text` lines; `Files` flattens to one `path ŌĆ” size` line per record (dir trailing `/`, symlink as `name -> target`).
- **`Files` carries `path` as a first-class field on every record.** That is the addressable value the downstream *paths* mission consumes. The marking/normalisation/permission-running is NOT in this mission ŌĆö the field is there to be marked, nothing more.
- Tool set: `Find` (discovery source), `Paths` (explicit-path source), `Read` (read stage), `Match` (filter), `Head`/`Tail`/`Range` (windowers), `Pipe`. `Grep` and `SearchFiles` are deleted ŌĆö their jobs are now `Match` and `Paths | Read`. `ReadFile` is the **non-pipe** side of reading and is never a pipe step; reading named files in a pipe is `Paths | Read`. The boundary is deliberate.

### claude-core reaches (scope was relaxed mid-mission to allow this)
- `walk` returns `FileRecord[]` (was `string[]`) ŌĆö it kept the type it already computed per entry instead of discarding it. Mid-traversal `readdir` failures are swallowed at the recursive call site (you never named those dirs); a top-level failure throws (fatal).
- `IFileSystem.find` returns records; `readlink` added across the interface, `NodeFileSystem`, and the test fakes. `FileRecord`/`FileType` live in claude-core (`fs/records.ts`) because `walk` must return them.
- Symlink target is the **one-hop** `readlink`, not `realpath` (which resolves the whole chain ŌĆö wrong shape for `name -> target`).

### The critical defect the gates missed ŌĆö pipe stages auto-denied (found in use, not by tests)
A pipe containing any stage was silently auto-denied with no prompt; only source-only pipes worked. `getPermission` (`apps/.../permissions.ts`) looks every pipe step up in the registered tool list, which holds only sources ŌĆö stages live solely in the pipe registry, so the lookup missed and `Deny` won the `Math.max`. Fix: a permission-only `permissionTools` list = registered tools + stages via `toStandalone`, used for resolution only and never put on the wire. The name-lookup is the right mechanism (a future `Delete` terminal must still be gated), so the fix makes the lookup *find* the stages rather than weakening the guard.

**The insidious half:** a missed lookup was surfaced to the model as *"Rejected by user, do not reattempt"* ŌĆö a human decision that never happened, so the model reasoned as if told no. `QueryRunner.ts` forwards the approval `reason` verbatim; the fix is to send a truthful one. Added `PermissionAction.NotFound = 3` (dominates the pipe `Math.max`); a genuinely-unknown tool now returns `Tool not found: <names>. This is a tool-lookup failure, not a user rejection.` A real `Deny` still sends no reason ŌåÆ the SDK's "Rejected by user" default, correct only for an actual deny.

### Post-build changes the SC made after using the tools
- **`Slice` deleted.** All four windowers shipped first on purpose, to gather usage data on which the model reaches for; the benchmark settled it ŌĆö windowers are `Head`/`Tail`/`Range`.
- **`Range` inverted bounds is a zod `.refine` schema error**, caught pre-flight, not a runtime throw.
- **Emptied files are dropped** (`windowGrain`): a file left with no content after a per-file stage produces no output ŌĆö grep parity, no empty path header.
- **`Match` follows grep:** dropped `context`, added `before`/`after` (`-B`/`-A`), each `int().min(0).default(0)`.
- **Regex validation via one shared schema.** `regexSchema` = `z.string().min(1).superRefine(...)` building `new RegExp` in a try/catch and surfacing the caught message via `ctx.addIssue` ŌĆö the value stays a string, the handler builds the RegExp at use. A `regexPattern(purpose, examples)` factory bakes the JS/ECMAScript-flavour note in once and attaches per-field `examples`. Applied to `Match.pattern`, `Find.pattern`, and `EditFile`'s `regex_text` (the shared `min(1)` dissolved the earlier fork that kept EditFile separate). The shared-schema form was a correction: an object-level `superRefine` doesn't reuse across fields.

### Deviations from the plan (compiler-led / codebase-fact, all legitimate)
- **No per-tool `reconcile` field.** TS can't infer `TIn` from a `StreamFor<TIn>` parameter, so a per-tool generic field wouldn't assign. One central `reconcile(model, upstream)` in `composable.ts`, applied by the Pipe and `toStandalone` ŌĆö the plan's "sole bridge" intent, more literally.
- **`pipe.ts` kept, trimmed to `PipeFilesSchema`.** The plan said delete it; `DeleteFile`/`DeleteDirectory` import `PipeFilesSchema` as their model input and Delete is out of scope.
- **`Read` drops the binary branch** ŌĆö binary-in-a-pipe is fatal.

### Traps for the next cast
- **`TsDiagnostics` serves stale results mid-refactor** ŌĆö it kept citing a removed field with byte-identical messages. Trust the real `tsc`/`tsup` build, not it.
- **turbo `type-check` gives a false green after app-only edits** (FULL TURBO cache). Run the package tsc directly: `pnpm --filter @shellicar/claude-sdk-cli exec tsc -p tsconfig.check.json`.
- **Don't run `test` and `build` concurrently against the same package** ŌĆö two tsup builds race on `dist` (`ENOENT unlink ŌĆ”js.map`). Build alone is clean.
- The mission's `## Agents` points at `fleet/agents/operators/*.md`, which does not exist. The role text was already in the system prompt, so nothing was lost ŌĆö standing brief defect, non-misdirecting.

### Deferred (tracked separately, not in this PR)
The `{summary, body}` tool-output envelope and `Ref`-as-a-pipe-source ŌĆö parked for a tracked issue post-ship, per the SC.

### Ship state
Phase 2 committed on `feature/composable-toolset-redesign` (`297f523` redesign + permission fix, `29bab48` Slice/Range/Match, `f8d391a`/`6fe9123`/`75e3c7e` regex). Courier added `changes.jsonl` + regenerated `CHANGELOG.md` for the three source-changed packages (claude-sdk-tools, claude-core, claude-sdk-cli), pushed, and opened the PR.
