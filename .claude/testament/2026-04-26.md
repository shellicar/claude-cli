# 07:59

## ReadFile: image/*, file-type detection, remove 500KB text limit

Three files changed: `schema.ts`, `types.ts`, `ReadFile.ts`.

### What changed

`InputMimeTypeSchema` is new — `['text/plain', 'application/pdf', 'image/*']`. The input schema now uses it instead of `SupportedMimeTypeSchema`. These two schemas serve different purposes and change for different reasons. `SupportedMimeTypeSchema` is about what the system can detect internally. `InputMimeTypeSchema` is what a caller declares as intent. They happened to share values before this, which was the bug.

`detectBlock` replaces `validateMagicBytes` + `detectMimeType`. It uses `file-type` (already a dependency, previously unused). Returns a discriminated union: `{ kind: 'text', lines }` or `{ kind: 'binary', mimeType, block }` or `null`. `case undefined` is the text path — file-type returning nothing means treat as text. Every other case is a supported binary type or `default: null`. The handler has no branching on `input.mimeType` after the call.

`MAX_FILE_BYTES` (500KB text limit) removed. It existed because without binary detection, a large binary read as text would dump garbage into context. `file-type` handles that now. The Ref tool handles large text files. No reason to block the read.

### DMG note

UDIF DMG stores its `koly` magic in the last 512 bytes. file-type reads from the header only, so it returns undefined. The bentobox DMG read as text and came back as a Ref. Accepted tradeoff — undetectable binary treated as text.

### SC design corrections

The SC corrected the design twice during this cast:

1. I proposed adding `image/*` to `SupportedMimeTypeSchema`. SC: wrong schema. The input schema and the supported-types schema look the same right now but serve different purposes. They need to be separate types.

2. I had `detectBlock` take `inputMimeType` and branch inside `if (!type)` before returning the text result. SC: why are there multiple code paths. The text case is just another case. `case undefined` returns the text result the same way other cases return binary results. No separate path.

The second correction is the one worth remembering. When a function produces multiple structurally different things in different branches, that's a signal the branches belong in separate functions or the return type should unify them. Here they unified — text lines and binary blocks are both just `DetectResult`.

### Tests

The existing magic byte tests used GIF content (ASCII-safe). That trap is documented in the 2026-04-25 testament. New tests for `image/*` and the `file-type` path will need the same care — or real binary fixtures.

## Phase 5 ruling: test rework

Three spec files reworked. Only the sections added in this PR were in scope.

### What the ruling required

Two problems in the original Phase 3/5 tests:

1. Multiple `expect` calls per `it` block. One test failing gives no information about which behaviour broke.
2. Property access inside `expect(...)`. The rule: `const actual = <terminal value>; expect(actual)`. The assignment holds the terminal value; nothing after `actual` inside `expect`.

The reference sections (`image/*`, `unsupported binary`, `mime type mismatch`) already in the file showed the correct shape.

### What changed

**`ReadFile.spec.ts` — `describe('createReadFile — binary files (mimeType)')`**: 5 tests → 15 tests. Each multi-assertion test split into individual tests. `toMatchObject` calls with multiple properties decomposed into separate assertions (one per property). Message checks from the original `toMatchObject` calls were dropped — the reference error tests check `error` flag and `attachments` only; message content is an implementation detail.

**`ToolRegistry.spec.ts` — `describe('ToolRegistry — attachments')`**: 3 tests → 10 tests. The first test (`toMatchObject` with 4 properties + 2 other assertions) became 6 tests. The transform test (3 assertions) became 3 tests. The `'blocks key is absent'` test already had 1 assertion but needed the `actual`/`expected` pattern applied.

**`QueryRunner.spec.ts` — `describe('QueryRunner — tool_result content array for binary outputs')`**: 1 test (4 assertions + `toMatchObject` with 4 properties) → 7 tests. For `document block is present`, I used `blocks.some((b) => b.type === 'document')` rather than finding the block and checking its type — that's a tautology since `.find((b) => b.type === 'document')` already proves the type. `some` tests presence; the source property tests verify the contents.

### Setup duplication

The reference sections repeat setup in each test rather than using `beforeEach`. I followed that pattern throughout, even for the heavy QueryRunner tests that re-run the full fake agent loop per assertion. Isolation over brevity.
