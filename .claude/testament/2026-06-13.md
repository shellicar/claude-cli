## Investigator — why the permissions notice prints on any config change (#347 / fix/permissions-print)

The 2026-06-09 testament (20:37 entry, "Permissions display: the on-change comparison is load-bearing") already diagnosed this bug. My job was to verify that account against the live code and the actual git history, not relay it. It holds up — and I pinned the specifics it left loose.

### What the code says (HEAD b05b4ed)

- Live gate is **whole-config inequality only**. `ConfigLoader.#reload()` dedupes on `JSON.stringify(previousConfig) === JSON.stringify(parsed)` (`ConfigLoader.ts:116`) then fires every listener (`:122-124`). The `main.ts` onChange (`entry/main.ts:212-219`) calls `spliceNotice(formatPermissionsDisplay(config.permissions))` unconditionally at `:214`. No permissions-specific compare exists in the live path.
- `ConfigChangeListener<T> = (config: T) => void` (`types.ts:83`) — listeners get only *next*, never *prev*. So any consumer-side gate has to hold its own baseline. This constrains the fix options.
- `formatPermissionChange.ts` no longer contains `formatPermissionChange` — only `formatPermissionsDisplay`. The filename is a stale artefact.

### What the history says (verified by diff, not inference)

The gate genuinely existed in committed history, but only on the feature branch and only briefly:
- `2c7ea94` 17:19:53 — added `formatPermissionChange`/`permissionsEqual` + a `stablePermissions` tracker in onChange that splices only on a real diff. Tests covered the pure function.
- `8e529f6` 17:21:17 — **the breaking commit.** Replaced the gated block with the unconditional splice. Direct feature-branch dev commit, ~2 min later. NOT a merge-back (the merge-backs are `5a81219`/`6b130d3`/`3fc146c`).
- `c5d2a48` (lint), `88d984d` (deleted the dead functions from source), `a790953` (deleted the dead tests).
- All squashed into `1676610` (`git rev-list --parents` → single parent `2979d43` = squash). #350 carried the unconditional onChange forward into `entry/main.ts` verbatim.

So the SC's "it worked during development" is real but never reached main — ~84 seconds of commit time on the branch. And even `2c7ea94`'s gate had a first-change quirk (`stablePermissions` null-seeds, so the first onChange always printed the full matrix regardless).
