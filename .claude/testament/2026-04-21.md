Branch `fix/server-tool-blocks-in-history`. The branch existed on origin with one commit (`b35aa5d Add prompt explaining bug`). Checked out cleanly, 1 ahead / 3 behind origin/main (fine for a feature branch).

### What the bug is

`StreamProcessor` tracks `server_tool_use` and `server_tool_result` accumulators in `BlockAccumulator` but never pushes them to `completed` at `content_block_stop`. Same for `redacted_thinking`: the `content_block_start` case just `break`s. `TurnRunner` builds the assistant message from `result.blocks`, so the wire message is stripped of these blocks. The API detects the modification on the next turn and rejects.

### What the scaffold does

Two test files changed:

**StreamProcessor.spec.ts** — four modifications + new describe block:
1. `'does not push server_tool_use or web_fetch_tool_result blocks to completed'` — changed `toHaveLength(0)` to `toHaveLength(2)`.
2. `'yields exactly one block after server tool use'` renamed to `'yields exactly three blocks after server tool use'`, count changed to 3.
3. `'multiple server tool invocations in sequence do not corrupt state'` — changed `result.blocks[0]` to `result.blocks[4]`. With 2 server_tool_use + 2 web_fetch_tool_result + 1 text, text is at index 4.
4. New describe `'StreamProcessor — conversation integrity'`: full 6-block web-search response (thinking, text, server_tool_use, web_search_tool_result, thinking, text), tests for count (6) and ordered types. Plus `'redacted_thinking block appears in completed'`.

**TurnRunner.spec.ts** — helper + new describe:
- `makeServerToolStream()` helper: yields a stream with server_tool_use and web_search_tool_result blocks, no text blocks.
- New describe `'TurnRunner — server tool block preservation'` with one test asserting `last.content` is an array with those two block types.

### What was done

**`types.ts`**: Added `ServerToolResultBlock` as a named exported type (union of the 7 server tool result type strings, plus `toolUseId` and `content`). `ContentBlock` expanded with `server_tool_use`, `ServerToolResultBlock`, and `redacted_thinking` variants.

Using a named `ServerToolResultBlock` type (rather than inlining it in `ContentBlock`) meant `StreamProcessor.ts` could reference `ServerToolResultBlock['type']` for the accumulator's `blockType` field — so TypeScript tracks the exact union rather than `string`, and the push to `completed` at stop requires no cast.

**`StreamProcessor.ts`**:
- `BlockAccumulator.server_tool_result`: added `blockType: ServerToolResultBlock['type']` and `toolUseId: string`. `name` kept for the event emit (mapped name like `'web_search'`, not the wire type).
- `BlockAccumulator` gains `redacted_thinking: { type: 'redacted_thinking'; data: string }`.
- `content_block_start` for server tool result cases: added `blockType: event.content_block.type, toolUseId: event.content_block.tool_use_id` — TypeScript narrows the fallthrough union so `.tool_use_id` is directly accessible without a cast (all 7 result block types have it). `event.content_block.content` was already accessible.
- `content_block_start` for `redacted_thinking`: replaced bare `break` with an accumulator entry, casting via `as unknown as { data: string }` (the SDK type doesn't expose `.data` directly through the union).
- `content_block_stop` for `server_tool_use`: added `completed.push(...)` before the emit.
- `content_block_stop` for `server_tool_result`: added `completed.push(...)` before the emit.
- `content_block_stop` added `redacted_thinking` case pushing to `completed`.

**`TurnRunner.ts`**: Added `BetaServerToolUseBlockParam` and `BetaRedactedThinkingBlockParam` to the import. `mapBlock` expanded with:
- `server_tool_use`: casts `b.name as BetaServerToolUseBlockParam['name']` (internal `name` is `string`, wire type requires a specific union), then uses `satisfies BetaServerToolUseBlockParam`.
- `redacted_thinking`: straightforward `satisfies BetaRedactedThinkingBlockParam`.
- All 7 server tool result types: fall-through producing `{ type: b.type, tool_use_id: b.toolUseId, content: b.content } as BetaContentBlockParam`. Cast needed because `b.content` is `unknown` but the wire param types require specific content shapes. The data came from the API so the shapes are correct.

**`StreamProcessor.spec.ts`**: `'the block after server tool use is the correct text content'` changed `blocks[0]` to `blocks[2]`. The Scaffolder's testament predicted this — with server_tool_use at [0] and web_fetch_tool_result at [1], text is now at [2].

### Design note

The Scaffolder's testament said `server_tool_result` in `ContentBlock` should store "the original block type name". The test `'blocks appear in the order emitted by the API'` checks `b.type === 'web_search_tool_result'` — so the `type` field on the ContentBlock must be the original wire type string, not a wrapper like `'server_tool_result'`. Using `ServerToolResultBlock` with a union `type` satisfies this and keeps TypeScript narrowing working in `mapBlock`.

## Courier: server tool blocks in history

### What was built

`StreamProcessor` was not pushing `server_tool_use`, server tool result blocks (`web_search_tool_result`, etc.), or `redacted_thinking` to the `completed` blocks array. `TurnRunner` builds the assistant message from `completed`, so those blocks were silently dropped. On the next turn the API detected the stripped message and rejected it.

**`types.ts`**: `ContentBlock` union expanded with `server_tool_use` (id, name, input), `ServerToolResultBlock` (typed union of 7 result type strings, plus toolUseId and content), and `redacted_thinking` (data). `ServerToolResultBlock` is a named exported type rather than inlined so `StreamProcessor` can reference `ServerToolResultBlock['type']` for the accumulator's `blockType` field.

**`StreamProcessor.ts`**: At `content_block_stop`, added `completed.push(...)` for `server_tool_use`, `server_tool_result`, and the new `redacted_thinking` case. The `redacted_thinking` start case previously just `break`d; now it adds an accumulator entry. The original event emits for display are unchanged. For server tool result blocks all their data comes in the start event, not via deltas; the accumulator stores `blockType` (original wire type string) and `toolUseId` alongside the existing `content` and `name`.

**`TurnRunner.ts`**: `mapBlock` expanded for the three new block kinds. `server_tool_use` casts `name` to `BetaServerToolUseBlockParam['name']` (internal type is `string`, wire type requires a specific union). All 7 server tool result types share a single fall-through case producing `{ type, tool_use_id, content }` cast to `BetaContentBlockParam`. `redacted_thinking` is straightforward.

One test modified by the Builder beyond what the Scaffolder expected: `'the block after server tool use is the correct text content'` changed from `blocks[0]` to `blocks[2]`. With server_tool_use at [0] and web_fetch_tool_result at [1], the text block is now at [2]. The Scaffolder's testament predicted this.

120 / 120 tests passing. Type-check passing.

### Key trap

The `server_tool_result` accumulator stores `name` (the mapped display name, e.g. `'web_search'`) for the event emit, and `blockType` (the original wire type string, e.g. `'web_search_tool_result'`) for the completed push. The distinction matters: `mapBlock` switches on `b.type` which comes from `ContentBlock.type`, which is the wire type string. Getting `name` and `blockType` confused would produce a broken switch case with no TypeScript error (both are strings).
