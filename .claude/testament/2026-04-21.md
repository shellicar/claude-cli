## Phase 1 scaffold: server tool blocks in history

Branch `fix/server-tool-blocks-in-history`. The branch existed on origin with one commit (`b35aa5d Add prompt explaining bug`). Checked out cleanly, 1 ahead / 3 behind origin/main (fine for a feature branch).

### What the bug is

`StreamProcessor` tracks `server_tool_use` and `server_tool_result` accumulators in `BlockAccumulator` but never pushes them to `completed` at `content_block_stop`. Same for `redacted_thinking`: the `content_block_start` case just `break`s. `TurnRunner` builds the assistant message from `result.blocks`, so the wire message is stripped of these blocks. The API detects the modification on the next turn and rejects.

### What the scaffold does

Two test files changed:

**StreamProcessor.spec.ts** — four modifications + new describe block:
1. `'does not push server_tool_use or web_fetch_tool_result blocks to completed'` — changed `toHaveLength(0)` to `toHaveLength(2)`.
2. `'yields exactly one block after server tool use'` renamed to `'yields exactly three blocks after server tool use'`, count changed to 3.
3. `'multiple server tool invocations in sequence do not corrupt state'` — changed `result.blocks[0]` to `result.blocks[4]`. With 2 server_tool_use + 2 web_fetch_tool_result + 1 text, text is at index 4.
4. New describe `'StreamProcessor — conversation integrity'`: full 6-block web-search response (thinking, text, server_tool_use, web_search_tool_result, thinking, text), tests for count (6) and ordered types. Plus `'redacted_thinking block appears in completed'`.

**TurnRunner.spec.ts** — helper + new describe:
- `makeServerToolStream()` helper: yields a stream with server_tool_use and web_search_tool_result blocks, no text blocks.
- New describe `'TurnRunner — server tool block preservation'` with one test asserting `last.content` is an array with those two block types.
