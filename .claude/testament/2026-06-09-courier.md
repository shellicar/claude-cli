# Courier — PR #349 (fix/streaming-render)

The git log and PR description tell you what changed. This testament records what
you need to think correctly about this code — things that aren't written anywhere
else and that will still be true when the next operator arrives.

---

## What a block is

A `Block` is a flat string in memory with a type. The `\n` characters in the content string
are intentional line breaks — things Claude actually broke a line at, or that the code
explicitly inserted. Visual wrapping is a completely separate concern: `renderBlockContent`
calls `wrapLine` at render time based on terminal width. The block content never knows about
column width.

This distinction matters for `spliceNotice`: splice at `lastIndexOf('\n')`, not at some
visual row boundary. The visual rows shift when the terminal resizes; the content's `\n`
characters don't.

## What "sealed" means

"Sealed" means a block has moved from `activeBlock` to `sealedBlocks`. That is all it means.
Sealed blocks are still in memory. They are still rendered every frame in the alt buffer.
They are mutable — `appendToLastSealed` and `setLastContent` both modify sealed block
content in place, and the next repaint picks up the change.

The main scroll buffer is a separate, unrelated concern. When `flushedCount` advances,
`renderBlocksToString` writes blocks to the permanent scroll history. That write is
one-way — after a block is flushed, updating its in-memory content won't appear in the
scroll. But the alt buffer (what the user sees right now) always re-renders from the
in-memory arrays, sealed or not.

This distinction caused confusion during this session. Several times I argued "you can't
update a sealed block" — this was wrong. The SC corrected it. The alt buffer renders
everything in memory every frame. Sealed is a position in an array, not a permanence
guarantee.

## What the Messages API actually returns for tools

Multiple `tool_use` content blocks in one message arrive **sequentially**. One closes
(`content_block_stop`) before the next opens (`content_block_start`). The "parallel" in
parallel tool use refers to how the user-side executes the results — not to how they
stream.

The API guarantees: when a message contains any `tool_use` blocks, `message_delta` will
have `stop_reason === 'tool_use'`. Always. This is what `tool_batch_end` is emitted on.
There is no guessing required.

A "tool batch" therefore has a clean boundary: it starts with the first `tool_use` or
`server_tool_use` `content_block_start` in a message, and it ends when
`stop_reason === 'tool_use'` arrives. The SDK is the right place to know this because it
sees both events. The CLI should not infer the batch boundary from its own block state.

Before this session, the CLI had `activeBlock?.type !== 'tools'` in `block_enter` to detect
"same batch vs new batch." That was the CLI inferring a message-level concept from block-level
state. It was fragile — it broke when `message_usage` sealed the block between turns.
`tool_batch_start`/`tool_batch_end` make the boundary explicit at the right level.

## Why approval display doesn't need its own block

`ToolObject` instances survive `message_usage`. They're in `#toolObjects` on the handler,
keyed by tool id. When `tool_approval_request` fires, the ToolObject for that tool is still
there — `message_usage` sealed the visual block but didn't clear the map.

The visual tools block is just a rendered snapshot of the ToolObjects' current states.
`#redrawTools()` rebuilds it from the map on every call. So updating the ToolObject's state
(via `resolve`/`approve`/`deny`) and calling `#redrawTools()` against the sealed block via
`setLastContent` is correct and sufficient. The sealed block's content in memory is updated;
the alt buffer repaint shows the new state.

Opening a new visual tools block for approval display creates duplication: the sealed block
(streaming states) and the new active block (approval states) both render as a continuation.
The right model is one block per tool batch, updated in place throughout its lifecycle.

## What's unresolved

**Notice indentation in a host block**: `spliceNotice` injects notice text into whatever
block is currently active. That block's renderer applies its own `CONTENT_INDENT` to every
line, including the spliced notice. The notice looks indented. Fixing this requires marking
spliced lines with a sentinel that `renderBlockContent` recognises and renders with zero
indent. Not done. The correct approach is a sentinel byte (something that never appears in
normal content) prefixed to injected lines, stripped and handled in `renderBlockContent`.

**Notice overwrite during tool use**: `setLastContent('tools', ...)` replaces the entire
tools block content. If a notice was spliced into the tools block and then a ToolObject
state changes (which calls `#redrawTools()`), the notice is overwritten. The tools block
content is built fresh from `#toolOrder` every time. Fixing this requires either tracking
notice insertions separately or re-splicing after every redraw. Not done.
