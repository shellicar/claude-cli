# Courier ‚Äî PR #349 (fix/streaming-render)

This session started as a Courier ship for the Maker's fixes, then became a live design and
implementation session with the SC. What was shipped is substantially larger than what the
mission brief described.

---

## Original scope (from the brief)

Three breaks introduced when `feature/streaming-tool-input` (#343) merged onto main's
refactored view layer:

1. **Break 1** ‚Äî Approval lookup key mismatch: `tool_approval_request` looked up by a
   fresh UUID rather than the tool_use id, so the resolved view never appeared.
2. **Break 2** ‚Äî `Flasher` never wired into `main.ts`, so approval flashing didn't fire.
3. **Break 3** (added mid-Phase 1 by SC direction) ‚Äî `renderConversation` re-highlighted
   every sealed block on every paint with no cache; per-delta full re-highlight cost.

All three were already implemented by the Maker. The Courier opened the PR and found CI
clean.

---

## What grew in session

### Block lifecycle ‚Äî the core architectural fix

The deeper problem: the block model only had `transitionBlock` (seal old, open new). Blocks
were never explicitly closed ‚Äî they closed implicitly when the next block started. This meant:

- After a tools turn, the active block stayed as `tools`
- The next turn's first event sealed it by opening something else
- If two tool turns ran back-to-back, the second turn's `tool_use_start` saw active already
  `tools`, hit a noop, and piled into the first turn's visual block

**Root cause identified in session**: the SC pointed out that two consecutive tool turns
from separate HTTP requests are two separate tool blocks. The block model had no concept of
"this turn ends here." The noop check `from !== 'tools'` was a workaround for the wrong
model.

**Fix 1** ‚Äî `message_usage ‚Üí completeActive()`: seal whatever is active at the end of every
turn. This immediately fixed the cross-turn tool block leakage.

**Fix 2 ‚Äî enter_block/exit_block events**: the SC pushed for explicit block lifecycle. Added
`enter_block(rawType)` and `exit_block(rawType)` events to `StreamProcessor`, emitted from
`content_block_start` and `contentBlock` (the `content_block_stop` equivalent). `main.ts`
wires them to the channel. `AgentMessageHandler` uses them to open/close blocks explicitly;
`transitionBlock` in delta handlers is removed.

**Fix 3 ‚Äî tool_batch_start/tool_batch_end**: the SC identified a deeper mismatch. Multiple
`tool_use` content blocks within a single message are sequential (one closes before the next
opens), but they belong to the same visual tools batch. Using `enter_block('tool_use')` per
content block forced an `activeBlock?.type !== 'tools'` guard in `AgentMessageHandler` to
detect "same batch vs new batch." The SC correctly called this a leaky abstraction ‚Äî the CLI
was inferring batch boundaries from block state rather than the SDK knowing them.

Fix: `StreamProcessor` emits `tool_batch_start` when the first `tool_use`/`server_tool_use`
content block starts in a message, and `tool_batch_end` when `message_delta.stop_reason ===
'tool_use'` fires. The Messages API guarantees `stop_reason === 'tool_use'` when any tool
blocks are present ‚Äî the SC confirmed this as an API fact. `AgentMessageHandler` handles
`tool_batch_start` to open the visual tools block and reset state; the `!== 'tools'` guard
is removed entirely.

### ToolObject emits ‚Üí no more approval duplication

**Problem found**: after `message_usage ‚Üí completeActive()`, the tools block was sealed. Then
`tool_approval_request` called `transitionBlock('tools')` to open a new block for approval
display. This created two visual tools blocks: one sealed (showing streaming/pending states)
and one active (showing approval states). Both rendered as a continuation ‚Äî visible as
duplication on screen.

**SC's diagnosis**: "we don't need a tool block when the approvals come through ‚Äî you just
find the previous tools and update them." The tools block is sealed but still in memory,
still rendered in the alt buffer every frame. Updating it in place is correct.

**Fix**:
- Added `setLastContent(type, text)` to `ConversationState`: replaces content of the most
  recent block of the given type, active or sealed, in place. The alt buffer repaint picks
  up the change.
- `ToolObject` now emits `change` on every state mutation (`appendInput`, `resolve`,
  `approve`, `deny`, `error`, `complete`).
- `AgentMessageHandler` subscribes to each new `ToolObject`'s `change` event on creation;
  the subscription calls `#redrawTools()`.
- `#redrawTools()` now calls `setLastContent('tools', ...)` instead of
  `setActiveBlockContent(...)`. It works on whichever tools block is most recent ‚Äî active
  or sealed.
- `tool_approval_request` no longer calls `transitionBlock('tools')`. No new block opened.
  The existing tools block (sealed after `message_usage`) is updated in place.

**Key insight on "sealing"**: the SC corrected a misunderstanding. "Sealing" (moving a block
to `sealedBlocks`) does NOT mean permanent/immutable. The alt buffer renders ALL blocks in
memory every frame. `appendToLastSealed` already demonstrated this. The main scroll buffer
flush (`flushedCount`, `renderBlocksToString`) is a separate concern that can handle the
"what happens to already-flushed notices" problem later.

### notice block type

**Problem**: when content arrives with no active block (errors, stop annotations), it was
silently dropped by `appendStreaming`.

**Fix**: `appendStreaming` auto-opens a `notice` block when called with no active block.
`renderConversation` suppresses the divider header for `notice` type ‚Äî content renders
inline without a label. `notice` blocks also render without `CONTENT_INDENT` (added later
in session after SC observed the indentation in a screenshot).

### spliceNotice

**SC's insight**: a notice mid-stream should appear at the last `\n` boundary in the current
block, so streaming content resumes cleanly after it. Blocks are just flat strings with
literal `\n` for intentional line breaks; visual wrapping is a render-time concern. Finding
`lastIndexOf('\n')` and splicing at that position is all that's needed.

**Implementation**: `ConversationState.spliceNotice(text)`:
- No active block ‚Üí opens a `notice` block.
- Active block with `\n` ‚Üí inserts `text\n` after the last `\n`; streaming continues after.
- Active block with no `\n` yet ‚Üí appends `\ntext\n` at end; streaming continues after.

**SC context**: `spliceNotice` unblocks the permissions config change display feature, which
was blocked because there was no way to surface dynamic config notifications mid-stream
without corrupting the response content.

**Known limitation**: when spliced into a non-notice block (e.g. meta or response), the
notice text inherits the host block's `CONTENT_INDENT`. Fixing this requires marking
spliced lines with a sentinel byte that `renderBlockContent` recognises and renders with
zero indent. Not addressed in this session ‚Äî scoped to a follow-on.

## The `notice` block during tool use

The SC observed that spliced notices are overwritten when `#redrawTools()` fires, because
`setLastContent('tools', ...)` replaces the entire tools block content. If a notice was
spliced into the tools block content and then a tool state changes, the notice disappears.

Not addressed in this session. The correct fix would track notice insertions independently
of tool render state, or re-splice after each redraw. Deferred.
