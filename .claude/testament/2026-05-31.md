# 02:08

Apostle for #239, iteration 3 — follow-up plan for the new locked decision: streaming view ends with a newline at `content_block_stop` for each tool-input block.

## What the new requirement adds

One locked decision bullet appended to the v4 set: at `content_block_stop`, append a newline to the streaming view so consecutive tool calls in a client batch render on separate lines.

## Design

`StreamProcessor` already tracks tool-input block indices in `toolInputIndices`. At `content_block_stop` it currently deletes silently. The fix: check `has` before deleting — if the index was in the set, emit `tool_use_input_stop` first.

The event travels the same four-file pipeline as the v4 events: `private/types.ts` → `StreamProcessor.ts` → `public/types.ts` → `main.ts` → `AgentMessageHandler.ts`. `AgentMessageHandler` handles it with `appendStreaming('\n')`.

## Interaction with existing completion mechanics

For server tools: `tool_use_input_stop` fires, `\n` is appended briefly, then `replaceFromMark('🌐 summary')` wipes from the mark (set before the name) to end-of-content. The `\n` is inside the replaced range and disappears. No special handling needed.

For client batches: the `\n`s from each tool's `content_block_stop` accumulate in the streaming content, then `replaceFromMark('')` at the first `tool_approval_request` clears them along with everything else from the batch-start mark. No special handling needed.

## What I did not surface as a stop note

The `has`-before-delete pattern is the minimal correct gate — `content_block_stop` fires for all block types (thinking, text, result blocks), and only tool-input indices were ever added to the set. If the has-check were wrong (e.g., checking size instead of index membership), a second concurrent block could mask the gate. Index-specific check is unambiguous.

# 16:04

Apostle for #239, iteration 4 — SC refined the newline locked decision: client tool-input blocks only, not server tools.

## What changed from v5

The v5 plan emitted `tool_use_input_stop` whenever any tool-input block index was in `toolInputIndices` (which tracks both `tool_use` and `server_tool_use`). The SC's refinement: server tools do not need the newline because `replaceFromMark` overwrites their entire streaming content at the `server_tool_use` event. Emitting a newline for server tools was harmless (it would be inside the replaced range) but unnecessary.

## Design

The minimum change: add `clientToolInputIndices = new Set<number>()` alongside the existing `toolInputIndices`. Only `tool_use` blocks add to `clientToolInputIndices`. `server_tool_use` blocks continue adding only to `toolInputIndices` (for `inputJson` gating). At `content_block_stop`, the `tool_use_input_stop` emission gates on `clientToolInputIndices.has(event.index)`.

`toolInputIndices` is not renamed or restructured — its only job is gating the `inputJson` subscriber, and that gate remains unchanged. The two sets have distinct responsibilities.

## Why `has`-before-delete on `clientToolInputIndices`

`content_block_stop` fires for all block types. The membership check is the precise gate: only indices that were explicitly added to `clientToolInputIndices` (i.e., `tool_use` blocks) will pass. The unconditional `toolInputIndices.delete` at the end cleans up both kinds.

## Tests

Two StreamProcessor tests instead of v5's one:
1. Client: emits `tool_use_input_stop` when a `tool_use` block stops
2. Server: does NOT emit `tool_use_input_stop` when a `server_tool_use` block stops

The second test is the meaningful addition — it directly exercises the client-only gate. All required fixtures (`toolUseStart`, `toolUseStop`, `serverToolUseStartLocal`, `serverToolUseStopLocal`) are already declared in the `tool input streaming` describe block.

One new AgentMessageHandler test (same as v5): `appends a newline to the streaming content`.

## Interaction with v4 completion mechanics

Server tools: stop fires, `clientToolInputIndices.has` is false, no event, `replaceFromMark` at `server_tool_use` replaces content. Clean.

Client batches: stop fires per tool, `\n` appended via `tool_use_input_stop`, all newlines cleared by `replaceFromMark('')` at first `tool_approval_request`. Clean.
