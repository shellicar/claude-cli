## Decisions made

**Local types in schema.ts to avoid circular imports.** `OperationSchema` and `PipelineSchema` need `z.ZodType<T>` annotations for recursive Zod schemas. `T` would normally come from `types.ts`, but `types.ts` imports from `schema.ts`. Defining local `CommandType`, `OperationType`, `PipelineType` in `schema.ts` breaks the cycle. biome's `noImportCycles` would flag the circular case at error level. `types.ts` stays a clean one-way import.

**`input_examples` is required on `ToolDefinition`.** The V1 `Exec.ts` includes it; the mission stub omitted it. TypeScript caught the missing field during type-check. Added a minimal example to the stub.

**V1 R3 quirk discovered.** The catalog predicted `results[0].stdout == ""` for R3 (pipeline: `echo hello | cat > /dev/null`). The actual V1 behavior is `stdout == "hello"`. In `execPipeline.ts`, `lastChild.stdout.on('data', ...)` is registered unconditionally before the redirect is set up. Both the capture buffer and the file receive the data. This differs from `execCommand.ts` which conditionally adds the listener. The characterisation test was corrected to assert `stdout == "hello"` and the scenarios.md was updated to document the quirk.

## For the Builder (phase 2)

The V2 contract lives in `test/ExecV2/scenarios.spec.ts`. Make those 87 failing tests pass without regressing the 14 passing ones.

Key implementation points:
- `normaliseTree` needed (mirrors `normaliseInput` but walks the AST recursively instead of the steps array)
- `builtinRules` are imported from `src/Exec/builtinRules.ts` ŌĆö walk every leaf in the pipeline tree
- Result array order is tree pre-order (left subtree before right at every node) ŌĆö deterministic even for concurrent `&`
- Pipefail: aggregate success for `|` is false if any segment failed
- `&` (concurrent): run left and right in parallel via `Promise.all`, collect results in pre-order
- Pipe source result capture: Command on left of `|` has empty stdout in result (stdout flowed to the pipe)
- R4 and NE2 refinements: R4 rejects `redirect.stream == 'stdout' | 'both'` on a direct-left-of-pipe Command; NE2 rejects `stdin` on a direct-right-of-pipe Command. Both should surface as `Invalid input: [...]` via `z.refine`/`z.superRefine` on `PipelineSchema` or `OperationSchema`
- The R3 V1 quirk (capture not suppressed with redirect in pipeline) should NOT carry over to V2 ŌĆö V2 should suppress capture for redirected pipe consumers
- `stripAnsi` and `trimEnd` apply to all result entries

The `byId` helper in the V2 spec throws if the id is not found ŌĆö useful diagnostic when a result is unexpectedly absent.

## Addendum ŌĆö schema rewrite (Zod v4 get accessor pattern)

The prompt was updated mid-cast to replace `z.lazy()` + `z.ZodType<T>` with Zod v4's `get` accessor pattern for recursive fields. The updated spec showed:

```ts
export const OperationSchema = z.object({
  op: z.enum([...]),
  get left() { return PipelineSchema; },
  get right() { return PipelineSchema; },
}).strict();
```

The spec claimed "The accessors run lazily on property access, so the forward reference to PipelineSchema resolves correctly even though const is not hoisted." This is true at PARSE time but not at CONSTRUCTION time.

**The TDZ problem:** `.strict()` calls `this.shape` on the ZodObject, which iterates the shape properties and calls the getters immediately. At that point `PipelineSchema` is in the temporal dead zone (`const` is not hoisted, and `OperationSchema` is declared before `PipelineSchema`). The ReferenceError is: `Cannot access 'PipelineSchema' before initialization`.

**Fix applied:** Removed `.strict()` from `OperationSchema`. Without `.strict()`, Zod only accesses the shape properties at parse time, when all `const` schemas are fully initialised. The practical effect: extra unknown fields on an operation object pass through instead of being rejected. Acceptable for the stub; the Builder can add strict validation with a different mechanism if needed.

The local TypeScript types (`CommandType`, `OperationType`, `PipelineType`) were also removed. The `get` accessor pattern lets Zod infer the recursive types directly ŌĆö no annotation needed and no circular import risk.
