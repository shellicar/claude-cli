Apostle for #239, iteration 3 — follow-up plan for the new locked decision: streaming view ends with a newline at `content_block_stop` for each tool-input block.

## What the new requirement adds

One locked decision bullet appended to the v4 set: at `content_block_stop`, append a newline to the streaming view so consecutive tool calls in a client batch render on separate lines.

## Design

`StreamProcessor` already tracks tool-input block indices in `toolInputIndices`. At `content_block_stop` it currently deletes silently. The fix: check `has` before deleting — if the index was in the set, emit `tool_use_input_stop` first.

The event travels the same four-file pipeline as the v4 events: `private/types.ts` → `StreamProcessor.ts` → `public/types.ts` → `main.ts` → `AgentMessageHandler.ts`. `AgentMessageHandler` handles it with `appendStreaming('\n')`.

## Interaction with existing completion mechanics

For server tools: `tool_use_input_stop` fires, `\n` is appended briefly, then `replaceFromMark('🌐 summary')` wipes from the mark (set before the name) to end-of-content. The `\n` is inside the replaced range and disappears. No special handling needed.

For client batches: the `\n`s from each tool's `content_block_stop` accumulate in the streaming content, then `replaceFromMark('')` at the first `tool_approval_request` clears them along with everything else from the batch-start mark. No special handling needed.

## What I did not surface as a stop note

The `has`-before-delete pattern is the minimal correct gate — `content_block_stop` fires for all block types (thinking, text, result blocks), and only tool-input indices were ever added to the set. If the has-check were wrong (e.g., checking size instead of index membership), a second concurrent block could mask the gate. Index-specific check is unambiguous.

`content_block_stop` fires for all block types. The membership check is the precise gate: only indices that were explicitly added to `clientToolInputIndices` (i.e., `tool_use` blocks) will pass. The unconditional `toolInputIndices.delete` at the end cleans up both kinds.

# 17:37

Apostle for #239, iteration 5 — phase 3 blank-during-approval fix.

## The bug

`tool_approval_request` sync arm calls `replaceFromMark('')`, erasing all streaming content (tool names + partial JSON). The async `#toolApprovalRequest` then asks for approval. During the approval wait, the tools section is blank. Phase 3 ("resolved view visible while user approves") never fires.

## Why a simple move doesn't work for multi-tool batches

The obvious fix: move `replaceFromMark(summary)` to `#toolApprovalRequest` before `await requestApproval()`. This works for one tool. For two concurrent tools:
- Both async pre-`await` portions run before any approval
- Tool1 shows its resolved view, tool2 immediately appends its
- Tool1's ` ✅\n` then appends to the END — after tool2's view

With pure `appendStreaming` semantics, there's no way to insert the status mark at a specific interior position without range replacement.

## The chain approach

Each tool's async waits for the previous tool's FULL cycle (show → approve → mark) before running its own. The chain is threaded through a new field `#clientBatchDisplayChain: Promise<void>`. Each `tool_approval_request` passes `prevChain` to `#toolApprovalRequest` and stores the returned Promise as the next tool's prevChain.

Result: tool2's resolved view appears after tool1's ` ✅\n`, so `appendStreaming(summary2)` naturally lands at the right position. The locked decision is satisfied per-tool: each resolved view is visible while the user approves that specific tool.

## Key timing insight

`await Promise.resolve()` in a resolved-promise chain creates one microtask tick. Auto-approve/deny paths have one tick (just `await prevChain`). Manual paths have two ticks (`await prevChain` + `await requestApproval()`). Tests need this many `await Promise.resolve()` calls to observe results.

## Open question for SC

The plan flags concurrent display (all tools visible simultaneously before any approval) as an alternative reading of the locked decision. The chain gives sequential display (tool2 visible only after tool1 approved). If concurrent is the intent, range replacement is needed — a bigger change.

# 19:22

Apostle for #239, iteration 6 — re-plan after SC blocked v5 with two corrections.

## The two corrections

1. **Architecture.** Each tool use is a class. State (name, partial input, completion status, approval status, resolved view) and rendering live on the instance. State updates → re-render. The mark/set/gate machinery in v4 exists because the design mutates one shared block; per-tool classes make the lifecycle obvious and rendering local.

2. **Plan format.** The v5 plan used a full-method replacement block for `#toolApprovalRequest`. Most lines in that method are identical between before and after — diff was the right format. Replacement is for full rewrites where context lines would be noise; that was not the case here.

## What the v5 picked wrong

v5 picked sequential display (chain the approvals; tool2 only appears after tool1 is fully approved). The locked decision plus the existing `ToolApprovalState` ←→ navigation UI both imply concurrent display — multiple tools in `#pendingTools` simultaneously, user works through them. v5's chain breaks the navigation UI and contradicts the natural read. v5's stop note §1 surfaced the ambiguity but the plan body committed to one reading anyway. The agent doc says: if you are about to invent a value, stop. The reading was the invention.

## Why the per-tool class falls out clean

Concurrent display is hard with a flat mutable string because tool2's status mark has to land at the right interior position regardless of when other tools' content changes around it. With per-tool classes, `tool.approve()` flips a phase, the redraw recomputes the entire client-tool region from `tools.map(t => t.render()).join('')`, and every status mark naturally lands beside its own resolved view. No chains, no microtask choreography, no offset arithmetic.

The locked-decision newline migrates: instead of `tool_use_input_stop` emitting a literal `\n`, every non-streaming phase's `render()` ends with `\n`. Consecutive tools still appear on separate lines; the rendering layer no longer needs to know about line separators.

## What survives, with stop notes

`#streamingMark` and `#clientBatchStarted` survive. They anchor the boundary between content that is owned by something else (a preceding server tool in a mixed batch) and content owned by the client-tool batch. The redraw replaces from that mark, not offset 0. If the SC's intent was that the mark also goes — implying a partitioned block or structured-state rendering — the plan needs revisiting. Surfaced as stop notes §1 and §2 rather than picked silently.

## Plan format application

`#toolApprovalRequest` uses a diff block. Signature change at the top, two unchanged-line blocks (logging, addPendingTool, getPermission, channel.send, removePendingTool), two changed tails (the display composition replaced with `state?.approve()` / `state?.deny()` + redraw). Most lines are context; the diff carries them as context lines.

`tool_approval_request` test describe block uses replacement. Every test changes shape (assertions move from `appendStreaming` to `replaceFromMark`, realistic lifecycle requires `streamTool` helper). No shared structural anchors with the previous tests — context lines would add noise.

The smaller test describe blocks (`tool_use_start`, `tool_use_input_delta`, `tool_use_input_stop`) use diff. Only a handful of lines change in each.

- `renderModel` in `src/view/renderStatus.ts` was fully rewritten to call `parseModelName`. `abbreviateModel` was deleted. Status bar shows `Name 4.6` or just `Name` when no version.

## Key traps

**`as string[]` cast (Item 3).** `ReturnType<typeof parseArgs>` widens `values.file` so `Array.isArray` narrows to `(string | boolean)[]`, not `string[]`. The plan didn't enumerate this consequence. The cast is safe at runtime (`type: 'string', multiple: true` guarantees string elements) but it's there. Future operators touching `values.file` should expect the widened type.

**Two variables named `parsed` in `main.ts`.** `let parsed: ReturnType<typeof parseArgs>` at the parseArgs site, and `const parsed = z.string().uuid().safeParse(resumeId)` inside the UUID check block. They shadow cleanly today. If the UUID check is ever lifted out of its block, the name collision becomes an error.

**IIFE vs guarded block.** An early plan draft used an IIFE for UUID validation (needed branching + a return value). The local idiom is one-liner extraction + guarded block at the same scope. When you see conditional logic that produces one value and needs to exit on bad input, reach for the guarded-block pattern, not a function wrapper.

**Duplication trap.** The Apostle's first draft had `resume()` duplicating the history-load fragment of `load()`. The signal to catch: prose that says a new method "mirrors" or "parallels" an existing one. Mirroring exactly is duplication. The correct shape is a shared private helper called from both.
