## Decisions made

**Local types in schema.ts to avoid circular imports.** `OperationSchema` and `PipelineSchema` need `z.ZodType<T>` annotations for recursive Zod schemas. `T` would normally come from `types.ts`, but `types.ts` imports from `schema.ts`. Defining local `CommandType`, `OperationType`, `PipelineType` in `schema.ts` breaks the cycle. biome's `noImportCycles` would flag the circular case at error level. `types.ts` stays a clean one-way import.

**`input_examples` is required on `ToolDefinition`.** The V1 `Exec.ts` includes it; the mission stub omitted it. TypeScript caught the missing field during type-check. Added a minimal example to the stub.

**V1 R3 quirk discovered.** The catalog predicted `results[0].stdout == ""` for R3 (pipeline: `echo hello | cat > /dev/null`). The actual V1 behavior is `stdout == "hello"`. In `execPipeline.ts`, `lastChild.stdout.on('data', ...)` is registered unconditionally before the redirect is set up. Both the capture buffer and the file receive the data. This differs from `execCommand.ts` which conditionally adds the listener. The characterisation test was corrected to assert `stdout == "hello"` and the scenarios.md was updated to document the quirk.

## Addendum — schema rewrite (Zod v4 get accessor pattern)

The prompt was updated mid-cast to replace `z.lazy()` + `z.ZodType<T>` with Zod v4's `get` accessor pattern for recursive fields. The updated spec showed:

```ts
export const OperationSchema = z.object({
  op: z.enum([...]),
  get left() { return PipelineSchema; },
  get right() { return PipelineSchema; },
}).strict();
```

The spec claimed "The accessors run lazily on property access, so the forward reference to PipelineSchema resolves correctly even though const is not hoisted." This is true at PARSE time but not at CONSTRUCTION time.

**The TDZ problem:** `.strict()` calls `this.shape` on the ZodObject, which iterates the shape properties and calls the getters immediately. At that point `PipelineSchema` is in the temporal dead zone (`const` is not hoisted, and `OperationSchema` is declared before `PipelineSchema`). The ReferenceError is: `Cannot access 'PipelineSchema' before initialization`.

**Fix applied:** Removed `.strict()` from `OperationSchema`. Without `.strict()`, Zod only accesses the shape properties at parse time, when all `const` schemas are fully initialised. The practical effect: extra unknown fields on an operation object pass through instead of being rejected. Acceptable for the stub; the Builder can add strict validation with a different mechanism if needed.

The local TypeScript types (`CommandType`, `OperationType`, `PipelineType`) were also removed. The `get` accessor pattern lets Zod infer the recursive types directly — no annotation needed and no circular import risk.

## How the pipe coordinator handles the four side-shape combinations

The mission listed four combinations (Cmd→Cmd, Op→Cmd, Cmd→Op, Op→Op). The current test catalog only exercises the first three. The implementation handles them via three mutually recursive variants of the executor — each variant differs only in what is overridden on the spawned child's stdio:

- **`executeTree`** — standalone: stdin from `cmd.stdin` literal, stdout captured.
- **`executeAsSource(pipeline, sink)`** — left side of a pipe: stdin from literal (or nothing), stdout pipes into `sink` with `{ end: false }`, no capture.
- **`executeAsConsumer(pipeline, stdinStream)`** — right side of a pipe (when the right is a Command): stdin from `stdinStream`, stdout captured normally.
- **`executeAsBridge(pipeline, pipeIn, pipeOut)`** — middle of a chain (R5: `(A|B)|C` — B): stdin from `pipeIn`, stdout pipes into `pipeOut`, no capture.

`executePipe(left, right)` is the orchestrator: spawn the right side first via `executeAsConsumer` so it is reading by the time `executeAsSource(left)` starts writing. After the left side resolves, `pt.end()` signals EOF to the right.

The `executeAsSource` variant has its own `|` branch — that handles the nested-pipe-as-source case (R5's `(A|B)` as the source for outer `C`). It mirrors `executePipe` but uses `executeAsBridge` for the right side instead of `executeAsConsumer`, because the right side of the inner pipe must forward its bytes to the *outer* sink.

`executeAsConsumer` and `executeAsBridge` only support a Command on that side. An Operation in the consumer/bridge position would need stdin threaded to the first-executed leaf within the subtree — no catalog scenario requires it, so the implementation throws with a clear message naming the unsupported case.

## Where R4 and NE2 landed

Both refinements live in `src/ExecV2/schema.ts` as a single `.superRefine()` on `OperationSchema`. They:

- check `val.op === '|'` and bail otherwise;
- inspect `val.left` / `val.right` for the `program` property to recognise a direct Command leaf (vs a group); and
- `ctx.addIssue({ code: 'custom', path: [...], message: '...' })` with messages that contain both `redirect` and `pipe` (R4) or `stdin` and `pipe` (NE2).

`.superRefine()` is safe with the recursive getter pattern that `.strict()` broke in phase 1 — `.superRefine()` wraps the schema in a `ZodEffects` that stores the function without inspecting the shape, so the getters are not triggered at construction time.

## Aggregate-exit semantics — the surprise that took the run from 99/101 to 101/101

The mission instruction says:

> 6. Aggregate `success = results.every(r => r.exitCode === 0)`.

This is **wrong for `||`**. O2 (`false || echo b`) and M1 (`(false && a) || b`) keep the failing left leaf in `results`, so a per-leaf reduction reports `success: false` while the catalog and the tests demand `success: true` (the right side handled the failure). The correct rule is `success = aggregateExit === 0`, where the aggregate flows from the operator's semantics.

The aggregate-exit rules across operators:

- **`&&`** — left's exit when left failed (right is skipped), else right's exit. Right's failure surfaces unchanged.
- **`||`** — left's exit (0) when left succeeded (right is skipped), else right's exit. Right's success "rescues" the aggregate.
- **`;`** — either side failing makes the whole non-zero. **This diverges from shell `$?` semantics**, which would be right's exit alone. The test catalog (C2: `false; echo b` → `success: false`) requires the unguarded combine. `combineUnguarded(leftExit, rightExit)` codifies this — left's exit if it failed, else right's.
- **`&`** — concurrent unguarded combine. Same `combineUnguarded` helper.
- **`|`** — pipefail. Aggregate non-zero if any leaf in the chain failed. My first attempt used `anyFailed ? rightExit ?? 1 : 0`, which **collapses to `0` when right exits zero** even though an upstream leaf failed (P3, ER3). Fixed to `anyFailed ? 1 : rightExit`.

`&&` and `||` are the only "guarded" operators — the structure itself handles the failure, so the failure doesn't propagate to `success`. Everything else is "unguarded" — any failure propagates.

## Behaviour discovered that the catalog didn't predict

- **Result-array order for short-circuited `&&` / `||`.** When the right branch is skipped, the skipped leaf has no result entry at all (not a synthetic one). This is consistent with the test expectations (A2 expects one result, O1 expects one result). The implementation falls out naturally — if we don't execute the branch, we don't add results for it.
- **`combineUnguarded` for `&` with one failure.** N2 (`false & true`) expects `success: false`. The shell-style "0 iff both 0" works for `&`, but using `combineUnguarded` (left's exit if non-zero, else right's) gives the same answer with a more diagnostic exit code (1, not a synthetic 1). Unified for symmetry with `;`.
- **`merge_stderr` in pipe-source mode (F2).** When `cmd.merge_stderr` is true and we are in pipe-source mode, both `child.stdout` and `child.stderr` pipe into the sink with `{ end: false }`. The sink stays open across both streams ending — the pipe coordinator's `pt.end()` after the left resolves is what closes it for the consumer. F2's `cat` receives both `out` and `err` interleaved (order is OS-dependent, but the test only checks containment).
- **ENOENT in a pipe (ER3).** When the left command fails with ENOENT, `child.stdout.pipe(sink, { end: false })` is set up but no data ever flows. The error handler resolves the result with `exitCode: 127`. When `pt.end()` then fires, the right side (cat) reads EOF and exits 0. No special handling needed; the natural flow produces the expected `[a:127, b:0]` results.

## Notes for downstream phases

- The handler in `ExecV2.ts` includes a `leavesAsV1` cast to satisfy `builtinRules[i].check`'s V1 Command parameter type. The cast is safe — V2 Command is a structural superset of V1 Command (adds `id`), so it satisfies every property the rules read. If a future change touches the rule signature, this cast may need revisiting.
- `executeAsConsumer` and `executeAsBridge` throw on Operations. The catalog doesn't exercise that path, but if a future scenario puts a group on the right of `|` (e.g. `a | (b; c)`), the implementation needs an "execute with stdin threaded to first-executed leaf" path. Pattern: walk the subtree to find the first leaf (pre-order), give it the stdin, execute the rest of the subtree normally.
- `.strict()` is still off `OperationSchema` (TDZ issue from phase 1). The `.superRefine()` does not bring it back. Extra unknown keys on operation nodes still pass through silently. If strict validation of unknown keys becomes important, a `.superRefine()` that checks `Object.keys(val)` against `['op', 'left', 'right']` would close the gap without re-triggering the TDZ.
