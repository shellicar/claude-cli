## Markdown render in the TUI — shipped (mission 2026-06-29_markdown-render)

Assistant `response` blocks now render as styled ANSI markdown in the terminal, opt-out
via config. Everything else (tool output, thinking, notices) is untouched.

### How it works

The render path has one choke point for sealed block content: `renderBlockContent`
(`view/renderConversation.ts`). It now branches — `markdown` on → `markdownContentLines`
(`model/markdown/markdownLayout.ts`), off → the existing `blockContentLines`. The gate is
`markdown.enabled && block.type === 'response'` (`blockRendersMarkdown`), so only assistant
prose is ever rendered; the fence splitter (`CODE_FENCE_RE` in `blockLayout.ts`) still serves
every other block type. Making markdown response-only rather than global is deliberate — a
global swap would have regressed fence highlighting in tool/thinking output.

Two files carry the feature:

- `model/markdown/markdownLayout.ts` — `marked.lexer` token walk, emitting a line array the
  wrapper can measure. It mirrors `blockContentLines`' **count-preserving `decorate` contract**
  (one output line per code line) so height is predictable and any plain-measurement path agrees
  with the highlighted render. `cli-highlight` is injected from the view as `getHighlighted`, not
  imported here — this layer stays pure.
- `model/markdown/palette.ts` — pure ANSI strings plus `box()` (boxed code block) and `link()`
  (OSC 8 hyperlink). Codes are copied verbatim from the mission's visual spec (`spec/spec.mjs`);
  that rendered output *is* the contract, which is why this owns its palette instead of using
  claude-core's ansi constants — **the spec's `dim` is `\x1b[90m` (bright-black fg), NOT
  claude-core `DIM` `\x1b[2m`.**

Config: `markdownSchema` in `cli-config/schema.ts` — `{ enabled, streaming }`, both default-on
opt-outs. `MarkdownConfig` in `types.ts`. `ConfigLoader` was added to the ViewModel + container
bag so the pure render fns read live flags; `PrimaryView` reads `configLoader.config.markdown`;
`main.ts` threads it into `flushSealedToScroll`. Hot-reload of the toggle works because the sealed
cache (`renderBlockContentCached`) keys on the `markdown` bool too.

### Decisions worth keeping

- **Streaming has its own flag** (`markdown.streaming`, gated under `enabled`). No special seam
  was needed: Primary repaints from state every frame, so a sealed block auto-re-renders once
  complete. The active streaming block keeps its `📝 ` emoji by swapping the first markdown line's
  indent for the emoji (`markdownContentLines` indents every line) — `mdLines[0].slice(activeIndent.length)`.
- **Scope for v1 was Primary/response only.** History and thinking markdown were deferred (SC
  decision on the plan's open flag). The layout lives in `model/` so a follow-up can reach it from
  the History measurement path without moving code.
- Tables and task lists fall through `marked`'s `default:` case as `t.raw` verbatim — matches the
  spec's "out of scope, passed through" cases exactly, no special handling.

### Traps

- **`marked`'s `space` token must emit one blank line (`''`), not `break`.** The boundary spec case
  (heading / blank / code box) needs the blank preserved; dropping it collapses the layout.
- **`box()` border width is label-aware: `innerW - 1 - labelWidth`, not a fixed `inner - 5`.** The
  fixed form only aligns for a 2-char label (`ts`, `md`) and overshot on longer labels like
  `plaintext`. This was an SC live-testing defect, not caught by the first test pass — the initial
  boxes looked right precisely because the two formulas coincide at label width 2.
- **Code boxes cap and wrap to terminal width.** `box` takes `termWidth` (the `code` handler passes
  the post-indent `cols`), caps inner width to `termWidth - 4`, and wraps each body line so long
  lines are seen, not clipped off the right edge. It wraps with `wrapLine` (claude-core reflow),
  **not** the spec's `hardWrap` — real code bodies carry SGR escapes and `hardWrap` byte-slices,
  which would split an escape. For plain ASCII the two produce identical splits, so the spec's
  plaintext long-line case still matches byte-for-byte. The unit test pins `box`'s exact bytes by
  hand rather than comparing `box` against itself, so it catches a formula regression instead of
  being a tautology.
- **`marked` is on `^18`;** `pnpm add marked` did not pull vite v8 into the lockfile (vite stayed
  `^7`, which the decorator toolchain requires — see project CLAUDE.md).
