## Phase 2 Maker, iteration 2 — code-box cap/wrap + label-aware border (mission 2026-06-29_markdown-render)

Iteration 1 shipped as `032cafe`. SC live testing found two code-block defects; the spec (`spec/spec.mjs`) was corrected to encode both. Fixed the renderer to match, re-verified, re-staged. All green: `pnpm type-check`, `pnpm build`, `pnpm test` (944 pass), `pnpm ci:fix` clean.

Both defects lived in one place — `box()` in `model/markdown/palette.ts`:

- **Label-aware border.** Old header dash count was `inner - 5`, which only aligns for a 2-char label; `plaintext` overshot. New count is `innerW - 1 - labelWidth`, matching the spec's `box()`. (For a 2-char label the two formulas coincide, which is why iteration 1's `ts`/`md` boxes looked right.)
- **Cap and wrap.** `box` now takes `termWidth` (default 80), caps the inner width to `termWidth - 4`, and wraps each body line to that width so a long line is seen rather than clipping off the right edge. The `code` handler in `markdownLayout.ts` passes `cols` (the width available after indent) as `termWidth`.
- **ANSI-aware wrap.** The spec's `hardWrap` byte-slices plain text for the demo only; real code bodies carry SGR escapes. `box` wraps with `wrapLine` (from claude-core reflow) instead — breaks on visible width, never splits an escape, adds no visible characters. For plain ASCII `wrapLine` and `hardWrap` produce identical splits, so the spec's plaintext long-line case matches byte-for-byte.

**Verification worth keeping.** Cross-checked my `box` formula against the spec's `box`+`hardWrap` in a throwaway node script (both copied in): identical output for the long-line (56), the `ts` box (80), and a trivial case. The long-line box is 4 lines all 56 wide (top, two wrapped body lines, bottom). The concrete `box` unit test (`box(['abcdefgh'], 'ts', 10)`) pins the exact bytes by hand — not via `box` on both sides — so it guards against a tautology and catches a formula regression.
