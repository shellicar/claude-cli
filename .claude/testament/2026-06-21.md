### What the mission assumed vs what is true

The mission said: "Add `better-sqlite3` to pnpm's `onlyBuiltDependencies`, reinstall." Two things wrong with that on this repo:

1. **`onlyBuiltDependencies` does not exist in pnpm 11.** pnpm 11 removed it (and `neverBuiltDependencies`, `ignoredBuiltDependencies`, `ignoreDepScripts`) and replaced them with `allowBuilds` (a map: `pkg: true|false`). The repo runs pnpm@11.8.0 (`packageManager` in root package.json). An `onlyBuiltDependencies` entry is silently ignored. This is why my first `pnpm rebuild` attempts were silent no-ops — better-sqlite3 was never actually approved.

2. **`ignoreScripts: true` (in pnpm-workspace.yaml) defeats the allowlist entirely.** Even with `allowBuilds: { better-sqlite3: true }`, a fresh install builds NO binding and emits no error. `ignoreScripts: true` is the blunt "no dependency build scripts at all" switch; the allowlist is inert underneath it.

### Verified behaviour matrix (all via fresh installs, moving node_modules aside)

- `ignoreScripts: true` + `allowBuilds: {better-sqlite3: true}` → install OK, **no binding built**.
- `ignoreScripts: true`, no allowBuilds → install **fails** `ERR_PNPM_IGNORED_BUILDS: better-sqlite3`.
- `ignoreScripts: false` + `allowBuilds: {better-sqlite3: true}` → **binding builds from source** (full node-gyp compile; no prebuilt exists for node 24.13.1 arm64), but install **fails** `ERR_PNPM_IGNORED_BUILDS: esbuild, lefthook` — pnpm 11 hard-errors on any build-needing dep not in the allowlist.

The native build itself is fine: `npm run install` inside the package dir compiles cleanly (4 benign warnings) and produces `build/Release/better_sqlite3.node`. The toolchain was never the problem.

## Phase 4 Maker — RESUMED and COMPLETED after SC decision

The SC resolved the blocker: `ignoreScripts: true` was the posture for "we have no built dependencies." Now that we have one (better-sqlite3), it goes. Constraint added: do NOT allow esbuild or lefthook builds.

Landed config (`pnpm-workspace.yaml`):

```yaml
ignoreScripts: false
allowBuilds:
  better-sqlite3: true
  esbuild: false      # explicit deny — silences ERR_PNPM_IGNORED_BUILDS without building
  lefthook: false     # explicit deny
```

Key pnpm-11 facts that cost time, recorded so the next person doesn't re-derive them:
- `onlyBuiltDependencies` is REMOVED in pnpm 11. Use `allowBuilds` (a map `pkg: true|false`). An `onlyBuiltDependencies` entry is silently ignored — `pnpm rebuild` becomes a silent no-op.
- `ignoreScripts: true` is a blanket override: it suppresses ALL dependency build scripts regardless of `allowBuilds`. You cannot keep it and build a native dep.
- With `ignoreScripts: false`, pnpm 11 HARD-ERRORS (`ERR_PNPM_IGNORED_BUILDS`) on any build-needing dep not in `allowBuilds`. Setting a package to `false` is the documented way to say "decided: don't build" and clears the error. esbuild and lefthook both need an entry (true or false) or install fails.
- pnpm rewrites `pnpm-workspace.yaml` on a partial/failed install (appends placeholder `allowBuilds` entries, collapses comment whitespace). Restore exact bytes with `git show HEAD:pnpm-workspace.yaml > pnpm-workspace.yaml`.
- pnpm caches build side-effects in its store, so a warm-cache install reuses the compiled `.node` without recompiling (instant, no gyp output). CI on a cold runner recompiles from source — no prebuilt binary exists for node 24.13.1 arm64.

Other changes:
- `SqliteObjectStore`: added `this.#db.pragma('busy_timeout = 5000')` beside the WAL/synchronous pragmas, and a `get db()` accessor so the test can read connection-level pragmas (busy_timeout is not persisted to the file, so the test needs the store's own connection).
- New `apps/claude-sdk-cli/test/SqliteObjectStore.spec.ts`: exercises the REAL on-disk store (temp dir, cleaned up). Construction proves the binding resolves (the guard that was missing); set/get round-trip; missing id → undefined; busy_timeout reads back 5000.

Verified: `pnpm type-check`, `pnpm test`, `pnpm build` all green; the 4 real-store tests pass; `better-sqlite3` loads in plain node from the app context with `busy_timeout=5000`. Staged, not committed — left the commit for the SC.
