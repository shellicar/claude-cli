# 09:08 — HistoryView rebuilt to the box model (Navigation rewrite)

The mission's `## Navigation` was rewritten: the one-line-row outline is replaced by a **box model**. Every block is a box (header divider + content); navigating, each box is collapsed and capped at a few lines (`...` marks more); the focused box is **centred** on the screen with neighbours bleeding off the top/bottom edges. `[open]` grows the focused box to its full content bounded by the screen; content taller than the screen scrolls inside a fixed `~~~` frame (header + frame stay, content slides by scrollOffset). Tools nest the same model. Only `HistoryView.render` changed — `HistoryViewState`, `historyKeyMap`, the handlers, and the actions are all unchanged (the state already carries focus path + contentOpen + scrollOffset, which is exactly what the box model consumes).

How the rebuild works (`view/HistoryView.ts`):
- `#blockCard` builds each block's lines. Focused+collapsed: `> ` gutter on every line, content capped via `#cap`. Focused+open: no gutter, `(open)` header, content via `#contentBox`, closing divider. Unfocused: flush, collapsed.
- `#centre` flattens all cards into one stack and slices a screen-height window so the focused card's midline sits on the screen midline; a clipped edge shows `...`. A stack shorter than the screen is shown whole.
- `#contentBox` is the shared fits-vs-frame helper: returns the lines whole when they fit the budget, else `[~~~, ...slice(offset), ~~~]` sliced by the clamped scrollOffset. Reused for block content and for an opened tool's input/output.
- The footer view bar is the last row; the body fills `rows - 1`, so `render` returns exactly `terminalState.rows` lines (full screen).

Decisions I made (the brief said build to the illustrations, don't reinterpret — these are realisations, not reinterpretations):
- **Gutter only while collapsed.** The collapsed-focused illustration shows `> ` on the focused box; the open illustrations show no gutter (the `(open)` marker carries focus). So `> ` is applied to focused+collapsed boxes and to the focused tool row, not to open boxes.
- **Open tool i/o as a labelled box, not the inline `input  { ... }` of the illustration.** I render `input` / `output` label lines with the value indented, run through `#contentBox`. Reason: the illustration's inline form drops scroll when the output is tall, which violates the explicit "tool i/o follows the box model — scrolled in a fixed frame when taller than the screen" rule. The box form preserves scroll; the inline form cannot. This is the one spot where faithfulness to the *model* (scrollable) outranks the literal inline glyphs.
- **`COLLAPSED_CONTENT_LINES = 6`** for the `~8 lines` cap (header + up to 6 content ≈ 7–8). Tunable constant.

The spec (`test/HistoryView.spec.ts`) was rewritten to the box model with exact-value `toContain` assertions (gutter line, `...` cap line, flush unfocused line, post-open revealed line, descended tool row, tool input/output lines), plus the footer-row and full-screen-height assertions. Still one assertion per test, no boolean folds (the tdd lesson from the prior iteration holds). Gates: `tsc` clean, `pnpm test` 9/9, `pnpm build` 5/5.

# 09:36 — Navigation Rules block + new keys

The mission's Navigation gained a **Rules** block and four keys (PgUp/PgDn/Home/End). Addressed all five points the Handler relayed:

1. **`...` only when there's more.** The collapsed cap (`#cap`) already conditioned on overflow; confirmed. The open-content window now adds a `...` line only when content is hidden in that direction (top marker when `offset > 0`, bottom marker when `offset + budget < length`).
2. **`~~~` never renders.** The `#contentBox` `~~~` bars are gone, replaced by the conditional `...` markers above. Pinned by a test asserting no rendered row contains `~`.
3. **Entry focuses the latest block; no state across exits.** `HistoryViewState.enterAtLatest(blockCount)` focuses the last block, folded. `ViewSelectHandler` calls it on F2 **only when entering from outside history** (re-pressing F2 in-history leaves focus alone). So each entry starts at the bottom; nothing is kept across an exit.
4. **Clamped.** `#move` and the new `#jump` (home/end) clamp at the range ends and no-op (no emit) at a boundary. Page actions reuse `#move` with a larger delta, same clamp. The focus index never runs past first/last.
5. **PgUp/PgDn + Home/End.** New `HistoryAction`s (`page-up/page-down/home/end`), mapped in `historyKeyMap` (mode-agnostic mapping; the action is mode-aware in `apply` — jump blocks on a list, slide content when open). Key layout matches the mission table.

The hard part was the **content scroll bottom-clamp**. The bound is the rendered (wrapped) content height minus the visible budget, which only the view's `renderBlockContent` knows. Rather than duplicate or store a sentinel, I extracted `view/historyContent.ts` (`historyOpenLines`, `historyContentBudget`) shared by the view (renders) and `HistoryNavHandler` (measures). The handler computes `maxScroll` and passes it to `apply`; the state clamps scroll to `[0, maxScroll]`. So Home/End/PgDn land exactly at the box edge — no dead-zone, no over-scroll past wrapped content. `apply`'s `maxScroll` defaults to `+Infinity` so list-action callers and the existing scroll unit tests need not supply it.

New deps wired in `container.ts`: `ViewSelectHandler(appModeState, historyViewState, conversationState)`, `HistoryNavHandler(historyViewState, conversationState, terminalState)`.

One realisation kept from the prior round: an opened tool's i/o renders as a labelled box (`input`/`output` label + indented value through the same window helper), not the illustration's inline `input { ... }` — because the inline form can't scroll when the output is tall, which the Rules require. `historyOpenLines` is the single source for both rendering and measuring it.
