## Making the SQLite persistence actually run (PR #359, Phase 4 + 5)

#359 shipped the persistence feature green, but the built CLI threw `Could not locate the bindings file` at startup: better-sqlite3's native binding was never installed, and the suite only ever exercised the in-memory `MemoryObjectStore`, so a store that never loaded passed CI. Three things fixed it — the binding install, a `busy_timeout` pragma, and a test of the *real* store.

### pnpm 11 and native build scripts (the install fix)

The binding wasn't installing because pnpm blocks dependency build scripts. The repo's posture was `ignoreScripts: true` in `pnpm-workspace.yaml` — the right call when there are *no* built dependencies, wrong once better-sqlite3 (which compiles a `.node`) is one. The SC's decision: drop `ignoreScripts`, allowlist the one dep, explicitly deny the rest.

Landed config:

```yaml
ignoreScripts: false
allowBuilds:
  better-sqlite3: true
  esbuild: false      # explicit deny
  lefthook: false     # explicit deny
```

pnpm-11 facts worth not re-deriving:

- `onlyBuiltDependencies` is **removed** in pnpm 11. It's replaced by `allowBuilds` (a `pkg: true|false` map). An `onlyBuiltDependencies` entry is silently ignored, and `pnpm rebuild` becomes a silent no-op.
- `ignoreScripts: true` is a blanket override — it suppresses all dependency build scripts regardless of `allowBuilds`. You can't keep it and build a native dep.
- With `ignoreScripts: false`, pnpm **hard-errors** (`ERR_PNPM_IGNORED_BUILDS`) on any build-needing dep not in `allowBuilds`. Setting a package to `false` is the documented "decided: don't build" that clears the error. esbuild and lefthook each need an entry (true or false) or install fails.
- No prebuilt better-sqlite3 binary exists for node 24 arm64, so a cold CI runner compiles from source (node-gyp). pnpm caches the built `.node` in its store, so a warm local install reuses it instantly with no gyp output — don't mistake that silence for "didn't build".
- pnpm rewrites `pnpm-workspace.yaml` on a partial/failed install (appends placeholder `allowBuilds` entries, collapses comment whitespace). Restore exact bytes with `git show HEAD:pnpm-workspace.yaml > pnpm-workspace.yaml`.

### busy_timeout

`SqliteObjectStore` sets `busy_timeout = 5000` beside the WAL/synchronous pragmas. The store is machine-wide, so two CLIs running at once are two concurrent writers; without the timeout the second gets `SQLITE_BUSY` and throws immediately. 5s wait-for-lock instead. No fallback or caching layer — SQLite errors surface to Claude as ordinary tool errors, by the SC's call; revisit only if failures prove frequent.

### The Database is injected, not constructed (SC correction)

The first cut gave `SqliteObjectStore` a `db` getter exposing the raw connection so the test could read `busy_timeout` back — production API widened for a test, punched through the two-method interface. The SC rejected it and fixed it at the root:

- `SqliteObjectStore`'s constructor takes a better-sqlite3 `Database`, not a path. It *configures* the injected connection (WAL / synchronous / busy_timeout pragmas, creates the `objects` table) and uses it. No `path`, no `mkdirSync`, no `new Database`, no getter. The better-sqlite3 import is now `import type`.
- `setup/container.ts` is the creation site: it builds the `~/.claude/persistence.db` path, runs `mkdirSync(dirname(path), { recursive: true })` (the directory must exist before `new Database`), news up the connection, and injects it. `createAppTools` only threads `IObjectStore` and never constructed the store, so the container was the sole site to move.
- The real-store test owns its own temp-file `Database` per test and reads `busy_timeout` back from that handle directly. `MemoryObjectStore` and `IObjectStore` are untouched.

### The test that should have existed

`apps/claude-sdk-cli/test/SqliteObjectStore.spec.ts` exercises the concrete on-disk store (temp dir, cleaned up in `afterAll`): construction proves the binding resolves (the guard that was missing), `set`/`get` round-trips, missing id → `undefined`, and `busy_timeout` reads back `5000`. This deliberately reverses the plan's "SqliteObjectStore is not unit-tested" decision — that omission is exactly what let a non-loading store ship green.

### Caching traps when verifying

- turbo returns `FULL TURBO` (cache hit) for type-check/build even after a source change, because it's content-keyed and the working tree equalled the index. Force a fresh run (`pnpm type-check --force`, or a direct `pnpm exec tsc --noEmit`) when you need to trust the result against staged edits.
- The `TsDiagnostics` language-service tool reported a stale constructor error on `container.ts` after the source was already changed. A direct `tsc --noEmit` was clean — trust tsc over the cached language service when they disagree.

## SEA runtime/execution split POC (sqlite-portability mission, Phase 1 Maker)

Built a minimal standalone SEA under `sea-poc/` to settle the one assumption the
SQLite-portability direction rests on. **Result: the split holds, both A and B PASS.**
A SEA bundled on Node 24 runs `node:sqlite` (v24.17.0 internally) while a spawned
`node -v` reports the shell's v20.20.2. Evidence and the working recipe are in
`sea-poc/FINDINGS.md` — read that first; it's the inheritable artefact.

### Things that cost time / would cost the next person time

- **Node 20 IS installed** (fnm v20.20.2), despite the first `fnm list` appearing
  to show only v22/v24 — that initial output was truncated. `fnm exec --using=20`
  works. Don't go installing a runtime; check with grep first.
- **`node:sqlite` is unflagged on Node 24.17** but flagged (`--experimental-sqlite`)
  on Node 22. This is the crux of why bundled-24 is the clean target: a SEA can't
  easily pass `node` flags to its own runtime.
- **macOS signing is load-bearing, not ceremony.** postject injection invalidates
  the signature; the strip-signature → inject → ad-hoc-sign bracket is mandatory or
  the OS kills the binary on launch. Order matters.
- **postject** isn't local; `npx -y postject` fetches it (needs network at build).

### Scope discipline

This is a throwaway spike, deliberately NOT touching the real CLI store code (the
mission was explicit). The `sea-poc/` dir is self-contained: `app.cjs`, `sea-config.json`,
`build.sh`, `FINDINGS.md`, plus a `.gitignore` keeping the ~100MB binary and blob
out of git. Only source + findings staged; build artefacts ignored.

## Swap claude-sdk-cli SQLite store: better-sqlite3 -> node:sqlite

Did the binding swap. Store now uses `node:sqlite`'s `DatabaseSync`; `better-sqlite3`
is gone from the dependency tree. Behaviour preserved exactly — same schema, queries,
pragmas, WAL/NORMAL/busy_timeout=5000.

### Files changed (mine)
- `src/persistence/SqliteObjectStore.ts` — `DatabaseSync`/`StatementSync` types; the
  three `.pragma('x = y')` calls become `.exec('PRAGMA x = y')` (node:sqlite has no
  `.pragma()` helper). `.exec`/`.prepare`/`.run`/`.get` map across unchanged.
- `src/setup/container.ts` — DI token `Database` -> `DatabaseSync`, construction
  `new Database(path)` -> `new DatabaseSync(path)`. mkdirSync stays (DatabaseSync also
  can't create in a missing dir).
- `test/SqliteObjectStore.spec.ts` — `DatabaseSync`; busy_timeout read-back is now
  `db.prepare('PRAGMA busy_timeout').get().timeout` (column name is `timeout`).
- `package.json` + `pnpm-lock.yaml` — `pnpm remove better-sqlite3 @types/better-sqlite3`.

### Verified
type-check, build, 4 store unit tests all green. Built `dist/main.js`: grep shows 0
better-sqlite3/bindings/NODE_MODULE_VERSION refs, 1 node:sqlite ref. CLI `--help`
launches under node 24, exit 0.

### THE WALL (surfaced to SC, not worked around)
The directive's final proof — feed `dist/main.js` to the SEA recipe, run from a node-20
shell — can't be done as-is. `dist/main.js` is **ESM** with `packages: 'external'`
(esbuild `format:'esm'`, bare `@shellicar/*` imports resolved from node_modules at
runtime). A Node SEA needs a **self-contained CommonJS** entry: ESM main isn't accepted,
and external packages aren't in the blob. Making it SEA-able needs `format:'cjs'` +
`packages:'bundle'` in `build.ts` — a build redesign.

Crucially: **`build.ts` is being edited by another session right now** (unstaged
`splitting:true->false`, a single-file SEA step). That's the SEA-readiness work in flight.
I did NOT touch build.ts (co-working) and did NOT stage it. The store swap is independent
of and unblocked by that work; the full-CLI-SEA proof waits on it.

Also note: `dist/main.js` now imports `node:sqlite`, which **does not exist on node 20**.
So the CLI can only run on node 24+ — which is the whole point of bundling node 24 into
the SEA. You cannot run the bundle directly under the shell's node 20; it must be the
embedded 24. (Phase-1 FINDINGS already proved node:sqlite works inside a real SEA.)

## Real CLI runs as an ESM SEA — the wall is cleared

The prior cast hit "THE WALL": dist/main.js is ESM with top-level await, and SEA
was thought to need CommonJS. The SC settled the direction with a hard
requirement: **the SEA must be ESM, not CJS** — don't transpile, don't move the
top-level `await main()`. CJS is a dead end (the project is heading ESM-only;
MCP is ESM-only). So the answer was never "make it CJS" — it was "build on a Node
that supports an ESM SEA entry."

### The crux fact (don't re-derive)

ESM SEA entry is gated on the **`"mainFormat": "module"`** field in sea-config.json.

- The value is **`"module"`**, NOT `"esm"`. Node 26 errors `"mainFormat" field
  of sea-config.json must be one of "commonjs" or "module"` if you write `esm`.
- **Node 24.17 silently ignores `mainFormat`** and runs the ESM entry as CJS,
  which dies with `SyntaxError: Cannot use import statement outside a module`.
  So the bundled runtime MUST be **Node 26+**. I installed v26.3.1 via fnm.
  (Didn't probe Node 25; 26 is proven, so build-sea.sh pins `--using=26`.)
- node:sqlite is unflagged on 26 too, so the store path stays clean.
- The proven recipe is otherwise identical to the POC: strip-sign -> postject
  inject -> ad-hoc re-sign. mainFormat is the only new ingredient.

### What was proven (empirically, run from a node-20 shell)

1. Real CLI SEA launches: `./dist/claude-sdk-cli --version` -> exit 0, prints
   version. No native-addon/ABI crash. Bundled runtime reports v26.3.1.
2. Real `SqliteObjectStore` under the SEA runtime (throwaway harness, now
   deleted): set/get round-trips, `journal_mode` = `wal`, `busy_timeout` = 5000.
   WAL intact. This exercised the actual store class, not a POC stand-in.

### What I changed (mine)

- `apps/claude-sdk-cli/build.ts` — kept the in-flight unstaged edits
  (`packages: 'external'` commented out so deps bundle in; `splitting: false`).
  These make dist/main.js a self-contained single ESM file (~7MB, 0 chunks),
  which a SEA requires (in ESM SEA, only builtins resolve — everything else must
  be in the bundle). Staged it as part of this work.
- `apps/claude-sdk-cli/sea-config.json` — new. main: dist/main.js,
  mainFormat: module, output + binary land in dist/ (already gitignored).
- `apps/claude-sdk-cli/build-sea.sh` — new. Full recipe: pnpm build -> blob on
  node26 -> copy binary -> codesign strip -> postject -> ad-hoc sign. Comments
  carry the two load-bearing constraints (node26 for mainFormat; macOS sign
  bracket). macOS arm64 only, matching the POC.

### Verified
type-check clean, 813 tests pass (incl. SqliteObjectStore). `bash build-sea.sh`
runs clean end to end and the resulting binary launches from a node-20 shell.

### Decisions made (not in the brief)
- Pinned the SEA runtime to Node 26 specifically (26.3.1). Forced by the
  mainFormat requirement; 24 demonstrably can't do it.
- Kept the unstaged build.ts edits rather than re-deriving a separate SEA-only
  esbuild config. The self-contained bundle is what SEA needs and is a sensible
  shape for the published bin too; a second config would just duplicate it.
- Proved the WAL store with a temporary harness importing the real store class,
  then deleted it. No test/source artefact left behind from that proof.
