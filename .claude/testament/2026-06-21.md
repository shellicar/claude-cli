## Making the SQLite persistence actually run (PR #359, Phase 4 + 5)

#359 shipped the persistence feature green, but the built CLI threw `Could not locate the bindings file` at startup: better-sqlite3's native binding was never installed, and the suite only ever exercised the in-memory `MemoryObjectStore`, so a store that never loaded passed CI. Three things fixed it — the binding install, a `busy_timeout` pragma, and a test of the *real* store.

### pnpm 11 and native build scripts (the install fix)

The binding wasn't installing because pnpm blocks dependency build scripts. The repo's posture was `ignoreScripts: true` in `pnpm-workspace.yaml` — the right call when there are *no* built dependencies, wrong once better-sqlite3 (which compiles a `.node`) is one. The SC's decision: drop `ignoreScripts`, allowlist the one dep, explicitly deny the rest.

Landed config:

```yaml
ignoreScripts: false
allowBuilds:
  better-sqlite3: true
  esbuild: false      # explicit deny
  lefthook: false     # explicit deny
```

pnpm-11 facts worth not re-deriving:

- `onlyBuiltDependencies` is **removed** in pnpm 11. It's replaced by `allowBuilds` (a `pkg: true|false` map). An `onlyBuiltDependencies` entry is silently ignored, and `pnpm rebuild` becomes a silent no-op.
- `ignoreScripts: true` is a blanket override — it suppresses all dependency build scripts regardless of `allowBuilds`. You can't keep it and build a native dep.
- With `ignoreScripts: false`, pnpm **hard-errors** (`ERR_PNPM_IGNORED_BUILDS`) on any build-needing dep not in `allowBuilds`. Setting a package to `false` is the documented "decided: don't build" that clears the error. esbuild and lefthook each need an entry (true or false) or install fails.
- No prebuilt better-sqlite3 binary exists for node 24 arm64, so a cold CI runner compiles from source (node-gyp). pnpm caches the built `.node` in its store, so a warm local install reuses it instantly with no gyp output — don't mistake that silence for "didn't build".
- pnpm rewrites `pnpm-workspace.yaml` on a partial/failed install (appends placeholder `allowBuilds` entries, collapses comment whitespace). Restore exact bytes with `git show HEAD:pnpm-workspace.yaml > pnpm-workspace.yaml`.

### busy_timeout

`SqliteObjectStore` sets `busy_timeout = 5000` beside the WAL/synchronous pragmas. The store is machine-wide, so two CLIs running at once are two concurrent writers; without the timeout the second gets `SQLITE_BUSY` and throws immediately. 5s wait-for-lock instead. No fallback or caching layer — SQLite errors surface to Claude as ordinary tool errors, by the SC's call; revisit only if failures prove frequent.

### The Database is injected, not constructed (SC correction)

The first cut gave `SqliteObjectStore` a `db` getter exposing the raw connection so the test could read `busy_timeout` back — production API widened for a test, punched through the two-method interface. The SC rejected it and fixed it at the root:

- `SqliteObjectStore`'s constructor takes a better-sqlite3 `Database`, not a path. It *configures* the injected connection (WAL / synchronous / busy_timeout pragmas, creates the `objects` table) and uses it. No `path`, no `mkdirSync`, no `new Database`, no getter. The better-sqlite3 import is now `import type`.
- `setup/container.ts` is the creation site: it builds the `~/.claude/persistence.db` path, runs `mkdirSync(dirname(path), { recursive: true })` (the directory must exist before `new Database`), news up the connection, and injects it. `createAppTools` only threads `IObjectStore` and never constructed the store, so the container was the sole site to move.
- The real-store test owns its own temp-file `Database` per test and reads `busy_timeout` back from that handle directly. `MemoryObjectStore` and `IObjectStore` are untouched.

### The test that should have existed

`apps/claude-sdk-cli/test/SqliteObjectStore.spec.ts` exercises the concrete on-disk store (temp dir, cleaned up in `afterAll`): construction proves the binding resolves (the guard that was missing), `set`/`get` round-trips, missing id → `undefined`, and `busy_timeout` reads back `5000`. This deliberately reverses the plan's "SqliteObjectStore is not unit-tested" decision — that omission is exactly what let a non-loading store ship green.

### Caching traps when verifying

- turbo returns `FULL TURBO` (cache hit) for type-check/build even after a source change, because it's content-keyed and the working tree equalled the index. Force a fresh run (`pnpm type-check --force`, or a direct `pnpm exec tsc --noEmit`) when you need to trust the result against staged edits.
- The `TsDiagnostics` language-service tool reported a stale constructor error on `container.ts` after the source was already changed. A direct `tsc --noEmit` was clean — trust tsc over the cached language service when they disagree.
