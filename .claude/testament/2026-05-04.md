## Phase 1 — Apostle: wrapped arrow nav (#319)

### What the terrain looks like

`EditorState.handleKey` owns all caret transitions. The `'up'` and `'down'` cases move by logical line (`#lines` index) with a `Math.min` col clamp. They have no knowledge of terminal width. The terminal width lives in `AppLayout.#screen.columns` and is never passed into `EditorState`.

Wrapping is computed in `renderEditor` (`apps/claude-sdk-cli/src/view/renderEditor.ts`). It calls `wrapLine(pfx + line, cols)` for each logical line. Both prefixes (`PROMPT_PREFIX = '💬 '` and `INDENT = '   '`) have visual width 3. Nothing about this is communicated back to `EditorState`.

### The fix is additive, not structural

The model does not need to change. `EditorState` stays as an array of logical lines. Two new public methods are added:

- `moveUpVisual(cols: number, prefixWidth: number): boolean`
- `moveDownVisual(cols: number, prefixWidth: number): boolean`

These compute visual row position using `prefixWidth + stringWidth(line.slice(0, cursorCol))`. `string-width` is already a direct dep of `apps/claude-sdk-cli`. The module-level `graphemeSegmenter` (already in the file) is reused by a new `colFromVisual` helper to map a target visual width back to a code-unit offset.

`AppLayout.handleKey` intercepts `'up'` and `'down'` in editor mode before they reach `editorState.handleKey`, calls the visual methods instead, and schedules a render.

### What stays unchanged

The existing `case 'up'` and `case 'down'` in `handleKey` are left in place. Many existing tests use `key('up')` in setup sequences (for `ctrl+k`, `ctrl+delete`, `delete` tests) and the `describe('EditorState — up')` section tests them directly. Removing them would break tests. In the live app they are never reached (AppLayout intercepts first), but they're alive in tests.

### The exported constant

`EDITOR_PREFIX_VISUAL_WIDTH = 3` is exported from `renderEditor.ts`. `AppLayout` imports it to pass to the visual nav methods. This makes prefix width a single edit point if the prefix ever changes.

### Test arithmetic (cols=10, prefixWidth=3)

Visual position of cursor: `3 + stringWidth(line.slice(0, cursorCol))`.
- Row 0: offsets 0–6 (visual pos 3–9)
- Row 1: offsets 7–16 (visual pos 10–19)
- Row boundary at offset 7 (visual pos 10 = 1×10).

Key test: `cursorCol=12` → `visualPos=15`, `row=1`, `colInRow=5`. After `moveUpVisual`: `targetPos=5`, `targetInLine=2`, `cursorCol=2`. Row 0 at `colInRow=5` is offset 2 (visual pos 5, offset=5-3=2). Verified.

### No stop condition

The model is clean enough for an additive fix. No structural change needed.

### What changed from the first plan

The architecture is identical (additive `moveUpVisual`/`moveDownVisual`, `AppLayout` interception, `EDITOR_PREFIX_VISUAL_WIDTH` export). The differences:

**Constructor added to `EditorState`.** No explicit constructor existed. The new optional constructor takes `{ lines, cursorLine, cursorCol }` and sets the private fields directly. Field initializers (`['']`, `0`, `0`) remain as defaults for `new EditorState()` with no argument. The getter doc comment updated to drop "Read-only".

**`case 'up'` and `case 'down'` removed from `handleKey`.** Lines 190–205. Once AppLayout intercepts, these are unreachable.

**Five existing tests rewritten.** The tests that used `key('up')` as cursor-positioning setup (not as the thing under test) are replaced with constructor calls:
- `delete — at EOL joins with next line` → `{ lines: ['ab', 'cd'], cursorLine: 0, cursorCol: 2 }`
- `ctrl+delete — at EOL joins with next line` → `{ lines: ['first', 'second'], cursorLine: 0, cursorCol: 5 }`
- `ctrl+k — at EOL joins with next line` → `{ lines: ['ab', 'cd'], cursorLine: 0, cursorCol: 2 }`
- `right — at EOL wraps to start of next line` → `{ lines: ['ab', ''], cursorLine: 0, cursorCol: 2 }`
- `right — at EOL wraps to next line index` → `{ lines: ['ab', ''], cursorLine: 0, cursorCol: 2 }`

**`describe('EditorState — up')` and `describe('EditorState — down')` blocks removed.** No longer have behaviour to assert.

**All new `moveUpVisual`/`moveDownVisual` tests use the constructor** for setup rather than key sequences, so they don't depend on other key handler correctness.
