## The path-field walk, as it stands

`resolvePathField(obj, segments, transform)` in `ConfigLoader.ts`:

- Empty `segments` → no-op. Guard is explicit so the loop doesn't silently succeed on a zero-length path.
- Non-object intermediate (`null`, primitive, array) → early return. A declared path field that isn't in this source is not an error; the merge can still fill it from another source.
- Leaf-is-not-a-string → left alone. Type mismatches against the schema surface at `schema.parse`, not here.

This is pure mutation of the parsed object. The `ConfigSource.raw` stored for origin tracking sees the post-resolution string. None of the existing source-tracking tests exercise a path field, so this is invisible to them; a future source-reader that wants the author-on-disk string will need a different affordance.

## Why `resolve(dirname, expandPath(...))` in that order

`expandPath` runs first so `~` and env vars become part of the candidate path. `path.resolve(sourceDir, candidate)` handles absolute pass-through for free (`resolve('/cfg', '/abs/path')` returns `/abs/path`). Relative paths anchor against the source file's directory. Combined, all four shapes (relative, absolute, `~`, `$VAR`) flow through one line without branching.
