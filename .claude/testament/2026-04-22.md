1. Hook `command` resolves paths (`~`, `$HOME`, relative) against the config file's directory. The config loader owns the resolution, not the hook code.

## One silent deviation and its fix

When adding `fs` to `ConfigLoaderOptions`, I told the SC I'd make it required. I then made it optional to avoid touching the 12 existing `new ConfigLoader(...)` call sites in tests and the one in `main.ts`. The SC caught it: "how is it optional if it's required?" and followed with "there is no point doing this feature if it's optional."

Fixed. `fs: IFileSystem` (no `?`) on `ConfigLoaderOptions`. All 13 call sites updated to supply `fs`. `pathFields` stays optional because not every config needs path fields, but `fs` is always there.

The lesson: when you say "I will add it as required", do that. Don't softly downgrade to preserve compatibility with existing code unless the SC said so.

## What Phase 2 is walking into

1. **`ConfigSource.raw` contents after resolution**. Resolution runs per-source during `#readAll()`, before `mergeRawConfigs`. The raw object stored in `sources[n].raw` can hold either pre- or post-resolution strings. The three existing source-tracking tests read `raw` values and need to keep passing. Pick: mutate a copy so `sources[n].raw` stays as authored on disk, or store post-resolution and update the source-tracking tests' expectations if needed. The sources tests I updated use single string fields (not paths), so they are not affected directly — but any future source-reader will see whatever you decide. Ask the SC before choosing.

2. **Resolution must run on reload**. `#reload()` calls `#readAll()`. If you put resolution inside `#readAll()` (or a helper it calls), both paths share it for free and the reload test passes without extra work.

3. **`NodeProcessLauncher` stdin wiring**. Currently `spawn(..., { detached: true, stdio: 'ignore' })`. For stdin: `{ detached: true, stdio: ['pipe', 'ignore', 'ignore'] }`, write the string, end. Keep `detached` + `unref()` for fire-and-forget. The `IProcessLauncher` boundary test covers the wire — do not invent new tests inside `NodeProcessLauncher.spec.ts` (there isn't one and the prompt does not ask for one).

4. **`expandPath` signature is `(value, fs)`**. Node `path.resolve(dir, ...)` handles the absolute-pass-through case for free: `resolve('/cfg', '/abs')` returns `/abs`. Combine `expandPath` then `path.resolve` with the source file's `dirname`.

5. **Main wiring in `apps/claude-sdk-cli/src/entry/main.ts`**. `NodeProcessLauncher()` is constructed on its usual line. No constructor change needed — Phase 2 only touches the body of `launch`.
