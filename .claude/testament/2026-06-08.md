# 22:03

Courier for #239. This PR introduces streaming tool input JSON display.

## What was built

Four phases per tool call, in SDK lifecycle order:

1. **Name appears** at `content_block_start` ŌĆö `tool_use_start` / `server_tool_use_start` events.
2. **Partial JSON streams** at `input_json_delta` ŌĆö `tool_use_input_delta` event, gated by `currentToolId`.
3. **Resolved view replaces streaming content** at completion ŌĆö `tool_approval_request` renders the summary synchronously before the async approval flow, so the resolved view is visible while the user approves.
4. **Status mark appended** after approval resolves.

Client tools also emit `tool_use_input_stop` at `content_block_stop`, which appends a newline so consecutive client tools in a batch each start on their own line.

## Architecture

**`ToolObject` state machine** (`model/ToolObject.ts`): seven phases (`streaming`, `streamed`, `pending`, `approved`, `denied`, `error`, `done`). Each object owns its tool id, name, streaming content, and resolved view. `render()` returns the display line; trailing `\n` in every non-streaming phase.

**Ordered tools block in `AgentMessageHandler`**: `#toolObjects: Map<string, ToolObject>` keyed by tool id, `#toolOrder: string[]` preserving emission order. Every state change calls `#redrawTools()`, which rebuilds the entire tools region by joining `#toolOrder.map(id => #toolObjects.get(id)?.render())` and writing it via `setActiveBlockContent`.

**`StreamProcessor` cursor**: `currentToolId: string | null` replaces the two-Set approach. Set at `content_block_start`, cleared at `content_block_stop`. The cursor value travels in every emitted event. No index arithmetic needed.

**`AppLayout.setActiveBlockContent(text)`**: replaces entire active block content from offset 0. Replaces the earlier `#streamingMark` / `markStreamingPosition` / `replaceFromMark` machinery.

## Key corrections during the arc

**Locked decision changed after v3 shipped**: the v3 design (streaming JSON coexists with resolved view, both visible) was what the plan said. The SC corrected: the resolved view *replaces* the streaming view at completion. This required adding `ConversationState.replaceActiveFromOffset` and `AppLayout` mark/replace methods (v4), then the full redesign to the ToolObject model (v8).

**Phase 3 was broken from v4 onward**: `tool_approval_request` sync arm called `replaceFromMark('')`, erasing streaming content before the async approval flow ran. The user saw a blank tools section while approving. Fixed in v8: `obj.resolve(summary)` + `#redrawTools()` fires synchronously before `void this.#toolApprovalRequest(...)`, so the resolved view is live during approval.

**Router translation failure**: a "narrows to status only" restatement of the SC's locked decision was wrong. The v2 plan over-cut based on it ŌĆö removed `formatToolSummary`, `server_tool_use` event, and narrowed the approval resolution arms. `Ref` and `Pipe` tools lost their resolved-view formatting (store-resolved hint, step display). Third plan reversed all deletions. Lesson: retraction-shaped or translation-shaped instructions from the Router need re-verification against the codebase, not pass-through acceptance.

**Per-tool class vs chain**: an earlier plan used a promise chain (tool2 waits for tool1 to fully complete before rendering). That contradicts the existing `ToolApprovalState` ŌåÉŌåÆ navigation UI, which shows all pending tools simultaneously. The ToolObject-per-tool design (v8) gives concurrent display: all tools render in the ordered list at the same time, each in its own phase.

## Test id typo

The `tool_approval_request` tests use `toulu_01` (missing `c`; real format is `toolu_01`). Both sides of each map lookup use the same string, so the lookup works. The typo is in the tests only, not in any production path.

## requireToolApproval = false gap

When `requireToolApproval` is false, no `tool_approval_request` event fires, so `replaceFromMark` / `#redrawTools` never runs. The streaming JSON persists with no resolved view replacing it. Pre-existing gap, carried forward, out of scope for this PR.

## Key decision: setActiveBlockContent on ConversationState

Our `AppLayout.setActiveBlockContent(text)` called `conversationState.replaceActiveFromOffset(0, sanitiseLoneSurrogates(text))`. The merged ConversationState will have `replaceActiveFromOffset` (from our side) + `appendStreaming` + events (from origin/main). We add `setActiveBlockContent` as a new method alongside `appendStreaming`, with the same sanitisation.

## Multi-batch annotation test issue

`#redrawTools()` replaces the entire active block content via `setActiveBlockContent`. If a second batch's `tool_use_start` fires while the active block is still 'tools' (no `message_text` between batches), `#redrawTools()` will overwrite any annotation the first batch's `message_usage` appended. The multi-batch delta tests must add `message_text` between batches to seal the first batch's block. This reflects real usage ŌĆö Claude always outputs text between tool batches.

## main.ts is not a listed conflict but needs updating

Origin/main's version of `main.ts` (the expected winner of the auto-merge) has two `processor.on` forwards for server tools without `id`. Our StreamProcessor emits those events WITH `id`. The forwards must be updated and four new forwards added for the streaming events.

# 00:28

Apostle for Phase 4 iteration 2 ŌĆö batch-sealing fix in the merge reconciliation plan.

## What the iteration-1 plan got wrong

The 23:48 entry says: "The multi-batch delta tests must add `message_text` between batches to seal the first batch's block. This reflects real usage ŌĆö Claude always outputs text between tool batches."

Both of those statements are wrong. The SC rejected the plan on exactly this point: consecutive tool-only turns with no text between them are common (this session produced many). Adding `message_text` to the test rewrites the test so the broken case is never exercised ŌĆö it hides the bug rather than fixing it.

## The actual bug

`message_usage` appends the batch annotation via `appendToLastSealed('tools', ...)`. At that point the active block IS the tools block, so the annotation goes into the active block content. Then `#usageBeforeTools = null`, `#toolObjects = new Map()`, `#toolOrder = []`. But `#activeBlock` stays as the 'tools' block ŌĆö it is not sealed.

When the second batch starts:
- `tool_use_start` ŌåÆ `transitionBlock('tools')` ŌåÆ **no-op** (same type already active)
- `#redrawTools()` ŌåÆ `setActiveBlockContent(newContent)` ŌĆö **overwrites the entire active block**, including the annotation

## The fix

Add `this.#conversation.completeActive()` at the end of the `message_usage` batch-close branch (after `#toolOrder = []`). `completeActive()` seals the active block and sets `#activeBlock = null`. The next batch's `transitionBlock('tools')` then opens a genuinely fresh block. `completeActive()` already exists on both the branch and origin/main's `ConversationState` ŌĆö no new method needed.

## The test correction

Three test changes:
1. "second independent batch produces its own annotation" ŌĆö removed the two-comment explanation that referenced `message_text` and removed the `message_text` line itself.
2. "computes the second batch delta from the post-first-batch usage" ŌĆö removed the `message_text` line.
3. New test: "first batch annotation survives second batch starting without text between" ŌĆö fires two batches back-to-back, no text, and asserts `conversationState.sealedBlocks.some(b => b.type === 'tools' && b.content.includes('+500'))`. This is the test the iteration-1 plan should have had.

## Why completeActive() is the right call site

The alternative would be to seal inside `transitionBlock` when the new type differs from the active type ŌĆö but `transitionBlock` already does that (it's just not called, because the no-op short-circuits first). The real problem is that after a batch ends, there's no event that forces a transition to a different type. `completeActive()` at `message_usage` is the natural closure: "this batch is done, seal what we have."
