## What was built

`--config <json>` flag added to `claude-sdk-cli`. Takes a JSON object string and applies it as a highest-precedence layer through the existing permissive config schema. Any config value can be overridden this way, as an alternative to the existing per-value flags like `--model`.

When `--config` is passed, the CLI prints the effective merged config at startup (via a second `showStartupBanner` call) so the user can see what resolved. This is intentional: the permissive schema silently strips unknown keys and coerces bad values, so without the print there is no feedback when a typo goes through.

**Files changed:**

- `apps/claude-sdk-cli/src/cli-config/parseConfigOverride.ts` (new): parses the raw flag string. Rejects malformed JSON (`JSON.parse` throws) and non-object payloads (top-level `z.object` throws on string/number/array). Returns the raw parsed object without running it through the permissive per-field validators, so partial overrides don't inject defaults.
- `apps/claude-sdk-cli/src/cli-config/formatEffectiveConfig.ts` (new): pure `effective config:\n` + JSON.stringify for the startup print.
- `packages/claude-core/src/Config/ConfigLoader.ts`: `overrides` option wired into `#readAll()`. The override is pushed last onto `sources` and last onto the merge layers, so it wins the fold and the reverse origin-walk finds it first.
- `packages/claude-core/src/Config/types.ts`: `ConfigLoaderOptions.overrides?: { origin: string; raw: Record<string, unknown> }`. The `ConfigSource` doc comment updated to note that `path` may be a synthetic label rather than a filesystem path.
- `apps/claude-sdk-cli/src/entry/main.ts`: `--config` flag added to `parseArgs`, parse block routes both error cases through `process.stderr.write` + `process.exit(1)` (same pipe as `--resume` validation), passes `{ origin: ':parameters:', raw: configOverride }` to `ConfigLoader`, second `showStartupBanner` gated to `--config`-present.
- `apps/claude-sdk-cli/src/help.ts`: `--config <json>` documented.

## How `--model` and `--config '{"model":"..."}` relate

`--model` does NOT flow through `ConfigLoader`. It lives in a standalone mutable slot (`const overrides: { model: string | null }`) resolved by `getEffectiveModel()` which layers the slot over `configLoader.config.model`. So `--config '{"model":"..."}'` sets the ConfigLoader layer, and `--model` wins over it via `getEffectiveModel()` — the dedicated flag wins for free, no conflict code.

## The SC's corrections during this mission

Initial plan required strict rejection of unknown/mistyped top-level keys in `--config`. SC dropped this: the config schema is permissive by design (`.catch()` everywhere), and `--config` should behave like a config file. Only malformed JSON and non-objects are rejected. The startup print of the effective config is the substitute: the user sees what resolved.

## Things that will confuse the next operator

**`sdkConfigSchema` is `any` at the type level.** The `.meta()` call on the schema in Zod 4.4.3 widens it. `configLoader.config` is effectively `any` everywhere. Pre-existing, out of scope, do not try to fix. Courier for #239. This PR introduces streaming tool input JSON display.

## What was built

Four phases per tool call, in SDK lifecycle order:

1. **Name appears** at `content_block_start` — `tool_use_start` / `server_tool_use_start` events.
2. **Partial JSON streams** at `input_json_delta` — `tool_use_input_delta` event, gated by `currentToolId`.
3. **Resolved view replaces streaming content** at completion — `tool_approval_request` renders the summary synchronously before the async approval flow, so the resolved view is visible while the user approves.
4. **Status mark appended** after approval resolves.

Client tools also emit `tool_use_input_stop` at `content_block_stop`, which appends a newline so consecutive client tools in a batch each start on their own line.

## Architecture

**`ToolObject` state machine** (`model/ToolObject.ts`): seven phases (`streaming`, `streamed`, `pending`, `approved`, `denied`, `error`, `done`). Each object owns its tool id, name, streaming content, and resolved view. `render()` returns the display line; trailing `\n` in every non-streaming phase.

**Ordered tools block in `AgentMessageHandler`**: `#toolObjects: Map<string, ToolObject>` keyed by tool id, `#toolOrder: string[]` preserving emission order. Every state change calls `#redrawTools()`, which rebuilds the entire tools region by joining `#toolOrder.map(id => #toolObjects.get(id)?.render())` and writing it via `setActiveBlockContent`.

**`StreamProcessor` cursor**: `currentToolId: string | null` replaces the two-Set approach. Set at `content_block_start`, cleared at `content_block_stop`. The cursor value travels in every emitted event. No index arithmetic needed.

**`AppLayout.setActiveBlockContent(text)`**: replaces entire active block content from offset 0. Replaces the earlier `#streamingMark` / `markStreamingPosition` / `replaceFromMark` machinery.

## Key corrections during the arc

**Locked decision changed after v3 shipped**: the v3 design (streaming JSON coexists with resolved view, both visible) was what the plan said. The SC corrected: the resolved view *replaces* the streaming view at completion. This required adding `ConversationState.replaceActiveFromOffset` and `AppLayout` mark/replace methods (v4), then the full redesign to the ToolObject model (v8).

**Phase 3 was broken from v4 onward**: `tool_approval_request` sync arm called `replaceFromMark('')`, erasing streaming content before the async approval flow ran. The user saw a blank tools section while approving. Fixed in v8: `obj.resolve(summary)` + `#redrawTools()` fires synchronously before `void this.#toolApprovalRequest(...)`, so the resolved view is live during approval.

**Router translation failure**: a "narrows to status only" restatement of the SC's locked decision was wrong. The v2 plan over-cut based on it — removed `formatToolSummary`, `server_tool_use` event, and narrowed the approval resolution arms. `Ref` and `Pipe` tools lost their resolved-view formatting (store-resolved hint, step display). Third plan reversed all deletions. Lesson: retraction-shaped or translation-shaped instructions from the Router need re-verification against the codebase, not pass-through acceptance.

**Per-tool class vs chain**: an earlier plan used a promise chain (tool2 waits for tool1 to fully complete before rendering). That contradicts the existing `ToolApprovalState` ←→ navigation UI, which shows all pending tools simultaneously. The ToolObject-per-tool design (v8) gives concurrent display: all tools render in the ordered list at the same time, each in its own phase.

## Test id typo

## What the iteration-1 plan got wrong

The 23:48 entry says: "The multi-batch delta tests must add `message_text` between batches to seal the first batch's block. This reflects real usage — Claude always outputs text between tool batches."

Both of those statements are wrong. The SC rejected the plan on exactly this point: consecutive tool-only turns with no text between them are common (this session produced many). ## The actual bug

`message_usage` appends the batch annotation via `appendToLastSealed('tools', ...)`. At that point the active block IS the tools block, so the annotation goes into the active block content. Then `#usageBeforeTools = null`, `#toolObjects = new Map()`, `#toolOrder = []`. But `#activeBlock` stays as the 'tools' block — it is not sealed.

When the second batch starts:
- `tool_use_start` → `transitionBlock('tools')` → **no-op** (same type already active)
- `#redrawTools()` → `setActiveBlockContent(newContent)` — **overwrites the entire active block**, including the annotation

## The fix

Add `this.#conversation.completeActive()` at the end of the `message_usage` batch-close branch (after `#toolOrder = []`). `completeActive()` seals the active block and sets `#activeBlock = null`. The next batch's `transitionBlock('tools')` then opens a genuinely fresh block. `completeActive()` already exists on both the branch and origin/main's `ConversationState` — no new method needed.

## The test correction

Three test changes:
1. "second independent batch produces its own annotation" — removed the two-comment explanation that referenced `message_text` and removed the `message_text` line itself.
2. "computes the second batch delta from the post-first-batch usage" — removed the `message_text` line.
3. New test: "first batch annotation survives second batch starting without text between" — fires two batches back-to-back, no text, and asserts `conversationState.sealedBlocks.some(b => b.type === 'tools' && b.content.includes('+500'))`. This is the test the iteration-1 plan should have had.

The alternative would be to seal inside `transitionBlock` when the new type differs from the active type — but `transitionBlock` already does that (it's just not called, because the no-op short-circuits first). The real problem is that after a batch ends, there's no event that forces a transition to a different type. `completeActive()` at `message_usage` is the natural closure: "this batch is done, seal what we have."

## What was built

ExecV2 executes commands as a recursive AST instead of V1's flat steps array. A `pipeline` is either a single `Command` (leaf, with `id` and `program`) or an `Operation` (node, with `op`, `left`, `right`). The five operators mirror shell semantics: `;` (unconditional sequence), `&&` (and), `||` (or), `&` (concurrent), `|` (pipe). Results are per-command, keyed by caller-assigned `id`, in tree pre-order.

Files added to `packages/claude-sdk-tools`:
- `src/ExecV2/schema.ts` - Zod schema for Command/Operation/Pipeline AST, with R4/NE2/NE3 refinements
- `src/ExecV2/executeTree.ts` - recursive executor with pipe coordinator
- `src/ExecV2/ExecV2.ts` - tool definition (name: `ExecV2`, handler wires normalise → validate → execute → strip)
- `src/ExecV2/types.ts` - TypeScript types derived from schemas
- `src/ExecV2/scenarios.md` - 35-scenario catalog (source of truth for both spec files)
- `src/entry/ExecV2.ts` - entry point mirroring V1's shape
- `test/Exec/scenarios.spec.ts` - 74 V1 characterisation tests
- `test/ExecV2/scenarios.spec.ts` - 102 V2 contract tests

CLI wiring in `apps/claude-sdk-cli`:
- `tools` config field added to `sdkConfigSchema` (defaults: `exec: false, execV2: true`)
- `createAppTools` accepts tools config and builds the exec tool list conditionally
- `entry/main.ts` passes `configLoader.config.tools` at the one call site
- `test/createAppTools.spec.ts` - 6 tests for conditional inclusion

## Schema: the TDZ trap

`.strict()` is intentionally absent from `OperationSchema`. Adding it calls `this.shape` at construction time, which fires the `get left/right` accessors while `PipelineSchema` is still in the temporal dead zone (`const` is not hoisted). The error is `Cannot access 'PipelineSchema' before initialization`. `.superRefine()` does not have this problem - it wraps the schema without inspecting the shape. The result: extra unknown keys on operation objects pass through silently. If strict validation matters later, a `.superRefine()` checking `Object.keys(val)` against `['op', 'left', 'right']` is the fix.

## `success` flag: use aggregate exit, not per-leaf

The `success` flag is `aggregateExit === 0`, not `results.every(r => r.exitCode === 0)`. The per-leaf reduction breaks `||`: O2 (`false || echo b`) and M1 (`(false && a) || b`) keep the failing left leaf in `results`, so per-leaf reports `success: false` while the catalog requires `true`. The aggregate-exit flows from the operator's semantics and gives the right answer for every catalog case.

## `;` aggregate diverges from shell `$?`

Shell `$?` after `a; b` is `b`'s exit code. The test catalog says otherwise: C2 (`false; echo b`) expects `success: false`. The `combineUnguarded` rule (left's exit if non-zero, else right's) matches the catalog. `&&` and `||` are the only "guarded" operators where the operator's own structure handles failure propagation; everything else is unguarded.

## Pipe coordinator

`executePipe` spawns the right side first (via `executeAsConsumer`) so it is reading before `executeAsSource(left)` starts writing. After left resolves, `pt.end()` signals EOF to the right. `executeAsConsumer` and `executeAsBridge` only support a Command on that side - an Operation in the consumer/bridge position throws. NE3 blocks the `cmd | (group)` shape at the schema (before the handler runs), so the runtime throw is defense-in-depth.

The `merge_stderr` case in pipe-source mode: when `cmd.merge_stderr` is true, both `child.stdout` and `child.stderr` pipe into the PassThrough sink. The sink stays open until `pt.end()` fires after the left side resolves.

## R3 V1 quirk (locked in characterisation tests)

V1's `execPipeline` registers `lastChild.stdout.on('data')` unconditionally before redirect setup. Both the capture buffer and the redirect file receive the output. So for R3 (`echo hello | cat > /dev/null`), the V1 result has `stdout == "hello"` even though the output was redirected. V2 suppresses the capture listener on pipe-source Commands - their `stdout` in results is always `""`. The V1 characterisation tests lock this quirk so phase 2 couldn't regress to it.

## `leavesAsV1` cast in ExecV2.ts

The handler casts V2 Command leaves to V1 Command type to pass them to `builtinRules[i].check`. The cast is safe: V2 Command is structurally a superset of V1 Command (adds `id`), so it satisfies every property the rules read. If the V1 rule signature ever changes, revisit this.

## NE3 was added mid-flight

After the initial Phase 2 pass, the SC decided `cmd | (group)` should be rejected at the schema rather than at runtime. NE3 was added as a third branch in `OperationSchema.superRefine()`: when `op === '|'` and `right` has an `op` field (Operation, not Command), add an issue at path `['right']`. The runtime throw in `executeAsConsumer`/`executeAsBridge` remains as a backstop.

- `apps/claude-sdk-cli`: 25 files / 513 tests

## V1 stays in the tree

V1 `Exec` is off by default (`tools.exec: false`) but stays in the codebase. Flip the config flag and restart to use it. Two tools cannot share a name - V1 is `Exec`, V2 is `ExecV2`. The coexistence is intentional; V1 is the fallback until V2 is proven in production.

# 21:38 — re-cut to a coordination layer (supersedes "The four moves" above)

The 21:07 plan put the coordinator into `controller/`. In conversation the SC
pushed on it ("if you move the view into controller, what is the view?" → "is it
actually a violation for view to depend on controller?"), and we worked the
architecture out together. The pivot, and why it matters for whoever builds this:

## What the discussion established

- The biome rules are **compile-time import** constraints; classic-MVC diagrams
  are **runtime collaboration**. Different axes. This codebase is not classic MVC
  — it cut the View↔Controller link and routes everything through the stores
  (unidirectional / reactive-store, Model as the hub). That's *why* it can be
  layered at all: the graph is a DAG pointing down to `model/`, not a cycle.
- `view ⊥ controller` is a **correct** rule (decisions 1 & 2: views render only,
  view and input-handling meet only at shared state). The violation is real, not
  pedantic. `Presentation`/`ViewHost` aren't views — they're the coordinator.
- The rules are **asymmetric**: `view ⊥ controller` exists, `controller ⊥ view`
  doesn't. Folding the coordinator into `controller/` (the 21:07 plan) only works
  by leaning on that gap. A dedicated layer lets you close it.

## The decision (SC-authorised, widens the "not a redesign" brief)

Dedicated **`app/` coordination layer** below `entry/`, above view/controller/
model. `Presentation`, `PrimaryPresentation`, `ViewHost`, **and `TerminalInput`**
move there. `controller/` drops to model-only and gains a new `biome.json`
forbidding `../view/*` and `../app/*`; `model/` and `view/` gain `../app/*`.

`app/` ≠ `entry/`: `app/` is reusable, unit-tested coordination machinery
(`ViewHost.spec.ts`); `entry/main.ts` is the run-once composition root. The
construction block in `main.ts` is **identical** under either approach — the only
difference is import sources and which `biome.json` can be written. That's the
whole argument for B: it's not about how `main.ts` reads, it's that `controller/`
becomes *provably* leaf-pure.

## The finding that "just move the classes" hides

Adding `controller ⊥ view` (the rule that justifies B) turns a currently-legal
import into an error: `EditorHandler.ts:7` imports `EDITOR_PREFIX_VISUAL_WIDTH`
from `view/renderEditor.ts`. So B is **not** only moving classes — that shared
constant must be relocated. New home: `model/editorLayout.ts` (bottom layer, both
sides import down). Key detail: `renderEditor` *defines but never consumes* the

the handler). So `renderEditor` just drops the export — the prefix strings stay
