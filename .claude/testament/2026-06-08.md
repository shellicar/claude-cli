## The layer model (what the code actually enforces)

The boundaries are per-directory `biome.json` files, not a single config:

- `src/model/biome.json` — `model/` must not import `../view/*` or `../controller/*`.
- `src/view/biome.json` — `view/` must not import `../controller/*`.
- `src/controller/` and `src/entry/` — no `biome.json`, unrestricted.

So the permitted direction is **controller → view → model**, entry on top. This
is the whole key to the relayering: a violating symbol is one sitting below the
layer it points at, and the fix is to move it to the layer that already implies
it. No `biome.json` edits, no suppressions — those would be silencing the lint,
which the mission forbids.

## The four moves

- `AppModeKey` (type) → `model/AppModeState.ts`. It's read by the `model/` store
  that holds the active value; defining it there inverts the dependency. It was
  only colocated with `Presentation` for convenience.
- `Presentation`, `PrimaryPresentation`, `ViewHost` → `controller/`. Each pairs a
  `View` (view/) with an `InputHandler` chain (controller/); only `controller/`
  may import both.

The trap worth remembering: `ViewHost` is **not** flagged by biome today. It
reaches `InputHandler` only transitively through `Presentation`, and biome checks
direct import paths. The moment `Presentation` moves to `controller/`, a
`ViewHost` left in `view/` would import `../controller/Presentation.js` — a fresh
violation. Moving `ViewHost` in the same step is what prevents it. biome's
direct-path checking means "find every violation" has to include the latent ones
that surface after the moves, not just what `pnpm run ci` prints now.

# 21:38 — re-cut to a coordination layer (supersedes "The four moves" above)

The 21:07 plan put the coordinator into `controller/`. In conversation the SC
pushed on it ("if you move the view into controller, what is the view?" → "is it
actually a violation for view to depend on controller?"), and we worked the
architecture out together. The pivot, and why it matters for whoever builds this:

## What the discussion established

- The biome rules are **compile-time import** constraints; classic-MVC diagrams
  are **runtime collaboration**. Different axes. This codebase is not classic MVC
  — it cut the View↔Controller link and routes everything through the stores
  (unidirectional / reactive-store, Model as the hub). That's *why* it can be
  layered at all: the graph is a DAG pointing down to `model/`, not a cycle.
- `view ⊥ controller` is a **correct** rule (decisions 1 & 2: views render only,
  view and input-handling meet only at shared state). The violation is real, not
  pedantic. `Presentation`/`ViewHost` aren't views — they're the coordinator.
- The rules are **asymmetric**: `view ⊥ controller` exists, `controller ⊥ view`
  doesn't. Folding the coordinator into `controller/` (the 21:07 plan) only works
  by leaning on that gap. A dedicated layer lets you close it.

## The decision (SC-authorised, widens the "not a redesign" brief)

Dedicated **`app/` coordination layer** below `entry/`, above view/controller/
model. `Presentation`, `PrimaryPresentation`, `ViewHost`, **and `TerminalInput`**
move there. `controller/` drops to model-only and gains a new `biome.json`
forbidding `../view/*` and `../app/*`; `model/` and `view/` gain `../app/*`.

`app/` ≠ `entry/`: `app/` is reusable, unit-tested coordination machinery
(`ViewHost.spec.ts`); `entry/main.ts` is the run-once composition root. The
construction block in `main.ts` is **identical** under either approach — the only
difference is import sources and which `biome.json` can be written. That's the
whole argument for B: it's not about how `main.ts` reads, it's that `controller/`
becomes *provably* leaf-pure.

## The finding that "just move the classes" hides

Adding `controller ⊥ view` (the rule that justifies B) turns a currently-legal
import into an error: `EditorHandler.ts:7` imports `EDITOR_PREFIX_VISUAL_WIDTH`
from `view/renderEditor.ts`. So B is **not** only moving classes — that shared
constant must be relocated. New home: `model/editorLayout.ts` (bottom layer, both
sides import down). Key detail: `renderEditor` *defines but never consumes* the
constant (it uses `PROMPT_PREFIX`/`INDENT` directly; the const exists only to feed
the handler). So `renderEditor` just drops the export — the prefix strings stay
view-private; only the width `3` moves. The `3`↔strings coupling is pre-existing
and accepted, cross-referenced by comments.

This is the one genuinely new design call in B, and it's the thing a mechanical
reading would miss and the build would catch as a red `controller ⊥ view`.

The plan file is fully re-cut to B (same path). The `AppModeKey → model/` move and
the non-layering-eight disposition carry over unchanged.
