Configurable tool permissions (#101). Shipped as two commits: seams and failing tests (Phase 1), then implementation (Phase 2).

## What was built

The hard-coded `permissions` const in `permissions.ts` is gone. `getPermission` now takes the permission matrix as a required parameter. The config schema has a `permissions` block (two zones: `default` for inside cwd, `outside`; each with `read`/`write`/`delete` as strings `approve`/`ask`/`deny`). Defaults reproduce the original matrix exactly.

The live-read wiring is in `main.ts`: `getPermissionMatrix` reads `configLoader.config.permissions` at call time (not snapshotted at construction) and maps config strings to `PermissionAction` via `actionFromConfig`. This is the same pattern as `getEffectiveModel` -- a permission config edit takes effect without a restart.

`AgentMessageHandler` gains `getMatrix: () => PermissionConfig` in its options. The handler calls `this.#getMatrix()` at approval time, so it always uses the current config.

`schema/sdk-config.schema.json` is a generated JSON-schema mirror of the Zod schema. It is regenerated by `pnpm --filter @shellicar/claude-sdk-cli run build` and must be staged alongside `schema.ts` changes.

## Traps

Tools with no file path (`Exec`, tools called with `input: {}`) resolve to the `default` zone. The `AgentMessageHandler` tests call `Find` and `DeleteFile` with `input: {}`, so they use default-zone permissions (Approve for read, Ask for delete). This was the pre-existing behaviour and was preserved.

`zonePermissionsSchema` has field-level `.catch('deny')` defaults on each action field. These only fire for partial zone objects (e.g. a zone with only one of the three fields present). No test exercises this path. The stub left them all-deny; the Builder left them as-is. Not a bug, just a gap.

`AgentMessageHandler.spec.ts` was not in the Phase 2 stage list but had to change: `getMatrix` became a required option, breaking the existing test factory. A stub (`getMatrix: () => defaultMatrix`) was added; the 35 existing test assertions are unchanged.

## String-to-enum mapping

Config stores action values as strings (`approve`/`ask`/`deny`). `PermissionAction` is a numeric TypeScript enum. The mapping lives in `main.ts`'s `actionFromConfig`: `approve` → `Approve`, `ask` → `Ask`, anything else → `Deny`. Future operators adding new zones or actions need to update this function as well as the Zod schema.
