## The two NodeConfigWatcher defects (now fixed)

The old `NodeConfigWatcher` had two defects:
1. **Absent file**: `watch()` threw if the file didn't exist yet â€” error was swallowed, file never picked up.
2. **Inode swap**: watched the file path, so editors that atomically replace the file (vim) left a stale watch on the old inode.

`NodeDirectoryWatcher` watches the parent directory instead of the file. Both defects vanish: the watch is set regardless of whether the file exists, and directory events fire regardless of inode swaps. Groups paths by parent directory â€” one `fs.watch` per directory, filtered by basename.

`ConfigLoader` picks up the fix automatically because it depends on the `IConfigWatcher` abstraction, not the concrete class. Only `main.ts` needed rewiring.

## Option B: per-source wire blocks

The locked design called for each source to be its own `system` block in the API array. The current SDK joined them into one. Option B meant changing `RequestBuilder` to spread them as separate entries. The Apostle surfaced this fork explicitly; the SC chose Option B.

`cache_control` goes on the last block only. Previously it was on the single joined block; the contract is identical at the API level.

Two `RequestBuilder` tests were rewritten to assert the per-source shape. The other 41 are unchanged.

## Buffer fallback deviation from the plan

The Apostle plan included a `filename.toString()` fallback for `Buffer` filenames in the `fs.watch` callback. Node's typed callback narrows to `string | null` (not `string | Buffer`) for this overload, so the else branch was `never`. Removed without functional impact.

## Schema stub trap (Scaffolder knowledge)

`z.any()` in a Zod v4 object schema fails parsing entirely when the field is absent. The Scaffolder used inverted boolean defaults (`false` instead of `true`) to force the defaults tests red, while keeping the schema structurally correct so unrelated tests (`initConfig.spec.ts`) stayed green. The Builder restored the correct `true` defaults.
