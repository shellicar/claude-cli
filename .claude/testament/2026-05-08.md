## Correction after Phase 1 debrief

The SC noted that `apps/claude-cli` should not receive voluntary dep updates in maintenance releases — it is legacy and not actively developed. The ncu-driven changes (claude-agent-sdk, zod, @types/node patches) were reverted and unstaged, and changes.jsonl entries for it were also reverted.

CLAUDE.md was updated to document the principle: apps/claude-cli is excluded from maintenance releases and dep updates, but not bumped unless a shared published package requires explicit version alignment.

The SC then ran `fix-mismatches`, which re-modified apps/claude-cli/package.json for syncpack consistency: workspace:* → workspace:^ for internal packages, the long-overdue @shellicar/mcp-exec pin (1.0.0-preview.6) migrated to workspace:^, and zod/@types/node aligned to the workspace high-water mark. These are not voluntary dep bumps — they are syncpack-forced consistency fixes that would accumulate as mismatches indefinitely if skipped. apps/claude-cli was re-staged with these changes only. No changes.jsonl entry was added.

Principle: legacy app skips voluntary ncu sweeps (no consumer benefit), but syncpack consistency stays. If a CVE in any transitive used by claude-cli ever lands, `pnpm audit` blocks the publish workflow on the active packages anyway — keeping legacy aligned with syncpack pays ahead for that case.

The staged set for Phase 2 covers all six packages: apps/claude-cli, apps/claude-sdk-cli, packages/claude-core, packages/claude-sdk, packages/claude-sdk-tools, packages/mcp-exec. apps/claude-cli is included because its package.json changed (via syncpack).
