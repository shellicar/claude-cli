## Phase 4 — Courier: wrapped arrow nav (#319)

### What shipped

Up/down arrows in `claude-sdk-cli` now move the caret between visual rows of wrapped input, instead of skipping over the wrapped portion to the next logical line. The fix is additive — the buffer/caret model still stores logical lines, the renderer still wraps for display, and `EditorState` is unchanged in shape. Two new methods do the visual-row arithmetic, and `AppLayout` calls them before `EditorState.handleKey` ever sees the keypress.

### How the pieces fit

- `EditorState.moveUpVisual(cols, prefixWidth)` and `moveDownVisual(cols, prefixWidth)` compute the current visual position via `prefixWidth + stringWidth(line.slice(0, cursorCol))`, derive the row and the column-in-row, and find the offset in the destination row whose visual width matches. The destination clamps to a grapheme boundary if the same column would land mid-character.
- `colFromVisual` is a module-level helper that walks graphemes via the existing `graphemeSegmenter`. The overshoot branch (`if (w + sw > targetVisual) return index`) is what gives wide characters their clamp behaviour.
- `EDITOR_PREFIX_VISUAL_WIDTH = 3` is exported from `renderEditor.ts`. `AppLayout` imports it and passes `this.#screen.columns` for `cols`. Single edit point if the prompt prefix ever changes width.
- `AppLayout.handleKey` intercepts `'up'` and `'down'` between the command-mode guard and the `editorState.handleKey` call, then `scheduleRender()` and returns.

### The trap that shaped this

`EditorState.handleKey` originally had `case 'up'` and `case 'down'` that moved by logical-line index. They became unreachable once `AppLayout` started intercepting, but a chunk of unrelated tests (`delete`, `ctrl+delete`, `ctrl+k`, both `right` cross-line tests) used `key('up')` purely as a cursor-positioning step before exercising something else. That backchannel kept dead branches alive for test convenience.

The fix: add an optional `EditorStateInit` argument to the constructor so tests can declare their starting state directly (`new EditorState({ lines, cursorLine, cursorCol })`), then remove the dead branches and rewrite the dependent tests to construct state instead of typing it. The constructor defensive-copies the `lines` array — without that, an external array mutation would propagate into editor state. `EditorStateInit` is a named module-scope type, not an inline literal, to match the codebase style.

The old `describe('EditorState — up')` and `describe('EditorState — down')` blocks went with the dead branches. The new contract lives under `describe('EditorState — moveUpVisual')` and `describe('EditorState — moveDownVisual')`.

### Coverage gap recorded for follow-up

Tests are ASCII-only except for two wide-char cases the SC added during Phase 1 review. `cursorCol` is a code-unit offset and `stringWidth(line.slice(0, cursorCol))` is a visual width; for ASCII they collapse to the same number, so a future edit that swaps one for the other could pass every test. The two wide-char tests (`'中'.repeat(7)` and `'a' + '中'.repeat(6)`) exercise the `colFromVisual` overshoot branch through the public methods. Not enough to fully close the gap; recorded so whoever picks it up can see the shape.

The Builder fixed them by hand because they were errors *in the new code*, not surrounding code: braces around two single-line `if` returns in `colFromVisual` (`useBlockStatements`), and `public` on the constructor (`useConsistentMemberAccessibility`). CI gate (`pnpm run ci`) blocks on these.

## Phase 5: Cleaner — mcp-exec interface alignment

### What broke and why

PR #328's CI failed type-check on `apps/claude-cli`. The errors were all from the same cause: `apps/claude-cli` was importing entities from `@shellicar/mcp-exec` that the new workspace package no longer exports, and one call site had a stale argument.

Specific errors:
- `ExecInputSchema` — not exported from mcp-exec (schema lives in claude-sdk-tools)
- `ExecInput` — same
- `expandPath` — moved from mcp-exec to `@shellicar/claude-core/fs/expandPath`, and signature changed: now requires `IFileSystem` instead of `{ home?: string }`
- `session.ts(158)` — `createExecServer({ cwd: process.cwd() })` called with an argument; the new version takes none

### What was changed

**Upstream packages (adding missing exports):**
- `packages/claude-sdk-tools/src/entry/Exec.ts`: added `ExecInputSchema` and `ExecInput` exports (the type already existed in `src/Exec/types.ts`)
- `packages/claude-sdk-tools/src/entry/fs.ts`: added `NodeFileSystem` export alongside the existing `nodeFs`
- `packages/mcp-exec/src/entry/index.ts`: re-exports `ExecInputSchema`, `ExecInput`, `NodeFileSystem`, `nodeFs` from claude-sdk-tools; renamed the local `ExecInput` alias to `ExecHandlerInput` to avoid the name collision

**Consumer changes in `apps/claude-cli`:**
- `matchRules.ts`, `ruleMatchesProgram.ts`: imports changed to claude-core for `expandPath`; function signatures changed from `home: string` to `fs: IFileSystem`; `expandPath(value, { home })` → `expandPath(value, fs)`
- `isExecAutoApproved.ts`: same expandPath change; uses `nodeFs` from mcp-exec for the production path
- `isExecPermitted.ts`: `home?: string` param → `fs?: IFileSystem`; defaults to `nodeFs`; `import { homedir }` removed
- `session.ts`: `createExecServer({ cwd: process.cwd() })` → `createExecServer()`

**Test changes:**
- `test/execPermissions.spec.ts`: added `TestFileSystem extends NodeFileSystem` that overrides `homedir()` and `getEnvVar('HOME')` to return the controlled test value `/home/testuser`; updated `match` and `permitted` helpers to pass `testFs` instead of the home string
- `test/autoApprove.spec.ts`: no changes needed — it uses `process.env.HOME ?? '/home/testuser'` which adapts to the real system, and `isExecAutoApproved` now uses `nodeFs` directly

### Key decisions

**Why TestFileSystem extends NodeFileSystem rather than implementing IFileSystem from scratch**: the mission says "do not craft a partial object or stand-in." A proper subclass that overrides only what the test needs is the right pattern. The test already used a fixed home string for a reason — these are coverage tests for the glob/expansion logic, not integration tests against the real filesystem.

**Why re-export through mcp-exec rather than having apps/claude-cli import from claude-sdk-tools directly**: apps/claude-cli's package.json doesn't list claude-sdk-tools as a dep. mcp-exec does. The dependency chain already exists; routing the new exports through mcp-exec avoids adding a new dep declaration to the legacy app.

**Why `ExecHandlerInput` rename**: the local type alias `type ExecInput = Parameters<...>` in index.ts conflicted with the new `export type ExecInput`. Renamed the local alias to `ExecHandlerInput` since it's only used inside the handler closure; external callers get the canonical `ExecInput` from the re-export.

### Verification

Build, type-check, lint, and tests all passed locally. The CI was failing specifically on `@shellicar/claude-cli#type-check`. All 11 errors listed in the CI log are addressed by these changes.
