# Session 2026-04-10

## Drift pattern to watch for (post-compaction reminder)

**The walk's purpose is: "what does an implementer need to know to execute the plan".**
**It is NOT: "does the code match the plan".**

`sdk-shape.md` is a refactor plan. The code does not match it. That is the entire
reason the plan exists. Writing out a table of "plan promises X / code delivers Y /
match ❌" is pointless: every row will be ❌, because that is what "refactor plan"
means.

### What this drift looks like in practice

When walking `Conversation.ts` earlier this session I produced a table mapping
plan-block promises (push, replace, insert, clear, cache-annotated, ...) to code
(push ✓, replace ❌, insert ❌, clear ❌, cache annotation ❌, ...) and then turned
each ❌ into a numbered question asking Stephen to confirm whether the missing
piece was "in scope". Every one of those questions was nonsense: the plan is the
scope. If the plan says Conversation provides `replace`, then the implementer adds
`replace`. There is no question to ask.

The failure mode: I was treating the code as the source of truth and the plan as
a document to be validated against it. That is exactly backwards. The plan is the
target. The code is the raw material being reshaped towards it.

### The correct shape of a walk-finding

For each file, the useful content is:

1. **What is in this file that an implementer needs to know when they touch it.**
   Surprises, edge cases, non-obvious invariants, things that will bite if missed.
   Example from `Conversation.ts`: `push()` merges consecutive user messages and
   drops the `id` tag on merge. An implementer adding `insert` has to decide
   whether `insert` enforces alternation or inserts verbatim; the existing
   behaviour is the precedent they will have to consider.

2. **Blast radius of the refactor action.** Call sites, test files, dependent
   code. Example from `ConversationStore.ts`: 9 call sites in tests, all using
   the no-arg form, so the test-side change is a mechanical type swap.

3. **Real questions with answerable actions** — things genuinely ambiguous in the
   plan, where an implementer cannot proceed without a decision. These are rare.
   Most "questions" are actually notes-to-self that I can resolve by reading
   another file in a later round, or are me asking the plan to validate itself
   against the code.

### The filter to apply before writing a question

Before writing "Question N: ...", ask:

- Is this something the plan already answers if I read it carefully? → not a question.
- Is this something I can answer myself by walking another file next round? → note-to-self, not a question.
- Is this me asking Stephen to confirm that the plan is the plan? → not a question.
- Is this genuinely ambiguous in the plan and blocking an implementer decision? → question.

This is the same discipline as the previous session's "point at a specific plan line"
rule, but for a different failure mode. The 2026-04-09 drift was importing outside
concerns. The 2026-04-10 drift is treating the code as ground truth. Both failures
dress up as careful analysis. Both are noise.

### Why this note exists

Context compaction has already happened once this session. When it happens again,
this section is here so the next compacted-me reads the rule before writing any
more walk output.

## Drift pattern 2 (this session, second half): the refactor is not a feature build

**The refactor's job is to extract the blocks the plan describes. It is NOT to**
**implement capabilities those blocks are described as having that the current**
**code does not yet provide.**

`sdk-shape.md` can be read two ways:

- "Block X is responsible for A, B, C, D" (a list of things the refactor has to
  implement).
- "Block X is the home A, B, C, D would live in if they existed" (a shape
  statement naming where future features go).

The second reading is the correct one for this refactor. When the plan says
"Stream processor tracks per-TTL cache split, tracks iterations, processes
streamed-or-otherwise responses", those describe where those responsibilities
would live once they exist. The current code does not track per-TTL split, does
not read `usage.iterations`, and does not process non-streamed responses. The
refactor does not add any of that. It creates the shape in which someone can
add them later.

### What this drift looks like in practice

Walking `MessageStream.ts`, I produced a list of "gaps an implementer will
fill": add a raw event pass-through event, add text block lifecycle events,
add tool input delta events, read `usage.cache_creation.ephemeral_5m_input_tokens`,
thread `usage.iterations` through the result, extend `MessageUsage` to carry
per-TTL subtotals, add a non-streamed entry point to the processor. Every one
of those is new functionality. None of it is refactor work. The behavioural
contract on the new interface describes the *current* behaviour of the class,
verbatim, not the aspirational behaviour named in the plan.

### The rule

When reading a block's description in `sdk-shape.md`, ask: does the current
code already do this? If yes, the refactor preserves it and the contract
describes it. If no, the refactor leaves it alone and the contract is silent
on it. New capabilities are follow-up work, not refactor scope.

This rule sits alongside the "plan is the target, not the code" rule above.
Both describe a reader of `sdk-shape.md` getting the relationship between
plan and code wrong, in different directions. The first drift reads the code
as truth and asks the plan to justify itself. The second drift reads the plan
as a feature list and asks the refactor to deliver features. Both are noise;
both have to be filtered out before anything useful comes out of a walk.

## Discipline: handling questions and answers

### Rule: answers get written into this log the moment they are made

Context compaction eats anything not written down. When Stephen answers a
question, the answer goes into this log *in the same turn it is given*, not
"later when I update the log". This is not optional.

Q5 below (the `AGENT_SDK_PREFIX` question) was answered in a prior session
and lost because the answer was never captured, and then the same question
was asked again in this session, and Stephen was (correctly) angry about it.

The rule: every time a question is answered, a PreviewEdit → EditFile →
commit cycle updates this log with the answer, in the same turn. Proceeding
without recording is a bug.

### Rule: the plan is also a confusion-prevention document

`sdk-shape.md` is not only a description of what we want to change. It is
also a document someone reads without the session log in their other hand.
If a question keeps coming up from a reader of the plan, it is a confusion
point, and the plan gets updated to answer it, even if the fact being
documented is an API-level constraint and not a design decision.

The `AGENT_SDK_PREFIX` question (Q5 below) is the worked example. It came
up in 2026-04-09 and Stephen said yes, update the plan. The plan was never
updated. It came up again in this session's RequestBuilder walk, and
Stephen was correctly angry that a question he had already answered for
updating the plan had resurfaced. The plan-edit commit is `0bdfa49`.

The rule: when a question's answer is "that is an API-level constraint
the SDK has no choice about", the constraint still goes in the plan,
because the plan reader will otherwise ask the same question. The plan
documents confusion points, not only change points.

### Rule: questions to Stephen are in plain English

If a question cannot be phrased without quoting a line from `sdk-shape.md`
or naming a principle by its title, the question is not ready to ask. Keep
refining until the question reads as plain English that makes sense without
the plan in the reader's other hand. Signs a question is not ready:

- starts with "Does the [principle-named-in-the-plan] apply to X"
- needs three bullet-point sub-options to be understood before anyone can
  answer it
- quotes more than one clause from the plan
- uses the word "cashes out"

The plain English version of Q4 below would have been: *"should the
consumer's compaction settings move from three flat fields into one grouped
object, as part of this refactor?"* That is a question anyone can answer.
The jargon version I actually wrote was a question only someone who had
re-read the plan that morning could answer.

### Rule: do not launder a decision as a question

If you already know the answer, answer it. Do not wrap a decision
in "Question: should we do X? My recommendation is X" framing.
Stephen cannot evaluate a recommendation without seeing the
alternatives and their tradeoffs. Either present a real choice with
both options' pros and cons on the table, or present a decision
with rationale. Anything in between is a decision decorated as a
question, which asks Stephen to validate a conclusion he has no
basis to evaluate.

Two signs of this failure:

- "Should X live at A or B? My recommendation: A." with no cons
  for A or pros for B laid out.
- "Does [principle] apply to Y? I would answer yes." where the
  answer is obvious and the question exists only to confirm the
  principle still holds.

The test: if Stephen reads the "question" and has no material to
evaluate it against, it is not a real question. If the answer is
clear to you, record the decision with reasoning directly. If the
answer is genuinely ambiguous, present both options with full pros
and cons, then either choose one yourself (with the chosen/rejected
labels) or ask for a call with the alternatives visible.

Example from this session. In the AgentRun walk I raised four
"questions" that were all decisions-with-recommendations. Two were
clear under existing discipline rules and did not need asking at
all (preserve tool-not-found asymmetry, do not add imperative
cancel). Two were real choices but presented without the opposing
side's pros and cons (abort signal assembly location, where
`transformToolResult` lives). Stephen called both failures out in
the same turn and asked for them to be re-presented as real
decisions. The four design decisions recorded later in this log
use the correct shape.

## Resolved questions (this session)

### Q3 (File 6, `RequestBuilder.ts`): plan wording for the return-value shape

**Resolution.** The Request builder block takes inputs and maps them to
what the HTTP request needs. The wording of the plan's return shape is not
a choice point. Whether the return value is called "headers", "request
options", or "the options bag" in prose is not something to adjudicate. No
plan edit.

### Q4 (File 6, `RequestBuilder.ts`): should `RequestBuilderOptions` be reshaped to grouped form in the refactor?

**Resolution.** No. Drift-pattern 2 applies: the refactor extracts blocks
and preserves current behaviour. The consumer-facing options stay flat.
Grouping `compactInputTokens` + `pauseAfterCompact` + `betas.Compact` into
`compaction: { enabled, inputTokens, pauseAfterCompact }` is a separate
change and is not in this refactor's scope.

(Self-resolved by applying the rule, not asked of Stephen. It should not
have been a question in the first place.)

### Q5 (File 6, `RequestBuilder.ts`; originally from 2026-04-09): is `AGENT_SDK_PREFIX` a consumer policy decision?

**Resolution.** No. The prefix is required by the Anthropic API. The SDK
cannot make calls without it. It is not a consumer policy decision, not
optional, and not overridable. It stays hardcoded, and the refactor does
not add a consumer-facing way to disable or replace it.

**Also documented in the plan as of commit `0bdfa49`.** The Request
builder block in `sdk-shape.md` now carries a sentence saying the system
prompt always begins with `AGENT_SDK_PREFIX`, that this is API-required,
and that it is not consumer-overridable.

**History of this question.** Asked in 2026-04-09, answered yes with
instruction to update the plan, plan was not updated. Re-asked in
2026-04-10 during the RequestBuilder walk. Stephen was (correctly) angry
that a question he had already answered had resurfaced. Both the session
log resolution above and the plan edit at `0bdfa49` landed in this turn.
The two discipline rules ("answers get written down immediately" and
"the plan is also a confusion-prevention document") were added in
response to this specific failure.

   per request). Public surface: `resolve(toolUse)` returning tagged
   union `{ kind: 'ok', tool, parsedInput } | { kind: 'not-found',
   name } | { kind: 'invalid-input', error }`, `execute(resolved,
   transformToolResult?)` returning a `tool_result` block, `schemas()`
   returning cached wire schemas for the request builder. Pure. No
   dependencies.

   Approval coordinator holds: `#pending` map, `#cancelled` flag, a
   reference to the channel (the piece that moves from AgentRun into
   the coordinator). Public surface: `requestApproval(toolUse):
   Promise<ApprovalResponse>` (internally generates id, sends
   outbound on channel, parks promise, returns it), `cancelled`
   getter, `handle(msg)` for inbound channel dispatch. Calls:
   `channel.send` for outbound approvals, the parked resolver
   functions on inbound response or cancel.

   Cross-block wiring: query runner holds the four things that cross
   turn boundaries (channel, coordinator, abort controller, loop
   counters). Turn runner holds nothing across turns. Tool registry
   holds state that is durable across queries (session-level).
   Approval coordinator holds state that is per-query (channel
   lifetime). The channel's inbound dispatch fans out from one
   handler to two targets (`coordinator.handle` and
   `abortController.abort()`), and that fan-out lives with the query
   runner because that is the block that owns both.

9. `packages/claude-sdk/src/private/AnthropicAgent.ts` (50 lines)
   plus `packages/claude-sdk/src/public/createAnthropicAgent.ts` (8
   lines) and `packages/claude-sdk/src/public/interfaces.ts` (20
   lines) — the misshapen Session block. Short walk; the untangling
   work lived in AgentRun, not here. Things an implementer needs to
   know:

   **Current shape.** Constructor takes `{authToken, logger,
   historyFile?}`, builds an `AnthropicClient` internally, an
   `AgentChannelFactory` internally, and a `ConversationStore` (with
   `historyFile`). `runAgent(options)` constructs a new `AgentRun`
   per call, passing the client, channel factory, logger, query
   options, and the shared history store. Four history methods
   (`getHistory`, `loadHistory`, `injectContext`, `removeContext`)
   delegate directly to `ConversationStore`.

   **The 14-field `RunAgentQuery` IS the "rebuild per call"
   problem.** Fields: `model`, `thinking`, `maxTokens`, `messages`,
   `systemPrompts`, `tools`, `betas`, `requireToolApproval`,
   `pauseAfterCompact`, `compactInputTokens`, `cacheTtl`,
   `transformToolResult`, `systemReminder`, `cachedReminders`. Every
   field except `messages` is work the consumer has to redo on
   every call, even though almost none of it actually changes
   between calls.

   **Refactor mapping. 11 durable, 3 per-query.** Plan Session block
   says "bundles client + conversation + durable config". Plan query
   runner block says "consumes per-query input once (the first turn
   gets `systemReminder`, subsequent turns do not)". Crossing the
   two, the split is:

   Durable (move to session construction):
   - `model`, `thinking`, `maxTokens`
   - `systemPrompts`, `tools`, `betas`
   - `requireToolApproval`, `pauseAfterCompact`, `compactInputTokens`
   - `cacheTtl`
   - `cachedReminders`

   Per-query (stay on the `session.query(input)` call):
   - `messages` — the user ask this query
   - `systemReminder` — one-shot, cleared after first turn (plan text)
   - `transformToolResult` — behavioural hook, semantically per-query
     (design decision 2 from the AgentRun walk)

   This split is not a new decision; it is derivation from the plan
   Session and query runner blocks. Recording it here because the
   implementer executing the refactor needs the mapping in one
   place.

   **`messages: string[]` is "new messages this query", not the
   whole conversation.** The Conversation lives on the Session.
   Per-query messages are pushed into it by the turn runner. The
   `string[]` shape exists because the consumer passes raw text and

   Session block exposes `.conversation` directly. Consumers then
   call `Conversation.push/remove/messages/cloneForRequest`
   themselves. The four methods vanish:
   - `getHistory()` becomes `session.conversation.messages`
   - `loadHistory(msgs)` becomes a loop of
     `session.conversation.push(msg)`
   - `injectContext(msg, {id})` becomes
     `session.conversation.push(msg, {id})`
   - `removeContext(id)` becomes `session.conversation.remove(id)`

   The only history-related test surface is `Conversation.spec.ts`
   (which tests `Conversation` directly, including the three
   `load()` call sites noted below). Deleting the four methods has
   no test fallout.

   `ConversationStore` is deleted.** Exactly one runtime caller
   (`ConversationStore.ts:30`). The method is explicitly marked
   "only `ConversationStore` should call this" in its JSDoc.
   Removing it is refactor scope (dead code removal). Three test
   call sites in `Conversation.spec.ts` at lines 219, 228, 236

   `push()` as part of the delete.

   rule ("SDK does not touch the filesystem") plus the
   `ConversationStore` delete together force the removal.

10. `packages/claude-sdk/src/private/Auth/` (20 files) plus
    `packages/claude-sdk/src/private/http/TokenRefreshingAnthropic.ts`
    (60 lines) — OAuth plus per-request refresh plumbing. Confirmation
    walk; the plan already captures the shape as the Client block's
    responsibility. Things an implementer needs to know:

    **20 files, most are tiny helpers.** `AnthropicAuth.ts` (55
    lines) is the orchestrator. Everything else is OAuth ceremony:
    `base64UrlEncode`, `generateCodeVerifier`, `generateCodeChallenge`,
    `generateState`, `buildAuthUrl`, `exchangeCode`,
    `refreshCredentials`, `waitForCallback`, `fetchProfile`,
    `parseTokenResponse`, `isExpired`, `schema`, `consts`, `types`,
    two error classes, plus `loadCredentials` / `saveCredentials` /
    `credentialsPath`. Per the "not every helper is a building
    block" principle these are wiring inside the Client block, not
    substitution surfaces in their own right.

    **`AnthropicAuth.getCredentials()` is the single entry point.**
    Load from file; if missing, run the login flow and enrich with
    `fetchProfile` then save; if expired, refresh then save;
    otherwise return as-is. `login()` has two modes: `'local'`
    spawns `execFile('open', [url])` and runs a local HTTP callback
    server on port 3001; `'manual'` prints the URL and reads a
    pasted code from stdin.

    **`TokenRefreshingAnthropic` is the mechanism that makes refresh
    invisible.** Subclasses the Anthropic SDK and overrides
    `apiKeyAuth` / `bearerAuth` to call a token-getter function per
    request. The Anthropic SDK constructor discards non-string auth
    values silently; the subclass captures the getter before
    `super()` drops it. Every HTTP call re-fetches the token, so an
    externally-driven refresh (via `getCredentials` detecting
    expiry) is picked up on the next call automatically. The
    refactor preserves this mechanism — the Client block ends up
    using it internally.

    **`AnthropicAuth` is exported as public API at
    `index.ts` line 31** despite living under `private/Auth/`. The
    file path says private, the export says public. Current
    consumer (the CLI at `apps/claude-sdk-cli/src/entry/main.ts`
    lines 58-63) uses the public shape: `new AnthropicAuth({
    redirect: 'local' })`, eager `await auth.getCredentials()` once
    to force login at startup, then an `authToken` closure that
    re-calls `getCredentials()` per request and returns the access
    token string.

    **Plan implication.** Client block "owns authentication (token
    acquisition, OAuth flow, token refresh)". That means
    `AnthropicAuth` becomes an internal collaborator of the Client
    block and stops being exported as public API. The current
    two-box consumer wiring collapses: instead of constructing
    `AnthropicAuth` separately and wiring a token closure, the
    consumer passes auth options directly into the Client (or
    Session) and the default assembly handles OAuth internally. A
    consumer that wants different auth substitutes the entire
    Client block — the substitution boundary is at the Client
    block's behavioural interface, not at an Auth sub-interface.
    Folding `AnthropicAuth` in is restructuring, not new capability,
    so it is in refactor scope.

    **Eager-login UX preservation.** The CLI's current shape calls
    `getCredentials()` eagerly before the interactive loop so the
    login prompt happens at startup, not on the first message. The
    refactor has to preserve this. The cleanest shape is an async
    factory: `const session = await createSession({...})` triggers
    the eager login inside construction, matches the "95% case is
    simple" principle, and hides the login step from consumers who
    do not care. An explicit `session.login()` method is the more
    mechanical preservation of the current two-step flow but adds
    a required method call for every consumer. Lazy-on-first-query
    is ruled out because it changes the UX (first message hits a
    2-5 second delay). Recommendation: async factory. Final call
    is the implementer's when the Session block's shape settles.

    **File I/O is exactly two functions.** `loadCredentials`
    (`readFile`) and `saveCredentials` (`writeFile`), both calling
    `credentialsPath()` which joins `~/.claude/.credentials.json`.
    This is the full extent of SDK-touches-files within the Client
    block and matches the plan's "one pragmatic exception" line
    verbatim.

    **Implementer notes that are not refactor scope but should be
    flagged:**
    - `saveCredentials` uses `writeFile` with no `mkdir`. On first
      run if `~/.claude/` does not exist, save fails. Not refactor
      scope but an implementer touching this file should know.
    - `execFile('open', [url])` is macOS/BSD-only. Linux uses
      `xdg-open`, Windows uses `start`. Not refactor scope.
    - The credentials path is hardcoded `~/.claude/.credentials.json`.
      Not configurable. Per "SDK knows the convention, not the
      filesystem" this is fine. Preserve.
    - `fetchProfile` enrichment (subscription type, rate limit tier)
      runs only on first login, not on refresh. It is reading
      account metadata, not pure OAuth. Preserve the shape.
    - Schema validation (`schema.ts`) runs on both load and save.
      Catches disk corruption on load and construction errors on
      save. Preserve.
    - Two named errors: `InvalidAuthorisationCodeError` (401 on
      code exchange) and `TokenExchangeFailedError` (anything
      else). Behavioural contract; consumers may catch them.
      Preserve.
    - `AnthropicClient.ts` JSDoc at line 15 claims to "own auth".
      It owns refresh TIMING via `TokenRefreshingAnthropic`, but
      not the OAuth flow itself. The JSDoc needs updating when the
      refactor touches the file, per the `#232` pointer-update
      note already in commit `be2554a`.

### Design decisions from the AgentRun walk

These four decisions are recorded here so the next session has them
in writing. Two are genuine choices between defensible alternatives;
two collapse to a single answer under existing discipline rules.

**Decision 1. Abort signal assembly stays at the turn runner.**

Option A (chosen): builder returns `{ body, headers }`, turn runner
merges `{ headers, signal }` before calling the client. Pro: the
builder stays a pure config-to-shape function; the signal is runtime
wiring, not config. Pro: the query runner owns the abort controller,
and the signal only crosses the turn runner boundary, not the
builder boundary. Con: "request options" is assembled in two steps
(headers at builder, signal merge at turn runner).

Option B (rejected): builder takes the signal as an extra input and
returns `{ body, requestOptions }` fully assembled. Pro: literal
reading of the plan's "builder returns request options". Pro: one
place assembles request options. Con: the builder stops being pure,
takes a runtime abort signal as input. Con: the builder starts
knowing about cancel wiring.

Rationale: the plan calls the builder "Pure function" and "Stateless"
explicitly. Option B breaks the purity the plan is asking for, and
the plan's "returns request options" phrasing is about the return
shape, not about which block owns every field of it. Signal stays
at the turn runner.

**Decision 2. `transformToolResult` stays per-query input.**

Option A (chosen): lives on the per-query input object, turn runner
passes it into `registry.execute(resolved, transformToolResult)` per
call. Pro: different queries can apply different transforms (a
"fetch file" query swaps file bodies for refs; a "show status" query
does not). Pro: registry stays stateless with respect to per-query
hooks. Con: extra parameter threaded through every `execute` call.

Option B (rejected): lives on the session as durable config, registry
reads it from the session. Pro: removes a parameter from `execute`.
Pro: consistent with other "rebuilt every call" fields that become
durable. Con: denies per-query transforms. Con: denies query-local
transformer state (e.g. a per-query ref cache).

Rationale: the other durable-config fields in the plan (model,
tools, cache TTL, betas) are structural. The transformer is a
behavioural hook, semantically more like the per-query user input
than like session structure. Per-query input, passed into
`registry.execute` per call.

**Decision 3. Preserve the tool-not-found channel-send asymmetry
during the refactor.**

Today, missing tool is silent on the channel (logs only); invalid
input broadcasts via `channel.send('tool_error')`. Under the
refactor-only rule (drift pattern 2: the refactor is not a feature
build), changing this is a behaviour change and does not belong in
the refactor. Preserve as-is, with the asymmetry flagged here so a
follow-up work item can find it.

The asymmetry may be intentional or an oversight. A plausible
intentional reading: missing tool is an SDK or consumer config bug
the consumer cannot react to live (the consumer wrote the tool
array in the first place, so a missing tool is their own bug and
there is nothing the running agent can do about it); invalid input
is a model behaviour the consumer may legitimately want to observe
(for retry strategy, error display, telemetry). That reading would
justify silence on tool-not-found and a broadcast on invalid input.
A future unification pass has to decide whether to accept that
reading or reject it; this log flags the question so the decision
is findable.

Follow-up candidate resolutions, not in this refactor:
- unify to both-broadcast (missing tool becomes a channel event too)
- unify to both-silent (invalid input becomes log-only)
- keep asymmetric, document the rationale at both call sites

**Decision 4. Do not add an imperative `cancel()` method to the
approval coordinator.**

Cancellation stays channel-only: an inbound `cancel` message
processed by `handle(msg)`. Adding an imperative method introduces
a second cancel surface with the same semantics, which means two
code paths to keep consistent. Under the refactor-only rule, this
is new API surface and not in scope. If a later feature needs
programmatic cancel, it is a separate change that has to think
carefully about consistency between the two paths.
