# Session 2026-04-10

## Drift pattern to watch for (post-compaction reminder)

**The walk's purpose is: "what does an implementer need to know to execute the plan".**
**It is NOT: "does the code match the plan".**

`sdk-shape.md` is a refactor plan. The code does not match it. That is the entire
reason the plan exists. Writing out a table of "plan promises X / code delivers Y /
match ❌" is pointless: every row will be ❌, because that is what "refactor plan"
means.

### What this drift looks like in practice

When walking `Conversation.ts` earlier this session I produced a table mapping
plan-block promises (push, replace, insert, clear, cache-annotated, ...) to code
(push ✓, replace ❌, insert ❌, clear ❌, cache annotation ❌, ...) and then turned
each ❌ into a numbered question asking Stephen to confirm whether the missing
piece was "in scope". Every one of those questions was nonsense: the plan is the
scope. If the plan says Conversation provides `replace`, then the implementer adds
`replace`. There is no question to ask.

The failure mode: I was treating the code as the source of truth and the plan as
a document to be validated against it. That is exactly backwards. The plan is the
target. The code is the raw material being reshaped towards it.

### The correct shape of a walk-finding

For each file, the useful content is:

1. **What is in this file that an implementer needs to know when they touch it.**
   Surprises, edge cases, non-obvious invariants, things that will bite if missed.
   Example from `Conversation.ts`: `push()` merges consecutive user messages and
   drops the `id` tag on merge. An implementer adding `insert` has to decide
   whether `insert` enforces alternation or inserts verbatim; the existing
   behaviour is the precedent they will have to consider.

2. **Blast radius of the refactor action.** Call sites, test files, dependent
   code. Example from `ConversationStore.ts`: 9 call sites in tests, all using
   the no-arg form, so the test-side change is a mechanical type swap.

3. **Real questions with answerable actions** — things genuinely ambiguous in the
   plan, where an implementer cannot proceed without a decision. These are rare.
   Most "questions" are actually notes-to-self that I can resolve by reading
   another file in a later round, or are me asking the plan to validate itself
   against the code.

### The filter to apply before writing a question

Before writing "Question N: ...", ask:

- Is this something the plan already answers if I read it carefully? → not a question.
- Is this something I can answer myself by walking another file next round? → note-to-self, not a question.
- Is this me asking Stephen to confirm that the plan is the plan? → not a question.
- Is this genuinely ambiguous in the plan and blocking an implementer decision? → question.

This is the same discipline as the previous session's "point at a specific plan line"
rule, but for a different failure mode. The 2026-04-09 drift was importing outside
concerns. The 2026-04-10 drift is treating the code as ground truth. Both failures
dress up as careful analysis. Both are noise.
