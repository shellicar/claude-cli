## SDK design reconstruction

Work started on `feature/usage-auditing`. That branch had added `RawEventWriter` and `AuditWriter` (calling `mkdirSync` and `appendFileSync`) inside the SDK — working around a design problem rather than solving it.

The session stepped back to ask why issue #232 existed at all. The six-step technical plan in #232 turned out not to contain the actual reasons the rewrite was needed. Those reasons had to be reconstructed from memory.

### The two underlying problems

**Problem 1: nothing was durable across queries.** The SDK was designed without any concept of a session, to avoid the persistence the official Claude Code SDK owns (session ids, session files, resume-by-id). Avoiding that persistence was correct. Avoiding it by refusing to have a session object at all was wrong. Every call to `runAgent` re-supplied ~15 fields, of which ~2 changed between calls.

The correct rule: no persistence or hydration of a session. Not no session. The SDK holds a session in memory for the life of the process. It never reads or writes it to disk.

**Problem 2: the SDK was touching files.** `ConversationStore` knows about file paths. `historyFile` is in `AnthropicAgentOptions`. The audit branch was adding file writers inside the SDK. None of that belongs in an API client library.
