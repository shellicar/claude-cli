### systemReminder bug fix — PR #228

Two bugs found and fixed before implementing CLAUDE.md loading:

**Bug 1:** `systemReminder` was passed to every `#getMessageStream` call inside the
agent loop, not just the first. Tool-result continuation turns were also receiving the
git delta block, so it would appear in the model context on every tool call, not
just the turn boundary.

Fix: consume `systemReminder` once before the loop (assign to local `reminder`), then
clear the query field. Pass `reminder` only to the first call; all subsequent calls get
`undefined`.

**Bug 2:** `AgentMessageHandler` stored `gitDelta` at construction time via
`AgentMessageHandlerOptions`. The delta was rendered on every `query_summary` message,
including multiple summaries within a single long tool sequence.

Fix: removed `gitDelta` from `AgentMessageHandlerOptions` entirely. Instead, `AgentRun`
includes `systemReminder` in the `query_summary` channel message itself. The handler
reads `msg.systemReminder` — stateless with respect to git delta. Added
`systemReminder?: string` to `SdkQuerySummary` in `public/types.ts`.

PR #228 opened on branch `fix/system-reminder-tool-result-injection`. Tests: 70 in

### CLAUDE.md loading — PR #229

Implemented issue #226. Design shifted from `systemPrompts` to `cachedReminders` during
implementation:

**Why `cachedReminders` instead of `systemPrompts`:** The Anthropic API requires that
cache-tagged blocks appear in the first user message (not the system prompt) for prompt
caching to apply. `cachedReminders` is a new optional field on `RunAgentQuery`. In
`AgentRun`, when the history has no user messages yet (fresh start or post-compaction),
they are prepended as a content block to the first user message with
`cache_control: { type: 'ephemeral' }`.

**Injection condition:** `!history.messages.some(m => m.role === 'user')` rather than
`history.messages.length === 0`. This correctly handles post-compaction state where the
assistant compaction block is present but no user messages exist. The Anthropic docs
confirm the API ignores all content before the compaction block, including the original
first user message that held the cached content.

**Load order:** `~/.claude/CLAUDE.md`, `<project>/CLAUDE.md`,
`<project>/.claude/CLAUDE.md`, `<project>/CLAUDE.local.md`. All present files included;
missing ones silently skipped.

**`ClaudeMdLoader`:** reads files at load time, exposes `getReminders(): string[]`.
Project-level files watched with 100ms debounce; home file read once at startup. Watcher
respects idle-only reload (same pattern as `SdkConfigWatcher`).

**Config:** `claudeMd.enabled` (default `true`) in `sdk-config.json` schema. Wired in
`main.ts` gated behind this flag.

# Session 2026-04-09 (SDK refactor planning)

Branch: `feature/sdk-refactor` (off `main`). Context was compacted earlier in the day; this section captures the design discussion before any code is written, so the plan survives another compaction.

## Trigger

Started from the usage-auditing work on `feature/usage-auditing` (raw events + `RawEventWriter` already landed as commits `424cb9e` and `d8d2555`). The next step was going to be session management, but before writing it the user asked to step back and question the overall SDK architecture. That question turned into a full rethink of the SDK shape. The rethink blocks session management cleanly, so it goes first on a new branch.

## What the current SDK shape actually is

Three classes in play:

- **`Conversation`** (`packages/claude-sdk/src/private/Conversation.ts`): pure in-memory message list. Holds `HistoryItem[]`, enforces role-alternation merge, and (critically for this refactor) **clears the array when a compaction block is pushed**.
- **`ConversationStore`**: file-backed wrapper around `Conversation`. Loads JSONL in constructor (calling `trimToLastCompaction` on load), saves atomically via tmp+rename on every push/remove.
- **`AnthropicAgent`**: constructs exactly one `ConversationStore` from `options.historyFile` in its constructor. That store lives for the agent's lifetime. `runAgent(options)` creates a new `AgentRun` that mutates the stored conversation.

CLI-side, `apps/claude-sdk-cli/src/runAgent.ts` wraps `agent.runAgent(...)` with a ~15-field `RunAgentQuery` of which only `messages` and `systemReminder` actually change turn-to-turn. The other 13 are constants recomputed on every call.

## Observations

1. **The configured-once vs per-run axis doesn't match usage.** In practice almost everything the CLI passes to `runAgent()` is session-scoped, not per-call. The split optimises for flexibility nobody uses.

2. **Persistence is baked into the SDK.** `historyFile` is an `AnthropicAgentOptions` field; the SDK does `readFileSync`, `writeFileSync`, `renameSync`. That forces decisions about file paths into the SDK when they're really consumer policy.

3. **Compaction conflates storage with transmission.** `Conversation.push()` clears the array when it sees a compaction block, and `ConversationStore.load()` truncates on load. Both are destructive. Once a compaction happens, pre-compaction messages are gone from memory *and* disk. Consequence: if the compaction turn itself errors, the session dies — there's nothing to roll back to. The user specifically called out wanting to recover from bad compactions.

4. **Cache control decorations live on stored messages.** `RequestBuilder` does a defensive spread-copy dance (`[...messages]`, `{ ...block, cache_control }`, etc.) purely to avoid leaking send-time markers into the stored conversation, because the same array is both storage and send-buffer. That pattern only exists because of the conflation.

5. **`RawEventWriter` / `EventParser` / `AuditWriter` are per-turn or module-level.** Wrong lifetime for anything session-scoped.

## Design agreed with the user

### SDK holds state, but never persists

- The SDK client holds the message array in memory and exposes it.
- No file paths, no session IDs, no JSONL inside the SDK.
- The CLI reads messages from disk on startup and hands them to the SDK. The CLI saves to disk after each turn by reading `client.messages`.
- Observation is pull-based: no `message_appended` events. The CLI reads the exposed array whenever it wants.

### Three-layer SDK shape (target)

- **`AnthropicClient`** — auth + HTTP + token refresh. Created once per process. Thin.
- **`AnthropicAgent`** — takes the client once, holds a `Conversation` instance, holds a config bag (model / tools / betas / cache settings / transformToolResult / cachedReminders / systemPrompts), exposes `configure(partial)` + `run({ systemReminder? })` + message operations. Represents one in-memory conversation. A "new conversation" = construct a new agent off the same client.
- CLI owns identity, persistence, audit.

### The compaction split (the important part)

**`Conversation` stores every message ever pushed, forever.** It never clears on compaction. Compaction blocks are data, not commands.

**Two message views on `Conversation`:**
- `messages` (read-only getter) — the full history, for CLI/TUI/audit/persistence.
- A clone method (name TBD — `snapshotForRequest` / `cloneForRequest` / `cloneApiMessages`) that returns a **deep-cloned slice from the last compaction forward**. That slice is the caller's to mutate freely.

`RequestBuilder` calls the clone method, gets its own working copy, mutates it in place (cache control, system-reminder injection) and sends. The defensive spread-copy pattern goes away because nothing is shared.

**Payoffs**:
- Compaction rollback works for free: `conversation.removeById(compactionMsgId)` makes the next `apiView` look further back. No dead sessions.
- Full TUI replay on startup becomes possible — the full conversation is still there.
- Audit trail is naturally complete (the file is the whole conversation, including pre-compaction history).
- On-disk format stays pristine — no `cache_control` fields polluting the JSONL.
- The "canonical intent" of the API (how compaction actually works) is expressed once, in the clone method.

### Naming decision

Picked `cloneForRequest()` over the other candidates (`snapshotForRequest()`, `cloneApiMessages()`). "clone" is explicit about the deep-copy guarantee and "for request" makes the intended use site obvious. `snapshot` implied immutability which is the opposite of what we want: the returned array is mutable and caller-owned.

### Scope discipline

Held the line on PR 1's narrow scope. The following are all deferred to later PRs and still accurate as planned: `AnthropicClient` extraction, `AnthropicAgent` reshape with `configure(partial)` / `run({ systemReminder? })`, deletion of `ConversationStore` from the SDK, removal of `historyFile` from `AnthropicAgentOptions`, CLI-side persistence layer, collapse of `apps/claude-sdk-cli/src/runAgent.ts`, session management, audit path derivation.

What's notable: the refactor turned out to be smaller than expected. The `AgentRun` call-site change was a single line; `RequestBuilder` simplifications were mostly deletion; all the behavioural weight sat in `Conversation.ts`. That is itself evidence the old design was wrong — the cache-control spread dance and the on-load trim and the compaction clear were all symptoms of the same confused ownership boundary. Separating stored state from API view eliminates all three at once.
