Implemented issue #226. Design shifted from `systemPrompts` to `cachedReminders` during
implementation:

**Why `cachedReminders` instead of `systemPrompts`:** The Anthropic API requires that

# Session 2026-04-09 (SDK refactor planning)

Branch: `feature/sdk-refactor` (off `main`). Context was compacted earlier in the day; this section captures the design discussion before any code is written, so the plan survives another compaction.

## Trigger

Started from the usage-auditing work on `feature/usage-auditing` (raw events + `RawEventWriter` already landed as commits `424cb9e` and `d8d2555`). The next step was going to be session management, but before writing it the user asked to step back and question the overall SDK architecture. That question turned into a full rethink of the SDK shape. The rethink blocks session management cleanly, so it goes first on a new branch.

## What the current SDK shape actually is

Three classes in play:

- **`Conversation`** (`packages/claude-sdk/src/private/Conversation.ts`): pure in-memory message list. Holds `HistoryItem[]`, enforces role-alternation merge, and (critically for this refactor) **clears the array when a compaction block is pushed**.
- **`ConversationStore`**: file-backed wrapper around `Conversation`. Loads JSONL in constructor (calling `trimToLastCompaction` on load), saves atomically via tmp+rename on every push/remove.
- **`AnthropicAgent`**: constructs exactly one `ConversationStore` from `options.historyFile` in its constructor. That store lives for the agent's lifetime. `runAgent(options)` creates a new `AgentRun` that mutates the stored conversation.

CLI-side, `apps/claude-sdk-cli/src/runAgent.ts` wraps `agent.runAgent(...)` with a ~15-field `RunAgentQuery` of which only `messages` and `systemReminder` actually change turn-to-turn. The other 13 are constants recomputed on every call.

## Observations

1. **The configured-once vs per-run axis doesn't match usage.** In practice almost everything the CLI passes to `runAgent()` is session-scoped, not per-call. The split optimises for flexibility nobody uses.

2. **Persistence is baked into the SDK.** `historyFile` is an `AnthropicAgentOptions` field; the SDK does `readFileSync`, `writeFileSync`, `renameSync`. That forces decisions about file paths into the SDK when they're really consumer policy.

3. **Compaction conflates storage with transmission.** `Conversation.push()` clears the array when it sees a compaction block, and `ConversationStore.load()` truncates on load. Both are destructive. Once a compaction happens, pre-compaction messages are gone from memory *and* disk. Consequence: if the compaction turn itself errors, the session dies — there's nothing to roll back to. The user specifically called out wanting to recover from bad compactions.

4. **Cache control decorations live on stored messages.** `RequestBuilder` does a defensive spread-copy dance (`[...messages]`, `{ ...block, cache_control }`, etc.) purely to avoid leaking send-time markers into the stored conversation, because the same array is both storage and send-buffer. That pattern only exists because of the conflation.

5. **`RawEventWriter` / `EventParser` / `AuditWriter` are per-turn or module-level.** Wrong lifetime for anything session-scoped.

## Design agreed with the user

### SDK holds state, but never persists

- The SDK client holds the message array in memory and exposes it.
- No file paths, no session IDs, no JSONL inside the SDK.
- The CLI reads messages from disk on startup and hands them to the SDK. The CLI saves to disk after each turn by reading `client.messages`.
- Observation is pull-based: no `message_appended` events. The CLI reads the exposed array whenever it wants.

### Three-layer SDK shape (target)

- **`AnthropicClient`** — auth + HTTP + token refresh. Created once per process. Thin.
- **`AnthropicAgent`** — takes the client once, holds a `Conversation` instance, holds a config bag (model / tools / betas / cache settings / transformToolResult / cachedReminders / systemPrompts), exposes `configure(partial)` + `run({ systemReminder? })` + message operations. Represents one in-memory conversation. A "new conversation" = construct a new agent off the same client.
- CLI owns identity, persistence, audit.

### The compaction split (the important part)

**`Conversation` stores every message ever pushed, forever.** It never clears on compaction. Compaction blocks are data, not commands.

**Two message views on `Conversation`:**
- `messages` (read-only getter) — the full history, for CLI/TUI/audit/persistence.
- A clone method (name TBD — `snapshotForRequest` / `cloneForRequest` / `cloneApiMessages`) that returns a **deep-cloned slice from the last compaction forward**. That slice is the caller's to mutate freely.

`RequestBuilder` calls the clone method, gets its own working copy, mutates it in place (cache control, system-reminder injection) and sends. The defensive spread-copy pattern goes away because nothing is shared.

**Payoffs**:
- Compaction rollback works for free: `conversation.removeById(compactionMsgId)` makes the next `apiView` look further back. No dead sessions.
- Full TUI replay on startup becomes possible — the full conversation is still there.
- Audit trail is naturally complete (the file is the whole conversation, including pre-compaction history).
- On-disk format stays pristine — no `cache_control` fields polluting the JSONL.
- The "canonical intent" of the API (how compaction actually works) is expressed once, in the clone method.

## Session: SDK design reconstruction (feature/usage-auditing to main)

### What actually happened this session

Work started on the `feature/usage-auditing` branch. That branch was used to illustrate a concrete problem: it had added `RawEventWriter` and `AuditWriter` (calling `mkdirSync` and `appendFileSync`) inside the SDK, and was piggybacking audit output on the channel event forwarding loop. The code was working around a design problem rather than solving it.

The session then stepped back to ask why issue #232 existed at all. The six-step technical plan in #232 and in earlier sections of this log turned out not to contain the actual reasons the rewrite was needed. Those reasons had to be reconstructed from memory. This is the section that records them.

### The two underlying problems (the actual why)

**Problem 1: nothing was durable across queries.** The SDK was designed without any concept of a session, to avoid the kind of persistence the official Claude Code SDK owns (session ids, session files, resume-by-id). Avoiding that persistence was correct. Avoiding it by refusing to have a session object at all was wrong. The result: every call to `runAgent` had to re-supply roughly fifteen fields of configuration, of which maybe two changed between calls. That is not avoiding persistence. That is making the consumer rebuild in-memory SDK state the SDK should have been holding.

The correct rule: no persistence or hydration of a session. Not no session. The SDK holds a session object in memory for the life of the process. It never reads or writes it to disk. If the consumer wants to resume, the consumer reads the conversation out and saves it. The SDK knows nothing about that flow.

**Problem 2: the SDK was touching files.** `ConversationStore` knows about file paths. `historyFile` is in `AnthropicAgentOptions`. The audit branch was adding file writers inside the SDK. None of that belongs in an API client library. Every file path, every `fs` call, every persistence decision belongs in the consumer.

### Authoritative document

`.claude/plans/sdk-shape.md` was created this session. It contains the full why, the principles that fall out of it, and the full building block list with responsibilities. That file is the source of truth going forward.

**The six-step technical plan in the earlier sections of this log and in issue #232 is superseded.** Do not use it to drive work. Re-derive the steps from the plan file.

### Key principles (summary, see plan file for detail)

- Substitution, not optionality. Replace a block with your own; do not expect on/off switches.
- The SDK understands the API. It does not understand the filesystem.
- Config describes usage, not API shape. `compaction: { enabled, inputTokens, pauseAfterCompact }`, not raw beta flags. The request builder translates usage config into body fields and beta headers. The client is the pipe, not the policy.
- Two-way messaging over `MessagePort` is wiring, not wrapping. It is the mechanism the default approval coordinator uses. If a consumer replaces that coordinator, the port goes with it.
- Observation and control are separate surfaces. Events for read-only observation; port for bidirectional control.
- The consumer owns the conversation. `Conversation` is a building block, not a private implementation detail.

### What happened

The walk did get done. The writeup of the walk was contaminated by invented requirements that the plan does not contain. Stephen caught it and rejected the analysis. The contamination has to be assumed to have affected the whole writeup, not just the one sentence that named it, because I did not notice the drift while writing.

### What was invented (do not repeat)

I wrote: "`MessagePort` imported from `node:worker_threads` is Node-only. The DOM/Web Worker `MessageChannel` exists globally on modern Node too; the import choice locks the SDK to Node-only environments."

Nothing in `.claude/plans/sdk-shape.md` says the SDK has to run on non-Node runtimes. Nothing in the two foundational problems (durable session, no filesystem) implies it. Nothing in any prior session log or issue raises it as a concern. This is a familiar refactoring instinct I imported from outside the plan and dressed up as analysis. That is exactly the failure mode the plan file's own "why this plan is written this way" section warns about: drifting back toward preserving or changing technical shapes without a reason tied to the why.

The rule for next session: before listing any file as problematic, point at the specific line or principle in the plan file that the file violates. If you cannot point at one, do not raise the issue.

### What was also mishandled

I flagged the `SdkMessage` union as a problem ("mixes lifecycle + content + usage + control + errors in one flat stream") and said "the plan wants observation and control split". Stephen's reply: "sorry i dont even understand, you said a bunch of stuff, but what does it mean? and why is this bad? what is your alternative? nothing about the two foundational issues have anything to do with the SdkMessage".

He is right that nothing about Problem 1 (durable session) or Problem 2 (no filesystem) requires changing `SdkMessage`. The plan does have a principle "observation and control are separate surfaces", but applying that to `SdkMessage` is an extrapolation, and I did not even make the extrapolation clearly, I just handwaved. Next session: if the observation-vs-control split implies a change to `SdkMessage`, I have to be able to state what the change is concretely and why the current shape fails. If I cannot, do not raise it.

### Corrections Stephen made to my other findings

**`pricing.ts` stays.** I suggested removing it on bundle-size grounds. Stephen: "hard disagree, this is what the official Claude SDK does as well. sure they change, how often? if you dont have this, how is anyone else going to estimate cost? the same way, with hard coded tables. keep it". This is the right correction. The cost of staleness is small and bounded; the value of shipping a working cost estimate is large. Drop this from the findings.

**`RunAgentQuery` split should not remove per-query override.** I said split durable config onto the session and put only user content per-query. Stephen's refinement: "yes, BUT, consider that they can AND will change per query, dynamic config, etc. dont make the SDK worse by preventing configuring them per query". So the shape is: durable defaults live on the session; per-query input can override any of them for that one query without touching the session defaults. Neither Problem 1 nor Problem 2 is served by making the SDK less flexible than it is today.

### Findings that survive the contamination (to be re-verified against the plan in the next session)

These held up against the plan on the first pass and I believe still hold, but next session I should re-walk them and for each write the explicit plan reference before acting:

- `ConversationStore.ts` is a direct Problem 2 violation. `readFileSync` / `writeFileSync` / `renameSync` / `historyFile` path. Plan file explicitly lists `readFileSync`, `writeFileSync`, `renameSync`, `historyFile` as things that must not appear in SDK sources. Blast radius: `AnthropicAgent.ts`, `AgentRun.ts`, `public/types.ts` (`historyFile?` field), `AgentRun.spec.ts` (seven test call sites).
- `Conversation.ts` is close to the plan's `Conversation` building block already. Main gap: it is `private/`, while the plan says "the consumer owns the conversation" and the block is described as public.
- `AnthropicClient.ts` matches the plan's `Client` block. Needs to be public. `IMessageStreamer` abstract class (8 lines, one method) is ceremony.
- `MessageStream.ts` matches the plan's `Stream processor` block. The content-block mapping is duplicated in `AgentRun.handleAssistantMessages`, which has to re-map `ContentBlock` to `BetaContentBlockParam` before pushing to history. The stream processor should emit something ready to push.
- `RequestBuilder.ts` is the plan's `Request builder` block. This is the file the "config describes usage, not API shape" principle lands on: `betas` as primary surface, `pauseAfterCompact` and `compactInputTokens` flat instead of grouped under `compaction: {...}`. The plan names this directly.
- `AgentRun.ts` is doing multiple jobs at once. Breaking it into turn runner + query runner + approval coordinator + tool registry is a shape the plan describes explicitly. This is real work, not extrapolation.
- `AnthropicAgent.ts` has the shape of the plan's `Session` block but is built wrong: it takes `historyFile` (Problem 2), `runAgent` takes fourteen fields per call (Problem 1), and ceremonial `injectContext` / `removeContext` / `loadHistory` / `getHistory` wrap access to a conversation the consumer should own directly.
- `pricing.ts` stays. Not a violation.
- Cost calculation done inline in `AgentRun` for the `message_usage` event is a placement detail, not a plan violation. Leave it unless it falls out naturally from splitting AgentRun.
