## Review

ExecV3's command logic is careful and matches its "bash, exactly" contract everywhere I traced it (precedence, short-circuit, exit status, grouping, redirect/merge, validation), and the 64-test suite is green. One foundational defect sits in the pipe lifecycle: when a downstream stage exits before draining its input — `cmd | head`, `cmd | grep -q`, everyday idioms — the call deadlocks and never returns, and the `timeout` the schema calls "always enforced" does not fire on that path. Reproduced live (`yes | head -n 1` still pending at 6 s under a 2 s timeout). The root cause is in exec-core's stream teardown and is shared with the opt-in V1/V2, but ExecV3 is now the default-on execution tool, so the failing path is the default path. Three comments below: the deadlock, the timeout overclaim, and the missing early-consumer-exit test that let it through. Secondary notes (timeout lower bound; default-on blast radius) are recorded in the Investigations.

## Comments

### C1 — `src/ExecV3/runPipeline.ts` (with exec-core `src/Executor.ts` `closeSinks`)

A pipeline whose downstream stage exits before draining its input deadlocks — the call never returns. `runPipeline` awaits `Promise.all(runs)`; when the consumer (`head`, `grep -q`, …) exits early, the inter-stage bridge is left with unread data and no reader, and exec-core's `closeSinks` awaits `finished(bridge)`, which never sees readable `end`. There is no SIGPIPE-equivalent teardown of the upstream producer.

Reproduced against the real tool: `echo hi | cat` returns in 6 ms; `yes | head -n 1` is still pending at 6 s with the loop kept alive, and a `timeout: 2000` does not end it. Everyday idioms (`cmd | head`, `cmd | head -nK`, `cmd | grep -q`) are affected, and ExecV3 is default-on. The fix likely belongs in exec-core (destroy the bridge / propagate the consumer's exit upstream rather than `end()` + `finished()` on an undrained sink); V1/V2 share the mechanism.

### C2 — `src/ExecV3/schema.ts` (timeout) and `schema.md`

`timeout` is documented as "always enforced", but on the pipe deadlock above the timeout does not fire (proven: a 2000 ms timeout, still hung at 6 s) — the run promise is stuck in exec-core's `closeSinks`, which the abort does not release. Either the deadlock fix restores the guarantee, or the wording should stop promising "always". Worth not shipping a safety claim the pipe path can't honour.

### C3 — `test/ExecV3/scenarios.spec.ts`

No scenario covers a pipe whose consumer exits before draining (all current pipe tests drain fully), which is why the deadlock above is invisible to CI. A `cmd | head` scenario with a bounded outer timeout, asserting prompt return, would guard the fix.

## Investigations

Actions: `comment` = surfaced as a Comment entry above; `flag` = recorded only (not posted). The SC may change any Action before dispatch.

### Investigation 1: Pipe deadlock when a downstream stage exits before draining
- Noticed: `runPipeline` spawns every pipe stage and awaits `Promise.all(runs)`; a downstream stage can exit before draining its input (e.g. `… | head`), leaving the upstream producer with a full bridge and no reader. 
- Traced: exec-core `Executor.run` pipes `child.stdout → bridge` with `{end:false}`, then on completion runs `closeSinks`, which calls `sink.end()` and awaits `finished(sink)` on every sink — including the inter-stage `PassThrough` bridge. With the consumer gone the bridge keeps unread data and never emits readable `end`, so `finished(bridge)` never resolves; there is no SIGPIPE-equivalent teardown of the producer. The same wiring is in V1 `execPipeline` and V2 `pump`. 
- Derived: proven live against the real tool — control `echo hi | cat` returned in 6 ms; `yes | head -n 1` was still pending at 6001 ms with the loop pinned open, and a 2000 ms `timeout` did not make it return (the bare run aborted with Node's "unsettled top-level await", exit 13). In the TUI, where the loop is always alive, the call hangs indefinitely with the producer running unbounded. 
- Concluded: a foundational defect for a pipe-capable tool — `cmd | head`, `cmd | head -nK`, `cmd | grep -q` all deadlock. Root cause sits in exec-core's `closeSinks` awaiting `finished()` on an undrained bridge and is shared with V1/V2, but ExecV3 is the new, default-on surface that brings pipes to users, so it is in scope. "Inherited" is not a pass: a new pipe tool that hangs on `| head` does not clear the bar.
- Action: comment

### Investigation 2: `timeout` "always enforced" is false on the deadlock path
- Noticed: `schema.md` and the schema describe `timeout` as "default 30000 … always enforced"; the handler passes `execSignal(signal, input.timeout)` to the executor as the sole bound on run time. 
- Traced: `execSignal` builds the bound from `AbortSignal.timeout(ms)`. On the Inv-1 deadlock the run promise is stuck inside exec-core `closeSinks` awaiting `finished(bridge)`; firing the signal group-kills the children but does not release that await. Proven: a 2000 ms timeout had not returned at 6000 ms. 
- Derived: "always enforced" fails exactly on the pipe path where a bound matters most. The 30 s default is presented as the safety net behind every run, yet it is the net with the hole. 
- Concluded: the contract overclaims what the implementation can deliver on the pipe path. Fixing Inv 1 (so the bound can fire) or correcting the claim both close it, but the gap between the promised "always" and the behaviour is itself a finding.
- Action: comment

### Investigation 3: No early-consumer-exit pipe test
- Noticed: the V3 scenario suite (59 tests) exercises pipes only where the consumer drains its full input (`echo|cat`, `printf|grep|wc`, `sh -c 'echo;exit1'|cat`); none has a downstream stage exiting early. 
- Traced: the adjacent V1/V2 suites share that shape — the `| head` idiom appears nowhere across Exec/ExecV2/ExecV3 scenarios. So the Inv-1 deadlock is invisible to CI, and the green 64-test V3 run never touches it. 
- Derived: this is the mechanism by which a foundational hang shipped behind a passing suite; one `cmd | head` scenario with a bounded outer timeout would have caught it. 
- Concluded: the missing test is not a nicety — it is the regression guard that belongs with the Inv-1 fix.
- Action: comment

### Investigation 4: Engine bash-list semantics
- Noticed: `engine.ts` implements bash `&&`/`||`/`;` precedence and short-circuit with a single `lastExit` accumulator plus skip-and-continue; `group.ts` splits the flat list into pipelines + connectors under an `i < commands.length - 1` guard. 
- Traced: walked ~15 operator combinations against bash (`a&&b||c`, `a||b&&c`, `a&&b;c`, `a;b&&c`, `a&&b&&c` with a mid failure, `a||b||c`, `true&&false||true`, `false||true&&false`, and pipe-mixed lists such as `a|b&&c|d`), plus the connector/index alignment for pipe-containing lists. The single accumulator is sufficient because bash `&&`/`||` are equal-precedence and left-associative, so the running exit status is all that is needed; skipped pipelines carry `$?` unchanged. Pipeline exit = last stage (no pipefail), matching bash default. 
- Derived: every traced case matches bash; the group spec's connector/indices expectations hold; `results[i] ↔ commands[i]` alignment is preserved and short-circuited slots are `null`. 
- Concluded: no divergence from the "bash, exactly" contract in the command-logic layer. Recorded as covered — no concern.
- Action: flag

### Investigation 5: Sink resolution, validation, path expansion
- Noticed: `resolveStageSinks` branches on `redirect.stdout` / `downstream` / `mergeStderr`; the `superRefine` enforces dangling-op, R4 (pipe + stdout redirect) and NE2 (stdin on a pipe target); `normalise.ts` runs `expandPath` over program/cwd/redirect paths, including an undefined `cwd`. 
- Traced: enumerated every reachable `resolveStageSinks` branch (single command, pipe source, pipe middle, terminal; with/without stdout redirect; stderr-to-file; stderr `&1` merge) against the intended bash mapping; checked which illegal shapes could reach the engine and crash `group`/`evaluate`; read `expandPath`'s overload — `expandPath(undefined)` returns `undefined`, so `cmd.cwd ?? ctx.cwd` falls back to the process cwd rather than an empty string. 
- Derived: sink resolution is correct in all reachable branches; no engine-crashing shape slips past the superRefine; the undefined-cwd path is safe. (V3 hard-rejects R4/NE2 where V1 silently ignored them — a deliberate improvement.) 
- Concluded: redirect/merge, validation completeness, and path expansion hold. Recorded as covered — no concern.
- Action: flag

### Investigation 6: `timeout` has no lower bound
- Noticed: `ExecV3InputSchema.timeout` is `z.number().max(600000).default(30000)` — a max but no `.min()`. 
- Traced: V1/V2 are also `.max(600000)` with no min (but `.optional()`, no default). `timeout: 0` → `AbortSignal.timeout(0)` (immediate abort, everything killed); a negative value makes `AbortSignal.timeout` throw a `RangeError`, surfacing as an unstructured tool error rather than the "loud, structured rejection" the design promises. 
- Derived: reachable only by an explicit bad value (default 30000), so low severity; the negative case is the sharper one as it breaks the structured-rejection contract. 
- Concluded: minor robustness gap, shared with V1/V2; `.int().positive()` (or `.min(1)`) closes it. Recorded — not worth a standalone PR comment.
- Action: flag

### Investigation 7: Default-on blast radius
- Noticed: cli-config flips `tools` default to `{exec:false, execV2:false, execV3:true}` and `advancedTools.enabled` false→true; the generated `sdk-config.schema.json` matches. 
- Traced: ExecV3 becomes the only execution tool registered by default (V1/V2 opt-in), and `input_examples` ride on `advancedTools` being on. So the Inv-1 deadlock and the Inv-2 overclaim ship to every default user through V3, where V1/V2's identical pipe flaw stayed behind an opt-in flag. 
- Derived: not a defect in itself, but it sets the severity of Inv 1/2 — the failing path is the default path. 
- Concluded: recorded as the amplifier on Inv 1/2's severity, not a separate defect.
- Action: flag
